# Anthropic authentication — API keys vs OAuth / JWT bearer tokens

This document explains the two authentication schemes the `anthropic`
provider needs to support, how `CLAUDE_CODE_OAUTH_TOKEN` fits in, and how
each scheme maps onto `StreamOptions` fields in `src/ai/registry.zig`.

The source for precedence and header shape is the official Claude Code
authentication page:

- https://code.claude.com/docs/en/authentication

The parts relevant to this provider are summarised below; when in doubt,
treat that page as canonical.

---

## The two wire shapes

Both schemes target the same endpoint —
`https://api.anthropic.com/v1/messages` — and the request body is
identical. Only the headers differ.

### 1. API key — the Console path

Used with a key issued from the Claude Console (the `sk-ant-api03-...`
shape). Sent as:

```
X-Api-Key: <key>
anthropic-version: 2023-06-01
```

This is what most Anthropic SDKs reach for by default and what the
`ANTHROPIC_API_KEY` environment variable maps to.

### 2. OAuth / JWT bearer — the subscription path

Used with an OAuth access token tied to a Claude Pro / Max / Team /
Enterprise subscription, or with a long-lived token generated by
`claude setup-token`. The full header set the provider sends:

```
Authorization: Bearer <token>
anthropic-version: 2023-06-01
anthropic-beta: claude-code-20250219,oauth-2025-04-20
user-agent: claude-cli/2.1.85 (external, cli)
x-app: cli
```

Only the first three are strictly "auth" headers. The last two
(`user-agent` and `x-app`) are **fingerprinting headers** — the
Messages API quietly rate-limits OAuth requests that don't look like
Claude Code's own traffic, even when the token itself is healthy. A
stock Anthropic-SDK or custom user-agent on the OAuth path routinely
triggers 429s that the same token never sees from the Claude Code CLI.
Match the fingerprint or don't use OAuth.

Only one of `X-Api-Key` and `Authorization` should be sent on a given
request. Do not send both.

### 2a. OAuth system-prompt contract

When bearer auth is used, the `system` field is also silently
validated. All non-Haiku models require the `system` field to begin
with the exact string:

```
You are Claude Code, Anthropic's official CLI for Claude.
```

…as either the entire plain `system` string, or the whole text of the
first element of a `system` array. User-supplied system-prompt content
goes in a **separate** second array element — concatenating the prefix
and your own prompt into one entry fails.

```json
"system": [
  {"type":"text","text":"You are Claude Code, Anthropic's official CLI for Claude."},
  {"type":"text","text":"…the driver's real system prompt…"}
]
```

Violating this yields HTTP 400 with a generic `"Error"` message (or, in
practice, heavy 429s). Haiku is exempt but behave as if it isn't.

Source: [anthropics/claude-code#40515](https://github.com/anthropics/claude-code/issues/40515).

The `anthropic.zig` provider applies this transformation automatically
whenever `options.auth_token != null`, so callers pass their own system
prompt verbatim — the provider handles prefix injection and dedup. See
the `buildRequestJson emits Claude Code system prefix…` unit tests.

## `CLAUDE_CODE_OAUTH_TOKEN` in particular

Quoting the precedence list from the Claude Code docs verbatim:

> `CLAUDE_CODE_OAUTH_TOKEN` environment variable. A long-lived OAuth
> token generated by `claude setup-token`. Use this for CI pipelines
> and scripts where browser login isn't available.

Key properties:

- **How to obtain.** Run `claude setup-token` in an environment where a
  browser login is available. The command walks you through OAuth
  authorization and prints the token to the terminal. It does **not**
  save the token anywhere — copy it yourself and set it via
  `export CLAUDE_CODE_OAUTH_TOKEN=<token>` wherever you need it.
- **Lifetime.** One year (from the docs: "generate a one-year OAuth
  token with `claude setup-token`").
- **Scope.** Inference only. A token minted this way cannot establish
  Remote Control sessions, and requires a Pro, Max, Team, or Enterprise
  plan.
- **Wire format.** It is an OAuth access token, sent as
  `Authorization: Bearer <token>`, with the
  `anthropic-beta: oauth-2025-04-20` header alongside. It is not an
  `sk-ant-...` API key — do not send it as `X-Api-Key`.

## Authentication precedence (relevant subset)

The docs define this order when several credentials are in the
environment simultaneously. The `anthropic` provider itself only ever
sees whatever the driver resolved and passed in `StreamOptions`, so
precedence is enforced **one layer up** — in
`coding/modes/print.zig::resolveProvider`. It must follow the published
order:

1. Cloud-provider credentials (`CLAUDE_CODE_USE_BEDROCK`, `_VERTEX`,
   `_FOUNDRY`) — routed to a different provider entirely; out of scope
   for this file.
2. **`ANTHROPIC_AUTH_TOKEN`** — sent as `Authorization: Bearer`. Used
   when routing through an LLM gateway or proxy that authenticates with
   bearer tokens rather than Anthropic API keys. This variable is
   general-purpose (any bearer-auth proxy), not subscription-specific.
3. **`ANTHROPIC_API_KEY`** — sent as `X-Api-Key`. Direct Anthropic API
   access with a Console key. In franky's non-interactive print mode
   the key is always used when present (matching Claude Code's
   `-p` / headless behaviour).
4. `apiKeyHelper` — a shell script that prints a key on stdout.
   Deferred in franky; mentioned here for parity with the docs.
5. **`CLAUDE_CODE_OAUTH_TOKEN`** — sent as `Authorization: Bearer` plus
   the OAuth beta header. The long-lived subscription token from
   `claude setup-token`.
6. Subscription OAuth credentials from `/login` — the interactive
   browser flow. Not applicable to franky today; if/when we implement
   the login flow (§Q), credentials end up stored in a
   claude-code-compatible location and feed back into this slot.

If `ANTHROPIC_API_KEY` is set alongside subscription credentials, the
API key wins (per the docs). Users who want their subscription honoured
must `unset ANTHROPIC_API_KEY`.

One footnote that matters for testing against the real API:

> If you have an active Claude subscription but also have
> `ANTHROPIC_API_KEY` set in your environment, the API key takes
> precedence once approved. This can cause authentication failures if
> the key belongs to a disabled or expired organization.

## How this maps onto `StreamOptions`

`src/ai/registry.zig` exposes two mutually-exclusive optional fields on
`StreamOptions`:

```zig
api_key:    ?[]const u8 = null, // → X-Api-Key
auth_token: ?[]const u8 = null, // → Authorization: Bearer
```

Contract for the `anthropic` provider:

- If `auth_token != null` → use bearer auth, emit `anthropic-beta:
  oauth-2025-04-20`, and ignore `api_key`.
- Else if `api_key != null` → use `X-Api-Key`.
- Else → close the stream with an `error_ev` of code `auth`.

Resolution happens in the driver (print mode), not in the provider. The
provider trusts whatever the driver filled in.

Suggested driver-side resolution order, matching Claude Code's
precedence list:

```text
auth_token =
    --auth-token CLI flag         (highest)
    ANTHROPIC_AUTH_TOKEN env var
    CLAUDE_CODE_OAUTH_TOKEN env var

api_key =
    --api-key CLI flag
    ANTHROPIC_API_KEY env var
```

With `ANTHROPIC_API_KEY` taking precedence over `CLAUDE_CODE_OAUTH_TOKEN`
to match the Claude Code docs. (`ANTHROPIC_AUTH_TOKEN` sits above
`ANTHROPIC_API_KEY` in the published order — a general bearer override
for proxies — so we honour that too.)

The provider CLI surface that this resolution feeds:

- `--api-key KEY` — plain API key. Becomes `StreamOptions.api_key`.
- `--auth-token TOKEN` — OAuth / JWT bearer. Becomes
  `StreamOptions.auth_token`. Covers both `ANTHROPIC_AUTH_TOKEN` and
  `CLAUDE_CODE_OAUTH_TOKEN` use cases; the distinction only matters for
  env-var fallback, not for the wire.

## Verifying a token by hand

A minimal curl that mirrors what the provider sends, including the
fingerprint headers and the system-prompt prefix:

```bash
curl https://api.anthropic.com/v1/messages \
  -H "Authorization: Bearer $CLAUDE_CODE_OAUTH_TOKEN" \
  -H "anthropic-version: 2023-06-01" \
  -H "anthropic-beta: claude-code-20250219,oauth-2025-04-20" \
  -H "user-agent: claude-cli/2.1.85 (external, cli)" \
  -H "x-app: cli" \
  -H "content-type: application/json" \
  -d '{
    "model": "claude-sonnet-4-20250514",
    "max_tokens": 64,
    "system": [
      {"type":"text","text":"You are Claude Code, Anthropic'"'"'s official CLI for Claude."}
    ],
    "messages": [{"role":"user","content":"say hi"}]
  }'
```

Common failure modes:

- **401 `authentication_error`** — token is actually an `sk-ant-api03-...`
  key (wrong scheme) or the subscription has lapsed.
- **400 `invalid_request_error` with generic `"Error"` body** — system
  prompt doesn't start with the required Claude Code prefix. Non-Haiku
  models silently enforce this.
- **429 on a healthy subscription** — you're missing the
  `user-agent: claude-cli/…` and/or `x-app: cli` headers. Anthropic
  rate-limits OAuth traffic that doesn't fingerprint as Claude Code,
  often immediately and hard, even when the token itself has plenty of
  quota left.

## What this package does *not* do

- **Token minting / refresh.** We consume tokens that the user already
  produced (via `claude setup-token` or a gateway). A full OAuth device
  flow is spec §Q and deferred.
- **Credential storage.** We never read or write `~/.claude/.credentials.json`
  or the macOS keychain. All credentials flow in via CLI flag or
  environment variable on a per-invocation basis.
- **`apiKeyHelper`.** The shell-script credential hook is a deferred
  feature; it would live here when added, in front of the env-var
  lookups in `resolveProvider`.
