# Documentation / specification archive

## Problem

Specs and design docs accumulate over time. Old versions, superseded
RFCs, and post-hoc brainstorms keep the project's history but
shouldn't pollute the model's default context — every grep / ls /
find walk that surfaces them costs tokens and invites confusion
between "current" and "historical" guidance.

`.gitignore` doesn't help here: archived docs are still in git, just
not part of the model's working set. We need a parallel mechanism
that hides them from default tool walks while keeping them readable
on deliberate request.

## Scope: what counts as "archive"?

Not every old document is archive material. A working categorization:

| Category | Examples | Archive? |
|---|---|---|
| **Frozen-but-cross-referenced spec** | `v0.md`, future `v1.md` after v2.x ships | **No** — current specs link into them; grep-driven research depends on them being visible. |
| **Shipped design RFCs** | `docs/design/v2.8-edit-tool-diff-view.md`, `v2.10-harness-enforced-guardrails.md` once they land | **Yes** — the implementation is the source of truth; the RFC is history. |
| **RFCs that didn't ship** | abandoned proposals, superseded brainstorms | **Yes** |
| **Active design RFCs** | the one you're reading now | **No** — under discussion. |

Loose rule: an RFC graduates to archive when (a) it shipped and the
spec/code reflects it, OR (b) the discussion concluded with "won't
do." Until then, it stays visible.

## Mechanism — 4 candidate shapes

### Option A — `.llmignore` only (declarative, no opt-in)

A new `.llmignore`-style file, parsed by the existing
`coding/gitignore.zig` matcher. Tools that walk the tree skip
matching paths. No model-side opt-in.

- **Pros**: cheapest. Mirrors `.gitignore` semantics. One concept.
- **Cons**: model has no recourse when it genuinely needs an
  archived file (e.g. "why was the v1 design this way?" — the only
  authoritative answer is the historical doc).

### Option B — convention path (`docs/archive/`)

A hard-coded directory that walkers skip by default. No new file
format.

- **Pros**: zero config; new tools work automatically.
- **Cons**: no flexibility for non-doc archive content (legacy code
  comments, retired test fixtures, etc.). Less expressive than globs.

### Option C — `.contextignore` + per-tool opt-in flag (recommended)

Same parser as option A, but each tree-walking tool gets a
`respectLlmignore: bool = true` parameter. Default behavior excludes
archived paths; the model can pass `false` for a deliberate access.

- **Pros**: composes with the existing `respectGitignore` pattern.
  Reuses `coding/gitignore.zig`. Gives the model an explicit escape
  hatch without adding a new tool.
- **Cons**: small per-tool API growth. Requires deciding whether
  `read` (which doesn't walk a tree) should also gate on it.

### Option D — dedicated `archive_read` tool

Default tools never see archived files; a separate `archive_read`
tool grants access on demand.

- **Pros**: cleanest separation. Model can't accidentally pull
  archived content via `grep` or `find`.
- **Cons**: adds tool surface. Model has to know the tool exists.
  Two tool flavors for the same operation feels like sprawl.

## Naming

`.llmignore` is readable but misleading — the files *are* readable
on demand, just not in the default working set. Alternatives:

- `.contextignore` — describes the actual semantics. **Preferred.**
- `.ai-context-deny` — explicit but verbose.
- `.archived` — paths-only, no glob support. Simpler but less
  expressive.
- `.modelignore` — concise, mirrors `.gitignore`. Could also work.

## Recommendation

**Option C with `.contextignore`.** Concretely:

1. Extend `coding/gitignore.zig` (or duplicate the matcher under a
   new name) so `.contextignore` files load with the same gitignore
   semantics.
2. Add `respectLlmignore: bool = true` to the tools that already
   accept `respectGitignore`: `ls`, `find`, `grep`. Default true.
3. Decide whether `read` also gates on it (see Q1 below).
4. Seed `.contextignore` at the repo root with `docs/archive/**` so
   there's a default home for archived material; teams that need
   their own conventions add patterns as needed.
5. As RFCs ship, move them to `docs/archive/<version>/<name>.md`
   (or keep in place and add a path-specific entry to
   `.contextignore`).

This composes with existing infrastructure and gives the model an
explicit, auditable escape hatch.

### Q1 — Should `read` respect `.contextignore`?

Path-based denials are cheap to enforce in `read`, but the question
is semantic: if the model already knows the exact archived path, is
denying the read on the default flag a feature or an annoyance?

- **A.** Yes — `read` gates on `respectLlmignore: bool = true` like
  the walkers. Consistent enforcement; model passes `false` on
  deliberate access.
- **B.** No — once the model has a specific path it's an explicit
  request; `read` always succeeds. `.contextignore` only governs
  *discovery* via tree walks, not direct access.
- **C.** Some other shape (describe).

**Your answer: yes**

### Q2 — Per-tool gate vs. per-prompt note

- **A.** Per-tool gate (Option C above): each tool enforces the
  exclusion at the matcher. Hard guarantee; the model literally
  can't see archived paths in default tool output.
- **B.** Per-prompt note: at session start, inject "the following
  paths are archived: …" into the system prompt. The model is
  trusted to avoid them but tools don't enforce.
- **C.** Both: tools enforce, system prompt also lists the archive
  roots so the model knows the escape hatch exists.

**Your answer:A**

### Q3 — Where do `v0.md` and a future `v1.md` (after v2.x ships) go?

- **A.** Stay where they are; they're cross-referenced spec, not
  archive.
- **B.** Move to `docs/archive/`; update cross-references to point
  at the archive path. Defaults exclude them but they're still
  readable on demand.
- **C.** Soft-archive: leave the file in place but add to
  `.contextignore`. Cross-references break only if those references
  themselves are loaded by an archive-respecting tool.
