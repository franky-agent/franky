# Writing a custom extension that registers a subagent preset

**Status:** Plan / how-to guide.

**Purpose:** Walk through building a custom Tier-1 extension for franky that
registers custom tools AND a custom `subagent` preset, so the parent LLM can
delegate Go-development tasks to a purpose-built sub-agent. Uses a "go-dev"
extension as the running example.

**Prerequisites:** v2.5 presets system (shipped) + a small SDK hook to expose
`PresetRegistry` through the `Host` view (Phase 1 below, not yet shipped).

## 1. Architecture overview

```
Mode driver (print/interactive/rpc/proxy)
  │
  ├── init PresetRegistry
  ├── registerBuiltinPresets(&reg)     ← ships "research", "code-audit", etc.
  ├── init Ext.Manager
  │     └── for each --extensions name:
  │           ext.init_fn(ext, &host)  ← **SDK hook runs here**
  │               ├── host.registerTool(...)     → adds to host_tools list
  │               ├── host.registerPreset(...)   → adds to PresetRegistry
  │               └── host.registerCommand(...)  → adds to slash_registry
  ├── buildParametersJson(&reg)   ← now includes custom presets!
  ├── build subagent.Ctx{ .presets = &reg, ... }
  └── wire subagent tool + list_presets tool
```

The key insight: **extensions register presets into the mode's `PresetRegistry`
BEFORE the registry is frozen and handed to `subagent.Ctx`**. The dynamic
`parameters_json` is built AFTER all extensions have run, so the preset `enum`
includes custom entries alongside built-ins.

## 2. Phase 1 — SDK hook: expose `PresetRegistry` via `Host`

### 2.1 Why this is needed

Currently the `Host` view (`src/coding/extensions.zig`) exposes:

```zig
pub const Host = struct {
    allocator: std.mem.Allocator,
    slash_registry: *slash.Registry,
    tools: *std.ArrayList(at.AgentTool),
    subscriptions: *std.ArrayList(Subscription),

    pub fn registerCommand(...) !void { ... }
    pub fn registerTool(...) !void { ... }
    pub fn subscribe(...) !void { ... }
};
```

There is **no `presets` field** and **no `registerPreset` method**. The
`PresetRegistry` is allocated by the mode driver and never handed to extensions.

### 2.2 What to add to `Host` (`src/coding/extensions.zig`)

Add a `presets` field and a `registerPreset` method:

```zig
// ─── in extensions.zig, imports ──────────────────────────────────
const subagent_mod = @import("tools/subagent.zig");

// ─── in Host struct ──────────────────────────────────────────────
pub const Host = struct {
    allocator: std.mem.Allocator,
    slash_registry: *slash.Registry,
    tools: *std.ArrayList(at.AgentTool),
    subscriptions: *std.ArrayList(Subscription),
    /// NEW — preset registry for subagent presets.
    /// Extensions register custom presets here in init_fn.
    presets: *subagent_mod.PresetRegistry,

    // ... existing methods stay ...

    /// NEW — register a custom subagent preset.
    pub fn registerPreset(self: *Host, p: subagent_mod.Preset) !void {
        try self.presets.register(p);
    }
};
```

### 2.3 What mode drivers need to change

Each of the 4 mode drivers (`print.zig`, `interactive.zig`, `rpc.zig`,
`proxy.zig`) needs **one extra line** when building the `Host`:

Currently (e.g. `print.zig` lines 376–413):

```zig
var preset_registry = PresetRegistry.init(allocator);
defer preset_registry.deinit();
try registerBuiltinPresets(&preset_registry);
// ... later ...
var subagent_ctx = .{
    .presets = &preset_registry,
    ...
};
```

The only change: pass `&preset_registry` into `Host` when running extensions.
The extension manager's `register` already calls `init_fn` — no new loop is
needed. The mode driver sets `host.presets = &preset_registry` before calling
`mgr.register(ext, &slash_registry)` (which internally builds the Host and
calls init_fn).

**Concrete change in each mode driver** — build the Host with the `presets`
field populated:

```zig
// Inside the extension-wiring block (e.g. print.zig ~line 268-ish):
var ext_manager = extensions_mod.Manager.init(allocator);
defer ext_manager.deinit();
if (cfg.extensions) |csv| {
    const names = try extensions_mod.Manager.parseOptIn(allocator, csv);
    defer allocator.free(names);
    for (names) |name| {
        const entry = ext_catalog.lookup(name) orelse {
            // warn and skip
            continue;
        };
        // The key: pass &preset_registry so the extension's init_fn
        // receives it via Host.presets. The Manager.register() builds
        // Host internally and calls init_fn — we must pass preset_registry
        // THROUGH the Manager so it reaches the Host constructor.
        //
        // Option A: pass &preset_registry to Manager.register as an extra arg.
        // Option B: build Host ourselves and pass init_fn the Host directly.
        //
        // Option A is simplest — add a `presets` param to Manager.register():
        try mgr.register(entry.factory(), &slash_registry, &preset_registry);
        //                                                      ^^^^^^^^^^^^^ NEW
    }
}
```

Alternatively, Option B (less invasive): add a `presets` field to `Manager`
that gets set before calling `register`:

```zig
// Set it once on the manager, and Manager.register() copies it into Host.
var ext_manager = extensions_mod.Manager.init(allocator);
ext_manager.presets = &preset_registry;   // NEW
// ... then register() as before — it reads self.presets when building Host.
```

**Recommendation: Option B.** It requires zero signature changes to
`Manager.register()` — just an extra `presets: ?*subagent_mod.PresetRegistry`
field on `Manager` that defaults to `null`. When non-null, the Host builder
passes it through. When null, extensions calling `host.registerPreset()` get a
compile error (null check) or a runtime panic.

```zig
// ─── 2. Preset builder ─────────────────────────────────────────────

/// The tools the "go-dev" preset selects from parent_tools + custom Go tools.
/// This is the `build_tools` closure on the Preset.
fn buildGoDevTools(
    allocator: std.mem.Allocator,
    parent_tools: []const at.AgentTool,
) anyerror![]at.AgentTool {
    // Step 1: select built-in tools by name (uses the existing selectTools helper)
    const selected = try subagent_mod.selectTools(
        allocator,
        parent_tools,
        &[_][]const u8{ "read", "write", "edit", "ls", "bash", "grep" },
    );
    errdefer allocator.free(selected);

    // Step 2: append the 3 custom Go tools
    const custom = [_]at.AgentTool{
        gofmtTool(),
        govetTool(),
        gotestTool(),
    };

    const total = try allocator.alloc(at.AgentTool, selected.len + custom.len);
    @memcpy(total[0..selected.len], selected);
    @memcpy(total[selected.len..], &custom);
    allocator.free(selected);  // no errdefer needed; we own total now
    return total;
}

// ─── 3. Extension entry point ──────────────────────────────────────

pub fn extension() ext.Extension {
    return .{
        .name = "go-dev",
        .version = "1.0.0",
        .init_fn = init,
    };
}

fn init(_: *ext.Extension, host: *ext.Host) ext.ExtError!void {
    // Register 3 custom tools with the host (makes them available
    // to the parent agent AND to any preset's build_tools).
    try host.registerTool(gofmtTool());
    try host.registerTool(govetTool());
    try host.registerTool(gotestTool());

    // Register the "go-dev" preset via the SDK hook (Phase 1 above).
    try host.registerPreset(.{
        .name = "go-dev",
        .description = "Go development: edit, format, vet, and test Go code.",
        .default_profile = "",
        .default_role = .code,
        .default_system_prompt =
        \\You are a Go development sub-agent. Your job is to help develop,
        \\review, and maintain Go code. You have the usual file tools
        \\(read, write, edit, ls, grep, bash) plus Go-specific tools
        \\(gofmt, govet, gotest). Format code before writing, run vet
        \\after edits, and run tests to verify correctness.
        ,
        .build_tools = buildGoDevTools,
    });
}
```

### 3.2 Key design points

1. **Tool factories (`gofmtTool()`, etc.) are `pub`** — tests can construct them
   independently of the extension lifecycle.

2. **`buildGoDevTools` uses `selectTools` + manual append** — it pulls wired
   tools from `parent_tools` by name (same pattern as `buildFileOpsTools`),
   then appends freshly constructed custom tool instances. The custom stubs
   don't need to come from `parent_tools` because they have no mode-driver
   context (no workspace scope, no session state). A future real implementation
   would wrap them in a context closure.

3. **`default_profile = ""`** — inherits the parent's profile, so the Go
   sub-agent uses whatever provider the user chose at startup.

4. **`default_role = .code`** — the sub-agent can execute shell commands
   (`bash`). The parent must be at least `code` role, else the spawn is
   rejected with `role_promotion_denied`.

5. **The `build_tools` list includes `bash`** — the sub-agent can run
   `go build`, `go test`, etc. via the shell. The Go-specific tools are
   convenience wrappers for common commands.

## 5. How the LLM discovers and invokes the preset

### 5.1 Discovery

When the extension is active, the `list_subagent_presets` tool returns the
`go-dev` preset alongside the four built-ins:

```json
[
  {"name":"research", ...},
  {"name":"code-audit", ...},
  {"name":"file-ops", ...},
  {"name":"bash-runner", ...},
  {"name":"go-dev","description":"Go development: edit, format, vet, and test Go code.","default_profile":"","default_role":"code","read_only":false}
]
```

The parent LLM sees the new entry and can decide to use it.

### 5.2 Invocation

The LLM calls `subagent` with:

```json
{
  "preset": "go-dev",
  "prompt": "Add a new handler to main.go that serves GET /health. Use gofmt after editing and run go test ./... to verify."
}
```

The `parameters_json` (dynamically generated) includes `"go-dev"` in the
`preset` enum, so the LLM knows it's a valid value.

### 5.3 Runtime dispatch

1. `subagent` tool receives the call.
2. `runSubagent` parses JSON, extracts `preset: "go-dev"`.
3. `ctx.presets.get("go-dev")` returns the custom preset.
4. `preset.build_tools(allocator, parent_tools)` calls `buildGoDevTools`:
   - Filters parent tools to [read, write, edit, ls, bash, grep].
   - Appends [gofmt, govet, gotest].
5. Returns a `[]AgentTool` of 9 tools.
6. The sub-agent is spawned with these 9 tools, the `.code` role, the
   Go-specific system prompt, and the parent's profile.

### 5.4 Custom tools in action

The sub-agent can call `gofmt`:

```json
{
  "name": "gofmt",
  "input": {"path": "handlers/health.go"}
}
```

Which currently returns `"gofmt: not yet implemented"`. A future version would
shell out to `gofmt -d handlers/health.go` and return the diff.

```zig
            for (ts) |t| if (std.mem.eql(u8, t.name, name)) return true;
            return false;
        }
    }.check;

    // All expected built-ins.
    try testing.expect(expectHas(tools, "read"));
    try testing.expect(expectHas(tools, "write"));
    try testing.expect(expectHas(tools, "edit"));
    try testing.expect(expectHas(tools, "ls"));
    try testing.expect(expectHas(tools, "bash"));
    try testing.expect(expectHas(tools, "grep"));

    // Custom Go tools.
    try testing.expect(expectHas(tools, "gofmt"));
    try testing.expect(expectHas(tools, "govet"));
    try testing.expect(expectHas(tools, "gotest"));

    // Must NOT include find (not selected).
    try testing.expect(!expectHas(tools, "find"));
}
```
