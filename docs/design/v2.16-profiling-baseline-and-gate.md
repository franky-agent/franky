# v2.16 — Profiling Baseline and Regression Gate

**Status:** draft — open for discussion
**Affects:** `src/bin/franky_profile.zig`, `build.zig` (profile step), new `src/bin/franky_baseline.zig` (if spun out), plus CI workflow files
**Related:** [docs/archive/profiling_guide.md](../archive/profiling_guide.md) (full pipeline reference), [v2.14 — Agentic Engineering](./v2.14-agentic-engineering.md) (CI gate is part of unattended QA)

---

## 1. Problem

`zig build profile` produces SVG flamegraphs and a summary of sizes — but the results are **eyeballed, not measured**. When a PR lands that makes the test suite 12% slower or 8 MB heavier, nobody notices until the flamegraph happens to be looked at. The existing differential workflow (`inferno-diff-folded`) requires manual before/after capture and manual interpretation.

We need:

1. **A repeatable baseline** — capture aggregate CPU and memory metrics from a known-good commit (`main`) and store them somewhere durable.
2. **A gate** — on every PR, run the same capture, compare against the baseline, and flag regressions above a threshold (WARN or FAIL).

The gate must be **cheap enough to run in CI** (a few minutes), **low-noise** (wide enough thresholds to not flake on benign variance), and **actionable** (tells you which metric regressed and by how much).

---

## 2. What already exists

`zig build profile` (defined in `build.zig` §214–234, implemented in `src/bin/franky_profile.zig`) drives the full pipeline:

- Builds an FP-preserved test binary (`zig build test-profile`).
- Captures CPU samples via `perf record --call-graph fp` (or dwarf fallback).
- Captures allocation traces via `heaptrack`.
- Renders five SVGs: `cpu.svg`, `mem-peak.svg`, `mem-leaked.svg`, `mem-allocations.svg`, `mem-temporary.svg`.
- Supports `--binary NAME`, `--mode cpu|mem|both`, `--freq`, `--call-graph`, `--out-dir`, `--no-keep-trace`, `--check`, `--list`.
- Supports compile-time test filtering via `-Dprofile-filter=PATTERN`.

What is **missing**:

- **Metric extraction** — nobody reads numeric totals out of the folded files.
- **Baseline storage** — no format, no location, no capture command.
- **Comparison logic** — no thresholds, no verdict output.
- **CI integration** — no workflow that runs the gate automatically.

## 3. Design

### 3.1 Metrics — what we extract from folded files

Each folded file is a text file where every line is `<stack> <count>` and the first line is `total <count>`. We extract:

| Source file | Metric key | Extraction | What it catches |
|---|---|---|---|
| `cpu.folded` | `cpu_total_samples` | `head -1`, second column | "code got slower overall" |
| `cpu.folded` | `cpu_unique_stacks` | `wc -l` minus 1 | "hot path fragmentation changed" |
| `mem-peak.folded` | `mem_peak_bytes` | `head -1`, second column | "worst-case RSS grew" |
| `mem-leaked.folded` | `mem_leaked_bytes` | `head -1`, second column | "new leak introduced" (should be 0) |
| `mem-allocations.folded` | `mem_allocation_count` | `head -1`, second column | "malloc churn increased" |
| `mem-temporary.folded` | `mem_temporary_bytes` | `head -1`, second column | "short-lived allocs grew" |

All extracted as plain JSON numbers. Extraction is a one-line `head -1` + `cut` + `wc -l` pipe, but the driver will do it natively to avoid shell dependencies.

### 3.2 Baseline format

A single JSON file, ~300 bytes, stored in the repo at:

```
baselines/<binary>/<optimize>/baseline.json
```

Example (`baselines/franky-test/ReleaseSafe/baseline.json`):

```json
{
  "version": 1,
  "created_at_ms": 1714992000000,
  "source_commit": "abc1234def",
  "binary": "franky-test",
  "optimize": "ReleaseSafe",
  "filters": [],
  "seed": "deadbeef",
  "runs": 1,
  "cpu": {
    "total_samples": 52341,
    "unique_stacks": 847
  },
  "mem": {
    "peak_bytes": 12345678,
    "allocation_count": 98765,
    "leaked_bytes": 0,
    "temporary_bytes": 123456
  }
}
```

Fields:

- `version` — schema version (start at 1; bump on breaking changes).
- `created_at_ms` — unix ms of capture.
- `source_commit` — git SHA of the baseline commit.
- `binary` / `optimize` / `filters` — capture context so the gate finds the right baseline.
- `seed` — pinned test-seed for reproducibility.
- `runs` — how many runs were median-ed (1 for single-run, 3+ for median-of-N).
- `cpu` / `mem` — the numeric metrics from §3.1.

### 3.3 Threshold configuration

Three tiers, selectable by flag or environment variable:

| Tier | Flag | CPU threshold | Mem threshold | Leaked bytes | CI behaviour |
|---|---|---|---|---|---|
| Strict | `--tier strict` | > 5% | > 5% | > 0 | FAIL (exit 1) |
| Normal | `--tier normal` (default) | > 15% | > 10% | > 0 | WARN (exit 0, print to stderr) |
| Permissive | `--tier permissive` | > 30% | > 25% | > 0 | WARN only |

**`leaked_bytes > 0` is always a FAIL** regardless of tier — it means the GPA leak checker missed something (C-malloc allocations from vendored deps, heaptrack-only-visible leaks). A non-zero leak is a real bug.

Thresholds are intentionally wide initially. Tightening happens after 2–4 weeks of real data shows what the noise floor is.

### 3.4 Gate workflow (two commands)

```
# Capture a baseline (run on main after every merge)
zig build profile -- --mode baseline-capture --seed deadbeef \
  --out-dir baselines/franky-test/ReleaseSafe

# Check a PR against a baseline
zig build profile -- --mode baseline-check \
  --baseline baselines/franky-test/ReleaseSafe/baseline.json \
  --seed deadbeef
```

Both re-use the existing FP-preserved test binary build and the perf/heaptrack pipelines. The difference is the post-processing: instead of rendering SVGs, they extract metrics and either write or compare JSON.

### 3.5 `baseline-capture` mode

After the perf/heaptrack captures complete:

1. Read `cpu.folded` first line → `cpu_total_samples`.
2. Count non-total lines in `cpu.folded` → `cpu_unique_stacks`.
3. For each of the four `mem-*.folded` files → read first line for the corresponding metric.
4. Build the JSON (§3.2).
5. Write to `<out-dir>/baseline.json`.
6. Print summary and artefact sizes (same as `--mode both` but also prints the JSON path).

Does **not** delete the `.zst`/`.folded`/`.svg` artefacts — the full SVG set is still useful for eyeballing. The user can add `--no-keep-trace` to strip raw traces.

### 3.6 `baseline-check` mode

1. Load baseline JSON from `--baseline PATH`.
2. Run the same binary with the same seed, capture CPU + mem.
3. Extract metrics from the new folded files.
4. For each metric, compute `delta = (new - baseline) / baseline`.
5. Compare against tier thresholds (§3.3).
6. Print a verdict table:

```
=== Performance Baseline Check ===
binary:        franky-test (ReleaseSafe)
baseline:      abc1234 (1714992000000)
seed:          deadbeef

Metric               Baseline     New      Delta   Verdict
───────────────────────────────────────────────────────────
cpu_total_samples     52341      55239     +5.5%  ⚠️ WARN
cpu_unique_stacks     847         862      +1.8%  ✅
mem_peak_bytes       12345678   13580246   +10.0% ⚠️ WARN
mem_allocation_count  98765      100123     +1.4%  ✅
mem_leaked_bytes      0           0         0.0%   ✅
mem_temporary_bytes   123456     129629     +5.0%  ✅

Result: ⚠️ WARN (2 metrics above normal threshold but below strict)
```

7. Exit code: 0 if all metrics pass, 1 if any metric hits the FAIL tier, 2 on infrastructure errors (missing binary, missing baseline, tool not found).

### 3.7 Determinism and variance

Tests are not perfectly deterministic:

- **CPU samples** fluctuate with kernel scheduler decisions, ASLR, and background load.
- **Heap peak** varies with page alignment and allocator bookkeeping.
- **Allocation count** is the most stable metric (same code → same malloc calls).

Mitigations:

1. **Pinned seed** — `--seed deadbeef` gives the same test order on every run. A hardcoded seed is fine for baselines; users can override with `--seed` on the check command if they want to test with different randomisation.
2. **Median-of-3** — `--runs 3` runs the binary 3 times and takes the median of each metric. This adds ~2× the runtime (3 runs instead of 1). Default is single-run for speed; median-of-3 is a `--precise` flag.
3. **Wide default thresholds** (15% CPU, 10% mem) — absorbs typical noise; only flags real regressions.
4. **CI isolation** — running in a dedicated CI runner with no other load reduces variance further. The thresholds assume CI-level isolation.

### 3.8 Where the logic lives

Two options:

**Option A — New mode in `franky_profile.zig` (preferred for MVP)**

Add `--mode baseline-capture` and `--mode baseline-check` to the existing `Options` enum. The existing build + capture pipeline is reused verbatim; only the post-processing differs. Add a `Baseline` struct with `load()`, `save()`, and `compare()` methods in a new `src/profile/baseline.zig` module.

Pro: zero new binaries, zero build.zig churn. The profile driver is already Linux-only and tool-checked.

Con: the `--mode` enum gets another two variants. But the driver is already designed for mode-based dispatch, so this is a natural extension.

**Option B — Separate `franky-baseline` binary**

A standalone binary that accepts only `capture` and `check` subcommands. Simpler CLI surface, no flag overload.

Con: needs its own build step in `build.zig`, its own FP-preserved binary dependency, its own tool check.

**Decision (proposal):** ✓ Option A for the MVP. The profile driver already has everything we need. If the flags get unwieldy, we can spin out later.

New file: `src/profile/baseline.zig` containing:

```zig
pub const Baseline = struct {
    version: u32,
    created_at_ms: u64,
    source_commit: []const u8,
    binary: []const u8,
    optimize: []const u8,
    filters: [][]const u8,
    seed: []const u8,
    runs: u32,
    cpu: struct {
        total_samples: u64,
        unique_stacks: u64,
    },
    mem: struct {
        peak_bytes: u64,
        allocation_count: u64,
        leaked_bytes: u64,
        temporary_bytes: u64,
    },

    pub fn save(self: *const Baseline, path: []const u8) !void;
    pub fn load(path: []const u8, allocator: std.mem.Allocator) !Baseline;
    pub fn compare(baseline: *const Baseline, current: *const Baseline, tier: Tier) CompareResult;
};
```



## 4. Out of scope (explicit non-goals for v2.16)

- **Full folded-diff comparison** (the `inferno-diff-folded` path). Aggregate metrics first; if they flag, the developer runs the full SVG pipeline manually. Adding folded-diff to the gate would mean bundling and calling `inferno-diff-folded`, which adds a Rust dependency and a CLI parsing step. Not worth it for v1.
- **Multi-binary baselines.** v1 captures only `franky-test` (the unit-test binary). Integration binaries (`franky-agent_loop_test`, etc.) are larger and slower; they can be added as a follow-up.
- **Multi-optimization-mode baselines.** v1 captures only `ReleaseSafe`. `Debug` and `ReleaseFast` baselines can be added later.
- **Per-test-function granularity.** The gate operates on overall test-suite metrics. Per-function breakdowns require per-function allocation tracking, which is a separate project.
- **Cross-architecture baselines.** The baseline records `source_commit` but not the target triple. If CI runners change architecture (x86_64 → aarch64), baselines must be re-captured. Detection of arch mismatch and a warning is a v2 improvement.
- **Automatic PR comments.** v2.16 specifies the exit codes and output format; the CI integration and comment-posting logic depends on which CI system is used and lives outside this design doc.
- **Webhook / dashboard.** The gate produces exit codes and text output. A dashboard that tracks baseline history is a separate project.

## 5. Phased implementation roadmap

```
Phase 1   Baseline capture + JSON I/O                 (1-2 days)
Phase 2   Baseline check + threshold comparison        (1-2 days)
Phase 3   CI integration + docs                          (1 day)
```

**Phase 1** — Add `src/profile/baseline.zig` with the `Baseline` struct, `save()`, `load()`, and helper functions. Add `--mode baseline-capture` to the profile driver. Test by capturing a baseline by hand.

**Phase 2** — Add `compare()`, `Tier` enum, threshold logic, verdict-printing. Add `--mode baseline-check` to the profile driver. Test by modifying a test to allocate more and verifying the gate catches it.

**Phase 3** — Wire the capture/check commands into the project's CI system, update `docs/archive/profiling_guide.md` with the baseline/gate sections, and add a `zig build profile -- --help` reference card. Ship the first baseline from `main`.
