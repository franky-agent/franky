# v2.14 — Agentic Engineering

**Status:** brainstorming — open for discussion
**Architectural posture:** **mostly sibling-app territory.** franky-core
gets a small surface of new primitives (mostly already in flight as
v2.10–v2.12); the trigger / source / report / oversight layers are
implemented as **independent Zig sibling projects** that depend on
`franky.sdk`, following the v1 §O / v2 §3 pattern set by `franky-do`
(Slack) and `franky-pods` (vLLM).

---

## 1. What "agentic engineering" means here

Today franky is interactive: a human types a prompt, watches the
stream, intervenes if something looks wrong. Agentic engineering is
the move to **unattended operation** — work arrives from somewhere,
gets done by an agent without a human in the live loop, and gets
reported back to a channel a human will check later.

The doc's four seed questions name the four pillars of that shape:

1. **Source** — where the work comes from, in what format.
2. **Reporting** — how results land in front of a human, in what format.
3. **Oversight** — how a human supervises a run that's already going
   or has already finished.
4. **Trigger** — what wakes the agent up to start a run.

Each pillar is largely a sibling-app concern. franky-core's job is to
make sure the primitives a sibling needs (hard budgets, stop
conditions, structured progress, branching discipline) are reliable
enough to trust under unattended operation.

## 2. Prerequisites — design docs that must land first

Unattended operation is **not safe** with today's franky. The model
will happily grind for hours on a doomed task, retry the same broken
edit twenty times, never emit `end_turn`, and crash on the first
provider 503. Four sibling design documents harden the loop and
**must ship before** this one can be implemented:

| Doc | What it adds | Why agentic engineering needs it |
|---|---|---|
| [v2.10 — Harness-Enforced Guardrails](./v2.10-harness-enforced-guardrails.md) | Compilation check after N edits, secret scanner, `finish_task` tool with auto-commit, per-session branching | Without these the agent can ship broken code, leak secrets, or never declare a task complete. The `finish_task` tool is the canonical "I'm done — here's a commit and a summary" handshake the sibling app reads. |
| [v2.11 — Model-Stuck-Detector](./v2.11-model-stuck-detector.md) | Repeat-error detection → hint → hard-stop escalation | Without this an unattended run loops on the same `edit_no_match` indefinitely. |
| [v2.12 — End-Turn Enforcement](./v2.12-end-turn-enforcement.md) | Periodic checkpoint injection, hard budget with forced summarisation, `end_turn_rate` observability | Without this the agent never yields and budget alone is the only stop. v2.12 makes "stuck" loud at the metric level too. |
| [v2.13 — Hosted Model Service Unavailability](./v2.13-model-service-unavailabl.md) | Tunable retry budget for transient provider failures, retry-progress events, optional fallback-model chain in settings | Without this an unattended run dies the moment Anthropic / Cloudflare / etc. has a 503 storm. The default 3-retry transport policy is fine for interactive use; agentic engineering needs longer budgets and visibility into retry state so siblings can surface "agent paused, retrying" rather than "agent crashed." |

Nice-to-have but not blocking:

- [v2.9 — `.contextignore`](./decided/2.9-documentation-speification-archive.md) (shipped) — keeps the agent's discovery surface focused; especially useful when an unattended agent grep/find/ls walks the workspace looking for work.

The dependency chain is therefore: **v2.10 + v2.11 + v2.12 + v2.13 → v2.14**.

## 3. Architecture — what's in core, what's in siblings

```
┌────────────────────────────────────────────────────────────────┐
│  Sibling apps (each a separate Zig project, depends on sdk)    │
│                                                                │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────────────┐    │
│  │  franky-do   │ │ franky-issues│ │  franky-cron         │    │
│  │  (Slack)     │ │ (GitHub/Lin.)│ │  (scheduled sweeps)  │    │
│  │ trigger:     │ │ trigger:     │ │ trigger:             │    │
│  │  socket-mode │ │  webhook     │ │  cron table          │    │
│  │ source:      │ │ source:      │ │ source:              │    │
│  │  thread msg  │ │  issue body  │ │  task spec on disk   │    │
│  │ report:      │ │ report:      │ │ report:              │    │
│  │  thread reply│ │  PR + comment│ │  PR or summary file  │    │
│  └──────┬───────┘ └──────┬───────┘ └────────┬─────────────┘    │
│         │                │                  │                  │
│         └────────────────┼──────────────────┘                  │
│                          │ embeds / spawns                     │
└──────────────────────────┼─────────────────────────────────────┘
                           ▼
┌────────────────────────────────────────────────────────────────┐
│  franky-core (this repo)                                       │
│                                                                │
│   sdk.zig             — programmatic embedding API             │
│   --mode rpc          — JSON-RPC over stdio for subprocess use │
│   --mode proxy        — HTTP/SSE for live oversight            │
│   permissions / role  — what tools can run unattended          │
│   sessions            — persist + resume + branch              │
│   v2.10–v2.12 work    — guardrails making unattended safe      │
└────────────────────────────────────────────────────────────────┘
```

**Rule of thumb for what lives where:**

- Anything that talks to an external service (Slack, GitHub, cron,
  Redis, SQS, …) → **sibling**.
- Anything that listens for external events (HTTP webhooks, queues)
  → **sibling**.
- Anything that affects the agent loop's safety, observability, or
  stop conditions → **franky-core** (v2.10–v2.12 territory).
- The `finish_task` tool, branching discipline, structured progress
  events, per-tool permission gates → **franky-core**.

This split keeps franky-core small and lets each integration evolve
on its own schedule with its own dependencies (Slack SDK, GitHub API
client, Redis driver, …) without bloating the core binary.

## 4. The four pillars

### 4.1 Source — where the work comes from

Spectrum from low-effort to ambitious:

| Source | Sibling project | Format the agent receives |
|---|---|---|
| File drop | `franky-cron` or `franky-watch` | YAML envelope + free-text body in `~/work/franky-tasks/<id>.yaml` |
| CLI queue | `franky-queue` | Same envelope, persisted in a SQLite/JSON store |
| Issue tracker | `franky-issues` | Issue title + body + labels, posted via webhook |
| PR review | `franky-pr` | PR title + body + diff + review comments |
| Chat | `franky-do` (Slack), future `franky-discord` | Thread-scoped message text |
| Message queue | `franky-worker` | Envelope pulled from Redis stream / NATS / SQS |

**Format convention** (proposal): each sibling normalises its trigger
into a small **TaskSpec** before invoking franky:

```yaml
task_id: "issue-1234"
goal: "Fix the regression in src/foo.zig that broke `zig build test`."
acceptance: |
  - `zig build test` passes.
  - The change is minimal and confined to src/foo.zig.
context:
  repo: org/repo
  base_branch: main
  workspace: /tmp/franky-tasks/issue-1234   # ephemeral worktree
constraints:
  max_turns: 30
  max_minutes: 20
  ask_on:
    - bash:rm
    - bash:curl
    - write_outside_repo
report_target:
  type: github_issue_comment
  ref: "org/repo#1234"
```

The body is free-text (the model parses meaning); the envelope is
structured so siblings and oversight tools can route, filter, and
budget without a model call.

### 4.2 Reporting — how it lands in front of a human

**Strong principle: report back to the source channel.** An issue
gets a comment on the issue. A PR gets a review on the PR. A Slack
thread gets a reply in the thread. Humans already check those
channels — don't make them check a new dashboard for the result.

Per-task artefact stack (each layer optional, ordered by canonicality):

1. **Branch + PR** — every code change lands on `franky/<task_id>`,
   never on the default branch. The PR description is the canonical
   summary. (Depends on v2.10 §5 auto-commit.)
2. **Markdown summary** posted to the source channel: what was tried,
   what worked, what's still unknown, time spent, tokens used.
3. **One-line status** in the channel envelope: `Done — opened
   #PR123` / `Blocked — needs API key for X` / `Failed — see logs`.
4. **Replay link** — URL to a `franky --mode proxy` instance that
   can scrub the full session transcript for post-mortem.

### 4.3 Oversight — how a human supervises

Three windows:

- **Pre-flight** (before the agent runs): per-tool policy declared
  in the TaskSpec or sibling config. `--prompts` + `--allow-tools` +
  `--deny-tools` already cover this; the sibling translates its
  config into those flags when it spawns the session.
- **Live** (mid-run): the sibling exposes the session's
  `--mode proxy` URL on its dashboard. Humans get an abort button,
  a steer / followUp box, a transcript scrub. (The proxy mode +
  reconnect-replay shipped in v1.16.0 already supports this.)
- **Post-flight** (after the run): the report-back artefact (§4.2)
  is the audit log. PR review is the canonical gate before code
  lands on `main`.

**Approval gates.** When a tool call sits outside the pre-declared
allowlist (e.g. `bash:rm`), the agent emits a
`tool_permission_request` event (already shipped per v1.11.4) and
the sibling routes it to the source channel: "I'd like to run
`rm -rf .zig-cache` — 👍 to allow." Slow path, but bounds the blast
radius. Skipping the prompt → deny → forced summary → reported as
"blocked: needed permission for X."

### 4.4 Trigger — what wakes the agent up

Triggers are pure sibling concerns; franky-core is unaware of them.
Realistic shapes per sibling:

| Sibling | Trigger mechanism |
|---|---|
| `franky-do` | Slack Socket Mode (already implemented; existing project) |
| `franky-issues` | GitHub webhook listener (Zig HTTP server in the sibling) |
| `franky-cron` | OS-level cron / systemd timer firing the sibling binary |
| `franky-watch` | `inotify` / FSEvents on a watched directory |
| `franky-worker` | Long-running process subscribed to a queue |

**The "infinite autonomous work" framing.** The realistic shape is
**not** a long-running daemon thinking continuously. It's:

```
trigger fires → pick exactly one task → run under hard budget →
  report → sleep until next trigger
```

Each run has a defined scope and a hard ceiling. Continuous
autonomous behaviour emerges from the **rate of triggers**, not from
a single never-ending session. This is a design constraint that
v2.10/v2.12 enforce at the loop level.

## 5. Cross-cutting design constraints

These apply to every sibling and shape the franky-core work needed:

1. **Branching discipline.** Every autonomous run creates
   `franky/<task_id>` and never pushes to a default branch. Comes
   from v2.10 §5 auto-commit; v2.14 cements it as a non-negotiable.

2. **Per-task isolation.** Each task gets its own session, its own
   workspace (probably a `git worktree` or fresh clone, ephemeral),
   its own log directory. Crashes / aborts don't pollute peers.

3. **Hard budgets, always.** Every run sets `max_turns` AND
   `max_turns_summarize: true` (v2.12). Without the second, a hit
   cap returns `agent_error{max_turns_exceeded}` and the sibling has
   nothing useful to report.

4. **Default-deny tool surface.** Autonomous mode defaults to
   `--role plan` (workspace edits, no shell) unless the TaskSpec
   explicitly opts up. Anything outside the workspace requires
   explicit allow.

5. **Replay-able by construction.** Every run writes its session
   transcript + http-trace to disk so post-mortems work even when
   the live oversight window has closed. v1.16.1's `--http-trace-dir`
   and v2.3.0's `franky fixture` pipeline already do most of this.

## 6. franky-core surface this needs

What v2.14 itself adds to franky-core (vs. what siblings build on
top). Should be **small** if v2.10–v2.12 land first:

- **Possibly**: a `franky.sdk.runTask(spec)` convenience that bundles
  "set up workspace + apply per-tool policy + run under budget +
  return the canonical artefact (commit + summary + status)" so
  every sibling doesn't re-implement the same plumbing.
- **Possibly**: a `task_id` concept in `Settings` / `Session` so
  log files and transcript paths can sort by task instead of by
  random ULID.
- **Documented**: a sibling-app reference / template repo showing
  the minimal scaffolding needed to consume `franky.sdk` for an
  unattended run with reporting back. (Mirrors how v2 §3.1
  documents `franky-do`.)

Most of the actual logic — webhook listeners, cron tables, queue
consumers, GitHub API clients, Slack integrations — stays out of
franky-core.

## 7. Open narrowing questions

Three questions whose answers shape the rest of the design.
Brainstorm pauses here until they're answered:

### Q1 — What's the smallest useful instance to design first?

Pick one to anchor the design. Options:

- **A.** `franky-issues` — agent watches GitHub Issues with label
  `agent-fix` and opens PRs. Highest leverage; canonical use case.
- **B.** `franky-cron` — nightly sweeps for known patterns (failing
  tests, stale dependencies, security advisories). Lower trust
  surface; easier to bound.
- **C.** Extend `franky-do` (existing) with auto-PR — Slack remains
  the trigger, but completion always opens a branch + PR. Builds on
  shipped infra.
- **D.** `franky-queue` — minimal CLI: `franky-queue add "task"` +
  `franky-queue worker`. No external integration; useful for solo
  users.

**Decision:** _____

### Q2 — Where does it run?

- **A.** User's laptop after `franky-<sibling> --daemon` (simple,
  user keeps the keys, dies when the laptop sleeps).
- **B.** A small server / VM the user owns (always-on, user keeps
  the keys).
- **C.** GitHub Actions / equivalent CI runner per task (ephemeral,
  no daemon, leverages existing CI auth).
- **D.** Container fleet (Kubernetes / Nomad — out of scope for
  early v2.14).

**Decision:** _____

### Q3 — What's the trust budget for v1?

- **A.** Read-only investigations. Agent can read, search, and
  comment but never writes. Lowest blast radius; useful for "is
  this issue still reproducible?" / "what's the right fix?" advice.
- **B.** Workspace edits + commit only (default). Agent commits to
  `franky/<task_id>`; never pushes. Human pulls + pushes.
- **C.** Push to `franky/<task_id>` and open a PR. Agent has a
  scoped token. The PR is the gate.
- **D.** Push to default branch. Probably not v1 — too risky.

**Decision:** _____

## 8. Next step

Once Q1–Q3 are answered, the brainstorm narrows to designing the
chosen smallest-useful-instance in detail: its TaskSpec schema, its
trigger plumbing, its report-back format, and what (if anything)
franky-core needs to add to support it. The other siblings then
follow the same shape.

Until then this stays a sketch — and v2.10/v2.11/v2.12 stay the
critical-path work that has to land first regardless of which
sibling ships first.
