# v2.14 — Agentic Engineering

**Status:** design review — 8 open questions identified from codebase audit (see §9)
**Architectural posture:** **mostly sibling-app territory.** franky-core
gets a small surface of new primitives (mostly defined by the
v2.10–v2.13 prerequisite docs); the trigger / source / report / oversight layers
are implemented as **independent Zig sibling projects** that depend on
`franky.sdk`, following the v1 §O / v2 §3 pattern set by `franky-do`
(Slack) and `franky-pods` (vLLM).

---

## 1. What "agentic engineering" means here

Today franky is interactive: a human types a prompt, watches the
stream, intervenes if something looks wrong. Agentic engineering is
the move to **unattended operation** — work arrives from somewhere,
gets done by an agent without a human in the live loop, and gets
reported back to a channel a human will check later.

The doc's four seed questions name the four pillars of that shape:

1. **Source** — where the work comes from, in what format.
2. **Reporting** — how results land in front of a human, in what format.
3. **Oversight** — how a human supervises a run that's already going
   or has already finished.
4. **Trigger** — what wakes the agent up to start a run.

Each pillar is largely a sibling-app concern. franky-core's job is to
make sure the primitives a sibling needs (hard budgets, stop
conditions, structured progress, branching discipline) are reliable
enough to trust under unattended operation.

## 2. Prerequisites — design docs that must land first

Unattended operation is **not safe** with today's franky. The model
will happily grind for hours on a doomed task, retry the same broken
edit twenty times, never emit `end_turn`, and crash on the first
provider 503. Four sibling design documents harden the loop and
**must ship before** this one can be implemented:

| Doc | What it adds | Implementation status (July 2026) | Why agentic engineering needs it |
|---|---|---|---|
| [v2.10 — Harness-Enforced Guardrails] | Compilation check after N edits, secret scanner, `finish_task` tool with auto-commit, per-session branching | **Largely implemented.** `finish_task` tool, `compilation_guard`, `GuardrailState`, auto-commit, branching (`franky/<ts>`), and `tool_permission_request` all present. No separate v2.10 doc file found — the functionality was built incrementally across several rounds. | Without these the agent can ship broken code, leak secrets, or never declare a task complete. The `finish_task` tool is the canonical "I'm done — here's a commit and a summary" handshake the sibling app reads. |
| [v2.11 — Model-Stuck-Detector] | Repeat-error detection → hint → hard-stop escalation | **Partially implemented.** `StuckDetector` struct exists in guardrails, wired into `GuardrailState.betweenTurns`. No separate v2.11 doc file found. | Without this an unattended run loops on the same `edit_no_match` indefinitely. |
| [v2.12 — End-Turn Enforcement](./decided/v2.12-end-turn-enforcement.md) | Periodic checkpoint injection, hard budget with forced summarisation, `end_turn_rate` observability | **NOT implemented.** `max_turns_summarize` exists but default is still `false` (decision says flip to `true` for proxy/interactive). No `checkpoint_enabled`, no `checkpoint_interval`, no `turns_since_last_yield` counter, no `end_turn_rate` / `checkpoints_fired` status-line fields, no `source` field on AgentMessage. The design is fully decided and ready to implement. | Without this the agent never yields and budget alone is the only stop. v2.12 makes "stuck" loud at the metric level too. |
| [v2.13b — Fallback Model Chain](./decided/v2.13b-fallback-model-chain.md) | Tunable retry budget for transient provider failures, retry-progress events, optional fallback-model chain in settings | **NOT implemented.** `provider_retry` event exists (Phase 1), but `fallback_models` / `fallback_scope` / `using_fallback` on the event payload are absent. No settings fields for fallback chain. | Without this an unattended run dies the moment Anthropic / Cloudflare / etc. has a 503 storm. The default 3-retry transport policy is fine for interactive use; agentic engineering needs longer budgets and visibility into retry state so siblings can surface "agent paused, retrying" rather than "agent crashed." |

Nice-to-have but not blocking:

- [v2.9 — `.contextignore`] (shipped) — keeps the agent's discovery surface focused; especially useful when an unattended agent grep/find/ls walks the workspace looking for work.

The dependency chain stands: **v2.10 ≈ v2.11 → v2.12 + v2.13b → v2.14**. v2.12 and v2.13b are fully designed (decided docs exist) but not yet implemented. v2.14 design should account for their shapes but shouldn't block on their implementation — they can be built in parallel.

## 3. Architecture — what's in core, what's in siblings

```
┌────────────────────────────────────────────────────────────────┐
│  Sibling apps (each a separate Zig project, depends on sdk)    │
│                                                                │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────────────┐    │
│  │  franky-do   │ │ franky-issues│ │  franky-cron         │    │
│  │  (Slack)     │ │ (GitHub/Lin.)│ │  (scheduled sweeps)  │    │
│  │ trigger:     │ │ trigger:     │ │ trigger:             │    │
│  │  socket-mode │ │  webhook     │ │  cron table          │    │
│  │ source:      │ │ source:      │ │ source:              │    │
│  │  thread msg  │ │  issue body  │ │  task spec on disk   │    │
│  │ report:      │ │ report:      │ │ report:              │    │
│  │  thread reply│ │  PR + comment│ │  PR or summary file  │    │
│  └──────┬───────┘ └──────┬───────┘ └────────┬─────────────┘    │
│         │                │                  │                  │
│         └────────────────┼──────────────────┘                  │
│                          │ embeds / spawns                     │
└──────────────────────────┼─────────────────────────────────────┘
                           ▼
┌────────────────────────────────────────────────────────────────┐
│  franky-core (this repo)                                       │
│                                                                │
│   sdk.zig             — programmatic embedding API             │
│   --mode rpc          — JSON-RPC over stdio for subprocess use │
│   --mode proxy        — HTTP/SSE for live oversight            │
│   permissions / role  — what tools can run unattended          │
│   sessions            — persist + resume + branch              │
│   v2.10–v2.12 work    — guardrails making unattended safe      │
└────────────────────────────────────────────────────────────────┘
```

**Rule of thumb for what lives where:**

- Anything that talks to an external service (Slack, GitHub, cron,
  Redis, SQS, …) → **sibling**.
- Anything that listens for external events (HTTP webhooks, queues)
  → **sibling**.
- Anything that affects the agent loop's safety, observability, or
  stop conditions → **franky-core** (v2.10–v2.13 territory).
- The `finish_task` tool, the `task_create` tool (§4.1.4),
  branching discipline, structured progress events, per-tool
  permission gates → **franky-core**.

The runner that watches a queue directory, atomic-claims taskfiles,
spawns franky sessions, and writes back artefacts is its own
sibling project (`franky-tasks` — see §4.1.3 for the filesystem
state machine). franky-core does not contain a queue runner.

This split keeps franky-core small and lets each integration evolve
on its own schedule with its own dependencies (Slack SDK, GitHub API
client, Redis driver, …) without bloating the core binary.

## 4. The four pillars

### 4.1 Source — where the work comes from

Spectrum from low-effort to ambitious:

| Source | Sibling project | Format the agent receives |
|---|---|---|
| File drop | `franky-tasks` (runner) + `franky-cron`, `franky-watch` (producers) | YAML envelope + free-text body in `<queue>/pending/<id>.yaml` |
| CLI queue | `franky-queue` | Same envelope, persisted in a SQLite/JSON store |
| Issue tracker | `franky-issues` | Issue title + body + labels, posted via webhook |
| PR review | `franky-pr` | PR title + body + diff + review comments |
| Chat | `franky-do` (Slack), future `franky-discord` | Thread-scoped message text |
| Message queue | `franky-worker` | Envelope pulled from Redis stream / NATS / SQS |
| **Agent itself** | (in-core `task_create` tool) | Same TaskSpec written into `<queue>/pending/` by the running session |

#### 4.1.1 The file-drop pattern is the canonical primitive

Every other source above ultimately normalises into the file-drop
shape: a YAML on disk picked up by a runner. Webhooks, cron, chat
mentions, and self-spawned follow-ups are all just *producers* that
land a YAML in the queue. The runner doesn't care who wrote the
file. This collapses six different siblings into one runner +
N small producers.

#### 4.1.2 TaskSpec format

```yaml
task_id: "issue-1234"
goal: "Fix the regression in src/foo.zig that broke `zig build test`."
acceptance: |
  - `zig build test` passes.
  - The change is minimal and confined to src/foo.zig.
context:
  repo: org/repo
  base_branch: main
  workspace: /tmp/franky-tasks/issue-1234   # ephemeral worktree
constraints:
  max_turns: 30
  max_minutes: 20
  role: plan                                # default if omitted
  ask_on:
    - bash:rm
    - bash:curl
    - write_outside_repo
created_by:
  source: human | cron | sibling | agent
  parent_task_id: null                      # set when source: agent
  parent_session_id: null                   # set when source: agent
  depth: 0                                  # parent.depth + 1 when source: agent
report_target:
  type: github_issue_comment
  ref: "org/repo#1234"
```

The body is free-text (the model parses meaning); the envelope is
structured so siblings and oversight tools can route, filter, and
budget without a model call.

#### 4.1.3 Filesystem state machine

The queue directory encodes lifecycle through atomic `rename()`
between sibling subdirs. No locks, no DB:

```
<queue>/
  pending/<id>.yaml      ← producers write here
  claimed/<id>.yaml      ← runner atomic-renames before processing
  done/<id>.yaml         ← + sibling result.json
  failed/<id>.yaml       ← + sibling failure.json
  blocked/<id>.yaml      ← agent yielded; needs human triage
```

A single `rename(pending/X, claimed/X)` is the atomic claim — two
runners watching the same queue can never both pick up the same
task. State transitions are visible to humans via `ls`.

#### 4.1.4 Agent-produced tasks (`task_create` tool)

A running franky session can emit new taskfiles via a structured
`task_create` tool in franky-core. This unifies four otherwise-
special cases under one mechanism:

| Use case | Without `task_create` | With `task_create` |
|---|---|---|
| Out-of-scope spin-off | scope creep or silent drop | new task in `pending/` |
| Hard guardrail hit | `agent_error{...}` and lose context | `blocked/` task with full context for human triage |
| Follow-up suggestions | prose buried in PR body | one task per follow-up, each with own scope/budget |
| Decomposition | one giant task with one big budget | parent → N children, each scoped tight |

The tool's parameter shape mirrors §4.1.2's TaskSpec, with
`created_by.source: agent` and `parent_task_id` /
`parent_session_id` / `depth` populated automatically by
franky-core (the model can't forge provenance).

**Loop-prevention rules** (enforced by the tool, not the model):

1. **Per-session cap.** A single session can call `task_create` at
   most `Settings.task_create_max_per_session` times (default: 10).
   The 11th call returns `tool_error{task_create_cap}`.
2. **Depth limit.** A task with `depth >= Settings.task_create_max_depth`
   (default: 5) cannot create children. Caps a runaway chain at
   A → B → C → D → E → (F refused).
3. **Scope-only-narrows.** Child task's `constraints.role` cannot
   be more permissive than the parent's; budgets cannot exceed the
   parent's remaining budget. Tool refuses with
   `tool_error{task_create_scope_escalation}`.
4. **Schema-validated.** Same JSON-schema validation as any other
   tool — typo'd fields are caught before the file lands.

These rules turn "agent self-tasking" from a footgun into a bounded
capability. Every taskfile in the queue carries provenance back to
either a human, a sibling, or a specific parent run, so runaway
chains are auditable.

### 4.2 Reporting — how it lands in front of a human

**Strong principle: report back to the source channel.** An issue
gets a comment on the issue. A PR gets a review on the PR. A Slack
thread gets a reply in the thread. Humans already check those
channels — don't make them check a new dashboard for the result.

Per-task artefact stack (each layer optional, ordered by canonicality):

1. **Branch + PR** — every code change lands on `franky/<task_id>`,
   never on the default branch. The PR description is the canonical
   summary. (Depends on v2.10 §5 auto-commit.)
2. **Markdown summary** posted to the source channel: what was tried,
   what worked, what's still unknown, time spent, tokens used.
3. **One-line status** in the channel envelope: `Done — opened
   #PR123` / `Blocked — needs API key for X` / `Failed — see logs`.
4. **Replay link** — URL to a `franky --mode proxy` instance that
   can scrub the full session transcript for post-mortem.

### 4.3 Oversight — how a human supervises

Three windows:

- **Pre-flight** (before the agent runs): per-tool policy declared
  in the TaskSpec or sibling config. `--prompts` + `--allow-tools` +
  `--deny-tools` already cover this; the sibling translates its
  config into those flags when it spawns the session.
- **Live** (mid-run): the sibling exposes the session's
  `--mode proxy` URL on its dashboard. Humans get an abort button,
  a steer / followUp box, a transcript scrub. (The proxy mode +
  reconnect-replay shipped in v1.16.0 already supports this.)
- **Post-flight** (after the run): the report-back artefact (§4.2)
  is the audit log. PR review is the canonical gate before code
  lands on `main`.

**Approval gates.** When a tool call sits outside the pre-declared
allowlist (e.g. `bash:rm`), the agent emits a
`tool_permission_request` event (already shipped per v1.11.4) and
the sibling routes it to the source channel: "I'd like to run
`rm -rf .zig-cache` — 👍 to allow." Slow path, but bounds the blast
radius. Skipping the prompt → deny → forced summary → reported as
"blocked: needed permission for X."

### 4.4 Trigger — what wakes the agent up

Triggers are pure sibling concerns; franky-core is unaware of them.
Realistic shapes per sibling:

| Sibling | Trigger mechanism |
|---|---|
| `franky-do` | Slack Socket Mode (already implemented; existing project) |
| `franky-issues` | GitHub webhook listener (Zig HTTP server in the sibling) |
| `franky-cron` | OS-level cron / systemd timer firing the sibling binary |
| `franky-watch` | `inotify` / FSEvents on a watched directory |
| `franky-worker` | Long-running process subscribed to a queue |

**The "infinite autonomous work" framing.** The realistic shape is
**not** a long-running daemon thinking continuously. It's:

```
trigger fires → pick exactly one task → run under hard budget →
  report → sleep until next trigger
```

Each run has a defined scope and a hard ceiling. Continuous
autonomous behaviour emerges from the **rate of triggers**, not from
a single never-ending session. This is a design constraint that
v2.10/v2.12 enforce at the loop level.

## 5. Cross-cutting design constraints

These apply to every sibling and shape the franky-core work needed:

1. **Branching discipline.** Every autonomous run creates
   `franky/<task_id>` and never pushes to a default branch. Comes
   from v2.10 §5 auto-commit; v2.14 cements it as a non-negotiable.

2. **Per-task isolation.** Each task gets its own session, its own
   workspace (probably a `git worktree` or fresh clone, ephemeral),
   its own log directory. Crashes / aborts don't pollute peers.

3. **Hard budgets, always.** Every run sets `max_turns` AND
   `max_turns_summarize: true` (v2.12). Without the second, a hit
   cap returns `agent_error{max_turns_exceeded}` and the sibling has
   nothing useful to report.

4. **Default-deny tool surface.** Autonomous mode defaults to
   `--role plan` (workspace edits, no shell) unless the TaskSpec
   explicitly opts up. Anything outside the workspace requires
   explicit allow.

5. **Replay-able by construction.** Every run writes its session
   transcript + http-trace to disk so post-mortems work even when
   the live oversight window has closed. v1.16.1's `--http-trace-dir`
   and v2.3.0's `franky fixture` pipeline already do most of this.

6. **Provenance on every taskfile.** No taskfile lands in `pending/`
   without a `created_by` field naming its origin (human, cron,
   sibling, or agent + parent_task_id). Runaway chains are
   identifiable; oversight tools can filter "show me everything
   spawned by task X."

## 6. franky-core surface this needs

What v2.14 itself adds to franky-core (vs. what siblings build on
top). Should be **small** if v2.10–v2.13 land first:

- **`task_create` tool** (§4.1.4) — structured tool the agent calls
  to emit a child taskfile. Validates the TaskSpec schema, stamps
  provenance (`created_by.source: agent` + `parent_task_id` +
  `parent_session_id` + `depth`), and writes to a configured
  `<queue>/pending/` directory. Enforces three caps:
  `task_create_max_per_session`, `task_create_max_depth`, and the
  scope-only-narrows rule. ~50 LOC of tool plumbing on top of
  existing schema validation.
- **`task_id` in `Settings` / `Session`** — log files, transcript
  paths, and branch names sort by task instead of by random ULID.
  Required by §5.5 (replay) and §5.6 (provenance).
- **`franky.sdk.runTask(spec)` convenience** (optional) — bundles
  "set up workspace + apply per-tool policy + run under budget +
  return the canonical artefact (commit + summary + status)" so the
  runner sibling doesn't re-implement the boilerplate. Could also
  be a thin SDK helper rather than a new entry point.
- **Documented sibling-app reference** — a template repo showing
  the minimal scaffolding needed to consume `franky.sdk` for an
  unattended run with reporting back. Mirrors how v2 §3.1 documents
  `franky-do`.

Most of the actual logic — webhook listeners, cron tables, queue
runner, GitHub API clients, Slack integrations — stays out of
franky-core. The runner that watches `<queue>/pending/`, claims via
atomic rename, drives the franky session, and reports back is its
own sibling project (`franky-tasks`).

## 7. Decided narrowing questions

Four questions whose answers shape the rest of the design.
These are already decided — recorded here for reference:

### Q1 — What's the smallest useful instance to design first?

**Decision:** B — `franky-cron`. Nightly sweeps for known patterns
(failing tests, stale dependencies, security advisories). Lower trust
surface; easy to bound. Serves as the reference implementation for
all subsequent siblings once shipped.

### Q2 — Where does it run?

**Decision:** A — Laptop daemon (`franky-cron --daemon`). Runs as a
launchd plist (macOS) or systemd user unit (Linux). Acceptable for
nightly tasks; a missed run because the laptop was asleep simply
retries on the next scheduled wake.

### Q3 — What's the trust budget for v1?

**Decision:** C — Push to `franky/<task_id>` and open a PR. Agent
holds a fine-grained GitHub PAT scoped to the target repo. The PR
is the human gate before anything touches the default branch.

### Q4 — `task_create` defaults

**Decision:** Higher caps — `max_per_session = 10`, `max_depth = 5`.
Allows more complex decomposition while still bounding runaway chains.
Scope-only-narrows remains a hard rule with no override.

## 8. franky-cron — anchor design detail

`franky-cron` is a standalone Zig binary that runs as a laptop
daemon and drives franky sessions on a schedule. It is the reference
implementation for all subsequent sibling projects.

### 8.1 Config format

`~/.franky-cron/config.yaml` — a crontab-like list of sweep tasks.
Each entry extends the base TaskSpec (§4.1.2) with a `schedule` field:

```yaml
github_token: "${GITHUB_TOKEN}"   # fine-grained PAT, or keychain ref

sweeps:
  - task_id: nightly-test-sweep
    schedule: "0 2 * * *"         # 2am nightly
    goal: |
      Run `zig build test`. If any tests fail, investigate the root
      cause, fix it, and open a PR. If tests pass, do nothing.
    context:
      repo: org/repo
      base_branch: main
    constraints:
      max_turns: 40
      max_minutes: 30
      role: code
    report_target:
      type: github_pr
      repo: org/repo

  - task_id: weekly-dep-audit
    schedule: "0 3 * * 1"         # Monday 3am
    goal: |
      Check for stale or known-vulnerable dependencies. If any are
      found, bump the affected entries and open a PR with a summary.
    constraints:
      max_turns: 20
      max_minutes: 15
      role: code
    report_target:
      type: github_pr
      repo: org/repo
```

The `schedule` field is a standard 5-field cron expression.
`task_id` must be unique within the config; it becomes the branch
name (`franky/<task_id>-<YYYYMMDD>`).

### 8.2 Daemon lifecycle

```
startup
  └─ parse config, validate GitHub token, register launchd/systemd unit

on each scheduled wake
  1. resolve which sweeps are due (cron expression check)
  2. for each due sweep:
       a. create a git worktree at /tmp/franky-cron/<task_id>-<date>
       b. write TaskSpec → <queue>/pending/<task_id>.yaml
       c. claim it (rename pending → claimed)
       d. spawn franky session (--mode rpc or sdk.runTask)
       e. wait for finish_task or budget exhaustion
       f. if changes committed → git push franky/<task_id>-<date>
                                → open GitHub PR via API
       g. rename claimed → done (or failed)
       h. clean up worktree
  3. sleep until next scheduled wake
```

Sweeps within one wake run **sequentially by default** — a laptop
daemon is unlikely to have spare capacity for parallel sessions.
A `parallel: true` flag per sweep can opt in to parallel runs for
known-cheap read-only tasks.

### 8.3 v1 sweep catalog

Three sweeps ship in v1. Each is a pre-written TaskSpec template
bundled with `franky-cron`:

| Sweep | Default schedule | What it does |
|---|---|---|
| `test-sweep` | nightly 2am | `zig build test` (or `npm test` / `go test` per detected stack); opens PR if failures found and fixable |
| `dep-audit` | weekly Monday 3am | Checks build deps for known CVEs / stale versions; opens PR with minimal bumps |
| `doc-drift` | weekly Sunday 4am | Finds public symbols with no docstring or obviously stale comments; opens PR with stubs filled in |

Users can override, disable, or add custom sweeps in `config.yaml`.

### 8.4 GitHub integration

`franky-cron` needs two GitHub API calls per completed sweep:

1. `POST /repos/{owner}/{repo}/git/refs` — push the branch (or
   `PATCH` if it already exists from a previous run).
2. `POST /repos/{owner}/{repo}/pulls` — open the PR with:
   - title: `[franky-cron] <task_id> <YYYY-MM-DD>`
   - body: the `finish_task` summary emitted by the session
   - base: the `base_branch` from the sweep config
   - head: `franky/<task_id>-<date>`

Token scope required: `contents: write`, `pull-requests: write` on
the target repo. No org-level permissions needed.

### 8.5 What franky-core needs (beyond v2.10–v2.13)

`franky-cron` can ship with only one new franky-core addition beyond
the v2.10–v2.13 prerequisites:

- **`task_id` in `Settings`** (§6) — so log files, transcript paths,
  and the committed branch name sort by task instead of by ULID.
  The cron binary sets `settings.task_id = "<task_id>-<date>"` when
  spawning the session.

The above assumes v2.12 and v2.13b land before franky-cron ships. If they
don't, franky-cron's v1 scope should be narrowed:

- **Without v2.12:** An unattended run that hits `max_turns` reports
  `agent_error{max_turns_exceeded}` with no useful artefact. Workaround:
  the sibling can set `--max-turns-summarize` as a CLI flag (the setting
  exists, just off by default).
- **Without v2.13b:** A 503 storm on the primary provider kills the run.
  Workaround: the sibling wraps the spawn with a watchdog that retries
  the entire session from scratch (losing progress) — acceptable for
  the nightly-test-sweep use case since each run is relatively short.
- **Without `--mode rpc`:** falls back to `--mode print` for the first
  version. The sibling gets `finish_task` output via exit status + stdout
  rather than a streamed event. Less elegant but workable.

The GitHub push + PR creation lives entirely in the sibling binary.

## 9. Open questions from design audit

The following questions surfaced during a codebase review (July 2026).
They are additional to the four narrowing questions in §7, which are
already decided. Resolving these is the next step before v2.14 can
move to implementation.

### 9.1 — TaskSpec serialization format: YAML vs JSON

§4.1.2 defines the canonical TaskSpec as **YAML**. However:

- The `task_create` tool (§4.1.4, §6) uses JSON parameters (all franky tools
  use JSON schema). If the tool writes YAML, it needs a YAML serializer that
  doesn't exist in the codebase today. The codebase has ad-hoc YAML parsers
  (`.franky-workflow.yaml` in `compilation_guard.zig`, skill frontmatter in
  `skills.zig`) but no writer.
- Settings, auth, and all other franky config use **JSON**.
- The sibling reading the queue files is a Zig program (not a person), so
  readability of the format is less important than the model parsing the
  body (which is free-text regardless of envelope format).

**Proposal:** Use **JSON** for the on-disk TaskSpec envelope. The `task_create`
tool emits JSON natively. Sibling runners read JSON natively. The human-edited
franky-cron config (§8.1) stays YAML by convention (it's edited by a person),
and the cron binary converts each sweep entry to a JSON TaskSpec when writing
`<queue>/pending/<id>.json`.

**What this affects:** §4.1.2 (envelope format), §4.1.3 (file extension
`.yaml`→`.json`), `task_create` tool implementation (JSON parameters
serialized to a JSON file), franky-cron §8.1 config (stays YAML, translates
to JSON TaskSpec internally).

### 9.2 — Runner architecture: is franky-cron a runner or a producer?

§4.1.3 sketches a **shared queue runner** (`franky-tasks`) that watches
`<queue>/pending/`, claims via atomic rename, and spawns franky sessions.
§8.2's franky-cron lifecycle does steps 2b–2f **inline** — it writes to
pending, immediately claims it, and spawns the session itself. These are
inconsistent.

**Option A — franky-cron is a self-contained runner.** It produces + claims
+ spawns + reports, all inline. The shared queue (§4.1.3) is only used by
producers that don't have their own runner (e.g. `franky-webhook`, which just
writes to `pending/` and lets `franky-tasks` pick up). franky-cron skips the
queue entirely if no other runner is consuming from it — it writes directly
to a local working directory, spawns the session, and reports.

**Option B — franky-cron is just a producer.** It writes to `pending/` and
`franky-tasks` (a separate daemon) does the claiming + spawning. franky-cron
then only needs to know the queue directory path. This enforces uniform
lifecycle for all tasks regardless of origin, at the cost of running two
daemons instead of one.

**Option C — franky-cron is the reference runner, and its lifecycle *is*
the queue.** Steps 2b–2d are the "inline queue" — the pending/ directory
exists for decoupling (so other producers can feed the same queue), but
the canonical runner implementation is built into franky-cron itself.
`franky-tasks` becomes a separate project only if a second runner
implementation is needed later.

**Proposal:** Option C. franky-cron is the canonical implementation of the
runner, not just a producer. The queue directory is used when other
producers are active; franky-cron's own sweeps pass through it for
consistency.

**What this affects:** §8.2 lifecycle steps (keep the queue rename steps),
§4.1.3 framing (franky-tasks deferred), and whether a separate
`franky-tasks` sibling project is needed for v1 (not needed).

### 9.3 — Subprocess vs embedding for sibling→franky session

§8.2 step 2d says "spawn franky session (`--mode rpc` or `sdk.runTask`)."
These are architecturally different:

- **`--mode rpc` (subprocess):** The sibling spawns a child franky process,
  communicates via JSON-RPC over stdio, parses streamed events, and handles
  process lifecycle (spawn, wait, kill on timeout, detect crash).
- **`sdk.runTask` (embedded):** The sibling links franky as a library,
  calls a single blocking function, and gets back the result synchronously.
  No subprocess management, no event parsing — but ties the sibling to
  franky's Zig version, memory model, and transitive dependencies.

**Proposal:** Use `--mode rpc` for v1. Reasons:
(1) The RPC mode already exists and is tested (see `src/coding/modes/rpc.zig`).
(2) It keeps sibling projects dependency-independent — a sibling written in
Go, Python, or another Zig version can still drive franky.
(3) `sdk.runTask` is an optimization for tight embedding, not a prerequisite
for unattended operation. It can be added in v2 once the RPC surface is
battle-tested and the exact convenience API is clear.

**What this affects:** §8.2 step 2d (use `--mode rpc` only, remove
"or sdk.runTask"), §6 (downgrade `sdk.runTask` from "optional" to
"deferred"), and the sibling's dependency scope.

**Decision:** _____

### 9.4 — `finish_task` → sibling handshake protocol

The sibling needs the agent's `finish_task` output (commit_message + summary)
to:
1. Push the branch to GitHub.
2. Create the PR with the summary as the body.
3. Report back to the source channel.

Three possible handshake mechanisms:

**Mechanism A — RPC event stream.** The `finish_task` call fires a
`finish_task` AgentEvent with `commit_message` and `summary` fields. The
sibling (running in `--mode rpc`) receives this as a JSON-RPC event
notification before the session exits. It then proceeds with GitHub push
+ PR creation.

**Mechanism B — Exit JSON line.** When the session ends (whether via
`finish_task` or budget exhaustion), franky prints a final JSON line to
stdout containing the result status, and (if finish_task was called) the
commit message and summary. The RPC mode already has a `done` notification;
this extends it with the payload.

**Mechanism C — Transcript file.** The sibling reads the session transcript
from disk after the process exits. No stream-time protocol needed, but adds
latency (wait for process exit, parse transcript) and requires the sibling to
know the transcript path.

**Proposal:** Mechanism B (exit JSON line). It's the simplest for the
sibling — wait for process exit, read the last line of stdout. It doesn't
require the sibling to maintain an event-parsing loop (unlike A), and
doesn't require filesystem access to the transcript directory (unlike C).
In `--mode rpc`, this extends the existing `done` notification with optional
`finish_task` fields.

**What this affects:** The RPC mode's `done` response shape, the sibling's
process-wait logic, and the `finish_task` tool's post-exit emission.

**Decision:** _____

### 9.5 — Queue directory configuration

The `task_create` tool (§4.1.4) writes to `<queue>/pending/`. The sibling
needs to know where this directory is. Configuration options:

**Option 1 — CLI flag (`--task-queue-dir`).** The sibling passes it when
spawning the franky session. If absent, `task_create` tool is not registered
(or returns a clear error).

**Option 2 — Settings.json field (`task_queue_dir`).** Read from the
settings layer. Same as other per-project configuration.

**Option 3 — Env var (`FRANKY_TASK_QUEUE_DIR`).** Matches the pattern of
other franky env vars.

**Proposal:** All three, with precedence CLI > env > settings (matching the
existing precedence convention for `max_turns` and retry settings). If none
are set, `task_create` is unavailable — the tool is omitted from the tool
list so the model never sees it.

**What this affects:** Settings struct (new `task_queue_dir` field), CLI
parser (new `--task-queue-dir` flag), the `task_create` tool registration
path, and the sibling's spawning command line.

**Decision:** _____

### 9.6 — `task_id` semantics: label vs unique identifier

The TaskSpec `task_id` field serves two roles that may conflict:

1. **Human-readable label** in log files, transcript paths, and branch names
   (`franky/<task_id>-<date>` per §8.1).
2. **Provenance chain** in `created_by.parent_task_id` (§4.1.2,
   §5.6) — must uniquely identify which parent spawned a child.

If two different tasks have the same `task_id` label (e.g. two issues both
labelled "fix-api-bug" from different repos), the provenance chain breaks.

**Proposal:** Two separate fields:

- `task_id` — human-readable label for logs and branches. Does NOT need
  to be globally unique; the sibling suffixes with `-<date>-<ULID>` to
  ensure branch-name uniqueness.
- `task_uid` — system-generated unique identifier (ULID), populated
  automatically when the taskfile is written (by the producer or runner).
  Used in `parent_task_uid` (renamed from `parent_task_id`) for provenance.

This separates display identity from provenance identity. The sibling only
needs to provide `task_id`; `task_uid` is generated by the runner or the
`task_create` tool.

**What this affects:** TaskSpec envelope (§4.1.2) adds `task_uid` (auto),
renames `parent_task_id` → `parent_task_uid`. The `task_create` tool
automatically stamps `task_uid` with a ULID. Log/transcript paths use
`task_id` for readability. Provenance uses `task_uid` for correctness.

**Decision:** _____

### 9.7 — `blocked/` state lifecycle

The `blocked/<id>.yaml` state (§4.1.3) is under-specified:

- Who moves a blocked task back to `pending/`?
- The session that hit the blocked condition ends. Does the blocked task
  carry full context (transcript, current state) so a human can resume?
- Is `blocked/` a terminal state (human must manually create a new task
  from the information), or is there a resurrection protocol?

§4.3 says "Skipping the prompt → deny → forced summary → reported as
'blocked: needed permission for X'" — implying the session terminates
and the report includes context. The task lands in `blocked/` as a record
of what was denied, not as a resumable checkpoint.

**Proposal:** `blocked/` is a **terminal** state in v1. When a session hits
an unapproved permission request:
1. The forced summary runs (if `max_turns_summarize` is true).
2. The session writes the TaskSpec + failure context to `blocked/<id>.yaml`.
3. The sibling reports "blocked" in its status line with a summary.
4. To retry, a human edits the TaskSpec (adds the needed permission to
   `constraints.ask_on` or `always_allow`), moves it back to `pending/`,
   and the runner picks it up.

No automatic resurrection. The `blocked/` directory is purely an audit trail
and handoff point for the human reviewer.

**What this affects:** §4.1.3 description of `blocked/`, the sibling's
handling of `tool_permission_request` events when `--prompts` is off,
and the `finish_task`/forced-summary pathway when a permission is denied.

**Decision:** _____

### 9.8 — `franky.sdk.runTask` surface design priority

§6 lists `sdk.runTask(spec)` as an optional addition: "set up workspace +
apply per-tool policy + run under budget + return the canonical artefact."
The design is incomplete — no parameter shape, return type, or workspace
setup steps are defined.

**Proposal:** Defer `runTask` to v2 of agentic engineering. The sibling
for v1 uses `--mode rpc` (§9.3 decision), which doesn't need `runTask`.
Once the RPC protocol, handshake shape, and workspace setup patterns are
battle-tested by franky-cron, a `runTask` convenience wrapper can be
designed from real usage data rather than from speculation.

**What this affects:** §6 removes `sdk.runTask` from the core surface
(moved to "future work"), and §8.2 step 2d no longer mentions it.
