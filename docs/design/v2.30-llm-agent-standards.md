# LLM Agent Standards — Franky Support Design

## 1. Problem

Franky is a production Zig 0.17-dev agent framework with its own config system
(CLI flags, `settings.json`, `skills/*.md`, `.contextignore`). The LLM agent
ecosystem has meanwhile converged on several open standards:

| Standard | Steward | File / Dir | Use |
|---|---|---|---|
| **AGENTS.md** | OpenAI · Linux Foundation | `AGENTS.md` (repo root) | Project instructions for coding agents |
| **.agents Protocol** | .agents Protocol | `.agents/` directory | One directory for MCP, skills, sub-agents, tasks, memories |
| **CLAUDE.md / .claude** | Anthropic | `CLAUDE.md`, `.claude/` | Project instructions, skills, sub-agents, MCP, rules |
| **.cursor/rules** | Cursor | `.cursor/rules/*.md` | Path-scoped rules with frontmatter |
| **Agentskills.io** | Anthropic | `SKILL.md` frontmatter | Skill format used by Hermes + others |
| **MCP** | Anthropic · Linux Foundation | `.mcp.json` | Tool discovery & invocation protocol |

**Gap:** Franky today ignores all of these. Every project must copy-paste
instructions into a flag or a franky-specific path. Skills and tools live in
franky-only locations. MCP servers are unsupported by design.

## 2. Goals

1. **Read AGENTS.md, CLAUDE.md, and `.agents/agents.md`** — inject as system
   prompt or instruction block, with proximity-based resolution.
2. **Scan `.agents/skills/`, `.claude/skills/` ** as
   additional skill roots (read-only).
3. **Interop with `.claude/settings.json` and `.agents/models.json`** for
   model/config sharing.
5. **Preserve existing franky paths** — standards are additional roots, not
   replacements. Franky's own config always wins on conflict.
6. **Zero overhead when no standard files exist** — no startup penalty,
   no mandatory dependencies.

## 3. Non-Goals (Phase 1)

- Full MCP client implementation (JSON-RPC tool invocation over stdio/HTTP).
  Phase 1 loads the config; Phase 2 adds the protocol runtime.
- Task scheduling (`.agents/tasks/`) or persistent memories
  (`.agents/memories/`). These are future phases.
- Hermes-style progressive skill disclosure (levels 0/1/2). Skills are
  always fully loaded when active.
- Writing standard files. Phase 1 is read-only.

## 4. Detailed Design

### 4.1 Root Layout — What Franky Scans

Scan order (highest → lowest precedence):

```
1. --system-prompt / --append-system-prompt   [CLI flag, current]
2. --skills <dir>                             [CLI flag, current]
3. --agents <dir>                             [NEW — explicit .agents/ override]
4. <workspace>/.franky/                       [current project config]
5. <workspace>/AGENTS.md                      [NEW — standard]
6. <workspace>/CLAUDE.md                      [NEW — standard]
7. <workspace>/.agents/                       [NEW — .agents protocol]
8. <workspace>/.claude/                       [NEW — Anthropic standard]
10. <workspace>/skills/                       [current workspace skills]
11. $FRANKY_HOME/                             [current user config]
12. ~/.agents/                                [NEW — global .agents]
13. ~/.claude/                                [NEW — global Claude config]
14. $HOME/.franky/                            [current fallback]
```

Files found at higher-precedence paths shadow same-named files at lower paths.
The merge is shallow-by-key for JSON, name-based for skills, path-prefix-based
for AGENTS.md sections.

### 4.2 AGENTS.md / CLAUDE.md — Instruction Loading

**What:** `AGENTS.md` (or `CLAUDE.md`) at any directory in the workspace tree.

**Resolution rules (matching the AGENTS.md spec):**

1. The closest `AGENTS.md` to a file being read/edited wins.
2. Root-level `AGENTS.md` applies to all files not covered by a closer one.
3. `CLAUDE.md` at the root is treated identically to `AGENTS.md` in content,
   but lower precedence (AGENTS.md wins on sibling conflict).
4. Explicit user instructions in the prompt override everything.

**Mechanism:**

- At startup, scan the workspace tree for `AGENTS.md` / `CLAUDE.md` files
  (respecting `.contextignore`).
- Parse each file into an `InstructionBlock { path, text, source } ` struct.
- When the agent processes a file (read/edit/write), resolve the nearest
  `InstructionBlock` via ancestor-path prefix match.
- Append the matching block's text to the system prompt under
  `## Project instructions (from AGENTS.md)`.

**Data structure:**

```zig
pub const InstructionBlock = struct {
    /// Absolute path of the AGENTS.md / CLAUDE.md file.
    source_path: []const u8,
    /// Full markdown content.
    content: []const u8,
    /// "agents.md", "claude.md", or ".agents/agents.md"
    kind: InstructionKind,
};

pub const InstructionKind = enum { agents_md, claude_md, dot_agents };

pub const InstructionForest = struct {
    allocator: std.mem.Allocator,
    blocks: std.ArrayList(InstructionBlock),
    /// Build from a workspace scan.
    pub fn scan(allocator: Allocator, io: std.Io, workspace_root: []const u8) !InstructionForest;
    /// Resolve the closest block to `file_path`.
    pub fn resolveClosest(self: *const InstructionForest, file_path: []const u8) ?InstructionBlock;
};
```

### 4.3 `.agents/` Directory — Protocol Support

Map `.agents/` files to franky concepts:

| `.agents/` file | Franky equivalent | Action |
|---|---|---|
| `agents.md` | Instructions | Merged into `InstructionForest` (kind: `.dot_agents`) |
| `models.json` | Profile presets | Seed profile registry with model configs |
| `skills/*/skill.md` | Skill files | Added as additional skill root (read-only) |
| `agents/*/agent.md` | Subagent presets | Seed subagent registry |
| `system-prompt.md` | System prompt override | Appended to system prompt if no CLI override |
| `memories/*.md` | *(Phase 2)* | — |
| `tasks/*/task.md` | *(Phase 2)* | — |

**Skill format interop:**

`.agents/skills/<name>/skill.md` uses the `agentskills.io` frontmatter format:

```markdown
---
id: code-review
name: Code Review Expert
description: Thorough code review of PRs
enabled: true
---

Review code changes for:
- Security vulnerabilities
- Performance implications
```

Franky's existing skill parser is extended to understand both formats:

- `name:` (franky native) → `name`
- `id:` (.agents protocol, used when `name:` absent)
- `description:` (both)
- `auto_apply:` (franky) → converted to `globs:` equivalent
- `enabled:` (.agents, defaults true)
- Unknown frontmatter keys are silently forwarded for forward-compat
Merge both formats into one.

### 4.4 `.claude/` and `.cursor/` — Ecosystem Interop

| `.claude/` file | Franky action |
|---|---|
| `CLAUDE.md` (project root) | Already handled via `InstructionForest` |
| `.claude/settings.json` | Merge selected keys into franky settings (project layer) |
| `.claude/skills/<name>/SKILL.md` | Added as skill root (read-only) |
| `.claude/agents/*.md` | Seed subagent presets |
| `.claude/rules/*.md` | Convert to skills (`.claude/rules/` → skill root) |
| `.claude/output-styles/*.md` | Ignored (franky has its own formatting) |
| `.mcp.json` | Load MCP server descriptors |

### 4.6 Skill Root Merge — Final Priority

After resolving all standard directories, the effective skill roots are:

```
1. --skills <dir>              [CLI, highest]
2. <workspace>/.franky/skills/ [franky project]
3. <workspace>/.agents/skills/ [.agents protocol]
4. <workspace>/.claude/skills/ [Claude interop]
5. <workspace>/.cursor/rules/  [Cursor interop — converted]
6. <workspace>/skills/         [franky workspace]
7. $FRANKY_HOME/skills/        [franky user]
8. ~/.agents/skills/           [global .agents]
9. ~/.claude/skills/           [global Claude]
```

Same `name` at multiple roots → higher-precedence wins.

### 4.7 Tool & Extension Merge

Currently tools are assembled from:

- Built-in set (role-filtered)
- Extension-contributed tools (from `--extensions`)
- MCP tools (post-Phase 2)

The merge order for the final tool list presented to the LLM:

```
1. Built-in tools (read, write, edit, bash, ls, find, grep, web_search, web_fetch)
2. Extension tools (from --extensions <csv>)
3. MCP tools (from mcp.json servers, Phase 2)
4. Subagent tools (filtered per subagent preset)
```

### 4.8 System Prompt Assembly — Revised Order

The existing `buildSystemPromptIo` function currently assembles:

1. Base system prompt (from disk or built-in default)
2. `--system-prompt` / `--append-system-prompt` override
3. Subagent hint block
4. Skills (active section, under `## Active skills`)
5. Review config block

Revised order with standard support:

1. Base system prompt
2. CLI overrides (`--system-prompt`, `--append-system-prompt`)
3. `.agents/system-prompt.md` content (if no CLI override)
4. Subagent hint block
5. AGENTS.md / CLAUDE.md instructions (closest to workspace root)
6. Active skills section (franky + standard roots merged)
7. Review config block (v2.16)
8. `--append-system-prompt` content (always last)

### 4.9 Settings Merge — Cross-Standard

Settings layers (existing):
```
CLI flags > project .franky/settings.json > user .franky/settings.json > built-in
```

Extended:
```
CLI flags > .franky/settings.json [project] > .claude/settings.json [project]
> .agents/models.json [project] > .franky/settings.json [user] > ~/.claude.json
> built-in defaults
```

Only shared keys are merged:
- `MCP servers` → McpRegistry (defered to v2)
- `model defaults` → profile registry
- `permissions` → permission store
- `env` → session environment

### Phase 1b — Skill root unification

Extend `skills.Roots` with a new field:

```zig
pub const Roots = struct {
    explicit_root: ?[]const u8,
    workspace_root: ?[]u8,
    user_root: ?[]u8,
    workspace_for_glob: ?[]const u8,
    /// NEW — directories from standard locations (read-only).
    standard_roots: []const []const u8 = &.{},

    pub fn resolveStandard(allocator: Allocator, workspace: ?[]const u8) ![]const []const u8 {
        var out: std.ArrayList([]const u8) = .empty;
        if (workspace) |p| {
            try out.append(allocator, try join(p, ".agents/skills"));
            try out.append(allocator, try join(p, ".claude/skills"));
            try out.append(allocator, try join(p, ".cursor/rules"));
        }
        if (homeDir()) |h| {
            try out.append(allocator, try join(h, ".agents/skills"));
            try out.append(allocator, try join(h, ".claude/skills"));
        }
        return out.toOwnedSlice(allocator);
    }
};
```

`loadAll` iterates `standard_roots` after the explicit/workspace/user roots,
collecting skills. Name dedup uses the existing logic (first seen wins).

### Phase 2 — MCP Runtime (future)

- Implement MCP JSON-RPC client over stdio/HTTP transports
- `McpRegistry.startAll(io)` spawns subprocesses, negotiates
- `McpRegistry.tools()` returns `[]AgentTool` for discovered tools
- Tools are merged into the final tool list at config resolve time

### Phase 3 — Sub-agents from standards (future)

- Load `.agents/agents/*/agent.md` into the subagent preset registry
- Load `.claude/agents/*.md` into the same registry
- Support per-agent `config.json` (tool overrides, model selection)

## 6. Backward Compatibility

| Existing feature | Impact |
|---|---|
| `--skills <dir>` | Unchanged. Still highest-precedence skill root. |
| `<workspace>/skills/` | Unchanged. Standard roots are lower priority. |
| `$FRANKY_HOME/skills/` | Unchanged. Standard roots are lower priority. |
| `--system-prompt` | Unchanged. Still beats AGENTS.md. |
| `.contextignore` | Unchanged. Standard scans respect it. |
| `settings.json` | Unchanged. New cross-standard merge is additive. |
| Existing build | No new dependencies. Pure Zig. |

## 7. Risks and Mitigations

| Risk | Mitigation |
|---|---|
| **Scan explosion** — scanning `.agents/`, `.claude/`, `.cursor/` in every project adds I/O | Scan is lazy (on first resolve + cached); empty dirs cost one `stat` each |
| **Skill name collisions** — `code-review` skill from franky vs `.agents` | Existing first-seen-wins dedup preserves current behavior; standard roots are lower priority |
| **AGENTS.md at every subdirectory** — scanning for 88 files like OpenAI's repo | Respect `.contextignore` + `.gitignore`; add a depth limit; log warning above N files |
| **`.mcp.json` with dangerous commands** | MCP servers are not auto-started in Phase 1; Phase 2 adds a permission gate |
| **CLAUDE.md semantic drift** — Anthropic may extend the format | Parse only the subset we understand; unknown frontmatter is silently forwarded |

## 9. Key Decisions

1. **Read-only, never write.** Franky never creates or modifies files in `.agents/`,
   `.claude/`. User intent stays with the tool that owns the directory.

2. **Precedence preserves existing behavior.** Franky's own flags and paths always
   beat standard files. Adding standard support never breaks an existing workflow.

3. **No mandatory scanning.** If no `AGENTS.md`, `.agents/`, `.claude/`, or `.cursor/`
   exists, the behavior is identical to today. Zero startup cost.

4. **MCP Phase is future.** Loading `mcp.json` describes what servers exist;
   Phase 2 adds the protocol runtime. This lets us ship the config layer without
   the complexity of subprocess management.

## 10. Open Questions

- Should `CLAUDE.md` at workspace root be treated as an `InstructionBlock` with
  proximity resolution, or only as a root-level instruction file? **Decision:
  proximity-resolved like AGENTS.md, but lower precedence.**
- When MCP Phase 2 starts server subprocesses, should they share the parent's
  environment or get a sanitized subset? **Deferred to Phase 2.**
- Should `.agents/memories/` be read and injected as system-prompt context?
  **Deferred to Phase 2/3.**
- Should we add a `/standards` slash command that shows which standard files
  were found and their merge status? **Yes — Phase 1b.**
