# franky-iikit — Intent Integrity Chain as an SDK consumer

**Status:** open for review  
**File:** `docs/design/open/v2.x-intent-integrity-chain.md`  
**Source inspiration:** [intent-integrity-chain/kit](https://github.com/intent-integrity-chain/kit) v2.10.0  
**Motto:** *Never trust a monkey — how to verify code without reading it.*

---

## 1. What this is

A design for **`franky-iikit`**, a standalone Zig project that implements the
Intent Integrity Chain methodology as an SDK consumer of `franky.sdk`. It does
**not** modify franky's internals — it wraps the agent loop from outside.

### Why standalone, not built-in

| Built-in (`src/agent/guardrails/`) | Standalone SDK consumer |
|---|---|
| Tied to franky's release cycle | Ships independently |
| Internal API (`loop.Config.guardrails`, `GuardrailState`) | Public SDK surface only (`AgentTool`, `agentLoop`, `Transcript`) |
| Config shared with franky's own guards (`.franky-workflow.yaml`) | Own config file (`.iikit.yaml`) |
| Can't be used with other AI agents | Wraps any franky SDK consumer |
| Single-binary deployment | Sidecar / CLI wrapper |

The IIKit methodology is **agent-agnostic** — it should work the same whether
the underlying agent is franky, Claude Code, or something else. A standalone
project keeps that independence.

### Relationship to IIKit

`franky-iikit` is a Zig port of the [Intent Integrity Kit](https://github.com/intent-integrity-chain/kit)
methodology. IIKit ships as a Tessl skill collection for Claude Code / Codex /
Gemini / Copilot; `franky-iikit` embeds the same phase machine natively in a
Zig binary that orchestrates a franky SDK agent.

```
IIKit (Tessl skills)              franky-iikit (Zig binary)
┌────────────────────┐            ┌──────────────────────────┐
│ /iikit-01-specify  │            │ Phase 1: spec            │
│ /iikit-02-plan     │            │ Phase 2: plan            │
│ /iikit-04-testify  │   port    │ Phase 3: testify         │
│ /iikit-07-implement│  ──────→  │ Phase 4: implement       │
│ verify-*.sh        │            │ verify-*.zig             │
│ .feature + hash    │            │ .feature + hash          │
└────────────────────┘            └──────────────────────────┘
```

## 2. Core concept: the Intent Integrity Chain

```
Intent ──▶ Spec ──▶ .feature ──▶ Steps ──▶ Code
       ↑       ↑          ↑          ↑
       │       │          │          └── step quality verified (no assert True)
       │       │          └────────────── hash locked (no tampering)
       │       └───────────────────────── @FR-XXX tags traced
       └───────────────────────────────── clarified until aligned
```

**Key principle:** No part of the chain validates itself. Tests are locked before
code is written. Step definitions are verified for coverage and quality. If
requirements change, you go back to the spec — you never edit tests to match
buggy code.

### Five mechanical layers

1. **Spec-first gating** — no code can be written until a spec artifact exists
   and is marked as reviewed.

2. **Hash-locked tests** — `.feature` files (and/or test files) are SHA256-hashed
   at the end of the spec phase. Any edit to a hashed file during the
   implementation phase is rejected. Hashes are stored separately from the files
   themselves (`.iikit/hashes/`) so the thing being verified doesn't verify itself.

3. **RED phase** — before any production code is written, the test suite must be
   run and must **fail**. This proves the tests can detect the absence of the
   feature. A test that passes before any code is written is a test that tests
   nothing.

4. **GREEN phase** — after production code is written, the test suite must
   **pass**. No code is complete until the tests it was written against report
   success.

5. **Step quality** — test bodies are inspected for anti-patterns: empty bodies,
   `assert True`, missing assertions. A test that can't fail is worse than no
   test at all.

### Defense in depth

- Pre-commit hooks prevent bypass at the git layer.
- CI workflows provide server-side enforcement.
- Every link in the chain is verified by a different mechanism than the link
  that produced it.

`franky-iikit` owns the **phase logic** and **verification**. It delegates
code generation to a franky agent (via `franky.sdk`). It does not modify franky's
loop — it wraps it.

### How it wraps the SDK

`franky-iikit` does not import `src/agent/guardrails/`. Instead, it implements
its own guardrails by wrapping the SDK's `AgentTool` interface:

```zig
// Pseudocode — not the exact API, but the shape.

const franky = @import("franky");
const sdk = franky.sdk;

pub const Iikit = struct {
    phase: Phase = .exploration,
    spec_gate: SpecGate,
    hash_lock: HashLock,
    step_quality: StepQuality,
    test_runner: TestRunner,

    /// Wrap an AgentTool so the iikit layer can intercept writes.
    pub fn wrapTool(tool: sdk.AgentTool) sdk.AgentTool {
        return .{
            .name = tool.name,
            .description = tool.description,
            .parameters_json = tool.parameters_json,
            .execute = makeExecute(tool.execute),
        };
    }

    /// Run the full workflow: spec → plan → testify → implement.
    pub fn run(self: *Iikit, allocator, io, registry) !void {
        while (self.phase != .done) {
            switch (self.phase) {
                .exploration => try self.runExploration(allocator, io, registry),
                .spec_done => try self.runSpecDone(allocator, io, registry),
                .red => try self.runRed(allocator, io, registry),
                .green => try self.runGreen(allocator, io, registry),
                .done => {},
            }
        }
    }
};
```

Each `runPhase` method configures an `sdk.loop.Config` with the appropriate
tools (wrapped for enforcement) and calls `sdk.agentLoop`. The phase transition
happens between loop runs, not inside the loop — so the iikit layer controls
flow, not the agent.

## 5. Phase detail

### 5.1 Exploration

**Goal:** Understand the problem. Gather requirements. Clarify ambiguities.

**Agent capabilities:** Read-only tools (read, grep, ls, web_search, web_fetch).
No edit/write/mv tools available.

**Spec gate:** None yet — there is no spec to check against.

**Transition to spec_done:** User runs `franky-iikit spec:done` OR the agent
creates a file matching `spec/**/*.md` with valid frontmatter (title + status:
reviewed).

**Verification:** The spec file must parse. Frontmatter must include `name`,
`description`, and at least one `FR-*` tag. If the spec references scenarios,
they must be well-formed Given/When/Then.

### 5.2 spec_done (testify)

**Goal:** Hash-lock test files. Generate Gherkin `.feature` files from the
spec's scenarios.

**Agent capabilities:** Write/edit allowed only for:
- `spec/**/*.md` (spec refinements)
- `**/*.feature` (Gherkin files)
- `**/steps/*.zig` (step definitions, if any)

**Hash lock:** Run `franky-iikit lock` (or auto-lock on entering this phase).
Scans all files matching `test_patterns` in `iikit.yaml`, writes SHA256 hashes
to `.iikit/hashes/`. From this point, any write to a hashed file that changes
its content is **rejected** by the tool wrapper.

**Transition to red:** User runs `franky-iikit phase:red` OR all of:
- Spec exists and is reviewed
- Test files are hash-locked
- `.feature` files exist (when BDD mode is enabled)
- Step definitions exist for all Gherkin steps (when BDD mode is enabled)

**Verification:**
- Hash lock file exists and matches current on-disk content.
- Step coverage: every Gherkin `Given`/`When`/`Then` line has a matching step
  definition file (run `verify-steps` dry-run).

### 5.3 RED

**Goal:** Write tests. Run them. Confirm they fail.

**Agent capabilities:**
- `edit`/`write` allowed ONLY for files matching `test_patterns`.
- `bash` allowed for running the test command.
- Any attempt to write production code (non-test files) is rejected with
  `tool_code = "red_phase_write_denied"`.

**Enforcement:** The `tool_wrapper.zig` intercepts every `edit`/`write` call.
If phase is RED and the target path does not match any test pattern, the tool
returns `is_error: true` with a message: "RED phase: only test files can be
written. Write to 'src/main.zig' denied."

**Test failure detection:** After the agent runs the test command via `bash`,
the tool wrapper checks the exit code. A non-zero exit code from a command
matching `test_command` pattern is captured as `test_failure_observed = true`.

**Transition to green:** User runs `franky-iikit phase:green` OR the tool
wrapper observes `test_failure_observed = true` AND the agent explicitly
requests advancement (via a dedicated `advance_phase` tool).

**Verification:**
- No production code files exist that differ from the locked spec baseline.
- Test failure was observed at least once.
- Hash lock is still intact (tests were not modified after being locked).

### 5.4 GREEN

**Goal:** Write production code. Run tests. Confirm they pass.

**Agent capabilities:** Full write access to all files. `bash` allowed.

**Enforcement:** No write restrictions, but the hash lock is still active —
if the agent modifies a test file, the write is **rejected** (tests are locked
until the phase is complete). To modify tests, the user must explicitly
`franky-iikit unlock`, which transitions back to spec_done (re-lock required).

**Test pass detection:** After the agent runs the test command, the tool wrapper
checks the exit code. Zero exit code = pass observed.

**Step quality verification:** On every test-file write (which should not happen
during GREEN — they're locked), and on every `finish_task`-equivalent call, scan
all test files for:
- Empty test bodies (`test "name" {  }`)
- Tautological assertions (`expect(true)`, `expect(true == true)`)
- Tests with no assertion calls at all

**Transition to done:** User runs `franky-iikit phase:done` OR:
- Tests pass (observed)
- Step quality passes (no violations)
- Hash lock intact
- User confirms

**Verification:**
- Tests pass.
- Step quality clean.
- Hash lock intact.
- Optional: spec → code trace (every `@FR-XXX` in spec appears in a commit
  message or code comment).

### 5.5 DONE

**Goal:** Final verification. Optional auto-commit. Summary.

**Actions:**
1. Run test suite one final time (force).
2. Run step quality verifier one final time (force).
3. If `auto_commit: true` in `iikit.yaml`, commit with a message that includes
   the FR-* tags from the spec.
4. Print summary: spec processed, tests passed, files changed, FR-* coverage.

**No further agent turns.** The loop is closed. Any subsequent run starts a new
cycle from exploration.

## 6. Tool wrapper

The tool wrapper is the **enforcement layer**. It wraps every `AgentTool.execute`
that comes from the SDK with a pre-check function.

```zig
// src/tool_wrapper.zig

pub const ToolWrapper = struct {
    phase: *Phase,
    spec_gate: *SpecGate,
    hash_lock: *HashLock,
    red_phase: *RedPhase,
    step_quality: *StepQuality,
    config: *Config,

    pub fn wrap(self: *ToolWrapper, tool: sdk.AgentTool) sdk.AgentTool {
        const original_execute = tool.execute;
        return .{
            .name = tool.name,
            .description = tool.description,
            .parameters_json = tool.parameters_json,
            .execute = struct {
                fn exec(
                    t: *const sdk.AgentTool,
                    allocator: std.mem.Allocator,
                    io: std.Io,
                    call_id: []const u8,
                    args_json: []const u8,
                    cancel: *sdk.Cancel,
                    on_update: sdk.agent_types.OnUpdate,
                ) anyerror!sdk.ToolResult {
                    // Pre-check
                    if (self.isWriteTool(tool.name)) {
                        const path = try extractPath(args_json);
                        try self.enforceWrites(path);
                    }

                    // Execute
                    const result = try original_execute(t, allocator, io, call_id, args_json, cancel, on_update);

                    // Post-check
                    if (self.isWriteTool(tool.name)) {
                        if (result.is_error) return result;
                        const path = try extractPath(args_json);
                        try self.enforceWrites(path); // re-check after write
                        // Step quality scan on test files
                        if (self.config.step_quality_enabled and matchesTestPattern(path)) {
                            self.step_quality.scan(path) catch |err| {
                                // Attach warning — non-fatal unless phase=done
                            };
                        }
                    }

                    // Detect test failure/pass on bash commands
                    if (self.isTestCommand(tool.name, args_json)) {
                        if (result.is_error) {
                            self.red_phase.recordTestFailure();
                        } else {
                            self.red_phase.recordTestPass();
                        }
                    }

                    return result;
                }
            }.exec,
        };
    }

    fn enforceWrites(self: *ToolWrapper, path: []const u8) !void {
        // G1: spec gate
        if (self.phase.* == .exploration and !isSpecPath(path)) {
            return error.SpecGateDenied;
        }
        // G2: hash lock
        if (self.phase.* == .red or self.phase.* == .green) {
            if (self.hash_lock.isLocked(path)) {
                try self.hash_lock.verifyUnchanged(path);
            }
        }
        // G3: red phase write restriction
        if (self.phase.* == .red and !matchesTestPattern(path)) {
            return error.RedPhaseWriteDenied;
        }
    }

    fn isTestCommand(tool_name: []const u8, args_json: []const u8) bool {
        // Match bash tool where args match configured test_command pattern
        _ = args_json;
        return std.mem.eql(u8, tool_name, "bash"); // simplified
    }
};
```

### What gets wrapped

| SDK Tool | Wrapped? | Enforcement |
|---|---|---|
| `edit` | Yes | Spec gate, hash lock, red-phase restriction |
| `write` | Yes | Spec gate, hash lock, red-phase restriction |
| `bash` | Yes (post-check) | Detect test failure/pass for phase transitions |
| `read`, `grep`, `ls`, `find` | No | Read-only — no enforcement needed |
| `finish_task` | Yes | Run final verification before allowing completion |
| `web_search`, `web_fetch` | No | Read-only |

The wrapper does **not** modify the agent loop. It wraps individual tools. The
loop sees only:
- Tool results (some with `is_error: true` when enforcement kicks in)
- Harness-synthesised events (phase hints injected as tool_execution_end events)

## 7. Verification modules

### 7.1 Hash lock (`src/hash_lock.zig`)

```zig
pub const HashLock = struct {
    /// Map from relative path to hex-encoded SHA256.
    hashes: std.StringHashMap([]const u8),
    hash_dir: []const u8,     // .iikit/hashes/

    pub fn lock(workspace_dir: []const u8, patterns: []const []const u8) !void {
        for each file matching patterns:
            compute SHA256
            store in hashes map
        write hashes to .iikit/hashes/lock.json
    }

    pub fn isLocked(path: []const u8) bool {
        return hashes.contains(path);
    }

    pub fn verifyUnchanged(path: []const u8) !void {
        const stored = hashes.get(path).?;
        const current = computeSha256(path);
        if (!std.mem.eql(u8, stored, current)) {
            return error.HashMismatch;
        }
    }
};
```

**Storage format** (`.iikit/hashes/lock.json`):
```json
{
  "version": 1,
  "created_at": 1715123456789,
  "patterns": ["**/*_test.zig", "**/*.feature"],
  "files": {
    "tests/test_auth.zig": "a1b2c3d4e5f6...",
    "tests/features/login.feature": "f6e5d4c3b2a1..."
  }
}
```

Hash lock is **not** stored inside the tracked workspace files — it lives in
`.iikit/` which is git-ignored by default (or git-tracked if the user opts in).
This prevents the circular-verification problem: the thing being verified
does not verify itself.

### 7.2 Step quality (`src/step_quality.zig`)

Inspects test file ASTs for quality signals. Language-aware: starts with Zig
(reuses `std.zig` parser), provides an external hook for other languages.

```zig
pub const QualityIssue = struct {
    path: []const u8,
    line: usize,
    kind: enum { empty_body, tautological_assert, missing_assertion },
    description: []const u8,
};

pub const StepQuality = struct {
    issues: std.ArrayList(QualityIssue),

    pub fn scanFile(path: []const u8) ![]QualityIssue {
        // 1. Parse file as Zig AST
        // 2. Find all `test "..." { ... }` blocks
        // 3. For each block:
        //    a. If body is empty or whitespace-only → empty_body
        //    b. If body contains `expect(true)` or similar → tautological_assert
        //    c. If body contains no `try expect*` or `try testing.expect*` → missing_assertion
        // 4. Return issues (empty = clean)
    }
};
```

### 7.3 Spec gate (`src/spec_gate.zig`)

```zig
pub const SpecGate = struct {
    spec_dir: []const u8,

    pub fn hasReviewedSpec() bool {
        // Scan spec_dir for .md files with frontmatter
        // Return true if at least one has status: reviewed
```

## 8. Config (`iikit.yaml`)

```yaml
# iikit.yaml — franky-iikit project configuration

# Phase enforcement
phase:
  initial: exploration          # Where to start on `franky-iikit run`
  auto_advance: true            # Allow automatic phase transitions

# Spec gate (G1)
spec:
  directory: spec/
  required_tags: true           # Require @FR-XXX tags in spec frontmatter
  require_reviewed: true        # Spec must have status: reviewed before proceeding

# Hash lock (G2)
hash_lock:
  enabled: true
  test_patterns:
    - "**/*_test.zig"
    - "**/*.feature"
    - "**/features/**"
  hash_dir: .iikit/hashes
  reject_mismatch: true         # Hard error vs advisory warning

# RED phase (G3)
red_phase:
  enabled: true
  test_command: "zig build test"
  # Commands whose exit code counts as a "test run" attempt
  test_commands:
    - "zig build test"
    - "cargo test"
    - "npm test"
  require_explicit_advance: false   # If true, user must run `franky-iikit phase:green`

# Step quality (G4)
step_quality:
  enabled: true
  hard_fail_on_finish: false
  # External script for non-Zig languages (receives file path, returns 0/1)
  external_check: ""

# Assertion integrity (G5)
assertion_integrity:
  enabled: true
```

## 10. Integration with franky SDK

### 10.1 Dependency

```zig
// build.zig.zon
.{
    .name = "franky-iikit",
    .version = "0.1.0",
    .dependencies = .{
        .franky = .{
            .url = "https://github.com/franky/franky/archive/<commit>.tar.gz",
            .hash = "...",
        },
    },
    .paths = .{""},
}
```

### 10.2 SDK surface consumed

| SDK export | Used for |
|---|---|
| `sdk.AgentTool` | Wrapping with enforcement layer |
| `sdk.ToolResult` | Returning enforcement errors |
| `sdk.loop.Config` | Configuring agent loop per phase |
| `sdk.agentLoop` | Running the agent within a phase |
| `sdk.Transcript` | Building conversation context |
| `sdk.Registry` | Model provider resolution |
| `sdk.Cancel` | Cancellation signalling |
| `sdk.loop.defaultConvertToLlm` | Message conversion |
| `sdk.types.Message` | Building system prompts per phase |

### 10.3 What we do NOT use

| Internal module | Reason |
|---|---|
| `src/agent/guardrails/` | We implement our own enforcement at the wrapper level |
| `src/coding/skills.zig` | We ship our own skills and pass via `--skills` flag |
| `src/coding/tools/` | We use the SDK `tools` module for tool registration |
| `loop.Config.guardrails` | That's for franky's built-in guards; we don't hook into them |

The boundary is clean: `franky-iikit` imports only `franky.sdk`. If the SDK
surface is insufficient, we extend the SDK rather than reaching into internal
modules.

### 10.4 Spawning the franky agent

`franky-iikit` can either:

**Option A: In-process (SDK embedding)**
```zig
var reg = sdk.Registry.init(gpa);
defer reg.deinit();
// ... register provider ...

var transcript = sdk.Transcript.init(gpa);
defer transcript.deinit();

var iikit = Iikit.init(.{ .phase = .exploration, .config = &cfg });
defer iikit.deinit();

while (iikit.phase != .done) {
    const tools = iikit.toolsForPhase(); // returns wrapped AgentTool slice
    const system = iikit.systemPromptForPhase();

    try sdk.agentLoop(gpa, io, .{
        .model = model,
        .system_prompt = system,
        .tools = tools,
        .registry = &reg,
        .transcript = &transcript,
        .cancel = &cancel,
    });

    try iikit.evaluatePhaseTransition(&transcript);
}
```

**Option B: Sub-process (CLI wrapper)**

`franky-iikit` invokes `franky` as a sub-process with phase-specific
`--system-prompt`, `--skills`, and `--tools` flags. Reads results from
stdout/transcript file. This is simpler but loses the in-process feedback
loop (tool result interception).

**Recommendation:** Ship Option A (in-process) as the primary path. Option B
is a fallback for when franky SDK embedding isn't available (e.g., wrapping
Claude Code directly). The in-process path is what makes `franky-iikit`
different from IIKit — it can intercept tool calls in real time rather than
relying on post-hoc verification scripts.

## 11. Open questions

### Q1 — Where does the phase state live?

**Options:**
- **A:** `.iikit/phase` file (plain text: `red`). Written by `franky-iikit`
  after each phase transition.
- **B:** Environment variable (`IIKIT_PHASE=red`). Ephemeral — lost on shell
  restart.
- **C:** Git notes (`.iikit/phase` committed). Survives clone, but pollutes
  git history.

**Recommendation:** A (`.iikit/phase` file). It's workspace-local, survives
`sleep`/`tmux`/editor restarts, and is easy to debug (`cat .iikit/phase`).
Git-ignored by default so it doesn't pollute the project repo.

**Decision:** _____

### Q2 — Should the agent have a `request_phase_advance` tool?

**Context:** Without it, phase transitions require a CLI command from the user.
The agent cannot autonomously say "I confirm the tests fail, please advance to
GREEN."

**Options:**
- **A:** Provide a dedicated `advance_phase` tool with a required `reason` field.
  The tool writes the reason to `.iikit/phase_request`; `franky-iikit run`
  picks it up and prompts the user for confirmation.
- **B:** No tool — phase transitions are always user-initiated via CLI.
  Aligns with "never trust the monkey" — the agent doesn't get to decide.
- **C:** Auto-advance when conditions are met (tests fail/pass observed), with
  `auto_advance: true` in config. User can override with `auto_advance: false`.

**Recommendation:** C (auto-advance opt-in) + A (advance_phase tool as the
mechanism). Default: `auto_advance: false` — the user must explicitly advance.
The agent can *request* advancement via the tool, but the request is queued for
user confirmation.

**Decision:** _____

### Q3 — How does step quality handle non-Zig files?

**Context:** The project might contain JavaScript, Python, Rust, or any language.

**Options:**
- **A:** Only scan Zig files natively. Provide `external_check` script hook for
  other languages. Document recommended linters per language (eslint, pytest,
  clippy).
- **B:** Only scan `.feature` files (Gherkin) — the BDD layer is language-agnostic.
  Step definitions in any language are the implementer's responsibility.
- **C:** Ship a small runner that detects language by extension and dispatches to
  a built-in heuristic (regex-based for simple cases).

**Recommendation:** A. Focus on Zig (franky's native language) + Gherkin (BDD
standard). External hook for everything else. The external hook is a shell
script that receives `(file_path, language_hint)` and returns JSON with quality
issues.

**Decision:** _____

### Q4 — Lock granularity: whole file or line range?

**Context:** A hash lock on an entire file means any whitespace change or
comment addition breaks the lock. This is too strict for long-lived branches.

**Options:**
- **A:** File-level SHA256. Simplest. User must explicitly unlock/re-lock if
  tests need legitimate edits.
- **B:** Section-level hashes. Hash only the `test "..." { ... }` blocks, not
  imports/comments/setup code. More complex parser work but allows legitimate
  non-test edits.
- **C:** Line-level hashes. Hash each scenario line in `.feature` files
  (IIKit's approach). Allows adding new scenarios without breaking existing
  hashes.

**Recommendation:** C for `.feature` files (aligns with IIKit) and A for
`*_test.zig` files (Zig test blocks may interleave with helper functions —
section-level is unreliable without a full AST). Document that users should
unlock/re-lock when test logic needs changes.

**Decision:** _____

### Q5 — Pre-commit hook: mandatory or opt-in?

**Context:** The pre-commit hook is the defense-in-depth layer. But it adds
friction — every commit runs verification, which takes time.

**Options:**
- **A:** Install automatically on `franky-iikit init`. User must explicitly
  remove if unwanted.
- **B:** Install only on `franky-iikit install hooks`. Explicit opt-in.
- **C:** Generate the hook script but don't install it. User decides.

**Recommendation:** B. The pre-commit hook is powerful but opinionated. Let the
user opt in explicitly. The CI workflow (`franky-iikit install ci`) is equally
important and should be the default recommendation in documentation.

**Decision:** _____

### Q6 — Relation to the IETF SPICE Intent Chain

**Context:** [draft-mw-spice-intent-chain-00](https://www.ietf.org/archive/id/draft-mw-spice-intent-chain-00.html)
(March 2026) defines a cryptographically verifiable intent chain for AI agent
content provenance. Franky's hash lock is isomorphic to an intent-chain
commitment.

**Options:**
- **A:** Ignore the IETF draft for now. The hash lock format is internal.
- **B:** Design `.iikit/hashes/lock.json` to be forward-compatible with the
  intent chain's data model (ordered logs, Merkle root, signed `input_hash` +
  `output_hash` pairs).
- **C:** Implement a lightweight intent chain exporter: `franky-iikit export
  intent-chain` produces a JSON log compatible with the draft.

**Recommendation:** B for storage format (plan for Merkle chaining of phase
commitments, even if not implemented yet). C is a future enhancement — not
needed at v0.1.0.

**Decision:** _____

### Q7 — Test command detection reliability

**Context:** The tool wrapper detects test runs by matching the `bash` command
against `test_commands` patterns. This is fragile — the agent might run tests
with flags (`zig build test -- --test-filter auth`), from a subdirectory, or
via a script.

**Options:**
- **A:** Pattern match on the command string. Accept wildcards
  (`zig build test*` matches any flags).
- **B:** Add an explicit `run_tests` tool that the agent must use instead of
  `bash` for test runs. Provides structured input/output.
- **C:** Heuristic: any `bash` command that exits non-zero and contains a
  substring from `test_commands` counts as a test-failure observation. Any
  `bash` command that exits zero and matches is a test-pass observation.

**Recommendation:** C (generous heuristic) + B (optional `run_tests` tool for
agents that support it). The heuristic is lossy but the cost of a false positive
(advancing phase when tests didn't actually run) is bounded by the next
verification step. The `run_tests` tool is the reliable path.

**Decision:** _____

### Q8 — Multi-project monorepos

**Context:** A single workspace might contain multiple independent components
(frontend + backend), each with its own spec and tests.

**Options:**
- **A:** Single `.iikit/` per workspace. One phase state for everything.
- **B:** Multiple `.iikit-<component>/` directories, each with independent phase
  state. `franky-iikit run --component auth` targets a specific component.
- **C:** Single `.iikit/` but with per-component phase tracking in
  `lock.json`. `"components": { "auth": "red", "api": "green" }`.

**Recommendation:** Start with A (simple). C is the next step if monorepo usage
emerges. B is over-engineered for v0.1.

## 12. Implementation sketch

### Phase 1 — Minimum viable (v0.1.0)

**Goal:** Can run `franky-iikit run` through a complete RED → GREEN cycle with
a faux provider.

**Files:**
- `src/main.zig`, `src/config.zig`, `src/phase_machine.zig`
- `src/tool_wrapper.zig` (wraps edit/write only)
- `src/hash_lock.zig` (file-level SHA256, no section-level)
- `src/workspace_state.zig` (`.iikit/` init + read + write)
- `skills/iikit-spec.md`, `skills/iikit-bdd.md`, `skills/iikit-implement.md`

**What works:**
- Phase transitions exploration → spec_done → red → green → done
- Spec gate (no writes before spec exists)
- Hash lock (reject writes to locked test files)
- RED phase (reject production code writes)
- Test failure/pass detection (basic bash command matching)
- Tool wrapping (edit/write interception)

**What's deferred:**
- Step quality scanning (v0.2.0)
- Pre-commit hook + CI generation (v0.3.0)
- Gherkin .feature parsing (v0.2.0)
- `advance_phase` tool (v0.2.0)
- Multi-component support (v0.4.0)
- IETF intent chain export (v0.5.0)

### Phase 2 — BDD + quality (v0.2.0)

- Step quality scanner (Zig AST + Gherkin)
- Gherkin `.feature` hash locking at section level
- Step coverage verification (every Gherkin step has a matching definition)
- `advance_phase` tool for agent-initiated phase requests
- External check hook for non-Zig languages

### Phase 3 — Defense in depth (v0.3.0)

- Pre-commit hook installation (`franky-iikit install hooks`)
- CI workflow generation (`franky-iikit install ci`)
- `franky-iikit verify` command suite (stateless CI checks)
- Auto-commit with FR-* tag traceability

## 13. Comparison with IIKit

| Aspect | IIKit (Tessl skills) | franky-iikit (SDK consumer) |
|---|---|---|
| Target agent | Claude Code, Codex, Gemini, Copilot, OpenCode | Any franky SDK consumer |
| Delivery | Tessl skill collection | Standalone Zig binary |
| Enforcement | `.feature` hash in git notes + `context.json` | `.iikit/hashes/lock.json` (file system) |
| Phase machine | Agent prompt instructions (human-readable) | Zig enum + tool wrapper (machine-enforced) |
| Tool interception | PreToolUse hook (Claude Code only) | SDK `AgentTool` wrapper layer |
| Step quality | Shell scripts (`verify-step-quality.sh`) | Zig AST parsing + shell hook |
| Pre-commit hook | Bash script installed by skill | Zig-generated hook |
| CI workflow | YAML template in skill files | YAML generated by `franky-iikit install ci` |
| Spec format | IIKit spec template (`.spec.md`) | Markdown with frontmatter (configurable) |
| Test format | Gherkin `.feature` (required) | Gherkin `.feature` + any test framework |
| Portability | Works with 5+ AI assistants | Works with any franky SDK embedder |

The core advantage of `franky-iikit` over IIKit is **machine enforcement**.
IIKit relies on the agent respecting its constitution prompt — a motivated
agent (or a buggy one) can bypass it. `franky-iikit` intercepts tool calls at
the SDK boundary and returns errors programmatically. The agent cannot edit a
locked test file even if it wants to — the tool wrapper rejects the write.
