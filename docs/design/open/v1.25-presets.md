# v1.25 design — subagent presets + SDK extensibility

This document captures the open design questions and proposed
defaults for the next milestone: a **preset** abstraction layered
on top of the v1.24 `subagent` tool, plus an SDK extension path
so consumers can register their own tools and presets without
forking franky.

**Review and edit inline.** Mark each question with one of:

- `✓ accept` — go with the proposed answer.
- `→ <new answer>` — override with what you want instead.
- `?` — needs more discussion before deciding.

---

## 0. Why presets

A **preset** is a named purpose-built sub-agent template. Two
overlapping use cases, one foundation:

### 0.1 — Workspace-local presets

A preset that bundles a curated subset of franky's built-in
filesystem tools + a default model + a default system prompt.
Runs against the parent's workspace; no external auth needed.

Examples:
- `research` — read-only filesystem inspection (read+ls+find+grep).
- `code-audit` — focused audit-only against a stated concern.
- `file-ops` — targeted edits with read+write+edit.
- `general` — back-compat: inherits the parent's full tool list
  (the v1.24 default behavior, kept so existing prompts don't
  break).

The pattern: the LLM picks a preset to scope the sub-agent's
capabilities + bias the model + bias the system prompt for a
specific kind of work. No secrets involved. No external
network.

### 0.2 — External-retrieval presets

A preset that wraps a deterministic CLI or HTTP client for an
external system. The auth, endpoint, account ID, and guardrails
all live INSIDE the tool's closure — the LLM never sees them
and can't tamper with them.

Examples:

- `k8s-read`: read-only kubectl querying a specific cluster.
  The tool knows the kubeconfig path, ships the auth token,
  refuses any verb that mutates state. LLM never sees the
  cluster URL or token.
- `datadog-query`: hits the Datadog logs/metrics API with a
  pre-provisioned API key. LLM never sees the key, never knows
  the account ID, can only POST to read endpoints.
- `github-read`: reads issues / PRs / files via `gh` or octokit
  with a fine-scoped PAT. Mutations rejected at the tool layer.
- `incident-io-read`: queries the on-call API for active
  incidents using the operator's API key.
- `bigquery-read`: runs SELECT queries against a designated
  project; rejects DDL/DML at the tool layer.

External-retrieval presets share four properties:

1. **Auth lives in the tool, never in the LLM context.** The
   SDK consumer authenticates at startup (env var, file,
   keyring, OAuth flow once); the resulting credential lives
   in the tool's closure. The LLM provides only the *query*.
2. **Guardrails are deterministic.** Read-only is enforced by
   the tool refusing write verbs at parse time, not by prompt
   engineering. A `k8s-read` tool that gets `kubectl delete pod`
   from the LLM rejects with a structured error and doesn't
   call kubectl at all.
3. **Retrieval-only.** The sub-agent's job is "answer this
   specific question" — pull the data, summarize, return. Heavy
   intermediate output (10k lines of pod YAML) stays in the
   sub-agent's transcript, never pollutes the parent's context.
4. **One tool per preset, mostly.** A `k8s-read` preset's only
   tool is the kubectl-read tool — sometimes plus a workspace-
   local helper like `grep` for filtering. The sub-agent isn't
   open-ended; it's a focused query handler.

### 0.3 — Same foundation

Both categories use the SAME `Preset` type (§2.1), the SAME
registry (§3.4), the SAME LLM-facing surface (`subagent` tool +
`preset` arg + system-prompt hint, §3.1-3.3). The difference
is purely what each preset's `build_tools` closure returns and
whether the closure carries credentials. A `research` preset
returns `[read, ls, find, grep]` — no auth state. A `k8s-read`
preset returns `[kubectl_tool]` where `kubectl_tool` was
constructed at SDK startup with a kubeconfig pointer in its
closure.

That gives us three things v1.24 cannot:

1. **Specialized scoping** — workspace presets restrict tools
   appropriately (no `bash` for an audit bot); external presets
   wrap a single deterministic client.
2. **Credential isolation** for external presets — the LLM
   never sees secrets, tokens, endpoints, or account IDs. SDK
   consumer authenticates once; the tool carries the closure.
3. **SDK extensibility** — the consumer wires `k8s-read` or
   their own custom presets without forking franky.

The hard tension to resolve:

> **How does the LLM know which preset to pick without us
> pre-announcing every preset's full tool schema in the parent's
> tool definitions?** Pre-announcement bloats context fast (5
> presets × 200-token schema = 1000 tokens of dead weight before
> the conversation even starts) AND leaks information the LLM
> shouldn't have (cluster URLs, account IDs).

The whole design pivots on §3.3 below — discovery without
announcement.

## 1. Goal + non-goals

### 1.1 In scope (v1.25.0)

- A `Preset` type + `PresetRegistry` for SDK-registered presets.
- `subagent` tool's `parameters_json` gains a `preset: enum`
  field (see §3.2). The sub-agent's tool list comes ENTIRELY
  from the preset's builder — not inherited from the parent
  (see §3.6).
- Concise discovery surface: 1-line-per-preset summary in the
  parent's system prompt; opt-in `list_subagent_presets` tool
  when the catalog grows >5 entries (§3.3).
- **Authentication / secrets architecture (§new-9):** auth
  happens at SDK consumer's startup; credentials live in the
  preset's `userdata`/closure; the `subagent` tool's args
  schema NEVER includes auth fields, endpoints, or anything
  secret. The LLM ONLY supplies the query.
- **Tool guardrails (§new-10):** preset declares safety
  properties (read-only, rate-limited, sandboxed); tool
  implementation enforces them. A read-only kubectl tool
  literally cannot dispatch a write verb.
- **External-tool packaging (§new-11):** how franky tools wrap
  CLIs (kubectl, gh, datadog) vs HTTP APIs. Tool implementation
  detail; the `AgentTool` shape stays unchanged.
- Built-in preset catalog (§2.2 — small starter set; richer set
  in v1.26+).
- Audit log for every external preset call (who, when, query,
  duration, summarized result-size). Operator-readable.

### 1.2 Out of scope (deferred)

- **External-retrieval preset built-ins** in franky core.
  `k8s-read`, `datadog-query`, etc. ship via SDK consumers —
  franky core stays vendor-neutral. The registration API is
  fully public so consumers can ship them in their own modules.
- **Multi-preset composition** ("preset C = tools from preset
  A + preset B"). YAGNI for v1.
- **Preset hot-reload, versioning.** Static at session init.
- **Tool-level RPC / out-of-process tools.** §N.4 Tier-2/3
  extension ABI is v2; v1.25 is in-process Zig only.
- **Per-preset settings.json overlays.** Each preset uses the
  ambient overlay; per-preset config is v2 if needed.
- **Per-user / multi-tenant access control** (some users see
  preset X, others don't). v1: process-wide registry. Multi-
  tenant proxy auth is its own milestone.
- **Semantic-search-by-purpose discovery** for >50 presets.
  Premature; one-liners scale to ~20.
- **Auth-token rotation in-process.** v1: assume the credential
  in the preset closure stays valid for the session. SDK
  consumer that wants in-flight rotation provides their own
  refresher.

## 2. Surface (proposed)

### 2.1 The `Preset` type

```zig
// src/coding/tools/subagent.zig — additions

pub const Preset = struct {
    /// Unique short name. Lowercase letters, digits, hyphen. The
    /// LLM uses this in the `preset` arg.
    name: []const u8,

    /// One-line purpose (≤100 chars). Goes into the parent's
    /// system prompt verbatim — shown to the LLM as the
    /// catalog hint. Be specific about WHAT this preset does, not
    /// HOW. Bad: "uses read and grep". Good: "audits a codebase
    /// for a single security or quality concern."
    description: []const u8,

    /// Default profile (model + provider). Sub-agent uses this
    /// when the `profile` arg is omitted in the tool call.
    /// Required — every preset must work without a manual
    /// profile pick.
    default_profile: []const u8,

    /// Default capability role for the sub-agent. Demoted-only
    /// against the parent's role per v1.24 §3.5. `read` for
    /// audit / research; `plan` for refactor; `code` only when
    /// shell access is genuinely needed.
    default_role: role.Role,

    /// Default system prompt for the sub-agent. Override-able
    /// via the tool's `system_prompt` arg per v1.24 §3.11.
    /// Should NOT mention the `subagent` tool (depth-1).
    default_system_prompt: []const u8,

    /// Builder closure. Receives the per-call allocator + a
    /// borrowed Ctx-like view. Returns the sub-agent's tool list.
    /// Owned by the allocator (sub-agent will deinit).
    /// IMPORTANT: do NOT include `subagent` itself — depth-1
    /// enforcement is the preset author's responsibility too.
    build_tools: *const fn (
        allocator: std.mem.Allocator,
        view: *const PresetContext,
    ) anyerror![]at.AgentTool,
};

pub const PresetContext = struct {
    workspace: ?*const tools_workspace.Workspace,
    environ_map: *std.process.Environ.Map,
    // ... other ambient context the preset's builder may need
};

pub const PresetRegistry = struct {
    allocator: std.mem.Allocator,
    presets: std.StringArrayHashMapUnmanaged(Preset) = .empty,
    pub fn register(self: *@This(), p: Preset) !void { ... }
    pub fn get(self: *const @This(), name: []const u8) ?Preset { ... }
    pub fn iterator(self: *const @This()) ... { ... }
};
```

### 2.2 Built-in catalog (proposed)

v1.25.0 ships **workspace-local** built-ins. **External-retrieval
presets** like `k8s-read` / `datadog-query` ship via SDK
consumers — they need org-specific auth and rate-limit policy
that doesn't belong in franky core.

**Built-in (workspace-local, no secrets):**

| Name | Purpose (1-liner shown to LLM) | Tools | Default profile | Default role |
|---|---|---|---|---|
| `general` | General delegation — runs with the parent's full tool set. **The v1.24 default; back-compat.** | parent's filtered tool list | (parent's) | (parent's) |
| `research` | Reads files, greps for patterns, summarizes findings within the workspace. | read, ls, find, grep | `gemini` | read |
| `code-audit` | Audits a stated quality / security concern read-only across the workspace. | read, ls, find, grep | `gemini` | read |
| `file-ops` | Performs targeted file edits given clear instructions. | read, write, edit, ls | (parent's) | plan |
| `bash-runner` | Runs a bounded shell task, reports stdout/stderr/exit. | bash, ls | (parent's) | code |

`general` is the v1.24 back-compat path: existing prompts that
call `subagent` without a `preset` arg keep working. See §3.6.

**SDK-registered (illustrative — built outside franky core):**

These are the kind of presets a consumer wires up at startup
in their own `main`. franky core doesn't carry kubeconfig
parsing or Datadog API quirks — that's the SDK consumer's
job:

| Name | Purpose | Tool(s) | Auth source | Guardrails |
|---|---|---|---|---|
| `k8s-read` | Inspects kubernetes resources (pods, services, events, logs) on the configured cluster. | kubectl-read | `KUBECONFIG` env or pre-loaded kubeconfig in tool closure | Verb allow-list: get, describe, logs, top, version, explain. Reject all others at parse time. |
| `datadog-query` | Queries Datadog logs / metrics / monitors. Read-only. | datadog-api | `DD_API_KEY` + `DD_APP_KEY` env loaded into tool closure at SDK startup | Endpoint allow-list: `/api/v1/logs/list`, `/api/v2/logs/events/search`, `/api/v1/query`, etc. POST-only to those paths. |
| `github-read` | Reads issues, PRs, files, releases on a designated org/repo set. | gh-read or octokit-read | Fine-scoped `GITHUB_TOKEN` (read:repo + read:issue + read:org) | Verb allow-list within the gh CLI / octokit; rejects mutations. |
| `incident-io-read` | Lists active incidents, on-call schedules, post-mortems. | incident-io-api | API key | Read endpoints only. |
| `bigquery-read` | Runs SELECT queries against a designated BigQuery project. | bigquery-api | Service-account JSON or ADC | Parses SQL — rejects DDL/DML; project allow-list. |

Each external preset has the same shape: ONE specialist tool
whose schema is `{query: string}` (or similar), with the
auth / endpoint / guardrails wrapped inside the tool's closure
— invisible to the LLM. franky core ships none of these but
the registration API is fully public; an example consumer
(`franky-ops` or similar) can ship the kubectl-read tool +
preset together as a separate Zig module.

### 2.3 Discovery surface

**System prompt addition (existing v1.24.1 hint extended):**

```
You also have a `subagent` tool. Pick a preset that matches the
work; sub-agents are isolated and return only their final answer:
  - general: general delegation — full tool set
  - research: read+grep, no writes
  - code-audit: audit-only, read-only
  - file-ops: targeted edits with read+write+edit
  - bash-runner: bounded shell tasks
Profiles (model selection, separate from preset): gemini, groq,
ollama, ...
```

That's 7 lines, ~70 tokens. Scales to ~12-15 presets before
becoming bloat — at which point the optional discovery tool
takes over (§3.3.1).

**Optional discovery tool (`list_subagent_presets`):**

When registered, returns a structured list of `{name, description,
default_profile, default_role}`. The LLM calls it on demand
("which preset fits this task?"). Off by default — only added
when the registry has >5 user-defined entries.

## 3. Open questions

### 3.1 — One `subagent` tool with a `preset` arg, OR one tool per preset?

**Proposal: one tool with `preset` arg.** The tool's
`parameters_json` declares `preset` as an enum at session-init
time, populated from the registry. LLM sees ONE `subagent`
tool with N preset choices listed in the enum (and one-liners
in the system prompt).

**Why not one-tool-per-preset:** N tools means N×schema bloat
in context. With 8 presets × 200-token schema = 1600 tokens of
permanent overhead. With one tool, the enum is ~50 tokens
regardless of preset count.

**Trade-off:** If the LLM wants to call multiple presets in
one turn (e.g. "spawn research AND code-audit in parallel"),
the parallel-tool path already handles that (multiple tool_use
blocks of the same tool name → parallel execution per §4.4).

**Decision:** _____

---

### 3.2 — Preset arg shape: enum, string, or "preset name OR purpose description"?

**Proposal: enum (string with allowed values).** The tool
schema's `preset` field is `{"type": "string", "enum": ["general",
"research", ...]}`. The enum is generated at session-init from
the registry's keys. LLM gets autocomplete-style guidance from
the schema.

**Alternative — free-form string:** "any string the LLM provides
gets routed by a fuzzy lookup or registered router". More
flexible but harder to debug ("why did 'security review' route
to `code-audit`?"). Risk of LLM hallucinating preset names that
don't exist.

**Trade-off:** enum is rigid (every preset add invalidates prompt
caches with the new enum entry); free-form is fragile. v1: enum.
v2: optionally enable "purpose lookup" tool that takes free-form
text and returns the matched preset name, leaving the actual
spawn through the strict enum.

**Decision:** _____

### 3.3 — Discovery: how does the LLM learn what each preset is for?

This is the central design question of the whole milestone.
Options:

**Option A — System-prompt hint with name + 1-line purpose.**
Same shape as v1.24.1's profile-list interpolation. ~7-15 lines
total. Lazy: the LLM only "looks up" a preset when it decides to
delegate.

**Option B — Optional `list_subagent_presets` tool.** Returns
structured details on demand. LLM calls when in doubt. Adds
~100 tokens of tool schema (one extra tool definition).

**Option C — Both.** Hint by default; tool only registered when
catalog grows large.

**Proposal: Option C with thresholds.**
- ≤5 presets → hint only.
- 6-15 presets → hint (truncated to top 5 most-used) + tool.
- 16+ presets → tool only (hint becomes a 1-liner pointing to
  the tool).

**Why not just always-on hint:** at ~20 presets the hint becomes
80-100 lines, dominating the system prompt.

**Why not just always-on tool:** tool is one extra schema
permanently. For small catalogs it's pure overhead; the LLM
will rarely call it when the system prompt already lists 4-5
presets clearly.

**Open sub-question (§3.3.1):** How does the LLM PICK a preset
when the user gives a vague ask ("review this code")? Today's
model: the LLM reads its system prompt, sees the descriptions,
picks the best match. We're betting the descriptions are good
enough. If they're not, the hit-rate falls and operators add
the discovery tool. Self-correcting.

**Decision:** _____

### 3.4 — Where does the preset registry live?

**Proposal: Mode-driver-owned, populated at session init.** Each
mode (print/proxy/rpc/interactive) initializes a `PresetRegistry`
on the session/binding struct, registers built-ins, then accepts
SDK consumer extensions BEFORE the registry is frozen and handed
to the `subagent.Ctx`.

Lifetime: same as the session struct. SDK consumers register
presets in their `main` between mode init and `mode.run()`. No
mid-session mutation (avoids prompt-cache invalidation racing
with active turns).

**SDK API sketch:**

```zig
// In the SDK consumer's main:
var cfg: cli.Config = ...;
try franky.coding.tools.subagent.registerBuiltinPresets(&registry);
try registry.register(.{
    .name = "my-custom",
    .description = "Custom audit preset for our codebase.",
    .default_profile = "gemini",
    .default_role = .read,
    .default_system_prompt = my_prompt,
    .build_tools = myBuildToolsFn,
});
// Then dispatch to franky's mode runner.
```

**Why session-scoped, not process-global:** different sessions
in proxy mode could (someday) want different preset sets. v1:
all sessions get the same registry. Extending later is purely
additive.

### 3.5 — What does the preset's `build_tools` callback receive?

**Proposal: a `PresetContext` struct** carrying the ambient
state typical tool factories need:

```zig
pub const PresetContext = struct {
    workspace: ?*const tools_workspace.Workspace,
    environ_map: *std.process.Environ.Map,
    // bash_state, read_ctx — whatever's needed; grow this struct
    // as new tool factories require new context. Each field is
    // optional so SDK consumers can ignore what they don't need.
};
```

Why a struct rather than positional args: the field set will
grow over time (every new core tool that needs ctx adds one);
struct keeps the callback signature stable.

**Open sub-question:** does the preset's `build_tools` need
access to the parent's tool list (for `general`-style "inherit"
behavior)? **Yes** — see §3.6. So `PresetContext` includes
`parent_tools: []const at.AgentTool`. Most presets ignore it;
`general` reuses it.

### 3.6 — How does the `general` preset coexist with the v1.24 "inherit parent's tools" semantic?

v1.24 has no preset concept; sub-agents inherit parent's tools
filtered by role. Adding presets would BREAK existing user
prompts that say "use the subagent tool" without a preset.

**Proposal:**
- `preset` is an OPTIONAL arg.
- When omitted → defaults to `general` (back-compat with v1.24).
- `general`'s `build_tools` returns the parent's tool list
  minus `subagent` itself — exactly what v1.24 does today.
- Existing user prompts continue to work unchanged.
- New prompts can opt into specialization with explicit `preset`.

**Why not break compat:** v1.24's surface is published; user
prompts and franky-do v0.3+ depend on it. Silent break = pain.

**Decision:** _____

### 3.7 — Tool definition: how does the LLM see the schema for the `preset` arg?

Two sub-questions:

**3.7.1 Static at session init, or dynamic per request?**
**Proposal:** Static. The `parameters_json` is built ONCE at
session init from the registry. Cached on the tool. The enum
list is fixed for the session.

**Why not per-request:** per-request schema generation defeats
prompt caching. The registry is frozen at init anyway, so
nothing's gained from per-request rebuild.

**3.7.2 Are preset descriptions embedded in the tool schema,
the system prompt, or both?**
**Proposal:** Names in the `preset` enum (so the LLM can name
them). One-line descriptions in the system prompt (so the LLM
knows what each one does). Detailed descriptions ONLY surfaced
via `list_subagent_presets`. This three-tier disclosure keeps
the per-message overhead low while making detail discoverable
on demand.

**Decision:** _____

---

### 3.8 — Custom tool registration: how do SDK consumers add new tools?

**Proposal: tools are just `at.AgentTool` values returned from
preset builders.** The SDK consumer writes their own tool
factory function exactly like franky's built-ins
(`read.tool()` etc.) — same signature, same execute callback
shape. The preset's `build_tools` returns a slice that includes
their tool.

**No tool registry per se** — tools are "owned" by the presets
that include them. If two presets both want a tool, they each
build it. Cheap (factories are small) and avoids cross-preset
coupling.

**Why not a global tool registry:** adds a layer of indirection
without a clear benefit at v1's scale. Presets are the unit of
extensibility; tools are implementation details inside presets.

**Decision:** _____

### 3.9 — How do we handle preset name collisions?

A built-in is `research`. SDK consumer registers `research` too.

**Proposal: SDK-registered wins (overrides built-in).** Same
precedence as the profiles system (§v1.24-design §3.2 — user
profiles override built-ins of the same name). Operators can
intentionally replace a built-in's behavior; we log a `warn`
when override happens so it's not silent.

**Alternative — error on collision.** Forces explicit naming.
Friendlier to debugging but worse for SDK consumers who want to
"swap in their better research preset".

**Decision:** _____

### 3.10 — Preset role-demotion: does the preset's `default_role` apply if the parent's role is lower?

Parent `--role plan`. Preset `code-audit` has `default_role =
read`. Sub-agent runs as `read` (correct demotion).

Parent `--role read`. Preset `bash-runner` has `default_role =
code`. v1.24 §3.5 says "demotion only — never promotion".

**Proposal:** when the preset's `default_role` is HIGHER than
the parent's role, the spawn fails with `error_kind:
"role_promotion_denied"`. SAME rule as the per-call `role` arg.
Preset author's responsibility to choose a default_role
appropriate for the lowest role this preset is expected to run
under, OR to have the operator override at spawn time.

**Per-call override:** the v1.24 `role` arg can still demote a
preset's default further (`bash-runner` defaults to `code`; user
demotes to `read` for an unusually-restricted run).

**Decision:** _____

### 3.11 — How do we test a preset registry?

**Proposal:** three test categories:

1. **Static schema**: assert that with N built-in presets, the
   tool's `parameters_json` is well-formed and contains the
   expected enum entries. Pin the regression that adding a
   preset doesn't drop existing ones.
2. **Dispatch**: a faux-provider sub-agent ALL-allowed; trigger
   `subagent({preset: "research", prompt: "..."})`; assert the
   sub-agent received the research preset's tool list (read +
   ls + find + grep, no write/edit/bash) and the research's
   default_profile + default_role + default_system_prompt.
3. **SDK extension**: register a custom preset with a custom
   tool, assert it shows up in the schema enum, assert
   `subagent({preset: "custom", ...})` invokes the custom tool.

**Decision:** _____

Current v1.24 description: `"Spawn an isolated sub-agent with

## 3.bis — Cross-cutting concerns (apply mostly to external presets)

### 3.14 — Credential / secret handling

**Proposal:** secrets stay entirely OUT of the LLM context.
Three mechanics, in order of preference:

1. **Closure capture** (preferred): SDK consumer reads the
   secret at startup (env / file / OS keyring), packages it
   inside the tool's `ctx` field. The tool's `execute` reads
   `ctx.api_key` etc. The LLM never sees the secret in the
   tool's `parameters_json` schema, never in args, never in
   results.
2. **Lookup by handle**: tool args carry an opaque handle
   (`{credential: "datadog-prod"}`). The tool resolves the
   handle to a real secret via a per-process credential
   store. Slightly more flexible (one tool can serve multiple
   accounts) but more moving parts. v1: closure capture only.
3. **OAuth / token refresh**: out of scope for v1.25 (deferred
   in §1.2).

**Schema rule:** preset authors MUST NOT add fields to the
sub-agent's `parameters_json` whose names suggest secret
material (`api_key`, `token`, `password`, `secret`,
`auth`, `kubeconfig`). v1 enforces this with a runtime check
at preset registration time — `register` returns
`error.SchemaContainsLikelySecret` if the JSON contains any
of those substrings as field names. Catches the obvious
foot-gun.

**Audit:** every external preset call gets logged at `info`
with `{preset, query_summary, duration_ms, result_bytes}` —
NEVER the credential, NEVER the full query if the schema
declares a query field as `sensitive: true`. Operators get
their own audit trail without LLM-output noise.

**Decision:** _____

### 3.15 — Guardrails

**Proposal:** preset declares safety properties as data; tool
implementation enforces them as code:

```zig
pub const SafetyClaims = struct {
    /// Read-only — tool implementation must reject any
    /// mutating verb / endpoint at parse time.
    read_only: bool = false,
    /// Per-call rate limit (operator-tunable). When exceeded,
    /// the tool returns a structured `rate_limited` error
    /// without making the upstream call.
    max_calls_per_minute: ?u32 = null,
    /// Sandbox required — tool refuses to run outside a
    /// `franky-zerobox` / Docker / similar sandbox.
    requires_sandbox: bool = false,
    /// Per-call output truncation — sub-agent's tool result
    /// is truncated to this many bytes before being added to
    /// the sub-agent's transcript. Prevents a 50 MB pod-yaml
    /// dump from blowing up the sub-agent's context.
    max_result_bytes: ?u32 = null,
};

pub const Preset = struct {
    // ... existing fields ...
    safety: SafetyClaims = .{},
};
```

The SafetyClaims are advisory metadata (the tool implementation
is the actual enforcer) but they're ALSO surfaced to the LLM in
the discovery surface — "this preset is read-only" makes the
LLM more comfortable picking it for risky-sounding queries.

**Rate-limit enforcement** sits next to the existing v1.24.4
concurrency cap pattern but is per-preset, not global.

### 3.16 — External-tool packaging (CLI vs HTTP vs SDK lib)

How does an SDK consumer actually wrap kubectl / Datadog as
a franky AgentTool?

**Three idioms, all valid:**

1. **CLI shell-out**: tool's `execute` runs `kubectl get ...`
   via `std.process.Child`, captures stdout/stderr, returns.
   The tool implementation parses LLM args into a verb +
   resource + namespace tuple (rejecting non-allow-listed
   verbs at parse time), constructs the command line itself,
   spawns. Auth comes from the kubeconfig path in the tool
   closure; LLM never gets to set it.
2. **HTTP API**: tool's `execute` makes an HTTPS request
   directly via `std.http.Client`. Cleanest for Datadog,
   GitHub REST, Incident.io, etc. Auth header from closure.
3. **Bundled Zig SDK**: an SDK consumer pre-compiles a
   provider's official SDK (or their own) and calls into it.
   Same shape as CLI/HTTP from franky's perspective; just
   nicer for the consumer.

franky core provides NONE of these directly — it provides
the `AgentTool` interface (`execute`, `parameters_json`) and
the `Preset` registration. The CLI/HTTP/SDK choice is the
consumer's. Common examples will live in a sibling module
like `franky-ops` once we have one.

**Helper utility we should add to franky core:** a tiny
`shell.zig` or `subprocess.zig` that wraps `std.process.Child`
with sane defaults (timeout, stdout/stderr capture, non-shell
exec) so consumer presets don't all reinvent it.
