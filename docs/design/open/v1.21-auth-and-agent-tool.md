## §1.5 — `auth.json` 0600 mode-check

**Status:** spec called this blocked on missing stdlib surface, but
`std.c.fstat` (macOS/BSD) and `std.os.linux.fstat` (Linux) are both
reachable today via the file's `.handle` field — same pattern we
already use for `std.posix.read(stdin.handle, ...)` in interactive
and rpc modes. So this is a quick win, **~30-50 LOC** including
tests.

### Q1.5.1 — Behavior on insecure mode

**Spec language:** "must refuse to read auth.json if the file's
permissions are more permissive than 0600."

**Proposed:** Return a new `AuthError.InsecureMode`. Loader treats
this as a hard error (not a warning) — the caller must handle it.
Print mode currently does `auth_state = auth_mod.load(...) catch
null;` which would silently swallow the error and fall back to env
vars; we should change that to `catch |e| switch (e) {
.InsecureMode => print_a_warning_and_exit, else => null }` so the
user sees *why* their auth.json was ignored.

**Why not auto-fix.** Silently `chmod 0600`ing a manually-edited
file would mask both user intent and a security incident. Refusing
loud is the right default.

**Decision:** _____

### Q1.5.2 — Windows behavior

**Proposed:** Skip the check entirely on Windows. POSIX mode bits
don't apply there; Windows uses ACLs, which aren't fungible with
0600 semantics. The spec doesn't mention Windows; I'm reading
"§H.1 must refuse on POSIX" as implicitly Unix-scoped.

**Decision:** _____

### Q1.5.3 — What counts as "insecure"

**Proposed:** Any of the bits in `0o077` set on the file
(group-read, group-write, group-execute, other-read, other-write,
other-execute). I.e. `(stat.mode & 0o077) != 0` → refuse. This
matches the spec's "more permissive than 0600" phrasing literally.

**Open subquestion:** what about the *user* execute bit (`0o100`)?
Strictly that's "more permissive" too if the file is a JSON config,
but it's a no-op for a non-script file. **Sub-proposal:** ignore
the user-execute bit (only check the bottom 6 bits). Matches what
`ssh` does for `~/.ssh/id_rsa`.

**Decision:** _____

### Q1.5.4 — Error message text

**Proposed:** When the loader refuses, the warning printed to
stderr (or returned via `AuthError.InsecureMode`) reads:

```
auth.json at /Users/.../auth.json has insecure permissions
(0644). franky refuses to read it. Fix:  chmod 0600 /Users/.../auth.json
```

i.e. include the actual mode (in octal) and the actual path so the
user can copy-paste the fix. **Decision:** _____

### Q1.5.5 — Test approach

**Proposed:** Add three tests in `auth.zig`:

1. `load: refuses 0644 file` — write the file via `createFile`,
   `fchmod` it to 0644, expect `AuthError.InsecureMode`.
2. `load: accepts 0600 file` — same but 0600, expect normal load.
3. `load: skips check on Windows` — `if (builtin.os.tag == .windows)
   return error.SkipZigTest;` for portability.

The fchmod call uses `std.c.fchmod` on macOS and
`std.os.linux.fchmod` on Linux (same dispatch as the load-side
check).

**Decision:** _____

## §7.1 — `agent` tool

**Status:** new eighth `coding` tool; spawns a child franky process
to delegate work, with the child structurally locked to a bounded
role. ~200-400 LOC of new code (tool + tests), no architectural
change. Composes with the v1.19.0 settings-layer overlay (project
settings can pre-seed the child's permission store).

### Q7.1.1 — Argument surface

**Proposed schema:**

```json
{
  "type": "object",
  "required": ["prompt"],
  "properties": {
    "prompt":        {"type": "string", "description": "What to ask the child agent."},
    "system_prompt": {"type": "string", "description": "Optional system-prompt override."},
    "timeout_ms":    {"type": "integer", "minimum": 1000, "description": "Hard timeout (default 300000)."}
  },
  "additionalProperties": false
}
```

**Deliberately omitted:** `profile`, `thinking`, `role`, `yes`,
`prompts`, `allow_tools`, `deny_tools`. Role is hard-locked (next
question). Thinking + profile inherited from parent env. Permission
flags would let the parent escalate the child's blast radius —
exactly what the tool is designed to *prevent*.

**Add later if real usage demands it:** `cwd` (start the child in a
specific working dir), `env` (inject extra env vars), `model` (one-
shot model override).

**Decision:** _____

### Q7.1.2 — Role tier classification

This determines who has access to the `agent` tool itself, and
whether children can recurse.

**Option A (my recommendation): `code` tier.**
- Parents running `--role code` or `--role full` can spawn agents.
- Plan-locked children cannot recurse (they don't have the tool).
- Naturally caps depth at 1. No env-var depth tracking needed.
- Simpler. Removes Q7.1.6 entirely.

**Option B: `plan` tier with depth cap.**
- Plan-locked children also have the tool, can recurse.
- Need `FRANKY_AGENT_DEPTH` env var to prevent runaway chains.
- More complex, more failure modes, but enables tree-of-agents
  patterns.

**Open question:** is multi-level delegation a real use case for
v1.21, or YAGNI? I lean YAGNI — single-layer delegation handles
"parent does coding, fans out research/analysis to plan-locked
helpers" which is the original motivating example. Recursion can
be added later by promoting the tool to `plan` tier (one-line
change in `role.zig`).

**Decision:** _____ _(if Option B, also resolve Q7.1.6)_

### Q7.1.3 — Output handling

**Proposed:** Child's stdout becomes the tool result text. stderr
is captured separately:

- If exit code = 0: stdout → result text. stderr → discarded
  (most franky stderr at info+ is noise the parent doesn't need).
- If exit code ≠ 0 OR timeout: stdout (if any) + a `[stderr]`
  section with the captured stderr → result text marked as
  `is_error: true`.

If the parent has `--log-file <path>` configured, write the child's
full stderr to that path verbatim regardless of exit status, so
debug traces aren't lost.

**Output size cap:** match the bash tool's 1 MiB stdout cap. Beyond
that, return a `tool_code = "agent_output_too_large"` error. We
don't want a runaway child to OOM the parent.

**Decision:** _____

### Q7.1.4 — Cancellation

**Proposed:** Use `std.process.Child.spawn()` (not
`std.process.run`) so we keep the pid. The tool's `execute` reads
the `cancel: *ai.stream.Cancel` parameter; on `cancel.isFired()` →
`SIGTERM` to the child's process group, wait 2s grace, then
`SIGKILL` if still alive. Mirrors how a shell handles Ctrl-C.

The 2s grace is enough for franky to flush its session/transcript
to disk (the v1.7.x persistence path is well-tested). If it
doesn't, `SIGKILL` cleans up; orphaned session files are
acceptable.

**Open subquestion:** do we also want to forward parent-side cancel
*deltas* (i.e. let the user abort the child mid-run via the parent's
`POST /abort` endpoint or interactive Ctrl-C)? **Sub-proposal:**
yes, by polling `cancel.isFired()` every ~250ms in the wait loop.
The bash tool already does this implicitly via its `timeout`.

**Decision:** _____

### Q7.1.5 — Cost & timeouts

**Proposed:** `timeout_ms` arg, default 300_000 (5 minutes). Hard
ceiling — cancellation flow above kicks in at expiry.

**Token budget: out of scope for v1.21.** Bounding tokens-per-turn
on the child would require a new franky CLI flag
(`--max-tokens-per-turn`?) that doesn't exist today; it's a
separate feature. The `timeout_ms` ceiling already provides a
soft cost cap (a runaway child can't burn budget for >5min).

**Decision:** _____

### Q7.1.6 — Recursion depth (only if Q7.1.2 → Option B)

**Proposed (only if Q7.1.2 picks Option B):** Env var
`FRANKY_AGENT_DEPTH`:
- Read on parent startup; default 0 if not set.
- Child invocation gets `FRANKY_AGENT_DEPTH = parent_depth + 1`.
- If depth ≥ 3 when the `agent` tool fires, refuse with
  `tool_code = "agent_depth_exceeded"`.

The cap of 3 is arbitrary but matches the "tree-of-agents 1 level
deep delegating to 1 more level deep" mental model. Beyond 3 is
almost certainly a confused chain.

**Decision (skip if Q7.1.2 → Option A):** _____

### Q7.1.7 — Auth / profile inheritance

**Proposed:** Child inherits the parent's full process environment
via `std.process.Child.env_map = null` (which means "inherit
parent's environ"). That covers:

- `auth.json` discovery (via `FRANKY_HOME` / `HOME`)
- API keys (`ANTHROPIC_API_KEY`, etc.)
- Active profile (via `FRANKY_PROFILE` if the parent's env has it)
- Sandbox detection (`FRANKY_*` envs)
- Cwd (child starts in parent's cwd unless `cwd` arg added later)

**No tool-arg overrides for any of the above** — keeps the surface
minimal. The parent decides at startup time which credentials to
use; the child inherits them transparently.

**Decision:** _____

### Q7.1.8 — How the parent finds the franky binary

**Proposed:** The `agent` tool needs to spawn `<franky-binary>
--mode print --role plan ...`. Resolution order:

1. `FRANKY_BIN` env var if set (escape hatch for tests + custom
   deploys).
2. `/proc/self/exe` on Linux, `_NSGetExecutablePath` on macOS to
   find the running binary's path.
3. Fallback: `argv[0]` resolution against `PATH`.

This ensures the child runs the *same binary* as the parent, not a
stale `franky` from `PATH`. Important for testing (faux provider
in tests) and for guaranteeing version parity between parent and
child.

**Decision:** _____

### Q7.1.9 — Test approach

**Proposed:** All tests use the faux provider (no real API calls).

1. `agent: spawns child with --role plan + faux replies` — set
   `FRANKY_BIN` to the test binary, fire the tool with a known
   prompt, assert the captured stdout contains the faux reply.
2. `agent: timeout fires SIGTERM` — set `timeout_ms = 100`, child
   sleeps via a faux script that takes longer; assert tool result
   marks `is_error: true` and includes a timeout message.
3. `agent: cancellation propagates SIGTERM mid-run` — fire
   `cancel.fire()` from a separate thread; assert the child exits.
4. `agent: nonzero exit captures stderr in result` — child run
   fails (e.g. unknown profile); assert result is_error + carries
   stderr text.
5. `agent: oversized output → agent_output_too_large` — child
   emits >1 MiB; assert tool_code is set, no full output captured.
6. `agent: depth cap` — only if Q7.1.2 → Option B. Set
   `FRANKY_AGENT_DEPTH=3`, fire the tool, assert refusal with
   `agent_depth_exceeded`.

**Decision:** _____
