# v2.19 — Design Documents panel in the web UI

A panel inside the web UI that lists the files under `docs/design/`, groups them
by lifecycle state, and pre-fills the composer with a context-aware prompt when
the user selects a document.

---

## Motivation

Design documents live in `docs/design/{open,decided}/`.  
Right now acting on them requires the user to:

1. Open their file manager or editor to find the file.
2. Manually compose a prompt that says "read X and implement it" or "let's discuss X".

Both steps are friction.  The web UI already knows the workspace root; it can
serve the doc list and pre-fill the right starter prompt automatically.

---

## Two prompt archetypes

**Decided doc** — a design that has been fully settled. The natural next action is
*implementation*. The pre-filled prompt should be:

```
Please implement the design described in docs/design/decided/<filename>.
Read the document first, then follow the implementation plan closely.
Start with the scope marked as v1 / first milestone.
```

**Open doc** — a design with unresolved questions. The natural next action is a
*design session*: reading the document together and driving each open question to
a decision. The pre-filled prompt should be:

```
Let's work through the open design questions in docs/design/open/<filename>.
Please read the document, summarise the open questions, and help me decide each
one. After every question has a decision mark it with "✓ decided:" and the
chosen answer inline.
```

Both prompts inject only the file *path*, not the full content. The agent reads
the file with the `read` tool at the start of the turn.

## Design questions

### Q1 — Where does the panel live?

**Option A — Right-side drawer (same pattern as the subagent panel)**  
A new `<aside id="design-panel">` mirroring `#subagent-panel`. A button in the
header (e.g. `📐` or `Design`) toggles it. Works well because the subagent-panel
infrastructure (CSS, open/close, `hidden` toggle) already exists.

**Option B — Left sidebar section below the session list**  
Extends `#sidebar` with a collapsible "Design docs" section. Lower click cost
(always visible if sidebar is open) but makes the sidebar taller.

**Option C — Modal/overlay triggered by `/design` slash command**  
Keeps the header clean; no persistent chrome. Fits the existing slash-command
pattern. Downside: requires keyboard or slash-command knowledge to open.

**Option D — Inline in the slash-command palette**  
`/design` in the palette expands to a sub-list of docs. Zero new chrome. But the
palette is ephemeral and loses any folder-tree structure.

**Proposed: Option A (right-side drawer)**  
The subagent-panel already establishes the right-side drawer pattern.  Reusing it
keeps the layout model simple.  The subagent panel and the design panel can
coexist because they are independently hideable.

**Decision:** ✓ decided: Option A — right-side drawer, mirroring the subagent panel.
Consider also the subagent (rarely happend will also be displayed there just consider it for the css and html style)

---

### Q2 — How are the docs fetched?

**Option A — New `GET /design-docs` API route**  
The proxy scans `docs/design/` at request time, returns JSON:

```json
{
  "docs": [
    { "path": "docs/design/decided/v2.17-self-restart.md",
      "name": "v2.17-self-restart",
      "category": "decided" },
    { "path": "docs/design/open/v1.21-auth-and-agent-tool.md",
      "name": "v1.21-auth-and-agent-tool",
      "category": "open" },
    { "path": "docs/design/v2.14-agentic-engineering.md",
      "name": "v2.14-agentic-engineering",
      "category": "root" }
  ]
}
```

The `category` field is derived from the path prefix (`decided/`, `open/`, or
neither → `"root"`).

**Option B — Embed the list at startup in the SSE `session_init` event**  
Avoids a separate HTTP round-trip. More coupling between proxy and the design
directory.

**Option C — Read via the existing `ls` / `read` tools on the fly**  
No new server route. But it requires an agent turn, which is too slow for
populating a UI panel.

**Proposed: Option A**  
Clean separation; the panel fetches on open (lazy) so the proxy only does work
when the user opens it.

**Decision:** ✓ decided: Option A — `GET /design-docs` route.

---

### Q3 — What triggers the fetch?

**Option A — On panel open**  
Fetch `GET /design-docs` every time the user opens the panel.  Always fresh; zero
startup cost.

**Option B — Once at page load, then cached**  
The list rarely changes during a session. A single fetch at startup is fine;
add a manual "Refresh" button for cases where the agent created a new doc.

**Proposed: Option A (fetch on open)**  
Simplest, always correct. The directory is small (tens of files) so latency is
negligible.

**Decision:** ✓ decided: Option A — fetch on every panel open.

---

### Q4 — Does the panel show a preview / summary?

**Option A — Title + category badge only (minimal)**  
The panel is a navigator, not a reader. Click → pre-fill → compose.

**Option B — First paragraph of the doc as a tooltip on hover**  
Gives context without leaving the panel.

**Option C — Inline preview that expands on click**  
Full reading experience in the panel. High implementation cost.

**Proposed: Option A for v1, Option B as a v2 enhancement**  
The panel's job is to trigger a conversation, not replace the editor.

**Decision:** ✓ decided: Option A — title + category badge only for v1; Option B (hover preview) deferred to v2.

---

### Q5 — What about docs at `docs/design/` root (neither decided nor open)?

These are designs in an intermediate state (written but not yet categorised).

**Option A — Show them under a third "draft" group**  
**Option B — Omit them from the panel**  
**Option C — Treat them as `open` for prompt purposes**  

**Proposed: Option A — show with a "draft" badge, use the open-session prompt**  
They exist, they're useful, hiding them creates confusion.

**Decision:** ✓ decided: Option A — show as a "draft" group with the open-session prompt archetype.

---

### Q6 — Does selecting a doc replace or append to the composer?

If the composer already has text, silently overwriting it is destructive.

**Option A — Always replace (with a confirmation if non-empty)**  
**Option B — Always append (safer, but may create garbled prompts)**  
**Option C — Replace only if composer is empty; show a toast if it wasn't**  

**Proposed: Option C**  
Matches how the session-list click behaves (it does not clear your in-progress
message).

**Decision:** ✓ decided: Option C — replace only if composer is empty; show a toast if it wasn't.

---

### Q7 — Slash command surface

Should `/design` be added to the command palette to open the panel?

**Proposed: Yes** — mirrors `/tools`, `/model`, etc. One-word command, no args.

**Decision:** ✓ decided: Yes — add `/design` to the slash-command palette.

---

### Q8 — Where is `docs/design/` resolved?

**Option A — Relative to the workspace root (the directory franky was started in)**  
**Option B — Configurable via `settings.json` key `designDocsPath`**  
**Option C — Auto-discover: walk up from workspace root until `docs/design/` is found**  

**Proposed: Option A**  
The proxy already knows the workspace root. A config option is a v2 concern.
If `docs/design/` does not exist the panel shows "No design docs found."

**Decision:** ✓ decided: Option A — relative to workspace root; missing directory shows "No design docs found."

- Closes the panel.

### Q9 — Does clicking a doc close the panel?

**Proposed: Keep open.** The user may want to click a second doc or compare
titles before sending. The first send closes nothing; the panel toggles
independently.

**Decision:** ✓ decided: Close on click — panel closes when a doc is selected, keeping the screen clear. (Overrides proposal.)

## Client-side implementation sketch

```js
// Toggle panel
designPanelBtn.addEventListener('click', () => {
    const hidden = designPanelEl.hidden;
    if (hidden) fetchAndRenderDesignDocs();
    designPanelEl.hidden = !hidden;
});

async function fetchAndRenderDesignDocs() {
    const data = await fetch('/design-docs').then(r => r.json());
    renderDesignPanel(data.docs);
}

function promptForDoc(doc) {
    const templates = {
        decided: `Please implement the design described in ${doc.path}.\nRead the document first, then follow the implementation plan closely.\nStart with the scope marked as v1 / first milestone.`,
        open:    `Let's work through the open design questions in ${doc.path}.\nPlease read the document, summarise the open questions, and help me decide each one.\nAfter every question has a decision mark it with "✓ decided:" and the chosen answer inline.`,
        root:    `Let's discuss the design in ${doc.path}.\nPlease read the document, identify any open questions or missing decisions, and help me flesh them out.`,
    };
    return templates[doc.category] ?? templates.root;
}
```
