# v3.0 — Context window improvements

> **Status:** Design doc — not yet implemented  
> **Last updated:** 2025-07

## Problem

After 220+ tool calls the request body for a single LLM call can exceed
**100k tokens** (≈1.7 MB of tool-result text + tool-call metadata). The
transcript grows monotonically — every user message, assistant reply,
tool call, and tool result is sent with every request.

This has three consequences:

1. **Latency increases** with every turn (more tokens to process).
2. **Cost increases** (input token pricing).
3. **Context window fills up**, eventually hitting the model's limit
   (typically 100k–200k tokens for Anthropic, 128k–1M for OpenAI/Google).

## Current mitigations (already shipped)

| Mechanism | Where | Effect |
|-----------|-------|--------|
| Per-text-block truncation (8KB) | `loop.makeToolResultMessage` | Each tool result content block is capped at 8 KB |
| Bash/Read/Grep output truncation | `truncate.zig` | 50 KB / 2000 line cap on tool output |
| Manual `/compact` | `compaction.zig` + slash handler | LLM-summarizes old messages, replaces them with a `compaction_summary` |
| Token-limit nudging | `loop.maybeNudge` | When `stop_reason == .length`, injects a "continue" prompt |
| Forced summary on max turns | `loop.runForcedSummary` | When turn budget exhausted, runs one final tool-disabled turn |

## Gaps (not yet addressed)

- **No automatic compaction** — `shouldTrigger` (≥80% / ≥92% of context
  window) exists but is never called from the agent loop.
- **No sliding window** — oldest messages are never dropped.
- **No tool-result offloading** — every tool result stays in the
  transcript forever.
- **No per-message eviction policy** — no way to say "keep last N tool
  results" without manual /compact.

## Proposed work

### 1. Tool-result offloading ✓ (shipped v3.0)

**Config:** `settings.json` field `tools.maxFullToolResults` (default 0 =
disabled).

When set to N > 0:

- Only the **N most recent** `tool_result` messages in the transcript
  carry full content to the LLM.
- Older tool results are replaced with a compact placeholder before
  serialization:
  `[tool result: <call-id> — <size> (offloaded)]`
- Non-tool-result messages (user, assistant, custom) pass through
  unchanged.
- The transcript itself is **not mutated** — only the copy sent to
  the LLM is transformed.

**Implementation:** `loop.offloadToolResults()` is called in `runTurn()`
after `convert_to_llm` but before building the `Context`. Wired through
`loop.Config.max_full_tool_results`.

### 2. Automatic compaction trigger (proposed)

**Config:** `settings.json` field `autoCompactThreshold` (default: none).

When the estimated transcript token count exceeds X% of the model's
`context_window` (e.g. 80%), the agent loop between turns runs the
`compaction.run()` summarization round-trip automatically.

This reuses the existing `compaction.zig` engine — same `selectSpan`,
`renderSpanAsPrompt`, `runSummarizer` path that `/compact` uses.

**Open questions:**
- Should this be blocking (turn waits for summarization) or fire-and-forget?
- How to handle summarizer failure mid-conversation?
- Should the user be notified via an agent event?

### 3. Sliding window / message eviction (proposed)

**Config:** `settings.json` field `maxTranscriptMessages` (default: none).

When the transcript exceeds this many messages, the oldest messages are
dropped (with the exception of preserving the first user message).

Simple alternative to compaction — no LLM call needed. Loses information
but is predictable and free.

### 4. Priority-based eviction (future)

Not all messages are equally valuable. A priority scheme might be:

- **Tier 0 (never drop):** System prompt, first user message, pinned
  messages.
- **Tier 1 (compact first):** Old tool results that reference tools
  no longer in the tool set.
- **Tier 2 (compact later):** Old assistant messages without tool calls.
- **Tier 3 (keep):** Recent messages (within tail budget).

The compaction engine already approximates this with its `selectSpan`
preservation rules (first user, tail budget, pinned, orphan-tool-pair).
This could be exposed as a configurable strategy.

## Implementation order

1. **Tool-result offloading** ✓ — shipped as `maxFullToolResults`
2. **Auto-compaction trigger** — wire `shouldTrigger` into the
   `between_turns` hook or the main loop
3. **Sliding window** — simple `maxTranscriptMessages` cap
4. **Settings.json UI + CLI flag** for all of the above

## settings.json schema additions

```json5
{
  "tools": {
    "maxFullToolResults": 5,           // keep full content for last 5 results
    // Future:
    // "autoCompactThreshold": 80,     // % of context window
    // "maxTranscriptMessages": 500    // hard cap on transcript size
  }
}
```

## Measuring impact

Each offloaded tool result replaces ~8 KB of text (after per-block
truncation) with ~80 bytes. For 200 offloaded tool results:

- **Before:** 200 × 8 KB = 1.6 MB → ~460k tokens
- **After:** 200 × 80 B = 16 KB → ~4.6k tokens

Saving: **~99 %** of tool-result token cost for stale entries.
