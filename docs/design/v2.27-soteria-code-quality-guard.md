## 1. What this design achieves

The model (franky) produces code changes; those changes land in files that may already be hotspots — high-complexity, high-churn, single-author files that are at risk of knowledge loss. Today the guardrails check **compilation** (`zig build`) and **stuck loops** (repeated identical edits), but not **whether the file being edited is already a quality problem**.

The Soteria guardrail adds:

1. **Pre-edit check** — before mutating a file, the guardrail checks whether the target is a known hotspot or has a negative complexity trend. If so, it injects a refactoring hint into the model's transcript.
2. **Post-edit check** — after a mutation pass, the guardrail re-scans the changed files and compares new metrics against the baseline. If complexity has increased, it injects a warning.
3. **Session summary** — on `finish_task`, the guardrail includes a quality delta in the summary so the sibling app or human reviewer can see whether the session improved or degraded the codebase.

All checks are advisory — they inject hints into the transcript, not hard blocks. The model can ignore a hint, but seeing the signal tends to shift its behaviour toward simpler code.

### Non-goals

- Do not replace `CompilationGuard`. Compilation is a hard gate; quality is a soft gate.
- Do not block the model from editing a hotspot file. The hint is informational.
- Do not add a new tool. The guardrail communicates through transcript hints, same as the stuck detector.
- Do not vendor or fork the upstream soteria binary. Use it as an external process (like `zig build`).

## 2. Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│ GuardrailState                                                  │
│  ┌──────────────────────┐   ┌────────────────────────────────┐  │
│  │ StuckDetector        │   │ SoteriaGuard (NEW)             │  │
│  │  → afterToolCall     │   │  → .soteria/thresholds.json   │  │
│  │  → betweenTurns hint │   │  → .soteria/history.jsonl     │  │
│  ├──────────────────────┤   │  → soteria scan --out /tmp/   │  │
│  │ CompilationGuard     │   │  → checkHotspot(path)         │  │
│  │  → afterToolCall     │   │  → checkRegression(path)      │  │
│  │  → betweenTurns hint │   │  → qualityDelta()             │  │
│  ├──────────────────────┤   └────────────────────────────────┘  │
│  │ FinishTaskState      │                                       │
│  │  → auto-commit       │                                       │
│  │  → quality summary   │                                       │
│  └──────────────────────┘                                       │
│  GuardrailState.betweenTurns() — runs soteria hints after       │
│  compilation / stuck checks, same priority slot.                │
└─────────────────────────────────────────────────────────────────┘
```

### Priority in `betweenTurns`

The existing `betweenTurns` order is:

1. finish_task pending → schedule compilation
2. finish_task compilation → run forced compilation → commit
3. **Compilation guard** triggered → run build → inject hint
4. **Stuck detector** pending hint → inject hint
5. **[NEW] Soteria quality hint** → inject regression/hotspot hint

Soteria runs last so it never competes with compilation or stuck handling. It is purely informational.

### Soteria binary lifecycle

The `soteria` binary lives at the workspace root (same as the existing `soteria` binary). The guardrail invokes it as a child process:

```sh
./soteria --out /tmp/soteria-scan-<ts>.json \
          --thresholds .soteria/thresholds.json \
          --history .soteria/history.jsonl
```

The JSON output is parsed into a `types.Report` struct. The guardrail then runs queries against the parsed report.

## 3. SoteriaGuard struct

```zig
const SoteriaGuard = struct {
    allocator: std.mem.Allocator,
    config: Config,
    /// Parsed thresholds from last scan (cached).
    thresholds: ?types.Thresholds,
    /// Last scan result, populated after a run.
    last_report: ?types.Report,
    /// Paths that have been flagged as hotspots this session
    /// (to avoid repeat-hinting the same file).
    flagged_paths: std.StringHashMap(void),

    pub const Config = struct {
        /// Path to the soteria binary (default: "./soteria").
        binary_path: []const u8 = "./soteria",
        /// Path to thresholds file (default: ".soteria/thresholds.json").
        thresholds_path: []const u8 = ".soteria/thresholds.json",
        /// Path to history DB (default: ".soteria/history.jsonl").
        history_path: []const u8 = ".soteria/history.jsonl",
        /// Whether to run a full scan on session start (seed data).
        scan_on_start: bool = true,
        /// Whether to check hotspot status before every edit.
        check_before_edit: bool = true,
        /// Whether to check regression after mutations.
        check_after_mutation: bool = true,
    };

    pub fn init(allocator: std.mem.Allocator, cfg: Config, io: std.Io) !SoteriaGuard { … }
    pub fn deinit(self: *SoteriaGuard) void { … }

    /// Run soteria scan and parse report. Called on session start and
    /// optionally after mutation passes.
    pub fn runScan(self: *SoteriaGuard, io: std.Io) !void { … }

    /// Load thresholds from JSON (fast, no scan needed).
    pub fn loadThresholds(self: *SoteriaGuard, io: std.Io) !void { … }

    /// Check whether `path` is a hotspot (orange/red zone) in the last
    /// scan. Returns a formatted hint string, or null.
    pub fn checkHotspot(self: *SoteriaGuard, path: []const u8) ?[]const u8 { … }

    /// After editing `path`, check whether indent_mean or other
    /// metrics moved in the wrong direction compared to the baseline.
    pub fn checkRegression(self: *SoteriaGuard, path: []const u8, io: std.Io) ?[]const u8 { … }

    /// Compute a quality delta summary for finish_task.
    pub fn qualityDelta(self: *SoteriaGuard, io: std.Io) ![]const u8 { … }
};
```

### Hooks into GuardrailState

The `afterToolCall` method gets a new arm: if the tool is an edit/write and `check_before_edit` is true, the guardrail calls `checkHotspot` on the target path. If a hotspot hint is returned, it's queued for injection at the next `betweenTurns`.

```zig
// In GuardrailState.afterToolCall:
pub fn afterToolCall(self: *GuardrailState, tool: *const at.AgentTool, call_id: []const u8, result: *const at.ToolResult) void {
    self.stuck_detector.afterToolCall(tool, call_id, result);
    self.compilation_guard.bumpIfMutation(tool.name);

    // NEW: queue hotspot check for edit/write targets
    if (self.soteria_guard.config.check_before_edit) {
        self.soteria_guard.queueHotspotCheck(tool, result) catch {};
    }
}
```

In `betweenTurns`, after the stuck detector check, the guardrail flushes any queued hints:

```zig
// NEW arm in betweenTurns, after stuck detector check:
// ── soteria quality hints ──────────────────────────────
if (try self.soteria_guard.betweenTurns(allocator, io, transcript, out)) {
    ai.log.log(.debug, "guardrails", "soteria_fired", "", .{});
    self.guardrail_fire_count += 1;
    return true;
}
```

## 4. Hint format

### Hotspot warning (pre-edit)

Injected when the model is about to edit a file in orange or red zone:

> **Code quality note:** `src/agent/loop.zig` is a known hotspot
> (hotspot score: 42.3 — 🔴 red zone, complexity trend: rising 🔺).
> Consider whether this change can be done in a smaller,
> lower-complexity file, or whether extracting a helper
> would reduce the complexity of this file.

### Regression warning (post-mutation)

Injected when a file's complexity increased after a batch of edits:

> **Code quality note:** `src/agent/loop.zig` complexity increased
> from 7.2 → 8.1 (indent mean). If this change adds new logic,
> consider extracting it to a separate module.

### Session summary (finish_task)

Injected as part of the finish_task summary:

> **Quality delta:** 3 files improved (complexity ↓), 1 file
> regressed (src/agent/loop.zig — hotspot score +5.1).

## 5. Implementation phases

### Phase 1 — Threshold loading + hotspot check (1-2 sessions)

1. Add `SoteriaGuard` struct to `src/agent/guardrails/`.
2. Wire `loadThresholds` to parse `.soteria/thresholds.json`.
3. Implement `checkHotspot` — reads path from edit/write tool, looks up in cached thresholds (no scan needed — thresholds are already computed).
4. Wire into `GuardrailState.afterToolCall` + `betweenTurns`.
5. Test with manual edits to known hotspot files.

**Deliverable:** model gets a hint when editing a file in orange/red zone.

### Phase 2 — On-demand scan + regression check (2-3 sessions)

1. Implement `runScan` — spawn `./soteria --out /tmp/scan.json` and parse result.
2. Call `runScan` on session start (if `scan_on_start`).
3. Implement `checkRegression` — compare new scan metrics against baseline for a changed file.
4. Wire regression check into `afterToolCall` (queues hint if complexity increased).
5. Add `.soteria/` to `.contextignore` so soteria doesn't scan its own data.

**Deliverable:** model gets regression hints when it makes a hotspot worse.

### Phase 3 — Quality delta in finish_task (1 session)

1. Implement `qualityDelta` — diff last scan vs. session-start scan.
2. Wire into `finish_task` summary (append to the summary string).
3. Cache the session-start scan result in memory for diffing.

**Deliverable:** every finish_task includes a quality summary line.

### Phase 4 — Build-system integration (1 session)

1. Add `zig build soteria` step to `build.zig` that builds the upstream binary from `vendor/soteria-guard/`.
2. Update `.franky-workflow.yaml` to include a `soteria` stage after `test` that runs the quality scan.
3. Document the build dependency in `CONTRIBUTING.md`.

**Deliverable:** CI runs soteria and reports quality regressions.

## 6. Interaction with existing guardrails

| Guardrail | Relationship |
|---|---|
| **CompilationGuard** | Compilation is checked first. If it fails, soteria hints are skipped (compilation is more urgent). |
| **StuckDetector** | Stuck hints fire before soteria hints. A stuck loop suppresses soteria hints to avoid noise. |
| **FinishTaskState** | Soteria quality delta is appended to the finish_task summary string. If the delta is negative, the model sees it before finalising the commit message. |

## 7. Upstream binary — platform compatibility

The existing `soteria` binary in the workspace root is built for arm64. The franky workspace may target different architectures in CI/dev. Two strategies:

| Strategy | Trade-off |
|---|---|
| **A: Rebuild from source** via `zig build` in `vendor/soteria-guard/` | Always matches the target arch, but adds a build step. Requires Zig 0.15.2+. |
| **B: Use any binary with matching ABI** (download release artifact) | Faster, but needs a download step and version pinning. Upstream has no CI releases yet. |

**Recommendation:** Strategy A for development (rebuild on `zig build`), with a plan to switch to Strategy B when upstream publishes release artifacts.

The `SoteriaGuard` config has a `binary_path` field. If the binary is missing, `runScan` returns an error that is logged as a warning and the check is skipped — the guardrail degrades gracefully.

## 8. Data file layout

The existing `.soteria/` files are written by the upstream binary. The guardrail only **reads** them:

- `.soteria/thresholds.json` — JSON object with p60/p85/p95 thresholds for each metric (see `types.Thresholds`).
- `.soteria/history.jsonl` — JSON-lines records from each scan, appended by the binary.
- `.soteria/coupling.json` — JSON array of `CouplingPair` objects (read by the guardrail but not used for hints — exposed for debugging).

The guardrail writes nothing to `.soteria/`. Temporary scan outputs go to the system temp directory.

## 9. Open questions

1. **How often to re-scan during a session?** Every mutation is too expensive (a scan takes ~2 seconds on this codebase). Options: (a) after every N mutations, (b) only on `finish_task`, (c) after each compilation pass. **Tentative:** only on `finish_task` for Phase 2, revisit in Phase 3.

2. **Should the hotspot check use the static thresholds file or run a full scan?** Static thresholds are sufficient for a pre-edit check — they tell us whether the file is in orange/red territory compared to the codebase distribution. A full scan is needed only for regression detection. **Decision:** thresholds file for pre-edit, full scan for post-edit.

3. **Should the guardrail track flagged paths to avoid repeat hints?** Yes — `flagged_paths` in `SoteriaGuard` prevents hinting the same file twice. Reset on `finish_task`.

4. **How to handle new files (no git history)?** Soteria assigns default metrics (0 revisions, 0 authors, etc.) to files without git history. The hotspot score will be low → green zone. Regression check compares the new scan against the session-start scan — a new file's "baseline" is its own initial metrics.

5. **What happens if the soteria binary is absent or wrong arch?** `runScan` returns an error, logged at `warn` level. The guardrail stays silent. No crash, no hint injection.

6. **Should soteria be vendored as a git submodule or fetched at build time?** Given the upstream is early (v0.1.0), a git submodule at `vendor/soteria-guard/` is simplest for now. Migrate to a Zig package dependency when the upstream publishes a package.

7. **What metrics should trigger a regression warning?** The most actionable single metric is `indent_mean` (approximates nesting depth/complexity). Secondary: `hotspot_score`. If either increases beyond a small noise threshold (e.g., +0.5 for indent_mean, +5 for hotspot_score), fire the hint.

8. **Should the coupling data be used for hints?** Not in v1 — coupling is informative but doesn't directly translate to an actionable refactoring hint. Expose in debug mode only.

## 10. Future directions (post-v1)

- **Auto-refactoring:** When soteria flags a function with complexity trend 🔺, the model could be prompted to refactor before adding new code. This is already partially covered by the v2.24 complexity reduction work.
- **CI gate:** A `.franky-workflow.yaml` stage that runs `soteria --out-markdown` and fails the build if any new file hits red zone.
- **Per-author quality trends:** Use the history DB to show whether a model session is improving or degrading the codebase over time (visible in the proxy dashboard).
- **Multi-language support:** soteria's `source_extensions` already covers .zig, .rs, .go, .py etc. The guardrail works unchanged for any language soteria supports.

## Appendix B — Soteria source structure (upstream v0.1.0)

```
src/
├── main.zig          — CLI entry, orchestration
├── types.zig          — FileMetrics, EvolutionMetrics, Signals, Thresholds, Report
├── scan.zig           — Static file scan (LOC, indent, comments)
├── git_log.zig        — Git history extraction
├── churn.zig          — Evolutionary metric computation
├── trend.zig          — Complexity trend over time
├── signals.zig        — Derived signals (hotspot, risk, congestion)
├── thresholds.zig     — Percentile calibration + zone assignment
├── coupling.zig       — Co-change coupling (git log --name-only)
├── report.zig         — JSON / Markdown report writers
├── ignore.zig         — .guardrailignore glob matching
└── complexity.zig     — Indentation-based complexity proxy
```

The guardrail only needs `types.zig` (parsed from JSON output) and `thresholds.zig` (the algorithm — already replicated in the guardrail's load logic). It does not import any upstream Zig code.
