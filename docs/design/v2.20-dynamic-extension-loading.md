# Dynamic extension loading — design document

**Status:** Draft / investigation.

**Related:**
- `src/coding/extensions.zig` — Tier 1 static extensions, Tier 2 placeholder
- `src/coding/extensions_builtin/catalog.zig` — compile-time catalog
- `src/coding/cli.zig` — `--extensions <list>` parsing
- `src/coding/rpc.zig` — JSON-RPC 2.0 + LSP framing (used by `--mode rpc`)
- `docs/spec/v0.md` — franky-golang spec, v0.4.0 roadmap

---

## 1. Motivation

Currently, franky supports only **Tier 1** extensions: compiled into the
binary at build time, opt-in at runtime via `--extensions <name>`. Adding
a new extension requires:

1. Writing the extension module (e.g. `go_dev.zig`)
2. Adding its import + entry to `extensions_builtin/catalog.zig`
3. Adding its import + re-export to `coding/mod.zig`
4. Recompiling the entire franky binary

Goal: allow a user to **drop a pre-compiled extension binary** into a
directory and have franky discover and load it at runtime, with **no
recompilation**. The extension runs as a **separate process** and
communicates over JSON-RPC 2.0 on stdin/stdout (LSP-style framing) —
the same wire protocol franky already uses for `--mode rpc`.

## 2. Current architecture (Tier 1 recap)

```
catalog.zig
  └── builtins: []const Entry = &.{
        .{ .name = "echo", .factory = echo_ext.extension },
      }

  └── pub fn lookup(name) ?Entry  // compile-time, fixed list

interactive.zig (runInteractive):
  └── ext_manager = extensions_mod.Manager.init(allocator)
  └── for each name in --extensions:
        entry = ext_catalog.lookup(name)
        ext_manager.register(entry.factory(), &slash_registry)
              └── runs init_fn(ext, &host)
                    └── host.registerTool(…)
                    └── host.registerPreset(…)
```

The `Extension` struct is function-pointer-based:

```zig
pub const Extension = struct {
    name: []const u8,
    version: []const u8,
    userdata: ?*anyopaque,
    init_fn: ?*const fn (ext: *Extension, host: *Host) ExtError!void,
    deinit_fn: ?*const fn (ext: *Extension) void,
};
```

For Tier 2 (dynamic) extensions this struct is still used, but instead
of direct function pointers, the init/deinit/execute paths go through an
RPC bridge that serialises calls to the extension process.

## 3. RPC process model

### 3.1 Why not shared libraries (`.so`/`.dylib`)

An earlier design considered `dlopen` + C-ABI trampolines (the model used
by HashiCorp go-plugin's predecessor). This was rejected for:

| Concern | Why not dlopen |
|---------|----------------|
| **Crash isolation** | A crash in a `.so` takes down franky. Extensions process untrusted input — process isolation is the right boundary. |
| **ABI coupling** | Even with a C-ABI bridge, struct layout, calling convention, and allocator discipline create subtle coupling between host and extension compilers. |
| **Platform distribution** | `.so`/`.dylib`/`.dll` are platform-specific; extension authors build per-OS. A standalone binary runs anywhere the target platform does. |
| **Debugging** | An extension is hard to test in isolation when it's a `.so`. A binary can be run standalone with JSON on stdin — trivial to pipe test input. |
| **Hot-reload** | Unloading a `.so` is risky (stale function pointers). Killing and respawning a process is safe and well-understood. |

The RPC process model (inspired by HashiCorp go-plugin, Docker CLI
plugins, and the Language Server Protocol) avoids all of these: the
extension is a **standalone binary** that speaks JSON-RPC 2.0 over
stdin/stdout with LSP Content-Length framing — the same protocol franky
already implements in `src/coding/rpc.zig`.

### 3.2 Protocol overview

```
┌─────────────────┐   JSON-RPC 2.0       ┌────────────────────┐
│  franky (host)  │ ◄─── stdin/stdout ──► │  go-dev-ext       │
│                 │   LSP framing         │  (any language)    │
│  Agent loop     │                       │  Tool executor     │
│  Tool dispatch  │                       │  Preset provider   │
│  Event bus      │                       │  Command handler   │
└─────────────────┘                       └────────────────────┘
```

All communication uses `src/coding/rpc.zig` primitives:
- `readFrame(bytes, cursor)` — parse LSP Content-Length framing
- `writeFrame(writer, payload_json)` — write framed JSON
- `parseRequest(allocator, body)` — parse a JSON-RPC 2.0 request
- `encodeResponse(allocator, resp)` — encode a JSON-RPC 2.0 response
- `encodeNotification(allocator, method, params)` — encode a notification

### 3.3 Lifecycle

```
franky startup
  │
  ├─ discover extensions in --ext-dir (or $FRANKY_HOME/extensions/)
  │
  └─ for each extension binary:
       │
       ├─ spawn: std.process.Child.run(&.{ext_path})
       │
       ├─ ──→ ext.handshake         (version negotiation)
       │  ←── {api_version, name, version, capabilities}
       │
       ├─ ──→ ext.init              (extension registers tools/presets/commands)
       │  ←── {ok, tools[], commands[], presets[], subscribe_events[]}
       │
       └─ extension is live (tools merged into AgentTool list)

during session:
  │
  ├─ on tool call:
  │    ──→ tool.execute(name, call_id, args_json)
  │       (optional ←── tool.update notifications for progress)
  │    ←── content[blocks], is_error
  │
  ├─ on cancel:
  │    ──→ tool.cancel(call_id)  (best-effort; process may be killed)
  │
  └─ on event (if subscribed):
       ──→ event.send(kind, json)  (notification, fire-and-forget)

franky shutdown:
  ──→ ext.deinit
  ──► close stdin  (triggers extension EOF → clean exit)
  ──► waitpid with timeout (e.g. 3 s)
  ──► SIGKILL if still alive
```

### 3.4 Protocol detail

#### 3.4.1 Handshake

```
──→  {"jsonrpc":"2.0","id":0,"method":"ext.handshake",
       "params":{"host_api_version":1}}

←──  {"jsonrpc":"2.0","id":0,"result":{
       "api_version":1,
       "name":"go-dev",
       "version":"1.0.0",
       "capabilities":["tool.execute","event.subscribe"]
     }}
```

The host checks `api_version` first. A mismatch rejects the extension
before any registrations happen.

#### 3.4.2 Init (registration)

The extension returns all registrations in the init response — no
reverse-RPC needed for v1.

```
──→  {"jsonrpc":"2.0","id":1,"method":"ext.init","params":{}}

←──  {"jsonrpc":"2.0","id":1,"result":{
       "ok":true,
       "tools":[
         {"name":"go","description":"Run go fmt/vet/build/test",
          "params_json":"{\"type\":\"object\",\"required\":[\"command\",...]}",
          "execute_method":"tool.execute"}
       ],
       "slash_commands":[
         {"name":"gofmt","description":"Format Go files in workspace"}
       ],
       "presets":[
         {"name":"go-dev","description":"Go development sub-agent",
          "default_profile":"ollama-deepseek-pro",
          "default_role":"code",
          "system_prompt":"You are a Go development assistant. ...",
          "tool_names":["go","read","ls","bash"]}
       ],
       "subscribe_events":["turn_start","turn_end"]
     }}
```

Fields:
- `tools[]` — each tool has `name`, `description`, `params_json`
  (JSON Schema), and `execute_method` (always `"tool.execute"` for v1).
- `slash_commands[]` — slash commands the extension handles. Each has
  `name` and `description`. Execution goes through `tool.execute` with
  the slash handler method.
- `presets[]` — sub-agent presets. `tool_names` is a list of tool names
  that the host selects from the parent's agent tool list (same
  mechanism as built-in presets in `subagent.zig`). No custom
  `build_tools` function — all presets filter from the parent's tool
  set.
- `subscribe_events[]` — extension subscribes to named agent events
  (notifications fire as `event.send`).

The host parses this response and populates:
1. The `Manager`'s tool list (merged with Tier 1 tools)
2. The slash registry
3. The preset registry
4. Event subscriptions

#### 3.4.3 Tool execute

```
──→  {"jsonrpc":"2.0","id":2,"method":"tool.execute","params":{
       "name":"go",
       "call_id":"c-1",
       "args_json":"{\"command\":\"fmt\",\"path\":\"/src/main.go\"}"
     }}

  (during execution, optional progress notifications:)

←──  {"jsonrpc":"2.0","method":"tool.update","params":{
       "call_id":"c-1",
       "json":"{\"status\":\"formatting...\",\"file\":\"main.go\"}"
     }}

←──  {"jsonrpc":"2.0","method":"tool.update","params":{
       "call_id":"c-1",
       "json":"{\"status\":\"checking vet...\"}"
     }}

  (final response:)

←──  {"jsonrpc":"2.0","id":2,"result":{
       "content":[
         {"type":"text","text":"formatted 3 files, vet passed"}
       ],
       "is_error":false,
       "tool_code":null
     }}
```

The result shape mirrors `agent/types.zig`'s `ToolResult`:
- `content: [{type, text}]` — content blocks (text-only for v1).
- `is_error: bool` — whether the tool failed.
- `tool_code: string | null` — error sub-code (e.g. `"edit_no_match"`).

Arguments (`args_json`) are a raw JSON string — the extension parses
its own schema.

#### 3.4.4 Cancel

```
──→  {"jsonrpc":"2.0","id":3,"method":"tool.cancel",
       "params":{"call_id":"c-1"}}

←──  {"jsonrpc":"2.0","id":3,"result":{"ok":true}}
      (or {"ok":false,"reason":"unknown call_id"})
```

Cancel is best-effort. If the extension doesn't respond within a
timeout (e.g. 2 s), the host sends SIGKILL and recovers the extension
process (respawns on next tool call or fails closed depending on policy).

#### 3.4.5 Event notification

```
──→  {"jsonrpc":"2.0","method":"event.send","params":{
       "kind":"turn_start",
       "json":"{\"turn_number\":5,\"model\":\"claude-3-5-sonnet\"}"
     }}
```

Notification (no `id`, no response). Only sent for event kinds the
extension listed in `subscribe_events` during init.

#### 3.4.6 Deinit

```
──→  {"jsonrpc":"2.0","id":4,"method":"ext.deinit","params":{}}

←──  {"jsonrpc":"2.0","id":4,"result":{"ok":true}}
```

After receiving the response (or timing out), the host closes stdin and
waits for the process to exit. If it doesn't exit within 3 s, SIGKILL.

### 3.5 JSON-RPC error codes

| Code | Meaning |
|------|---------|
| `-32600` | Invalid request (malformed JSON, missing method, etc.) |
| `-32601` | Method not found (extension doesn't implement it) |
| `-32603` | Internal error (extension-side exception) |
| `-32000` | Extension not initialised (tool call before handshake) |
| `-32001` | Unknown tool name |
| `-32002` | Unknown call_id |
| `-32003` | Tool execution timed out |

### 3.6 Extension binary

An extension is a standalone binary. It reads JSON-RPC 2.0 frames
(LSP Content-Length) from stdin and writes responses to stdout. Stderr
is free for logging (captured by franky and forwarded to the host's
logger).

Minimal extension in any language:

```python
#!/usr/bin/env python3
"""go-dev extension for franky."""
import sys, json

def read_frame():
    """Read one LSP frame from stdin."""
    ...

def write_frame(obj):
    """Write JSON-RPC frame to stdout."""
    ...

state = {"initialised": False}

while True:
    req = read_frame()
    if req is None:
        break
    method = req["method"]
    params = req.get("params", {})

    if method == "ext.handshake":
        write_frame({"jsonrpc":"2.0","id":req["id"],"result":{
            "api_version":1,"name":"go-dev","version":"1.0.0",
            "capabilities":["tool.execute","event.subscribe"]
        }})
    elif method == "ext.init":
        state["initialised"] = True
        write_frame({"jsonrpc":"2.0","id":req["id"],"result":{
            "ok":True,
            "tools":[{"name":"go","description":"Run go tools",
                      "params_json":"{...}","execute_method":"tool.execute"}],
            "presets":[{"name":"go-dev","description":"...",
                        "default_profile":"ollama-deepseek-pro",
                        "default_role":"code",
                        "system_prompt":"You are a Go dev assistant.",
                        "tool_names":["go"]}],
            "subscribe_events":[]
        }})
    elif method == "tool.execute":
        name = params["name"]
        call_id = params["call_id"]
        args = json.loads(params["args_json"])
        # ... run go fmt/vet/build/test ...
        write_frame({"jsonrpc":"2.0","id":req["id"],"result":{
            "content":[{"type":"text","text":"ok"}],
            "is_error":False
        }})
    elif method == "ext.deinit":
        write_frame({"jsonrpc":"2.0","id":req["id"],"result":{"ok":True}})
        break
    else:
        write_frame({"jsonrpc":"2.0","id":req["id"],
                     "error":{"code":-32601,"message":"not implemented"}})
```

Zig, Go, Rust, or any language that can read stdin and write stdout
works identically.

## 4. Loading approaches (deferred)

### 4.1 Wasm/WASI sandboxing (Tier 3)

**How it works:**
1. Extension is compiled to `.wasm` targeting WASI
2. Host embeds a Wasm runtime (e.g. `wasmtime` C API)
3. Host calls an exported `franky_extension_get` Wasm function
4. Host registers trampolines (callbacks that the extension invokes)
5. Wasm linear memory is bridged: host marshals strings in/out

**Pros:**
- **Sandboxed** — extension can't access the filesystem or network
  unless the host explicitly grants capabilities
- **Cross-platform** — same `.wasm` runs on Linux, macOS, Windows
- **Version-independent** — no host/compiler ABI coupling
- **Safe crash isolation** — a crashing extension doesn't take down
  franky

**Cons:**
- **Significantly more complex** — need a Wasm runtime, memory bridge,
  marshaling layer, capability granting
- **Performance overhead** — function calls cross the Wasm boundary,
  strings must be copied in/out of linear memory
- **Limited stdlib access** — extension can't use `std.process.run`,
  filesystem ops, networking without explicit host function imports
- **No existing Wasm runtime in Zig stdlib** — would need to vendor
  or bind to a C library (e.g. `wasmtime.h`, `wasmer.h`)
- **Dependency cost** — adds a large C library dependency to franky

### 4.2 Zig modules loaded via `@import` at runtime

**Not currently possible.** `@import` is evaluated at compile time.
There is no `@runtimeImport` or equivalent in Zig.

## 5. What must change in franky core

### 5.1 `src/coding/extensions.zig` — `Manager` additions

```zig
pub const Manager = struct {
    // ... existing fields ...

    // ── Dynamic loading (Tier 2 — RPC process) ────────────────────
    /// Directory scanned for dynamic extensions at startup.
    ext_dir: ?[]const u8 = null,
    /// Loaded dynamic extension processes.
    loaded: std.ArrayList(LoadedDynamicExt) = .empty,

    pub fn discoverAndLoad(self: *Manager, dir: []const u8) !void;
    pub fn loadDynamic(self: *Manager, path: []const u8) !void;
};
```

`discoverAndLoad`:
1. Scans `ext_dir` for executable files (see §7 for naming)
2. For each, calls `loadDynamic`

`loadDynamic`:
1. Spawns the extension binary as a child process (`std.process.Child`)
2. Sends `ext.handshake` — rejects on API version mismatch
3. Sends `ext.init` — parses the registration response
4. Populates tool/slash/preset registries from the response
5. Registers event subscriptions

```zig
const LoadedDynamicExt = struct {
    /// Child process handle.
    process: std.process.Child,
    /// Unique name returned by the extension.
    name: []const u8,
    /// Version string from handshake.
    version: []const u8,
    /// Capabilities advertised (e.g. "tool.execute", "event.subscribe").
    capabilities: [][]const u8,
    /// Allocator used for the protocol bridge (freed on deinit).
    allocator: std.mem.Allocator,
    /// Pending tool.execute calls keyed by call_id → cancel flag.
    pending_calls: std.StringHashMapUnmanaged(*ai.stream.Cancel),
};
```

### 5.2 `src/coding/ext_rpc.zig` — new module

New file. The RPC bridge between franky and an extension process.

Responsibilities:
- **`RpcExtClient`** struct — wraps `std.process.Child`, manages
  the JSON-RPC message loop (read/write framing on stdin/stdout)
- **`handshake(client)`** — send `ext.handshake`, validate version
- **`init(client)`** — send `ext.init`, parse registrations
- **`executeTool(client, name, call_id, args_json, cancel, on_update)`** —
  send `tool.execute`, stream `tool.update` notifications, return result
- **`cancelCall(client, call_id)`** — send `tool.cancel`
- **`sendEvent(client, kind, json)`** — send `event.send` notification
- **`deinit(client)`** — send `ext.deinit`, close stdin, wait

The client needs a dedicated reader thread (or async read loop) that
receives messages from the extension's stdout and dispatches them:
responses to waiting callers, `tool.update` notifications to the
progress callback, unexpected messages to a log.

### 5.3 `extensions.zig` — wrap RPC tools into `AgentTool`

When `loadDynamic` registers tools from the init response, it creates
`AgentTool` entries whose `execute` function is a trampoline:

```zig
fn rpcExecuteTrampoline(
    tool: *const at.AgentTool,
    allocator: std.mem.Allocator,
    io: std.Io,
    call_id: []const u8,
    args_json: []const u8,
    cancel: *ai.stream.Cancel,
    on_update: at.OnUpdate,
) anyerror!at.ToolResult {
    const client: *RpcExtClient = @ptrCast(@alignCast(tool.ctx.?));
    return try client.executeTool(tool.name, call_id, args_json, cancel, on_update);
}
```

### 5.4 `Host` struct — no opaque handles needed

Since registration data flows in one direction (extension → host via
the `ext.init` response), there is no reverse-RPC and no opaque handle.
The `Host` struct remains Zig-only for Tier 1 extensions; dynamic
extensions interact purely through the protocol.

### 5.5 Mode drivers — wire dynamic loading

Currently extension loading only runs in `interactive.zig`. All four
modes need the extension loading path:

| Mode | Current state | Change |
|------|---------------|--------|
| `interactive.zig` | Loads Tier 1 from catalog | Add `discoverAndLoad` after catalog |
| `print.zig` | No extension loading | Add catalog + dynamic loading |
| `rpc.zig` | No extension loading | Add catalog + dynamic loading |
| `proxy.zig` | No extension loading | Add catalog + dynamic loading |

### 5.6 `cli.zig` — add `--ext-dir` flag

```zig
/// `--ext-dir <path>` — directory to scan for dynamic extensions.
ext_dir: ?[]const u8 = null,
```

### 5.7 `build.zig` — no C header needed

The earlier dlopen design required shipping a C header
(`include/franky/ext.h`). The RPC process model needs **no build system
changes** — the protocol is self-describing JSON. An optional SDK can
be published separately as a library (e.g. `franky-ext` Python/Rust/Go
package) but is not part of the franky build.

### 5.8 `src/coding/mod.zig` — add `ext_rpc` re-export

```zig
pub const ext_rpc = @import("ext_rpc.zig");
```

## 6. Protocol versioning

The protocol carries an integer API version (`host_api_version` in
handshake, `api_version` in the response). Increment it on any
breaking change:

| Version | Changes |
|---------|---------|
| 1 | Initial — `ext.handshake`, `ext.init`, `tool.execute`, `tool.cancel`, `event.send`, `ext.deinit` |

Backward-compatible additions (new optional fields, new methods) do
not bump the major version. The extension ignores unknown methods;
the host ignores unknown fields.

## 7. Search path & discovery

### File naming

Extension binaries must match the platform's executable naming
convention:

- `*.franky_ext` — Linux, macOS, BSD
- `*.franky_ext.exe` — Windows (deferred)

The host checks executable permission (`std.posix.access` with `X_OK`)
on Unix before attempting to spawn.

### Default search path

```
$FRANKY_HOME/extensions/              (if FRANKY_HOME is set)
~/.franky/extensions/                 (default fallback)
```

Franky scans these at startup (before the mode driver enters
interactive/print/rpc) for files matching the platform pattern.

Overridable via `--ext-dir <path>`.

### Security: scope restriction

- Only `--ext-dir` or `$FRANKY_HOME/extensions/` are scanned.
- `$CWD` and `$PATH` are **never** searched (prevents `franky_ext` in
  a compromised directory from being auto-loaded).
- The host validates that the file is an executable regular file
  (not a symlink to / arbitrary binary in a different location).
- **No `sudo` / setuid inheritance** — the extension process inherits
  the host's uid/gid and capabilities.

## 8. Unloading & lifecycle

When the extension manager deinits:

```zig
pub fn deinit(self: *Manager) void {
    // 1. Call deinit_fn for each Tier 1 extension (same as today)
    for (self.extensions.items) |*ext| if (ext.deinit_fn) |fnptr| fnptr(ext);

    // 2. For RPC process extensions: graceful shutdown
    for (self.loaded.items) |le| {
        le.client.sendDeinit() catch {};
        le.client.closeStdin();
        // Wait with timeout; kill if stuck
        const pid = le.process.id;
        _ = std.posix.waitpid(pid, 0) catch {
            std.posix.kill(pid, std.posix.SIG.KILL) catch {};
            _ = std.posix.waitpid(pid, 0) catch {};
        };
    }
}
```

Key design decisions:
- **No hot-reload in v1** — unloading then re-loading is possible but
  not a first-class feature (tool references are cached in the agent
  loop's tool list, which assumes stable function pointers).
- **Extensions loaded at startup, not mid-session** — all discovery
  happens before the agent loop starts. The tool list remains
  immutable during a session.
- **Crash recovery** — if an extension process dies during a tool call,
  the host returns a `ToolResult` with `is_error=true` and
  `tool_code="ext_crashed"`. The extension is **not respawned**
  mid-session (the tool list is immutable). A warning is logged.
  Future versions may respawn on the next call.
- **Hanging extension** — tool execution has a timeout (configurable
  per extension, default 120 s). If the extension doesn't respond
  within the timeout, the host sends `tool.cancel` (best-effort) and
  then kills the process.

## 9. Security considerations

| Concern | Mitigation |
|---------|------------|
| Malicious extension | Same as Tier 1 — the extension has the same capabilities as the host user (filesystem, network, subprocess). Dynamic loading adds no new privilege. |
| Process isolation | Extension runs in a **separate process** with its own address space. A crash or memory corruption does not affect franky. |
| Untrusted input | Extension may receive user-controlled `args_json`. Extension is responsible for validating its own inputs (same as any tool). |
| Extension impersonation | Scan only `--ext-dir` or `$FRANKY_HOME/extensions/`, never `$CWD` or `$PATH`. Validate executable permissions. |
| IPC snooping | stdin/stdout are pipe fd's inherited by the child; no other process can inject messages (on Unix). |
| Fork bomb | Host enforces a configurable limit on concurrent extension processes (default 8). |
| Zombie processes | Host always `waitpid`s the child on deinit, with a kill-timeout as backstop. |

## 10. Implementation phases

### Phase 1 — RPC process loading (Tier 2)

Deliverable: a user can run `franky --ext-dir ~/.franky/extensions/` and
franky loads any executable `*.franky_ext` files found.

| File in franky core | Change |
|---------------------|--------|
| `src/coding/ext_rpc.zig` | **New** — `RpcExtClient`, protocol methods, message loop |
| `src/coding/extensions.zig` | Add `Manager.loadDynamic`, `discoverAndLoad`, `LoadedDynamicExt` |
| `src/coding/modes/interactive.zig` | Wire `--ext-dir` after static catalog |
| `src/coding/modes/print.zig` | Wire extension catalog + dynamic loading |
| `src/coding/modes/rpc.zig` | Wire extension catalog + dynamic loading |
| `src/coding/modes/proxy.zig` | Wire extension catalog + dynamic loading |
| `src/coding/cli.zig` | Add `ext_dir` field + `--ext-dir` flag parsing |
| `src/coding/mod.zig` | Add `ext_rpc` re-export |

**Estimated effort:** ~3-4 days (moderate). Core loading logic is
straightforward with `std.process.Child` and the existing `coding/rpc.zig`
primitives. The RPC message loop (the reader thread that dispatches
notifications while waiting for responses) is the trickiest part.

### Phase 2 — Wasm/WASI (Tier 3)

Deliverable: franky can load `.wasm` extensions alongside native
extensions. Requires a Wasm runtime dependency.

| File | Change |
|------|--------|
| `src/coding/ext_wasm.zig` | **New** — Wasm runtime bridge |
| `build.zig` | Add Wasm runtime dependency (e.g. wasmtime C API) |
| `src/coding/extensions.zig` | Add `.wasm` discovery path |

**Estimated effort:** ~2-3 weeks (substantial). Requires evaluating and
vendoring a Wasm runtime, implementing the memory bridge, testing across
platforms.

### Phase 3 — Extension SDK / tooling

Deliverable: a CLI scaffold/build/package tool for extension authors.

```
franky-ext init go-dev          # scaffold extension
franky-ext build go-dev         # compile to binary
franky-ext package go-dev       # produce .tar.gz with manifest
franky-ext install pkg.tar.gz   # install into $FRANKY_HOME/extensions/
```

An extension SDK library (Python package, Rust crate, Go module, Zig
package) that handles JSON-RPC framing and the protocol methods so
extension authors only write the business logic.

## 11. Interaction with franky-golang

For the franky-golang project specifically:

**Option A — Move into franky core's catalog (simplest)**

Register `go-dev` in `extensions_builtin/catalog.zig`. Users activate
with `--extensions go-dev` — no separate binary needed, but requires the
extension to be baked into the binary. This is practical for a reference
extension that ships with franky.

**Option B — Publish as a dynamic extension (post-Phase 1)**

The `go-dev` extension is compiled to a standalone binary and
distributed. Users drop it into `~/.franky/extensions/`. No recompilation
needed. Works in any language (Go, Zig, Python, Rust, etc.).

**Option C — Both**

Ship `go-dev` as a built-in (Option A) AND publish it as a downloadable
binary for users who build franky from source without the go-dev catalog
entry.

## 12. Open questions (deferred to implementation)

1. **RPC message loop architecture** — synchronous reader thread +
   async callback dispatch, or a single-threaded select loop? The
   answer depends on whether extensions are expected to send
   notifications concurrently with responses. For v1, the simpliest
   approach is: a dedicated reader thread that receives messages,
   dispatches `tool.update` to the progress callback, and signals
   the response future for tool results.

2. **Extension stdout buffering** — the child process may buffer stdout
   (especially in C/Rust if stdout is not set to unbuffered). The
   extension SDK must explicitly set stdout to unbuffered or flush after
   every frame. Document this requirement in the protocol spec.

3. **Concurrent tool calls to the same extension** — can franky call
   `tool.execute` on the same extension process while a previous call
   is still running? For v1, no — one call at a time per extension
   process. The reader thread needs to associate notifications with
   the correct call_id even if concurrent calls are added later.

4. **Event forwarding volume** — `event.send` notifications fire on
   every turn. For extensions that subscribe to `message_update`
   (every streaming token), the volume could be high. V1 limits
   subscriptions to structural events only (`turn_start`, `turn_end`,
   `tool_execution_start`, `tool_execution_end`, `agent_error`).
   Token-level streaming is deferred.

5. **Extension logging** — stderr from the extension process should be
   captured and forwarded to franky's logger at debug level, tagged
   with the extension name. Implemented via `std.process.Child.stderr`
   pipe.

6. **Windows support** — deferred. `std.process.Child` works on Windows,
   but stdin/stdout pipe semantics and process signalling (SIGKILL)
   differ. Document Windows as a follow-up.

## 13. Recommendation

**Start with Phase 1 (RPC process loading).** The protocol is simple
(JSON-RPC 2.0 with LSP framing — already implemented). The process
model gives crash isolation, language freedom, and easy debugging for
extension authors. The per-call latency (IPC + JSON serialisation) is
negligible compared to LLM inference time.

The existing `src/coding/rpc.zig` provides the framing, parsing, and
encoding primitives out of the box — this design reuses them directly.

**Wasm (Phase 2) is the right long-term approach** for sandboxed
extensions, but should wait until franky has a proven extension
ecosystem and user demand justifies the runtime dependency.

For now, continue with **v0.4.0 catalog integration (Option A)** as the
immediate next step — it ships `--extensions go-dev` without dynamic
loading, and the RPC process design can mature in parallel.
