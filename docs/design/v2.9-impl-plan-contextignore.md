# v2.9 — `.contextignore` — Implementation Plan

**Source design:** [docs/design/2.9-documentation-speification-archive.md](./2.9-documentation-speification-archive.md)

**Decisions baked in (from inline answers + follow-up):**

- **Q1** — `read` enforces `.contextignore`. **Yes.**
- **Q2** — Per-tool gate at the matcher (Option A). **Unconditional** — no per-call escape hatch. The model cannot pass a flag to bypass it.
- **Q3** — Soft-archive: file stays in place, path added to `.contextignore`. Frozen specs (`v0.md`, future `v1.md`) explicitly listed.
- **Q4** — Filename: **`.contextignore`**.
- **Q5** — Status front-matter on design RFCs (`Status: draft|shipped|abandoned`). **Auto-mover skipped per user edit** — convention only, manual file moves.

**Affects:**
`src/coding/gitignore.zig`, `src/coding/tools/{ls,find,grep,read,write,edit}.zig`, repo-root `.contextignore`, every existing file in `docs/design/`.

---

## 1. Architecture

The matcher in `coding/gitignore.zig` is filename-agnostic except for two hard-coded `.gitignore` literals (line 150 and 171 of `loadFromTree`). The cleanest path is to **parameterise the filename** and let the same code load both kinds of stack. Tools that already accept `respectGitignore: bool` keep that flag for the gitignore stack only. The contextignore stack loads **unconditionally** alongside it — there is no schema flag, no tool argument, no way for the model to disable it.

No new module. No duplication of the matcher. The `gitignore.zig` filename stays — the existing API stays — and a thin convenience function loads `.contextignore` patterns through the same machinery.

```
coding/gitignore.zig
  ├── loadFromTree(allocator, io, scan_root)                 (existing — defaults to ".gitignore")
  ├── loadFromTreeNamed(allocator, io, scan_root, filename)  (new — generalised)
  └── Stack.match(...)                                       (existing, filename-agnostic)

tools/{ls,find,grep,read,write,edit}.zig
  ├── load gitignore stack    if respectGitignore=true   (existing flag, model-controllable)
  ├── load contextignore stack ALWAYS                    (no flag, no override)
  └── path is excluded if either stack matches
```

## 2. Phases

### Phase 1 — Matcher infrastructure (~80 LOC, additive)

**Files:** `src/coding/gitignore.zig`

Steps:

1. Extract the filename literal from `loadFromTree` into a parameter. Refactor:
   ```zig
   pub fn loadFromTree(allocator, io, scan_root) !Stack {
       return loadFromTreeNamed(allocator, io, scan_root, ".gitignore");
   }
   pub fn loadFromTreeNamed(allocator, io, scan_root, filename) !Stack { ... }
   ```
   `loadOne` already takes a `base_dir` argument — extend it to also take the filename.
2. Existing tests for `loadFromTree` still pass unchanged (the wrapper preserves the public API).
3. **New tests** in the same file:
   - `loadFromTreeNamed` with `.contextignore` produces the same matcher behaviour as with `.gitignore`.
   - Both files can coexist: a tree with one of each yields independent stacks.
   - Empty `.contextignore` (file exists, zero rules) returns an empty stack — no error.

**Acceptance:** `zig build test` passes; the new helper is callable from any tool module.

### Phase 2 — Tool plumbing — tree walkers (~30 LOC, repeated 3×)

**Files:** `src/coding/tools/ls.zig`, `find.zig`, `grep.zig`

Each tool already follows this shape:
```zig
const respect_gitignore: bool = root.object.get("respectGitignore") ... default true;
... if (respect_gitignore) load gitignore stack ...
```

Add unconditional contextignore loading next to it (no flag, no schema change):
```zig
// Always load — no model override.
const ctx_stack = try gitignore.loadFromTreeNamed(allocator, io, root, ".contextignore");
defer ctx_stack.deinit();
```

Path-exclusion check inside the walk becomes: `gitignored OR contextignored → skip`. The `gitignored` half still honours `respectGitignore`; the `contextignored` half is always on.

**No schema changes.** `parameters_json` for these tools is unchanged — there is no new field for the model to discover or set.

**Tests** (one per tool):
- `tool: contextignored entries are hidden` — fixture with `.contextignore` listing `private/**`; assert `private/foo.txt` is not in the output.
- `tool: contextignore is independent of respectGitignore` — same fixture, run with `respectGitignore: false`; assert contextignored entries are still hidden.
- `tool: hard-skips .git regardless of either stack` (existing — should still pass).

**Acceptance:** With a fixture containing `docs/archive/old.md` and a repo-root `.contextignore` listing `docs/archive/**`, `ls -R`, `find pattern=*`, and `grep` all hide `old.md` regardless of any tool argument the model passes.

### Phase 3 — Tool plumbing — single-path tools (~25 LOC, repeated 3×)

**Files:** `src/coding/tools/read.zig`, `write.zig`, `edit.zig`

`read`/`write`/`edit` operate on a single path, not a tree walk. The integration is simpler:

1. **No schema change.** No flag.
2. Before doing IO, load the contextignore stack rooted at the workspace root and call `Stack.match(resolved_path)`.
3. If matched → return `tool_error{contextignored, "path is in .contextignore — archived/historical content not available to the model"}`. **Hard refusal**, no override.

This is also the place where path-resolution-vs-symlink subtlety lives — match against the workspace-relative path produced by `path_safety.zig`, not the user's input. Symlinks resolved into a contextignored target also refuse.

**Tests:**
- `read: refuses contextignored path with structured error`
- `write: refuses to create at contextignored path`
- `edit: refuses to modify contextignored file`
- `read: refuses contextignored path even when reached via symlink that lands inside an ignored tree`

**Acceptance:** Every call to `read`/`write`/`edit` against a contextignored path returns `tool_error{contextignored}`. There is no argument the model can pass to make the call succeed.

### Phase 4 — Seed the repo `.contextignore` (~5 lines)

## 3. Known limitation: `bash` is the one channel that can still reach archived paths

`bash` runs arbitrary shell commands. The harness can't intercept what the shell reads — `cat docs/spec/v0.md`, `awk … docs/spec/v0.md`, or `cd docs && cat spec/v0.md` all work regardless of any matcher we add to the Zig tool layer.

Because the rest of `.contextignore` is unconditional, this is the **single** loophole. Two ways to harden it (both v2 work):

1. **Path-prefix sniff before exec.** Tokenise the command, check each token against the contextignore stack. Refuses obvious cases (`cat docs/spec/v0.md`) but doesn't catch indirection (`cat $(echo docs/spec/v0.md)`, `cd docs/spec && cat v0.md`).
2. **Sandbox the bash exec under a chroot/namespace** that hides contextignored paths from the underlying filesystem. Only viable under `zerobox` / Docker / similar.

**v1 decision:** Accept as a known limitation. Document in `bash.zig`'s doc comment and in this plan. Most realistic threat model: an honest model that genuinely doesn't know archived paths exist won't try to `bash`-read them. A model that has seen them in prior context and chooses to bypass via shell is operating beyond what a content-addressed gate is meant to defend.

## 6. Configuration

| Layer | Mechanism |
|---|---|
| Per-call | **None.** No tool argument can disable enforcement. |
| Per-session | **None for v1.** A future `--no-contextignore` CLI flag for human inspection / debugging is possible but not in scope (the model still wouldn't see it). |
| Repo-level | The `.contextignore` file itself + per-directory `.contextignore` files anywhere in the tree (mirrors `.gitignore`'s nested-rules semantics) |

The **only** way to add or remove paths from the matcher is to edit the `.contextignore` files in the repo. That edit is itself a deliberate human action, captured in git history.

## 7. Out of scope (explicit non-goals)

- **`bash` enforcement** — see §3. Trust boundary, not security boundary.
- **Auto-move on `Status: shipped`** — explicitly removed per user's edit.
- **System-prompt mention of `.contextignore`** — rejected by Q2=A; the gate is structural, the model doesn't need a prompt nudge.
- **`web_search` / `web_fetch`** — they don't touch the local filesystem.
- **Subagent inheritance** — sub-agents already inherit the parent's tool surface; the flag plumbs through automatically.

## 8. Acceptance for the whole feature

1. `zig build test` passes (all new tests + every existing test).
2. With seed `.contextignore` in place, the default tool walks (`ls`, `find`, `grep`) and path-targeted tools (`read`, `write`, `edit`) hide `docs/archive/**` and the frozen specs.
3. Each of those tools accepts `respectContextignore: false` and surfaces / acts on the path when passed.
4. Every design RFC carries a `Status:` line.
5. The 2.9 design doc + this plan move to `docs/design/decided/` in the same PR that ships the feature.
