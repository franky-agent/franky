## 1. Problem

`stop_reason` is the model's own decision about why it stopped generating:

- `tool_use` ‚Äî "I want a tool result, resume me"
- `end_turn` ‚Äî "I'm done, yielding to the user"
- `max_tokens` / `stop_sequence` ‚Äî externally bounded

`end_turn` is **not a tool the harness invokes**. It's the model's choice to compose a final message instead of emitting another tool call. So *"100 turns, zero `end_turn`s"* means the model never once concluded "I should stop and report back" ‚Äî every sample, the next-most-likely token sequence was another tool call.

### Why the model never yields

Five reinforcing pressures, all observable in the wild:

1. **No completion predicate.** "Debug the Go vet test" has no observable "done" state until the test passes. The model can always justify one more reproducer.
2. **Sunk-cost / persistence prior.** Post-training rewards finishing tasks; abandoning before success is penalised. After 50 failed attempts, "give up and report" looks worse than "try once more" even when it shouldn't.
3. **No repetition awareness.** The model doesn't notice it's hit the same shape of API error 5 times. Each error is processed as a fresh local problem.
4. **`end_turn` competes with tool calls in the same sampling step.** If "next try a different API" has higher logit than "summarise and stop," it just keeps going. There is no separate "should I stop?" decision.
5. **No interlocutor.** In a proxy session there is no user message between turns, so the model never gets the natural "does the human want me to keep going?" signal.

Smaller models (e.g. `deepseek-v4-flash`) are more prone to this ‚Äî weaker meta-cognition, less robust "I'm stuck" recognition.

## 2. What already exists in the harness

Before adding new mechanisms, document what's already wired so the new design doesn't duplicate it.

### 2.1 Hard turn cap (`max_turns`)

`Config.max_turns` (default **50**) caps the loop. When exhausted:

- `on_max_turns` hook can `.extend(N)` (additive) or `.stop`.
- Without a hook, or after `.stop`, `agent_error{max_turns_exceeded}` fires and the loop closes.

Lives in `agent/loop.zig:257`.

### 2.2 Forced summarisation (`max_turns_summarize`)

`Config.max_turns_summarize: bool = false` (loop.zig:272) ‚Äî when true, after the cap is exhausted **and** any `on_max_turns` extensions are spent, the loop runs **one final tool-disabled turn** with a synthetic user message:

> *"Tool budget exhausted (N turns). No more tools ‚Äî produce a final report in plain text: what you tried, what's still unknown, what you'd do next. Do not call any tools."*

Implemented at `loop.zig:runForcedSummary` (line 428). This is exactly the *"Hard budget with forced summarisation"* intervention from the problem statement ‚Äî it's already there, just **off by default**.

### 2.3 Repeat-error escalation (v2.11)

`v2.11-model-stuck-detector.md` proposes a stuck detector that fires after N consecutive same-error tool calls and either injects a hint or triggers a hard stop. That subsumes the *"Repetition-triggered escalation"* intervention from the problem statement.

**Status:** designed in v2.11, not yet implemented. v2.12 should not redesign it ‚Äî it should defer to v2.11.

## 3. The new mechanism: periodic checkpoint injection

The two existing levers (hard cap + repeat-error escalation) trigger only at **boundaries**: end of budget, or after a specific failure pattern. They don't help in the long middle of a run where the model is steadily progressing-but-not-converging.

The genuinely new intervention is a **periodic, time-based meta-reflection prompt** ‚Äî fired every N turns regardless of error state ‚Äî that forces the model to enumerate its current hypothesis space. This forces a meta-step into the context that the model wouldn't generate on its own.

### 3.1 Trigger

Every `checkpoint_interval` turns **without an `end_turn`** since the last checkpoint (or since session start), inject a synthetic user message *before* the next LLM call.

Counter resets:

- when the model emits `end_turn` (it yielded ‚Äî no need to nudge);
- when a checkpoint fires (so the next checkpoint is N turns later, not N turns after session start).

This means:

- **Cooperative model** that yields naturally ‚Üí never sees a checkpoint.
- **Stuck model** that tool-loops ‚Üí sees a checkpoint every N turns.

### 3.2 The checkpoint message

Proposed text (rendered as a `user`-role message in the transcript):

> *"You've made N tool calls without yielding. Before the next action, enumerate in plain text:*
>
> *  1. Hypotheses you've ruled out (and why).*
> *  2. Your current hypothesis and the evidence for it.*
> *  3. One alternative hypothesis worth trying.*
>
> *If the same hypothesis has failed twice already, switch approach or stop and report. You may call tools after this ‚Äî but only if your reasoning above warrants it."*

This is an information injection, not a tool gate ‚Äî the model can still call tools after the checkpoint. The hypothesis is that the meta-step itself shifts the next-token distribution toward `end_turn` for stuck states without blocking productive ones.

### 3.3 Where it fires

The natural integration point is the existing `between_turns` hook (loop.zig:243), same as v2.10's compilation guard and v2.11's stuck detector. Specifically:

```
Agent (src/agent/agent.zig)
  ‚îî‚îÄ‚îÄ checkpoint_state: u32 turns_since_last_yield

agentLoop (src/agent/loop.zig)
  ‚îú‚îÄ‚îÄ after a turn ends:
  ‚îÇ     if turn produced end_turn: turns_since_last_yield = 0
  ‚îÇ     else: turns_since_last_yield += 1
  ‚îÇ
  ‚îî‚îÄ‚îÄ between_turns hook:
        if turns_since_last_yield >= checkpoint_interval and config.checkpoint_enabled:
          append synthetic user message (the checkpoint prompt)
          turns_since_last_yield = 0
```

### 3.4 Configuration

| Setting | Default | Description |
|---|---|---|
| `checkpoint_enabled` | `false` | Master toggle. Off by default to avoid altering existing-session behaviour. |
| `checkpoint_interval` | `10` | Turns without `end_turn` before injecting the meta-prompt. |
| `checkpoint_prompt` | (built-in) | Override the prompt text per-session/project. |

Plumbing in lockstep with `max_turns` (CLI flag, `Settings.json`, profile field, `Agent.Config` SDK field).

## 5. Observability ‚Äî measure `end_turn` rate per session

The whole problem only matters if we can see it. Add three counters to the per-session metrics already maintained for the web UI status line:

| Counter | Definition |
|---|---|
| `turns_total` | already tracked |
| `turns_with_end_turn` | increment when `stop_reason == end_turn` |
| `checkpoints_fired` | increment when a checkpoint is injected (¬ß3) |

Surface as `end_turn_rate = turns_with_end_turn / turns_total` in the existing usage/status stream. Below ~5% on long sessions is a strong signal the model is in a non-yielding loop.

These counters also make experimentation possible: if we change `checkpoint_interval` or the prompt text, we can A/B against the rate directly.

## 6. What's explicitly *not* in scope

- **Repeat-error stuck detection** ‚Äî covered in [v2.11](./v2.11-model-stuck-detector.md). v2.12 does not redesign it; the two interventions compose (a stuck pattern triggers v2.11; a long non-yielding run triggers v2.12).
- **Compilation / secret guardrails** ‚Äî covered in [v2.10](./v2.10-harness-enforced-guardrails.md). Orthogonal ‚Äî v2.10 tries to keep the model from doing damage; v2.12 tries to keep it from running forever.
- **Auto-commit & `finish_task` tool** ‚Äî also v2.10. A `finish_task` tool *terminates* the loop, which is a strong yield signal but only when the model chooses to call it. v2.12 addresses the case where the model does **not** call it.
- **Probability-of-yielding nudges in the system prompt.** Tempting ("Always emit `end_turn` after a successful test pass"), but system-prompt edits collide with role-prompt and skill-prompt machinery. Defer until we have data showing checkpoints alone aren't enough.
- **Per-model heuristics** (e.g. tighter intervals for `deepseek-v4-flash`). Single global default first; tune by profile in v2 if needed.

## 7. Open questions

### Q7.1 ‚Äî Default `checkpoint_interval`

10 turns is a guess. Anthropic Claude routinely yields well under 10; small models routinely don't. Setting it too low pesters good models; too high lets bad ones drift.

**Proposed:** 10 for v1. Re-tune once we have `end_turn_rate` data from the wild.

**Decision:** _____

### Q7.2 ‚Äî Should the checkpoint prompt mention "stop"?

The proposed text says *"‚Ä¶switch approach or stop and report."* That's an explicit nudge toward `end_turn`. Risk: a still-productive model misreads it as "the user wants me to stop now."

**Proposed:** Keep the "stop and report" clause but conditionally ‚Äî only when the same hypothesis has failed twice. The checkpoint is a hypothesis-enumeration prompt first; the stop nudge is a fallback the model should weigh against its own evidence.

**Decision:** _____

### Q7.3 ‚Äî Should the checkpoint be a `user` message or a `system` message?

`user`-role injections are visible in the transcript and survive replay; `system`-role nudges are stronger but cleaner-looking.

**Proposed:** `user` role. Reasons: (a) a `system` mid-conversation breaks the API contract for several providers; (b) the model already responds well to user-role nudges (we use one for `runForcedSummary`); (c) the transcript stays a faithful replay record without hidden context.

**Decision:** _____

### Q7.4 ‚Äî Counter reset on partial yield

If a turn produces `stop_reason == max_tokens` (the model ran out of output but didn't conclude), should `turns_since_last_yield` reset?

**Proposed:** No ‚Äî only `end_turn` resets the counter. `max_tokens` is an external bound, not a deliberate yield. `stop_sequence` follows the same rule.

**Decision:** _____

### Q7.5 ‚Äî Interaction with `max_turns_summarize`

If both are enabled and the loop hits `max_turns`, the forced summary runs (¬ß2.2). Should pending checkpoints fire first, or get skipped?

**Proposed:** Skip pending checkpoints once `max_turns` is exhausted ‚Äî the forced summary is a stronger signal and would conflict with the checkpoint's "you may call tools after this" framing.

**Decision:** _____

### Q7.6 ‚Äî Visibility in the web UI

Should checkpoint injections render as a special card type, or as a regular `user` message? Currently injected user messages render identically to user-typed ones, which could mislead the human.

**Proposed:** Add a `synthetic: true` marker on injected user messages (transcript-level field, ignored by the model). The web UI renders them with a distinct style ‚Äî e.g. a small "üõü harness checkpoint" badge ‚Äî so the human knows the harness inserted it. Same machinery would dress up `runForcedSummary`'s injection retroactively.

## 9. Edge cases & non-goals

| Case | Handling |
|---|---|
| Model emits text + tool call in the same turn | Counts as non-yield; only `stop_reason == end_turn` resets. |
| Sub-agent loop | Sub-agents have their own `max_turns`; checkpoint injection should be **per-loop**, not shared. |
| Compaction mid-run | Compaction summarises old turns; the synthetic checkpoint message survives intact (it's a regular transcript message). |
| User explicitly disables tools mid-session | Without tools, every turn yields by definition ‚Äî checkpoint never fires. |
| Cancellation between checkpoint inject and next LLM call | The injected message persists in the transcript; on resume, the model sees it as the most recent turn, which is the desired effect. |
| Token-budget cap (vs. turn-budget cap) | Out of scope for v1. `max_turns` is the primary lever; a token cap is a separate design. |
