# v2.9 — Documentation / specification archive

**Status:** shipped — 2026-04

## Problem

Specs and design docs accumulate over time. Old versions, superseded
RFCs, and post-hoc brainstorms keep the project's history but
shouldn't pollute the model's default context — every grep / ls /
find walk that surfaces them costs tokens and invites confusion
between "current" and "historical" guidance.

`.gitignore` doesn't help here: archived docs are still in git, just
not part of the model's working set. We need a parallel mechanism
that hides them from default tool walks while keeping them readable
on deliberate request.

## Scope: what counts as "archive"?

| Category | Examples | Archive? |
|---|---|---|
| **Frozen-but-cross-referenced spec** | `v0.md`, future `v1.md` after v2.x ships | **No** — current specs link into them; grep-driven research depends on them being visible. |
| **Shipped design RFCs** | `docs/design/v2.8-edit-tool-diff-view.md`, `v2.10-harness-enforced-guardrails.md` once they land | **Yes** — the implementation is the source of truth; the RFC is history. |
| **RFCs that didn't ship** | abandoned proposals, superseded brainstorms | **Yes** |
| **Active design RFCs** | the one you're reading now | **No** — under discussion. |

Loose rule: an RFC graduates to archive when (a) it shipped and the
spec/code reflects it, OR (b) the discussion concluded with "won't
do." Until then, it stays visible.

## Final decision

**`respectedFiles`**: `.contextignore` files, parsed through the existing
`gitignore.zig` matcher (parameterised to load either `.gitignore` or
`.contextignore` stacks).

**Enforcement**: **Unconditional.** No tool argument, no flag, no escape
hatch. Every filesystem-touching tool (`ls`, `find`, `grep`, `read`,
`write`, `edit`) hard-refuses paths matched by the contextignore stack.
The gitignore stack remains model-controllable (`respectGitignore` flag);
the contextignore stack loads alongside it and is always on.

**Naming**: `.contextignore`

**Migration trigger**: `Status:` front-matter line on every design RFC
(`draft|shipped|abandoned`). No auto-mover — convention only, manual
file moves.

**Frozen specs**: Soft-archive — file stays in place, path explicitly
listed in `.contextignore`. (Q3: answer C.)

**`bash` loophole**: Accepted as a v1 known limitation — `bash` can
still read archived paths via shell commands; the harness can't
intercept arbitrary shell I/O.

## Architecture

The matcher in `coding/gitignore.zig` is parameterised so the same code
loads both `.gitignore` and `.contextignore` stacks:

```
coding/gitignore.zig
  ├── loadFromTree(allocator, io, scan_root)                 (existing — defaults to ".gitignore")
  ├── loadFromTreeNamed(allocator, io, scan_root, filename)  (new — generalised)
  └── Stack.match(...)                                       (existing, filename-agnostic)

tools/{ls,find,grep,read,write,edit}.zig
  ├── load gitignore stack    if respectGitignore=true   (existing flag, model-controllable)
  ├── load contextignore stack ALWAYS                    (no flag, no override)
  └── path is excluded if either stack matches
```

No new module. No duplication of the matcher. No schema changes to any
tool — there is no new field for the model to discover or set.

## Phases

### Phase 1 — Matcher infrastructure (~80 LOC)

**Files:** `src/coding/gitignore.zig`

Extract the filename literal from `loadFromTree` into a parameter:
```zig
pub fn loadFromTree(allocator, io, scan_root) !Stack {
    return loadFromTreeNamed(allocator, io, scan_root, ".gitignore");
}
pub fn loadFromTreeNamed(allocator, io, scan_root, filename) !Stack { ... }
```
New tests: coexistence of both stacks, empty `.contextignore`.

### Phase 2 — Tool plumbing — tree walkers (~30 LOC × 3)

**Files:** `src/coding/tools/{ls,find,grep}.zig`

Add unconditional contextignore loading alongside the existing
`respectGitignore`-gated gitignore load:
```zig
const ctx_stack = try gitignore.loadFromTreeNamed(allocator, io, root, ".contextignore");
defer ctx_stack.deinit();
```
Path exclusion: `gitignored OR contextignored → skip`.

### Phase 3 — Tool plumbing — single-path tools (~25 LOC × 3)

**Files:** `src/coding/tools/{read,write,edit}.zig`

Before IO, load the contextignore stack rooted at the workspace root
and call `Stack.match(resolved_path)`. If matched → return
`tool_error{contextignored}`. Hard refusal, no override.

### Phase 4 — Seed the repo `.contextignore`

Create `/.contextignore` with:
```
# v2.9 — paths hidden from default tool walks and IO. Enforcement is
# unconditional: no tool argument can disable this.

docs/archive/
docs/spec/v0.md
# Add docs/spec/v1.md here once v2.x ships.
```
Plus `docs/archive/` directory (empty `.gitkeep`).

### Phase 5 — Status front-matter convention

Every file in `docs/design/` gets a `Status: shipped|draft|abandoned`
line at the top. No automation — convention only.

## Known limitation: `bash` loophole

`bash` runs arbitrary shell commands and can still read archived paths
(`cat docs/spec/v0.md`, etc.). V2 hardening candidates: path-prefix
sniff before exec, or chroot/namespace sandbox. V1 decision: accept as
a trust-boundary limitation (not a security boundary).

## Configuration

| Layer | Mechanism |
|---|---|
| Per-call | **None.** No tool argument can disable enforcement. |
| Per-session | **None for v1.** A future `--no-contextignore` CLI flag for human inspection is possible but not in scope. |
| Repo-level | The `.contextignore` file + per-directory `.contextignore` files anywhere in the tree (mirrors `.gitignore`'s nested-rules semantics) |

The **only** way to add or remove paths is to edit `.contextignore` in
the repo — a deliberate human action captured in git history.

---

## Out of scope

- **`bash` enforcement** — trust boundary, not security boundary.
- **Auto-move on `Status: shipped`** — convention only.
- **System-prompt mention of `.contextignore`** — gate is structural.
- **`web_search` / `web_fetch`** — don't touch the local filesystem.
- **Subagent inheritance** — sub-agents inherit parent's tool surface automatically.
