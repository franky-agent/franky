# v2.6 — SubAgent Tool Execution Progress

**Status:** decided — ready for implementation  
**Affects:** `src/coding/tools/subagent.zig`, `src/agent/proxy.zig`,
`src/coding/modes/proxy.zig`, `src/coding/modes/rpc.zig`,
`src/coding/modes/web/app.js`, `src/coding/modes/web/style.css`  
**Deferred to v3:** interactive mode progress display

---

## 1. Problem

When the parent agent calls the `subagent` tool, the user sees:

```
tool: subagent   running…
```

…then silence for up to several minutes while the sub-agent runs its
own turns, calls its own tools, and produces its own reasoning. The
parent's event stream shows nothing until `tool_execution_end` fires
with the final result.

This is especially visible in the web UI (proxy mode) but affects
any consumer of the SSE event stream.

---

## 2. Goals

1. Surface sub-agent progress to the user without modifying the
   parent agent's **context window** (no new messages injected into
   the parent's Transcript).
2. Reuse existing wire protocol — no new SSE event types required.
3. Keep the web UI **clean**: sub-agent activity is collapsible and
   hidden by default so the main conversation stays readable.
4. Work in proxy and rpc modes. Interactive mode deferred to v3.
   Print mode skipped (single-turn stdout, no event bus).

---

## 3. Architecture

### 3.1 `Agent.subscribe` — the tap point

`Agent` already exposes a pub subscription API:

```zig
pub fn subscribe(self: *Agent, handler: SubscribeHandler, userdata: ?*anyopaque) !u32
pub fn unsubscribe(self: *Agent, id: u32) void
```

The handler fires synchronously (under `subs_mutex`) from the
sub-agent's worker thread for every `AgentEvent` the loop emits.
This is the tap point — no changes to `agentLoop` required.

### 3.2 `tool_execution_update` — the carrier

`AgentEvent.tool_execution_update` already exists on the wire:

```zig
tool_execution_update: struct {
    call_id: []const u8,   // the PARENT's subagent call id
    update_json: []const u8,
},
```

`update_json` is an opaque JSON blob the UI renders as it wishes.
Sub-agent progress is encoded into this field.

### 3.3 `progress_fn` and `verbose_progress` in `subagent.Ctx`

Add three fields to `subagent.Ctx`:

```zig
/// Called from the sub-agent's worker thread for each forwarded
/// event. Must be thread-safe. Null → no forwarding.
progress_fn: ?*const fn (
    userdata: ?*anyopaque,
    call_id: []const u8,
    update_json: []const u8,
) void = null,
progress_userdata: ?*anyopaque = null,
/// When true, also forward message_update (text + thinking) deltas.
/// Default false — structural events only.
verbose_progress: bool = false,
```

In `runSubagent`, before `sub_agent.prompt(...)`:

```zig
var fwd: ForwardState = .{
    .io        = io,
    .allocator = allocator,
    .call_id   = call_id,
    .ctx       = ctx,
    .turn      = std.atomic.Value(u32).init(0),
};
const sub_id = if (ctx.progress_fn != null)
    try sub_agent.subscribe(subagentProgressHandler, &fwd)
else
    null;
defer if (sub_id) |id| sub_agent.unsubscribe(id);
```

### 3.4 Events forwarded

Structural events always forwarded. Text/thinking deltas only when
`verbose_progress = true`:

| Sub-agent `AgentEvent`  | Forwarded?                     | `update_json` shape |
|-------------------------|--------------------------------|---------------------|
| `turn_start`            | always                         | `{"kind":"turn_start","turn":N}` |
| `tool_execution_start`  | always                         | `{"kind":"tool_start","name":"X","cid":"Y"}` |
| `tool_execution_end`    | always                         | `{"kind":"tool_end","name":"X","cid":"Y","ok":true}` |
| `turn_end`              | always                         | `{"kind":"turn_end","turn":N}` |
| `agent_error`           | always                         | `{"kind":"error","msg":"..."}` |
| `message_update` text   | only when `verbose_progress`   | `{"kind":"text_delta","block":N,"delta":"..."}` |
| `message_update` thinking | only when `verbose_progress` | `{"kind":"thinking_delta","block":N,"delta":"..."}` |
| `message_start/end`     | never                          | no user value |
| `tool_execution_update` | never                          | no nested forwarding |

All `kind` values use readable names (not short abbreviations) for
debuggability.

### 3.5 Thread safety

`subagentProgressHandler` fires from the sub-agent's worker thread.
`broadcastEvent` in `proxy.zig` must hold its mutex for the full
duration of stamp + ring-write + fan-out — not just fan-out — so
concurrent calls from multiple sub-agent threads cannot interleave
frames. **Verify this before wiring and add a concurrent-broadcast
test** that spawns two goroutines emitting events simultaneously and
asserts no frames are torn.

### 3.6 `update_json` encoding

Build `update_json` with minimal allocation:
- Fixed-shape events (`turn_start`, `turn_end`, `tool_start`,
  `tool_end`): `std.fmt.bufPrint` into a stack buffer.
- `agent_error` (variable-length message): `allocPrint`.
- `text_delta` / `thinking_delta` (verbose path): `allocPrint`.

### 3.7 Mode wiring

**proxy.zig** — wire `progress_fn` in the `subagent.Ctx` built at
session init:

```zig
.progress_fn     = subagentProgressForward,
.progress_userdata = session,   // *Session
```

`subagentProgressForward` encodes a `tool_execution_update`
`AgentEvent` and calls `session.broadcastEvent(...)`.

**rpc.zig** — same wiring; rpc clients already consume
`tool_execution_update` events.

**interactive.zig** — deferred to v3 specification.

**print.zig** — skipped (no event bus).

## 4. Web UI integration — collapsible sub-agent panel

### 4.1 DOM structure

When `tool_execution_start` fires for a `subagent` tool, extend the
existing `.tool-card` to include a progress panel:

```html
<div class="tool-card tool-card-subagent" data-call-id="abc">
  <div class="tool-head">
    tool: <span class="tool-name">subagent</span>
    <span class="tool-status">running…</span>
    <button class="subagent-toggle" aria-expanded="false">▶</button>
  </div>
  <div class="subagent-log" hidden>
    <div class="subagent-entry">turn 1</div>
    <div class="subagent-entry">→ bash <span class="sa-status">running</span></div>
    <div class="subagent-entry sa-indent">→ bash <span class="sa-status done">done</span></div>
    <div class="subagent-entry">turn 2</div>
    …
  </div>
</div>
```

Panel starts collapsed (`hidden`). Toggle button flips
`aria-expanded` and removes/adds `hidden`.

### 4.2 Detecting `subagent` tool calls

When `tool_execution_start` fires with `data.name === 'subagent'`:
- Add class `tool-card-subagent` to the card.
- Append `.subagent-log` (hidden) and the toggle button.
- Create a per-card `Map<cid, HTMLElement>` for tool-entry tracking.

The tool name `'subagent'` is the canonical const from
`subagent.zig` — stable, no wire change needed.

### 4.3 Rendering `tool_execution_update`

New event handler:

```js
es.addEventListener('tool_execution_update', (e) => {
    noteEvent();
    const data = parseData(e.data);
    if (!data || !data.callId) return;
    const card = toolCards.get(data.callId);
    if (!card || !card.classList.contains('tool-card-subagent')) return;
    const log = card.querySelector('.subagent-log');
    if (!log) return;
    const upd = typeof data.updateJson === 'string'
        ? JSON.parse(data.updateJson) : data.updateJson;
    appendSubagentEntry(card, log, upd);
});
```

`appendSubagentEntry(card, log, upd)` dispatches on `upd.kind`:

| `kind`           | Action |
|------------------|--------|
| `turn_start`     | Append `turn N` entry |
| `tool_start`     | Append `→ <name> running` entry; store element in per-card `saToolEntries` map keyed by `upd.cid` |
| `tool_end`       | Look up entry by `upd.cid`; update status text to `done` or `error`; add `.is-error` class on error |
| `turn_end`       | no-op |
| `error`          | Append `⚠ <msg>` entry in error colour; auto-expand panel if collapsed |
| `text_delta`     | (verbose path) Append/extend a `text` entry for the current block |
| `thinking_delta` | (verbose path) Append/extend a `thinking` entry for the current block |

Sub-agent tool entries are tracked per-card in `saToolEntries`
(`Map<cid, HTMLElement>`) so parallel tool calls update the correct
row.

### 4.4 Toggle behaviour

- Panel starts **collapsed**.
- Toggle button: `▶` collapsed / `▼` expanded. Styling deferred to v3.
- On parent `tool_execution_end` for the subagent call: update
  `.tool-status` to `done` or `error`; leave panel state unchanged.
- **Auto-expand** when `upd.kind === 'error'` and panel is collapsed.
