# LLM Agent Standards — Franky Support (Not Yet Implemented)

This document tracks the portions of the [v2.30 design](./v2.30-llm-agent-standards.md)
that have **not** been implemented. The completed items live in
[v2.30-implemented.md](./v2.30-implemented.md).

---

## 1. `--agents <dir>` Flag Not Wired Into Scanning

**Design ref:** §4.1 (root layout), §5 (implementation plan — Phase 1a)

**Status:** Flag is parsed by `cli.zig` into `cfg.agents_path` but never read
by `InstructionsState.scan()` or any workspace-resolution logic.

**What's missing:**
- When `--agents <path>` is set, `InstructionsState.scan()` should use the
  given path instead of `<workspace>/.agents/` for the `.agents/agents.md`
  file and the `.agents/skills/` root.
- The same override should propagate to `skills.Roots.resolveStandard()`.

**Effort:** Small — threading through config and one argument to `scan()`.

---

## 2. `.cursor/rules/*.md` Parsing — `parseCursorRule()` Not Written

**Design ref:** §4.5, §4.6, §10.8

**Status:** The `.cursor/rules/` path is registered as a standard skill root
in `skills.Roots.resolveStandard()`, but there is no `parseCursorRule()`
function to convert Cursor frontmatter into franky's `Skill` struct.

**Cursor → Franky mapping (specified but unimplemented):**

| Cursor field | Franky field |
|---|---|
| `globs` | `auto_apply` (array, same format) |
| `description` | `description` |
| `enabled` | Stored as `enabled:` frontmatter (ignored when false) |
| Title (first `# ` heading) | `name` |
| No `name` in Cursor rules | Extracted from first `# ` heading or filename stem |

**What's missing:**
- A `parseCursorRule(allocator, source_path, text) !Skill` function (analogous
  to `parseSkill` but using Cursor's frontmatter keys).
- Wiring into the skill loader so `.cursor/rules/*.md` files are actually
  parsed by the new function rather than the existing `parseSkill`.
- Tests for Cursor rule parsing, precedence, and edge cases.

**Effort:** Medium — write a new parser + integrate into skill loading path.

## 3. `.claude/settings.json` Merge Not Implemented

**Design ref:** §4.4, §4.9, §10.5

**Status:** No code reads `.claude/settings.json` at startup.

**Specified merge keys:**

| `.claude/settings.json` key | Franky target |
|---|---|
| `model` | `profile_registry.default_model` seed |
| `permissions` | `permission_store` allow/deny lists |
| `allowedTools` | `permission_store.allow_list` |
| `env` | Session environment overlays |
| `systemPrompt` | Instruction block (merged as `.claude` kind, lowest priority) |

**What's missing:**
- File read + JSON parse of `<workspace>/.claude/settings.json`.
- Selective key extraction and injection into the config resolver.
- Integration with the settings layering order (§4.9):
  `CLI flags > .franky/settings.json [project] > .claude/settings.json [project]
  > .agents/models.json [project] > .franky/settings.json [user] > ~/.claude.json
  > built-in defaults`

**Effort:** Medium — JSON parsing + config resolver integration.

## 4. `.agents/models.json` Loading Not Implemented

**Design ref:** §4.3, §4.9, §10.6

**Status:** No code reads `.agents/models.json`.

**Specified schema:**

```json
{
  "defaultProvider": "anthropic",
  "defaultModel": "claude-sonnet-4-6",
  "profiles": {
    "code": {
      "provider": "anthropic",
      "model": "claude-sonnet-4-6",
      "contextWindow": 1000000,
      "maxOutput": 8192
    }
  }
}
```

**What's missing:**
- File read + JSON parse of `<workspace>/.agents/models.json`.
- Mapping profile entries to franky's `Profile` struct in `settings.json`.
- Integration with the settings layering order (§4.9).

**Effort:** Medium — JSON parsing + profile registry integration.

## 5. MCP Config Loader — `src/coding/mcp.zig` Not Created

**Design ref:** §4.7, Phase 1c, §10.7

**Status:** The file `src/coding/mcp.zig` does not exist.

**Specified API:**

```zig
pub const McpRoots = struct {
    project: ?[]const u8 = null,  // workspace-root .mcp.json
    claude: ?[]const u8 = null,   // .claude/.mcp.json
    agents: ?[]const u8 = null,   // .agents/mcp.json
};

pub const McpConfig = struct {
    servers: std.StringHashMap(McpServerDescriptor),
    pub fn load(allocator, io, workspace_root) !McpConfig;
    pub fn deinit(self: *McpConfig) void;
};
```

**What's missing:**
- File creation, JSON parsing, `McpServerDescriptor` struct.
- Discovery of `.mcp.json` at all three standard locations.
- Integration into the config resolver (tool list assembly).
- (Phase 2 — MCP runtime — is deferred entirely.)

**Effort:** Medium — new module.

## 6. `.agents/system-prompt.md` Not Loaded

**Design ref:** §4.3, §4.8 step 3

**Status:** The file `system-prompt.md` inside `.agents/` is documented in the
design as overriding the base system prompt, but no code reads it.

**What's missing:**
- After loading the base system prompt (disk default or built-in), check for
  `<workspace>/.agents/system-prompt.md`. If present and no CLI `--system-prompt`
  override is in effect, replace base with its content.
- Integration into `buildSystemPromptIo()`.

**Effort:** Small.

---

## 7. Cross-Standard Settings Merge Not Integrated (§4.9)

**Design ref:** §4.9

**Status:** The extended settings layering in §4.9 is a design doc only. The
config resolver (`config.zig`) does not iterate `.claude/settings.json` or
`.agents/models.json`.

**Specified order:**
```
CLI flags > .franky/settings.json [project] > .claude/settings.json [project]
> .agents/models.json [project] > .franky/settings.json [user] > ~/.claude.json
> built-in defaults
```

**What's missing:**
- A merge loop in `config.resolve()` that reads each overlay and applies shared
  keys (MCP servers, model defaults, permissions, env).
- Conflict resolution logic (higher-precedence keys survive).

**Effort:** Medium — config resolver refactor.

---

## 8. Per-File Proximity Resolution Not Implemented (§4.2 Phase 2)

**Design ref:** §4.2, §10.4

**Status:** `InstructionForest.resolveClosest()` returns `null`. Only the
root-level `AGENTS.md` / `CLAUDE.md` / `.agents/agents.md` is rendered into
the system prompt at session start.

**What's missing:**
- Per-file resolution: when the agent processes a file (read/edit/write), find
  the nearest `InstructionBlock` via ancestor-path prefix match.
- Either re-assemble the system prompt on context switch, or provide a tool
  the model calls to fetch subdirectory-specific instructions.
- (Spec design note: per-file resolution is deferred to Phase 2 to keep the
  current PR scoped to root-level render only.)

**Effort:** Large — requires loop integration at each file-access point.

## 9. Phase 2 — MCP Runtime

**What's missing:**

- `McpRegistry.startAll(io)` spawns subprocesses, negotiates.

**Effort:** Large — deferred after Phase 1c config loader.

## 10. Phase 3 — Sub-agents from Standards

**Design ref:** Phase 3 sections

**What's missing:**
- Load `.agents/agents/*/agent.md` into the subagent preset registry.
- Load `.claude/agents/*.md` into the subagent preset registry.
- Respect the `.agents/` agent format (frontmatter-based) vs. `.claude/` agent
