# v2.5 design — subagent presets + SDK extensibility

**Status: decided.** All open questions resolved. Ready for implementation.
Originally drafted as v1.25; promoted to v2.5 when v1.x concluded at v1.31.0
and the v2.x line opened at 2.0.0 (2026-05-01).

---

## 0. Why presets

A **preset** is a named purpose-built sub-agent template. Two
overlapping use cases, one foundation:

### 0.1 — Workspace-local presets

A preset that bundles a curated subset of franky's built-in
filesystem tools + a default model + a default system prompt.
Runs against the parent's workspace; no external auth needed.

Examples:
- `research` — read-only filesystem inspection (read+ls+find+grep).
- `code-audit` — focused audit-only against a stated concern.
- `file-ops` — targeted edits with read+write+edit.
- `bash-runner` — bounded shell task with bash+ls.

The pattern: the LLM picks a preset to scope the sub-agent's
capabilities + bias the model + bias the system prompt for a
specific kind of work. No secrets involved. No external
network.

### 0.2 — External-retrieval presets (v3 scope)

A preset that wraps a deterministic CLI or HTTP client for an
external system. Auth, endpoint, account ID, and guardrails
live INSIDE the tool's closure — the LLM never sees them.

Examples: `k8s-read`, `datadog-query`, `github-read`, `incident-io-read`,
`bigquery-read`. These are deferred to v3 (credential handling, §3.14;
external packaging idioms, §3.16).

### 0.3 — Same foundation

Both categories use the SAME `Preset` type (§2.1), the SAME
registry (§3.4), the SAME LLM-facing surface (`subagent` tool +
`preset` arg + `list_subagent_presets` discovery tool, §3.3).

## 1. Goal + non-goals

### 1.1 In scope (v2.5.0)

- `Preset` type + `PresetRegistry` for SDK-registered presets.
- `subagent` tool's `parameters_json` gains a `preset: enum` field,
  generated dynamically at session-init from the registry. The sub-agent's
  tool list comes ENTIRELY from the preset's `build_tools` closure.
- `list_subagent_presets` tool — always registered; LLM calls it to
  discover what each preset does. System-prompt hint deferred to v3 (§3.3).
- **Breaking change**: `preset` is now a required arg. The implicit
  v1.24 "inherit parent tools" behaviour is gone; callers must pick a
  preset explicitly. Documented in the migration guide.
- **`SafetyClaims`** struct on `Preset` (§3.15): `max_result_bytes`
  enforced via existing truncation tools; other fields advisory metadata.
- Built-in catalog (§2.2): `research`, `code-audit`, `file-ops`, `bash-runner`.
- Role-promotion guard: preset's `default_role` > parent role → spawn
  fails with `role_promotion_denied` + hint naming the required parent role.
- SDK extensibility: consumers register custom presets before `mode.run()`;
  SDK-registered preset overrides built-in of same name (warn log on override).
- Collision policy: SDK preset wins, `warn` log when overriding a built-in.

### 1.2 Out of scope (deferred)

- **System-prompt hint with threshold logic** (hint-only for ≤5, both for
  6–15, tool-only for 16+). Deferred to v3. v2.5 uses only the
  `list_subagent_presets` tool.
- **External-retrieval preset built-ins** (`k8s-read`, `datadog-query`, etc.).
  Deferred to v3. Registration API is fully public so consumers can build
  them now against the SDK.
- **Credential / secret handling** (§3.14). Deferred to v3.
- **External-tool packaging idioms** (CLI shell-out, HTTP API, bundled SDK).
  Deferred to v3.
- **`PresetContext` shape** (§3.5). Deferred — no built-in preset needs
  ambient context beyond `parent_tools` at v2.5. Add fields when a built-in
  or SDK preset actually needs them.
- **Multi-preset composition, hot-reload, versioning, per-preset config
  overlays, per-user ACLs.** YAGNI.
- **Updated `subagent` tool description** (§3.13). Deferred until
  `list_subagent_presets` tool ships and can be referenced from the description.
- **Auth-token rotation in-process.** v3.

## 2. Surface

### 2.1 The `Preset` type

```zig
// src/coding/tools/subagent.zig — additions

pub const SafetyClaims = struct {
    /// Truncate sub-agent tool results to this many bytes before
    /// adding to transcript. Uses the existing truncate tool impl.
    max_result_bytes: ?u32 = null,
    /// Advisory — declared here for future discovery surface / audit.
    read_only: bool = false,
    requires_sandbox: bool = false,
    max_calls_per_minute: ?u32 = null,
};

pub const Preset = struct {
    /// Unique short name. Lowercase letters, digits, hyphen.
    name: []const u8,
    /// One-line purpose (≤100 chars). Returned by list_subagent_presets.
    description: []const u8,
    /// Default profile (model + provider). Used when `profile` arg omitted.
    default_profile: []const u8,
    /// Default capability role. Demotion-only against parent role.
    default_role: role.Role,
    /// Default system prompt. Override-able via tool's `system_prompt` arg.
    default_system_prompt: []const u8,
    /// Returns the sub-agent's tool list. Must NOT include `subagent`.
    build_tools: *const fn (
        allocator: std.mem.Allocator,
        parent_tools: []const at.AgentTool,
    ) anyerror![]at.AgentTool,
    /// Safety metadata. `max_result_bytes` is enforced; rest is advisory.
    safety: SafetyClaims = .{},
};

pub const PresetRegistry = struct {
    allocator: std.mem.Allocator,
    presets: std.StringArrayHashMapUnmanaged(Preset) = .empty,

    pub fn register(self: *@This(), p: Preset) !void { ... }
    pub fn get(self: *const @This(), name: []const u8) ?Preset { ... }
    pub fn iterator(self: *const @This()) ... { ... }
};
```

`build_tools` receives `parent_tools` (the parent's filtered tool list, minus
`subagent` itself) so presets that want to compose with the parent's tool set
can do so without a full `PresetContext`. Full `PresetContext` (environ_map,
workspace, etc.) deferred to §3.5.

### 2.2 Built-in catalog

v2.5.0 ships **workspace-local** built-ins only. External-retrieval
presets ship via SDK consumers.

| Name | Purpose (returned by list_subagent_presets) | Tools | Default profile | Default role |
|---|---|---|---|---|
| `research` | Reads files, greps for patterns, summarises findings within the workspace. | read, ls, find, grep | `gemini` | read |
| `code-audit` | Audits a stated quality / security concern read-only across the workspace. | read, ls, find, grep | `gemini` | read |
| `file-ops` | Performs targeted file edits given clear instructions. | read, write, edit, ls | (parent's) | plan |
| `bash-runner` | Runs a bounded shell task, reports stdout/stderr/exit. | bash, ls | (parent's) | code |

The v1.24 `general` preset (inherit parent tools) is **not shipped**.
The `preset` arg is now required; existing prompts must be updated to
name a preset explicitly. See migration notes.

### 2.3 Discovery surface

**`list_subagent_presets` tool** — always registered alongside `subagent`.
Returns structured JSON: `[{name, description, default_profile, default_role,
safety}]`. LLM calls on demand when choosing which preset fits the task.

Tool schema (static — defined once):
```json
{
  "type": "object",
  "properties": {},
  "required": []
}
```

Response shape (example):
```json
[
  {"name":"research","description":"Reads files, greps for patterns, summarises findings within the workspace.","default_profile":"gemini","default_role":"read","read_only":true},
  ...
]
```

System-prompt hint with threshold-based fallback is deferred to v3.

## 3. Resolved questions

### 3.1 — One `subagent` tool with a `preset` arg, OR one tool per preset?

**→ One tool with `preset` enum arg.**

N-tools approach bloats context (N × 200-token schema = constant overhead).
One enum field is ~50 tokens regardless of preset count. Parallel-tool
execution already handles multiple concurrent presets in one turn.

### 3.2 — Preset arg shape: enum, string, or free-form?

**→ Enum.** `{"type": "string", "enum": [...]}` generated at session-init
from the registry. Free-form with fuzzy routing deferred; if needed, a v3
purpose-lookup tool converts free-form text to a valid preset name and the
actual spawn still uses the strict enum.

### 3.3 — Discovery: system-prompt hint, tool, or both?

**→ `list_subagent_presets` tool only.** System-prompt hint with threshold
logic (≤5 hint-only, 6–15 both, 16+ tool-only) deferred to v3.

The tool adds ~100 tokens of schema (one definition); hint logic adds
templating complexity for marginal gain at small catalog sizes.

### 3.4 — Where does the preset registry live?

**→ Mode-driver-owned, populated at session init.**

Each mode (print/proxy/rpc/interactive) initialises a `PresetRegistry`,
calls `registerBuiltinPresets`, then accepts SDK consumer extensions
BEFORE the registry is frozen and handed to `subagent.Ctx`. No
mid-session mutation.

```zig
// SDK consumer's main:
var registry = PresetRegistry.init(allocator);
try subagent.registerBuiltinPresets(&registry);
try registry.register(.{ .name = "my-custom", ... });
// then dispatch to mode runner
```

### 3.5 — What does `build_tools` receive?

**→ Deferred.** v2.5 passes only `parent_tools: []const at.AgentTool`.
`PresetContext` struct (environ_map, workspace, etc.) added when a built-in
or SDK preset actually needs ambient state beyond parent tools.

### 3.6 — Back-compat for v1.24's implicit tool inheritance?

**→ Break compat; document it.**

`preset` is a required arg. The implicit "inherit parent tools" path
(v1.24 default) is removed. Existing prompts must name a preset. Migration
note: use `file-ops` for write-heavy delegation, `research` for read-heavy,
`bash-runner` for shell tasks.

### 3.7 — Static vs dynamic schema; description placement?

**→ Static at session init.** `parameters_json` built once from the registry
at session start. Preset names live in the enum; descriptions live in the
`list_subagent_presets` response, not in the tool schema.

### 3.8 — Custom tool registration model?

**→ Factory functions (`at.AgentTool`).** Tools are owned by the presets
that build them. The `Preset.build_tools` closure returns a slice of
`at.AgentTool`. No global tool registry; tools are implementation details
inside presets. SDK consumers write tool factories using the same shape as
built-ins (`read.tool()` etc.).

### 3.9 — Preset name collision policy?

**→ SDK-registered wins.** Overrides the built-in of the same name;
`warn` log when this happens. Matches the profiles system precedence rule.

### 3.10 — Preset role demotion semantics?

**→ Parent role wins; error with hint.**

If `preset.default_role > parent.role` → spawn fails with
`role_promotion_denied`. Error includes a `hint` field naming the minimum
parent role required by this preset (e.g. "preset `bash-runner` requires
parent role `code` or higher; current parent role is `read`").

Per-call `role` arg can still demote a preset's default further.

### 3.11 — Test plan?

**→ Static schema + dispatch tests.**

1. `registerBuiltinPresets` populates `research`, `code-audit`, `file-ops`,
   `bash-runner` with correct defaults.
2. Dynamic `parameters_json` enum lists every registered preset name.
3. Faux sub-agent: `preset: "research"` → tool list is exactly
   `[read, ls, find, grep]` (no write/edit/bash).
4. `list_subagent_presets` tool call returns all registered presets.
5. SDK custom preset: registered → appears in enum + dispatch invokes builder.
6. Collision: SDK preset overrides built-in, warn log fires.
7. Role promotion denied: parent=read, preset default_role=code →
   `role_promotion_denied` error with hint.
8. `max_result_bytes` on `SafetyClaims` → sub-agent tool results truncated
   via existing truncation impl.

### 3.12 — Naming?

**→ `Preset`.**

### 3.13 — Update `subagent` tool description?

**→ Deferred** until `list_subagent_presets` tool is available and can be
referenced in the description text.

### 3.14 — Credential / secret handling?

**→ Deferred to v3.** v2.5 ships built-in workspace-local presets only;
no external auth needed. Closure-capture pattern, schema secret-field check,
and audit log deferred to v3 when external presets are scoped.

### 3.15 — Guardrails (`SafetyClaims`)?

**→ Ship `SafetyClaims`; enforce `max_result_bytes` now.**

`SafetyClaims` struct ships on `Preset` with `max_result_bytes` wired to
the existing truncate tool implementation. Other fields (`read_only`,
`requires_sandbox`, `max_calls_per_minute`) are advisory metadata for v2.5;
enforcement deferred to the external-preset milestone.

### 3.16 — External-tool packaging idioms (CLI vs HTTP vs SDK lib)?

**→ Deferred to v3.** franky core provides `AgentTool` interface + `Preset`
registration. CLI shell-out / HTTP API / bundled-SDK packaging patterns and
the `subprocess.zig` helper land in v3 with the external-preset work.

## 5. Migration notes

v1.24 callers used `subagent` without a `preset` arg and received the parent's
filtered tool list. v2.5 removes that path:

| Old behaviour | v2.5 replacement |
|---|---|
| Inherits parent tools | Pick the closest built-in preset |
| General delegation (all tools) | Use `file-ops` or `bash-runner` depending on task |
| Read-heavy analysis | `research` or `code-audit` |
| File writes | `file-ops` |
| Shell tasks | `bash-runner` |

The `preset` enum field is now required in `parameters_json`; any call that
omits it will fail argument validation with `invalid_args`.
