# v2.8 — Edit Tool Diff View

**Status:** draft — open for discussion  
**Affects:** `src/coding/tools/edit.zig`, `src/agent/proxy.zig`, `src/coding/modes/web/app.js`, `src/coding/modes/web/style.css`, `src/coding/modes/web/index.html`  
**Depends on:** v2.7 subagent panel (overlay infrastructure is already in place)

---

## 1. Problem

The edit tool currently returns a plain-text line-diff like:

```
  [0] - const provider = data.provider || '?';
  [0] - const model = data.model || '?';
  [0] - el.textContent = 'role: ' + role + ' · ' + provider + ':' + model;
  [0] + el.textContent = 'role: ' + role;
```

This is:
- Hard to scan visually — no color coding
- Missing line numbers and surrounding file context
- Does not highlight intra-line word-level changes
- Forces the model (not the user) to be the primary consumer of the diff

Users reviewing edits need to quickly understand *what changed* and *where*.
A visual diff — like GitHub/GitLab's unified or side-by-side view — would
make the edit tool output dramatically more usable.

---

## 2. Goals

1. **Show a visual diff in the tool card** when the `edit` tool completes — colored additions/deletions, line numbers, file path header.
2. **Support both line-by-line (unified) and side-by-side** views, with an easy toggle.
3. **Preserve existing text summary** for the model's context window — the diff view is an *enhancement* of the tool's `resultText`, not a replacement. The model still sees the plain text in its transcript.
4. **Zero build-step overhead** — plain JS, no bundler, no React. Libraries loaded from CDN.
5. **Reuse existing tool-card infrastructure** — the result panel (`tool-result-log`) is already collapsible. We replace its content with rendered HTML instead of plain text.
6. **No wire-protocol changes** — reuse `details_json` (already part of `ToolResult`) to carry the unified diff string to the client. Add `detailsJson` to the `tool_execution_end` SSE frame.
7. **Syntax highlighting** for the code in the diff (optional but high value).

---

## 3. Design decisions

### Q1: Where to compute the diff — server vs client?

| Option | Description | Verdict |
|---|---|---|
| **A: Server generates unified diff string** | `edit.zig` builds a proper unified diff (from original ↔ final text) and sends it in `details_json`. Client renders with `diff2html`. | ✅ **Chosen** — cleaner separation, lighter client, single library on the front-end |
| B: Server sends old+new text | Client computes diff with `jsdiff` and renders with `diff2html`. | Avoided — requires 2 client libraries, more JS code |
| C: Fully client-side | `app.js` would need to track original file content and compute diff. | Avoided — no state management for file contents on the client |

### Q2: Diff algorithm on the server

| Approach | Description | Verdict |
|---|---|---|
| **LCS-based line diff** | Compute LCS of lines between `original` and final buffer, emit standard unified diff format | ✅ **Recommended** — gives context lines, handles multiple changes correctly |
| Per-edit hunk reconstruction | For each individual edit, emit a hunk using the diff_rows we already track | Simpler to implement but loses global context and doesn't handle adjacent edits merging correctly |
| Shell out to `diff` | Spawn `/usr/bin/diff` | Avoided — portability, latency, no control |

Decision: Implement a **minimal LCS-based line diff** in Zig (≈100 LOC). This
produces a standard unified diff string with 3 context lines. We don't need
Myers — a simple dynamic-programming LCS is fine for the file sizes the edit
tool handles (< 1000 lines).

### Q3: How to deliver the diff to the client

The `ToolResult` struct already has `details_json: ?[]const u8` — opaque
metadata not shown to the model. We use this:

```json
{
  "diff": "@@ -1,5 +1,3 @@\n context line\n-removed line\n+added line\n context line\n",
  "path": "src/main.zig"
}
```

The `tool_execution_end` SSE frame currently only includes `resultText` and
not `details_json`. We need to add `detailsJson` to the SSE encoding in
`proxy.zig`.

### Q4: Side-by-side vs unified (line-by-line) default

| View | Pros | Cons |
|---|---|---|
| Unified (line-by-line) | Compact, familiar from `git diff`, default in diff2html | Harder to scan for wide changes |
| Side-by-side | Easier to see word-level changes side-by-side | Takes more horizontal space |

Decision: Default to **unified (line-by-line)** with a toggle button to
switch to **side-by-side**. diff2html supports both with the
`outputFormat` config option.

### Q5: Auto-expand vs collapsed

Current behavior: tool result panel is collapsed by default (user clicks `▶`
to expand), except on error where it auto-expands.

Decision: **Always auto-expand the result panel for `edit` tool calls.**
The diff is the primary output the user needs to review. For non-edit tools,
keep current collapsed-by-default behavior.

### Q6: Rehydration

When the page reloads, `appendFinalizedToolCard` renders all past tool cards
from transcript data. The `details_json` is not currently available in the
transcript replay. We need to ensure the edit diff is preserved across page
reloads.

Options:
- A: Store `details_json` in the transcript messages alongside `ToolResult`
- B: Recompute the diff from `resultText` on rehydration
- C: Skip rehydration for the diff view (fall back to plain text)

Decision: **Option A** — modify the transcript storage to include
`details_json` in tool result messages. This is the correct long-term
solution. For v2.8 initial implementation, **Option C** (graceful fallback
to plain text on rehydration) is acceptable as a minimum — the diff is shown
live during the session, and plain text is still usable on reload.

### Q7: TUI (terminal) diff view

The TUI editor (`src/tui/editor.zig`) and diff renderer
(`src/tui/diff_renderer.zig`) are separate from the edit tool output. The
edit tool's result text goes to the model's context window and the terminal
user sees the same plain text summary.

Decision: **Defer to a future version.** The TUI already has an ANSI diff
renderer for the screen-buffer layer. A separate design is needed for
showing line-level diffs in the terminal conversation pane.

## 4. Architecture

### 4.1 Data flow

```
edit tool called
  ├─ read file → original_text
  ├─ apply edits → final_text
  ├─ compute unified diff between original_text and final_text
  ├─ return ToolResult {
  │     content[0].text = plain text summary (unchanged)
  │     details_json = { "diff": "...", "path": "..." }
  │     is_error = false
  │   }
  └─ loop.zig encodes tool_execution_end SSE event
       └─ proxy.zig → includes detailsJson in SSE payload

tool_execution_end received in app.js
  ├─ endToolCall(data.callId, data.isError, data.toolCode, data.resultText, data.detailsJson)
  ├─ detect edit tool by checking detailsJson.diff exists
  │   ├─ YES → renderEditDiffView(panel, detailsJson)
  │   │         └─ diff2html renders unified diff → HTML inserted into panel
  │   └─ NO  → panel.textContent = resultText (current behavior)
  └─ auto-expand panel for edit tools (toggle ▼, panel visible)
```

### 4.2 Server-side: unified diff generation (edit.zig)

A new function `computeUnifiedDiff` in `edit.zig`:

```zig
fn computeUnifiedDiff(allocator: std.mem.Allocator, original: []const u8, modified: []const u8, path: []const u8) ![]u8 {
    // 1. Split both texts into line slices
    // 2. Compute LCS of lines (simple O(n*m) DP for n,m ≤ 2000)
    // 3. Trace back to find changed regions
    // 4. Emit standard unified diff format:
    //      --- a/<path>
    //      +++ b/<path>
    //      @@ -start,count +start,count @@
    //       context
    //      -removed
    //      +added
    // 5. Include up to 3 lines of context around each hunk
}
```

The unified diff format is well-defined and parseable by diff2html:

```
--- a/src/main.zig
+++ b/src/main.zig
@@ -10,7 +10,7 @@
     const x = 1;
-    const y = 2;
+    const y = 3;
     const z = 4;
```

The diff is stored in `details_json` as a JSON string (properly escaped).

If computing the diff fails (OOM, too many lines), fall back gracefully:
include `{"diff": null, "path": "..."}` and the client shows plain text.

    toolCards.delete(callId);

    // Subagent cards — unchanged
    const saLog = el.querySelector('.subagent-log');
    if (saLog) {
        // ... existing subagent logic, unchanged ...
        return;
    }

    // Regular tool cards
    const panel = el.querySelector('.tool-result-log');
    if (!panel) return;

    // Check for diff view payload
    if (!isError && detailsJson) {

        try { details = JSON.parse(detailsJson); } catch (_) {}

#### 4.6c New function: `renderEditDiffView`

```js
function renderEditDiffView(cardEl, panelEl, toggleEl, details) {
    cardEl.classList.add('is-edit');

    // Auto-expand the result panel
    if (toggleEl) {
        toggleEl.setAttribute('aria-expanded', 'true');
        toggleEl.textContent = '▼';
        panelEl.removeAttribute('hidden');
    }

    // Add view-toggle button (unified ↔ side-by-side)
    let currentMode = 'line-by-line';  // unified default
    const viewToggle = document.createElement('button');
    viewToggle.className = 'diff-view-toggle';
    viewToggle.textContent = 'Side-by-side';
    viewToggle.addEventListener('click', () => {
        currentMode = currentMode === 'line-by-line' ? 'side-by-side' : 'line-by-side';
        viewToggle.textContent = currentMode === 'line-by-line' ? 'Side-by-side' : 'Unified';
        renderDiff(panelEl, details.diff, currentMode, details.path);
    });

    // Insert toggle next to the tool head
    const head = cardEl.querySelector('.tool-head');
    if (head) head.appendChild(viewToggle);

    // Initial render
    renderDiff(panelEl, details.diff, currentMode, details.path);
}

function renderDiff(panelEl, diffString, mode, filePath) {
    const config = {
        outputFormat: mode,         // 'line-by-line' or 'side-by-side'
        drawFileList: false,        // single file, no file list
        fileContentToggle: false,
        matching: 'lines',
        highlight: true,
        rawTemplates: {
            // Override the file-header template to use our path
        },
    };

    // Set the file path so diff2html shows it
    const fullDiff = '--- a/' + (filePath || 'file') + '\n+++ b/' + (filePath || 'file') + '\n' + diffString;

    try {
        const html = Diff2Html.html(fullDiff, config);
        panelEl.innerHTML = html;

        // Apply syntax highlighting after render if highlight.js is available
        if (typeof hljs !== 'undefined') {
            panelEl.querySelectorAll('.d2h-code-line-ctn').forEach(el => {
                hljs.highlightElement(el);
            });
        }
    } catch (e) {
        // Fallback: show plain text on render failure
        panelEl.textContent = diffString;
    }
}
```

## 5. Step-by-step implementation

### Step 1 — `edit.zig`: capture original text and compute unified diff

- In `applyEdits()`, after reading the file, keep `original` as a field
- After all edits succeed, call `computeUnifiedDiff(allocator, original, buf.items, path)`
- Store result in `details_json` alongside `{"edits":N}`
- If diff computation fails, set `details_json` to `{"diff":null,"path":"..."}`

**Acceptance:** edit tool returns `details_json` with a valid unified diff string.

### Step 2 — `edit.zig`: implement `computeUnifiedDiff()`

- Split original and modified into lines (preserve trailing newline semantics)
- Compute LCS with a simple DP table (O(n*m) where n,m ≤ 2000 — fine for source files)
- Trace back to identify changed/unchanged regions
- Emit hunk headers with proper line numbering
- Include 3 lines of context before and after each change
- Handle edge cases: empty files, no changes, trailing newlines, binary-ish content

**Acceptance:** Unit tests verify generated diff parses by diff2html (we can
test by checking expected hunk structure in the output).

### Step 3 — `proxy.zig`: include `detailsJson` in `tool_execution_end`

- In `encodeEventJson`, check `e.result.details_json`
- If present, append `,"detailsJson":<raw json blob>` to the SSE payload
- If null, omit the field

**Acceptance:** SSE frame for `tool_execution_end` includes `detailsJson`
when the tool result has one.

### Step 4 — `index.html`: add diff2html CDN scripts

- Add `<script src="...diff2html.min.js">` before `app.js`
- Add `<link rel="stylesheet" href="...diff2html.min.css">` in `<head>`
- (Optional) Add highlight.js CDN for syntax highlighting

**Acceptance:** `typeof Diff2Html !== 'undefined'` in browser console.

### Step 5 — `style.css`: diff2html overrides

- `.tool-card.is-edit .tool-result-log` — zero padding, horizontal scroll
- `.tool-card.is-edit .d2h-file-header` — hidden (redundant with tool head)
- `.tool-card.is-edit .d2h-code-line` — compact padding
- `.diff-view-toggle` — small button next to tool head
- Dark-theme compatibility: ensure `.d2h-code-side-line` colors work with
  the existing CSS variable theme

**Acceptance:** diff renders inside tool card without overflow, matches dark theme.

### Step 6 — `app.js`: pass `detailsJson` through SSE handler

- Add `data.detailsJson` parameter to `endToolCall()`
- Add `detailsJson` parameter to `appendFinalizedToolCard()`
- Both default to `null` (backward compatible)

**Acceptance:** No crashes on existing tool results that lack `detailsJson`.

### Step 7 — `app.js`: implement `renderEditDiffView()` and `renderDiff()`

- Detect edit tool by `detailsJson.diff` presence
- Render with `Diff2Html.html()`
- Add unified/side-by-side toggle button
- Fall back to plain text on render error or missing diff data
- Auto-expand the result panel for edit tools

**Acceptance:** Edit tool card shows a colored diff with line numbers
instead of plain text on successful edit.

### Step 8 — `app.js`: rehydration support

- In `rehydrate()`, pass stored `detailsJson` from transcript to
  `appendFinalizedToolCard()`
- (Minimal v1) If `detailsJson` is not available in transcript, fall back
  to plain text gracefully

**Acceptance:** On page reload, edit tool cards from past turns show plain
text summary (no crash). A follow-up can store and replay `detailsJson`.

### Step 9 — `edit.zig`: add fallback for failed diff computation

- Wrap diff computation in a `catch` block
- On failure, set `details_json = null` and continue normally

**Acceptance:** Edit succeeds even if diff computation panics or OOMs.
Tool result text is still produced.

## 7. Edge cases & non-goals

### 7.1 Edge cases

| Case | Handling |
|---|---|
| **Edit fails** (no_match, ambiguous) | `is_error=true`, no `details_json` diff, red card + error text as before |
| **Empty file** | `edit_file_empty` error — no diff attempted |
| **Single-line edit** | Unified diff shows one-line hunk with context |
| **Whitespace-only changes** | diff2html will highlight the changed whitespace in red/green. No special handling needed. |
| **Very large file (>2000 lines)** | Cap LCS table size; fall back to simple before/after dump if exceeded |
| **Binary-ish content** | Unified diff won't be meaningful; skip diff and show plain text |
| **Multiple edits in one call** | LCS handles all changes together — one unified diff for the whole file |
| **Page reload** | Fall back to plain text summary (no `details_json` in transcript yet) |
| **Network failure loading CDN** | `typeof Diff2Html === 'undefined'` → fall back to plain text |
| **Dark theme mismatch** | Test diff2html CSS vars against existing theme; override `.d2h-code-line-*` colors if needed |

### 7.2 Non-goals

- **TUI terminal diff** — deferred to a separate design. The ANSI
  `DiffRenderer` already handles screen-buffer diffs; the conversation-pane
  diff is a different problem.
- **Interactive diff review** (accept/reject individual edits) — out of scope
- **Diff persistence across page reload** — acceptable for v1 to show plain
  text on rehydration
- **Word-level diff highlighting inside diff2html** — diff2html supports
  this via `matching: 'words'` but it's computationally expensive. Default
  to `'lines'` and revisit if requested.
- **Edit tool's `applyEdits` called from non-web contexts** — the unified
  diff is only produced when the tool is invoked via the proxy/session
  pipeline (which is always the case in practice). Pure library callers
  still get the plain text summary.

## 8. Open questions for discussion

1. **LCS vs Myers diff?** Simple LCS (DP, O(n²)) is fine for files up to
   ~2000 lines. Files larger than that are unlikely for find/replace edits.
   We cap at 2000 lines and fall back gracefully. OK? Yes

2. **Syntax highlighting?** diff2html can use highlight.js for code blocks.
   We can either:
   - A: Vendor highlight.js + a language pack (extra JS weight)
   - B: Use diff2html's built-in `highlight: true` which depends on
     highlight.js being available globally
   - C: Skip highlighting for v1
   I lean **B** — include highlight.js from CDN alongside diff2html; if it
   fails to load, diff2html still renders colored diff without syntax
   highlighting. 
   
   Reconsideration: We already have prism.js used for markdown code blocks highlighting.
   Can we use this instead of adding highlight.js? That would be ideal to avoid extra dependencies.
   If not then lets replace prism with highlight.js for both markdown and diff highlighting to avoid multiple libraries.
   If we will use highlight.js then we should vendor it to ensure consistent experience and avoid CDN issues.

3. **Default view mode: line-by-line or side-by-side?** I propose
   **line-by-line** (unified) as default since it's more compact and
   familiar from `git diff`. The toggle button lets users switch.
   Decision: Yes line-by-line*

4. **Should we store `details_json` in the transcript for rehydration?**
   This requires changes to the transcript storage schema. Worth doing now,
   or defer to a follow-up?
   - **Proposal:** Defer. The live session experience is the primary UX
     win. Rehydration showing plain text is acceptable for v1.
   Decision: Lets do it now. It's a bit more work but ensures the diff view is preserved across reloads, which is important for user experience.

5. **Naming**: Should the `details_json.diff` field contain just the hunk
   body (without `---/+++` headers) or the full diff?  
   - **Proposal:** The full diff including `---/+++` headers. This lets
     diff2html parse it directly and gives us the file path for display.
     We prepend the headers in `computeUnifiedDiff`.
   Decison: Yes
