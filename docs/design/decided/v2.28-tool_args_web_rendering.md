# Tool Args Web Rendering

**Status:** implemented — v2.28 tool args web rendering deployed
**Decision date:** 2025-01-21  
**Scope:** web UI (interactive mode) only — no API or backend changes  
**Affected files:**
- `src/coding/modes/web/app.js` — tool card rendering logic
- `src/coding/modes/web/style.css` — tool card & per-tool layout styles
- `src/coding/modes/web/index.html` — structural template updates (if required)
- `src/coding/modes/web/wireframe-*.html` — reference wireframes (this design drives them)

## Current State

Tool arguments are currently rendered as raw JSON inside a generic card:

```html
<div class="tool-card">
  <div class="tool-head">
    tool: <span class="tool-name">read</span>
    <span class="tool-status">done</span>
    <button type="button" class="tool-result-toggle" aria-expanded="false">▶</button>
  </div>
  <div class="tool-args">{"path":"docs/design/tool_args_web_rendering.md"}</div>
  <div class="tool-result-log" hidden=""></div>
</div>
```

**Problems with the current approach:**
- Raw JSON is hard to scan quickly, especially for nested or long values
- No visual hierarchy — every argument has equal weight
- File paths, commands, and diffs (the most important info) are buried in escaped strings
- Long values (e.g., `write.content`, `subagent.prompt`) blow up the card height
- No per-tool optimization — `edit` diffs and `bash` commands are rendered the same way as simple `read` args

## Design Goals

1. **Scannability** — a user should identify what the tool did in < 2 seconds
2. **Per-tool clarity** — each tool's arguments have a natural visual representation (path → file header, command → terminal block, edit → diff)
3. **Progressive disclosure** — show the most important info by default; hide or truncate the rest
4. **Accessibility** — keyboard-navigable, screen-reader friendly, proper contrast
5. **Consistency** — all tool cards share the same chrome (header, status, toggle, result log) even if the args body varies

## Decisions

### Default wireframe
**Wireframe 2 (Domain-Specific Cards) on desktop, Wireframe 3 (Compact) on mobile.**  
Desktop has enough horizontal space for side-by-side diffs and terminal blocks; mobile needs single-line summaries to prevent wall-of-text scrolling. A CSS breakpoint (`@media (max-width: 640px)`) toggles between the two modes on the same DOM.

### Syntax highlighting
**Yes, use the existing Prism.js setup** (`src/coding/modes/web/prism.js` + `prism-tomorrow.css`).  
Language detection from the `lang` field in code-fence metadata (already supported by the hand-rolled Markdown renderer). For `write.content`, detect language from file extension in `path`; fall back to plain text.

### Diff rendering for `edit`
**Unified / inline diff as default.** Side-by-side is available on desktop when the panel is wide enough (> 480 px), but the canonical render is unified — consistent with the existing `wireframe-edit-diff-view.html` and avoids horizontal-scroll fatigue.

### Handling many `edits[]`
**Collapse after 3 edits** with a "+ N more edits — expand" button. Each edit is a self-contained diff hunk; expanding reveals the rest inline (no pagination). This keeps the card height bounded while preserving full context when needed.

### `subagent.prompt` Markdown rendering
**Yes, render as Markdown.** The prompt often contains headings, lists, and code blocks; the existing hand-rolled Markdown renderer in `app.js` handles this safely (XSS-escaped, no external deps).

### JSON copy / raw mode
**Always provide a "View raw JSON" toggle** at the bottom of every tool card. This switches the rendered args view back to the current raw-JSON display for debugging and copy/paste.

### Animation
**150 ms ease** for expand/collapse transitions (matching the existing sidebar transition timing in `style.css`). Respect `prefers-reduced-motion` by disabling transitions when the media query is active.

## Per-Tool Rendering Strategy

| Tool | Primary Info | Secondary Info | Rendering Approach |
|------|-----------|---------------|-------------------|
| `read` | `path` | `limit`, `offset` | File header bar with path as clickable link; offset/limit as small badges |
| `write` | `path` | `overwrite` flag | File header bar + "will overwrite" warning badge if true; content in collapsible code block with syntax highlight |
| `edit` | `path`, `edits[]` | — | File header bar + diff view per edit (unified inline; side-by-side on wide panels). Collapse after 3 edits |
| `bash` | `command` | `description`, `cwd` | Terminal-style block with `$` prompt; description as muted subtitle; cwd as working-directory pill |
| `subagent` | `preset`, `max_turns` | `prompt`, `timeout_ms` | Tag/badge for preset; prompt in collapsible Markdown block; metadata as small pills |
| `grep` | `pattern`, `path` | `regex`, `maxMatches` | Search bar aesthetic — pattern in monospace, path below, toggle flags as pills |
| `find` | `pattern`, `cwd` | `limit` | Similar to grep — pattern prominent, directory context below |

## Wireframe 1: Structured Table (Generic Fallback)

A clean, consistent key-value table with syntax-highlighted values. Used for any tool without a bespoke renderer and as the "raw JSON" alternative view.

```
┌──────────────────────────────────────────┐
│ tool: read                    done   [▶] │
├──────────────────────────────────────────┤
│ Argument      Value                      │
│ ──────────────────────────────────────── │
│ path          docs/design/...            │
│ limit         2000                       │
│ offset        0                          │
├──────────────────────────────────────────┤
│ [Result log — collapsed]                 │
└──────────────────────────────────────────┘
```

**Pros:** Simple to implement, consistent, easy to parse for structured data fans  
**Cons:** Still somewhat generic; long values need truncation; no visual affordance for "this is a file path" vs "this is a command"

## Wireframe 3: Compact / Inline-Expandable (Mobile Default)

Single-line summary that expands into full detail. Optimized for dense chat histories and narrow screens.

```
┌──────────────────────────────────────────┐
│ [▶] read  docs/design/...     done       │
│     ─────────────────────────────────────│
│     limit: 2000  offset: 0               │
├──────────────────────────────────────────┤
│ [Result log — collapsed]                 │
└──────────────────────────────────────────┘
```

Collapsed state shows only tool name + primary argument. Expanded state renders the full domain-specific card (Wireframe 2) or the structured table (Wireframe 1) depending on screen width.

**Pros:** Minimal vertical space; fast to scroll through long sessions; works on mobile  
**Cons:** Requires an extra click to see details; primary-arg extraction logic needed per tool

## Interaction & Accessibility Details

- **Long values:** Any value > 3 lines or 200 chars gets truncated with a "Show more" button. `content` and `prompt` are always collapsed by default.
- **Copy actions:** Each argument row has a subtle copy button on hover (or always-visible on touch). The entire args block can be copied as JSON for power users via the "View raw JSON" toggle.
- **Clickable paths:** If `path` or `cwd` refers to a workspace file, render as `<a>` linking to the file view.
- **Keyboard:** Toggle button is focusable with `Enter`/`Space`. Expandable sections use `aria-expanded`. Tab order: toggle → copy buttons → show more → result log.
- **Result log:** Keep the existing toggle pattern but align it visually with the args section.
- **Status:** "done", "error", "running" should have distinct colors + icons (✅ ❌ ⏳).
- **Mobile:** Cards stack vertically; terminal/diff blocks scroll horizontally instead of wrapping.
- **Motion:** 150 ms ease transitions; disabled when `prefers-reduced-motion` is active.

## Implementation Notes

- **Rendering layer only:** This is a frontend change. The backend continues to emit the same JSON args; the web UI maps tool name → renderer.
- **Renderer registry:** A `toolRenderers: Map<string, (args: any) => VNode>` allows per-tool overrides. Unregistered tools fall back to the generic table (Wireframe 1).
- **Schema-driven fallback:** If we later add tool schemas, the generic renderer can auto-generate labels/inputs from the schema without hardcoding.
- **Existing assets:** Leverage the already-vendored `prism.js` and `prism-tomorrow.css` for syntax highlighting, and the hand-rolled `Markdown` module in `app.js` for prompt rendering.
