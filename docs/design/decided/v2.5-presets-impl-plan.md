## Step 1 — `SafetyClaims` + `Preset` + `PresetRegistry` types

**File:** `src/coding/tools/subagent.zig`

Add three structs before the existing `Ctx` definition:

```zig
pub const SafetyClaims = struct {
    max_result_bytes: ?u32 = null,
    read_only: bool = false,
    requires_sandbox: bool = false,
    max_calls_per_minute: ?u32 = null,
};

pub const Preset = struct {
    name: []const u8,
    description: []const u8,
    default_profile: []const u8,
    default_role: role_mod.Role,
    default_system_prompt: []const u8,
    build_tools: *const fn (
        allocator: std.mem.Allocator,
        parent_tools: []const at.AgentTool,
    ) anyerror![]at.AgentTool,
    safety: SafetyClaims = .{},
};

pub const PresetRegistry = struct {
    allocator: std.mem.Allocator,
    presets: std.StringArrayHashMapUnmanaged(Preset) = .empty,

    pub fn init(allocator: std.mem.Allocator) PresetRegistry {
        return .{ .allocator = allocator };
    }

    pub fn deinit(self: *PresetRegistry) void {
        self.presets.deinit(self.allocator);
    }

    pub fn register(self: *PresetRegistry, p: Preset) !void {
        const gop = try self.presets.getOrPut(self.allocator, p.name);
        if (gop.found_existing) {
            std.log.warn("subagent: preset '{s}' overrides built-in", .{p.name});
        }
        gop.value_ptr.* = p;
    }

    pub fn get(self: *const PresetRegistry, name: []const u8) ?Preset {
        return self.presets.get(name);
    }

    pub fn names(self: *const PresetRegistry, allocator: std.mem.Allocator) ![]const []const u8 {
        var out = try allocator.alloc([]const u8, self.presets.count());
        for (self.presets.keys(), 0..) |k, i| out[i] = k;
        return out;
    }
};
```

**Acceptance:** `zig build test` still passes; new types visible via
`@import("tools/subagent.zig").Preset`.

## Step 2 — Built-in preset catalog

**File:** `src/coding/tools/subagent.zig`

Add `registerBuiltinPresets` after the registry type. Each builder returns a
freshly allocated slice owned by the sub-agent's allocator.

Four built-ins:

| Preset | Tools | default_profile | default_role |
|---|---|---|---|
| `research` | read, ls, find, grep | `gemini` | read |
| `code-audit` | read, ls, find, grep | `gemini` | read |
| `file-ops` | read, write, edit, ls | `""` (inherit parent) | plan |
| `bash-runner` | bash, ls | `""` (inherit parent) | code |

`file-ops` and `bash-runner` use `""` as `default_profile` — empty string
signals "inherit parent's profile" in the dispatch logic (Step 4).

**Tool selection — filter from `parent_tools`, not bare factories.**
The mode driver already wires contexts: `read.toolWithCtx(&read_ctx)` carries
workspace scope; `bash.toolWithStateAndWorkspace(&bash_ctx)` carries session
state + workspace. These wired tools live in `parent_tools`. Preset builders
must FILTER by tool name from `parent_tools` rather than calling `read.tool()`
or `bash.tool()`, which are unconfigured stubs.

```zig
fn selectTools(
    alloc: std.mem.Allocator,
    parent_tools: []const at.AgentTool,
    names: []const []const u8,
) ![]at.AgentTool {
    var out = try alloc.alloc(at.AgentTool, names.len);
    var n: usize = 0;
    for (names) |want| {
        for (parent_tools) |t| {
            if (std.mem.eql(u8, t.name, want)) {
                out[n] = t;
                n += 1;
                break;
            }
        }
    }
    return alloc.realloc(out, n);
}

const research_tools = &[_][]const u8{ "read", "ls", "find", "grep" };
const file_ops_tools  = &[_][]const u8{ "read", "write", "edit", "ls" };
const bash_runner_tools = &[_][]const u8{ "bash", "ls" };

fn buildResearchTools(alloc: std.mem.Allocator, parent: []const at.AgentTool) ![]at.AgentTool {
    return selectTools(alloc, parent, research_tools);
}
// code-audit reuses buildResearchTools (same tool set, different prompt/role)
// file-ops and bash-runner follow same pattern

pub fn registerBuiltinPresets(reg: *PresetRegistry) !void {
    try reg.register(.{
        .name = "research",
        .description = "Reads files, greps for patterns, summarises findings within the workspace.",
        .default_profile = "gemini",
        .default_role = .read,
        .default_system_prompt = "You are a research assistant. Read, grep, and summarise — do not write files.",
        .build_tools = buildResearchTools,
        .safety = .{ .read_only = true },
    });
    // ... code-audit, file-ops, bash-runner
}
```

**Acceptance:** Unit test (Step 7, case 1) passes — registry contains all four
presets with correct fields.

## Step 4 — Wire preset into `runSubagent`

**File:** `src/coding/tools/subagent.zig`

### 4a — `ParsedArgs`

Add `preset: []const u8` (required field). Remove any default for profile
that assumed it was required — profile becomes optional (uses preset's
`default_profile` when omitted).

```zig
const ParsedArgs = struct {
    preset: []const u8,      // required
    profile: ?[]const u8,    // optional — falls back to preset.default_profile
    prompt: []const u8,
    timeout_ms: u64,
    max_turns: u32,
    role: ?role_mod.Role,
    system_prompt: ?[]const u8,
    ...
};
```

Parse error for missing `preset` → `error.MissingPreset` → `invalid_args`
with message `"missing required field 'preset'"`.

### 4b — Preset lookup

After parsing, look up the preset in `ctx.presets`:

```zig
const preset = ctx.presets.get(parsed.preset) orelse {
    return errorResult(allocator, .preset_not_found,
        "preset '{s}' is not registered; call list_subagent_presets to see available presets",
        .{parsed.preset});
};
```

Add `preset_not_found` to `ErrorKind`.

### 4c — Role check with hint

Use preset's `default_role` when `parsed.role` is null, then apply the
existing demotion check:

```zig
const effective_role = parsed.role orelse preset.default_role;
if (!ctx.parent_role.atLeast(effective_role)) {
    return errorResult(allocator, .role_promotion_denied,
        "preset '{s}' requires role '{s}' or higher; parent role is '{s}'",
        .{ preset.name, effective_role.toString(), ctx.parent_role.toString() });
}
```

Update `ErrorKind.role_promotion_denied.hint()` to reference the preset name
when available (thread it via the errorResult path or make hint dynamic).

### 4d — Effective profile

```zig
const effective_profile = parsed.profile orelse
    if (preset.default_profile.len > 0) preset.default_profile else ctx.parent_profile;
// Call applyProfile only when effective_profile is non-empty.
// Empty → sub-agent inherits the same settings/env defaults as the parent.
```

Add `parent_profile: []const u8` to `Ctx`. Mode drivers pass `cfg.profile orelse ""`
(the `--profile` value the parent session was started with, or empty string when the
parent used no profile). This lets `file-ops` / `bash-runner` follow the operator's
provider choice without hardcoding a profile name in the preset.

### 4e — Build sub-agent tool list

Replace the current "copy parent_tools minus subagent" logic:

```zig
const sub_tools = try preset.build_tools(allocator, ctx.parent_tools);
defer allocator.free(sub_tools);
```

### 4f — Effective system prompt

```zig
const effective_system_prompt = parsed.system_prompt orelse preset.default_system_prompt;
```

### 4g — `max_result_bytes` truncation

After the sub-agent completes and before packaging the `ToolResult`, apply
truncation when `preset.safety.max_result_bytes != null`:

```zig
if (preset.safety.max_result_bytes) |limit| {
    const result = truncate_mod.truncateHead(final_text, .{ .max_bytes = limit });
    // use result.content; if result.truncated, append a notice
```

## Step 6 — Mode driver wiring (4 files)

**Files:** `src/coding/modes/print.zig`, `proxy.zig`, `rpc.zig`, `interactive.zig`

For each mode, in the session-init block (where `subagent_ctx` is built today):

```zig
// 1. Init registry
var preset_registry = subagent_mod.PresetRegistry.init(allocator);
defer preset_registry.deinit();
try subagent_mod.registerBuiltinPresets(&preset_registry);
// SDK hook: consumers call registry.register() before this point via
// a passed-in callback or by extending the mode's init function.

// 2. Build dynamic parameters_json (once)
const subagent_params_json = try subagent_mod.buildParametersJson(
    allocator, &preset_registry);
defer allocator.free(subagent_params_json);

// 3. Wire into Ctx
var subagent_ctx = subagent_mod.Ctx{
    ...existing fields...,
    .presets = &preset_registry,
    .parameters_json_owned = subagent_params_json,
    .parent_profile = active_profile_name,
};

// 4. Add list_presets tool alongside subagent tool
const final_tools = blk: {
    const slice = try allocator.alloc(at.AgentTool, filtered_tools.len + 2);
    @memcpy(slice[0..filtered_tools.len], filtered_tools);
    slice[filtered_tools.len] = subagent_mod.toolWithCtx(&subagent_ctx);
```

## Step 7 — Tests

**File:** `src/coding/tools/subagent.zig` (existing test block)

Eight test cases matching §3.11 of the design:

1. `registerBuiltinPresets` populates `research`, `code-audit`, `file-ops`,
   `bash-runner` with correct `default_role`, `default_profile`, `safety`.
2. `buildParametersJson` enum contains exactly `["research","code-audit","file-ops","bash-runner"]`
   after `registerBuiltinPresets`.
3. Faux sub-agent: `preset:"research"` → sub-agent receives tool list with
   exactly 4 tools named read, ls, find, grep (no write/edit/bash).
4. `executeListPresets` returns a JSON array with all 4 built-ins.
5. SDK custom preset: `registry.register(.{.name="x",...})` →
   `buildParametersJson` enum includes `"x"`, dispatch invokes custom builder.
6. Collision: register `"research"` again → `warn` log fires, new definition
   wins.
7. Role promotion denied: build a `Ctx` with `parent_role=.read`, pick
   `bash-runner` (default_role=.code) → result is `role_promotion_denied`
   with hint text containing `"bash-runner"` and `"code"`.
8. `max_result_bytes`: preset with `safety.max_result_bytes = 10`, sub-agent
   produces 100-byte final_text → packaged result is truncated to ≤10 bytes.

## Order of steps

Steps 1–2 can be done together (pure type additions, no behaviour change).
Step 3 is a prerequisite for Step 6.
Step 4 is the core behaviour change — do after Step 2 so builders exist.
Step 5 is independent of Step 4.

Step 7 can be written alongside Steps 2–4 (TDD-friendly).
Step 8 is last.
