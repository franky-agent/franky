## Remaining scope

### Interactive mode support (v2) — ✅ done

The terminal loop in `src/coding/modes/interactive.zig` checks the
`restart_requested` flag between turns. When the flag is set:

1. Persist session state (transcript, settings).
2. Close any listeners / IO handles.
3. Call `restart.spawnAndExit(io)` — same mechanism as proxy mode.

The interactive mode already has a main loop structure; the change is adding a
flag poll before each turn prompt.

The `/restart` slash command was added alongside the flag check.

### `finish_task` guardrail extension — `restart: bool = false` (v2) — ✅ done

In `src/agent/guardrails/finish_task.zig`, added a `restart: bool = false`
parameter to the `finish_task` tool call. When `true`, after the termination
check passes and the final compilation succeeds, the guardrail sets the session's
`restart_requested` flag.

The `GuardrailState` struct gained a `restart_requested: ?*std.atomic.Value(bool)`
field that the mode driver wires to its session-level flag.

### Integration test — PID changes after restart (v2) — ✅ done

A multi-step test that:

1. Starts franky in proxy mode (unique port).
2. Captures the PID of the running process.
3. Issues `POST /restart`.
4. Waits for the old process to exit (polls `/health` until it fails).
5. Polls `/health` on the same port until the new process responds.
6. Verifies the new PID differs from the old PID.

Implemented as `test/restart_test.zig`.

### Watchdog / `franky --as-watchdog` (v3)

Approach D from the original design — a tiny bootstrap that spawns franky,
waits for it to exit, and re-spawns it when exit code signals "restart".

```
watchdog ──► spawn franky args...
              │
              ├── franky finishes a feature, recompiles itself
              │   └── calls finish_task with restart flag
              └── franky exits(42)
         watchdog sees exit(42) ──► spawn franky args... (new binary)
```

**Pros:**
- Absolutely clean restart — no signal-safety issues, no fd games
- Works across binary replacements (the watchdog re-reads `argv[0]` from PATH)
- Systemd / Docker already provide this pattern (`Restart=always`)

**Cons:**
- Need a wrapper binary or script (`franky-restartd` or `franky --as-watchdog`)
- Two processes in memory during the handoff
- Marginal complexity for the restart logic

### Port-conflict test (v3)
