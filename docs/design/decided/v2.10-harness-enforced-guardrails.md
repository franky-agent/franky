# To guide the Model — Harness-Enforced Guardrails

**Status:** complete — ready for implementation

---

## Overview

The harness enforces four classes of guardrails to keep the model productive and avoid runaway loops:

1. **Compilation Guard** — run build+test after N edits; inject failures as advisory hints.
2. **Stuck Detector** — detect repeated same-error patterns; inject escalating hints.
3. **Secret Scanner** — deferred to [v2.15 secret management](./v2.15-secret-management.md).
4. **Auto-Commit** — opt-in structured task completion with `finish_task` tool.

v2.11 (model-stuck-detector) is merged into this document. §3 below is the authoritative stuck-detector spec.

## §2 — Compilation Guard

Run compilation (and tests on success) after a threshold of source-code mutations. On failure, inject a structured hint as a harness-synthesised tool call.

### Q2.1 — Where does the check fire?

**Decision:** ✓ accept — the `between_turns` hook in `agent/loop.zig`. After a turn ends (tools executed, results collected) but before the next LLM call.

### Q2.2 — What counts as "an edit"?

**Decision:** ✓ accept — both `edit` and `write` tool invocations increment the mutation counter. `bash` invocations do not count. The counter resets after a successful compilation.

### Q2.3 — Which build command to run?

**Decision:** `.frank-workflow.yaml` in the workspace root defines the build workflow. Format:

```yaml
# .frank-workflow.yaml
stages:
  - name: build
    command: "zig build"
    timeout_ms: 120000

  - name: test
    command: "zig build test"
    timeout_ms: 60000
    needs: [build]          # only runs if 'build' succeeds
```

**File format rules:**
- `stages` is a list of stage objects.
- Each stage has `name` (string, unique), `command` (shell string), optional `timeout_ms` (integer, default 120 000).
- `needs` is an optional list of stage names that must succeed before this stage runs. Stages without `needs` run first (or in parallel with each other in future).
- For v1: stages are run sequentially in topological order. Parallelism is a v2 aspiration.
- If `.frank-workflow.yaml` is absent, the harness falls back to auto-detection:
  1. `build.zig` → `zig build`
  2. `Makefile` / `makefile` → `make`
  3. `Cargo.toml` → `cargo build`
  4. Otherwise → skip (no recognised build system).

The compilation guard only runs the stages needed for compilation feedback. In v1 it runs all stages in order (build → dependent test). Individual stage selection is a v2 feature.

### Q2.4 — How are failures presented to the model?

**Decision:** ✓ accept with clarification — the compiler stdout/stderr is injected as a **harness-synthesised tool call** (not a user message). The loop constructs a synthetic `tool_execution_end` event with `is_error = true`, `tool_code = "compilation_failed"`, and content:

```
The workspace no longer compiles. Here is the compiler output:

```
<stderr/stdout>
```

This is advisory. You can keep editing or fix it now.
```

The model sees a familiar tool-result shape and knows it can act on it or defer. The harness-synthesised call has a fixed `call_id` of `"harness:compilation_guard"`.

### Q2.5 — Compilation run: blocking or background?

**Decision:** ✓ accept — v1 is blocking with a 120 s timeout. On timeout, emit "compilation timed out" and let the model proceed. Background compilation is a v2 aspiration.

### Q2.6 — Threshold: fixed or configurable?

**Decision:** ✓ accept — fixed at 5 mutations for v1. Promoted to `settings.json` field `compilation_guard_threshold` once the feature stabilises.

### Q2.7 — Auto-recompile after fix?

**Decision:** ✓ accept — the harness re-runs compilation after the *next* batch of edits crosses the threshold again. No per-tool compile.

### Q2.8 — False positive: mid-refactor states

**Decision:** ✓ accept — the hint is advisory. The message says "You can keep editing or fix it now." The model decides urgency; further tool calls are never blocked.

## §3 — Stuck Detector (merged from v2.11)

Detect when the model calls the same tool with similar args and gets the same error repeatedly. Inject a structured hint after a threshold; no hard stop in v1.

### Q3.1 — What constitutes "similar"?

**Decision:** tool_name + tool_code + arg hash.

- **tool_name**: the name of the tool (e.g. `edit`, `bash`).
- **tool_code**: the sub-code from `ToolResult.tool_code` (e.g. `edit_no_match`, `bash_timeout`).
- **arg hash**: SHA-256 of the canonical (sorted-key) JSON serialisation of the tool's args. This catches the model passing the same file path with slightly different edit content.

All three must match for the counter to increment. If any differs, the counter resets.

### Q3.2/Q3.3 — Thresholds

**Decision:** Hint at **5** consecutive matching errors. No hard stop in v1 — hard-stop escalation is deferred to v2.

The counter resets on *any* successful tool call (any tool, no error).

The hint message injected into the transcript (as a harness-synthesised tool call, same pattern as §2):

```
It seems you are stuck in a loop: tool '{name}' with similar arguments has returned
error '{tool_code}' {N} times in a row. Please check whether you are using the tool
correctly or whether there is an issue with your logic before continuing.
```

`call_id` = `"harness:stuck_detector"`.

### Q3.4 — Where does the detector live?

**Decision:** Guardrails directory, hybrid hook-struct approach.

File: `src/agent/guardrails/stuck_detector.zig`

Exposes two functions wired directly into the loop's hook slots (no new vtable):

```zig
pub fn afterToolCall(
    userdata: ?*anyopaque,
    tool: *const at.AgentTool,
    call_id: []const u8,
    result: *at.ToolResult,
) void

pub fn betweenTurns(
    userdata: ?*anyopaque,
    transcript: *Transcript,
) bool   // returns true = keep going (hint injected), false = stop (never in v1)
```

The `GuardrailState` aggregate struct (in `guardrails.zig`) owns a `StuckDetector` and wires these functions into the loop `Config`.

### Q3.5 — Non-error stuck patterns

**Decision:** Defer to v2. Error-only detection covers the most common and harmful cases.

## §5 — Auto-Commit on Task Completion

### Q5.1 — How does the harness know a task is done?

**Decision:** A new `finish_task` tool (not a magic text signal).

Tool schema:

```json
{
  "type": "object",
  "required": ["commit_message", "summary"],
  "properties": {
    "commit_message": {
      "type": "string",
      "description": "Git commit message for this task's changes."
    },
    "summary": {
      "type": "string",
      "description": "What was done, what remains unknown, what to do next."
    }
  },
  "additionalProperties": false
}
```

`terminate: true` — the loop stops after this tool fires.

### Q5.2 — When does the commit happen?

**Decision:** After `finish_task` fires, the harness runs all active guardrails one final time:

1. Compilation check (must pass — blocking).
2. If compilation fails → do not commit; inject the compiler error; loop continues until the model fixes it (loop is not stopped by `terminate` in this case — treated as advisory continuation).
3. If compilation passes and `--auto-commit` is active → `git add -A && git commit -m "<commit_message>"`.

**Branch safety:** if auto-commit is enabled, the harness checks at **session start** whether the current branch is the default branch (typically `main` or `master`). If so, it creates a new branch `franky/<session-ulid>` before any edits are made. The model never commits directly to the default branch.

### Q5.3 — Branching strategy

**Decision:** Option A — one branch per session, `franky/<session-ulid>`. Sequential auto-commits on that branch. Created at session start if `--auto-commit` is active and the current branch is the default.

### Q5.4 — Opt-in mechanism

**Decision:** Off by default. Opt-in via:
1. CLI flag: `--auto-commit`
2. `settings.json`: `"auto_commit": true`
3. `.frank-workflow.yaml`: `auto_commit: true` (project-level override — see §9)

### Q5.5 — Commit message

**Decision:** Model provides the commit message via the `finish_task` tool call. Harness validates: non-empty, printable ASCII. Used verbatim.

## §6 — Event Model

### Q6.1 — How do guardrail diagnostics surface to consumers?

**Decision:** Extend `agent_error` rather than adding a new event kind.

Changes to `src/ai/errors.zig` / `src/agent/types.zig`:

```zig
pub const ErrorCode = enum {
    // Existing
    tool_runtime,
    provider_error,
    internal,
    max_turns_exceeded,
    // New — guardrail diagnostics (non-fatal when is_fatal = false)
    compilation_failed,
    stuck_pattern,
};

pub const ErrorSource = enum {
    llm,        // originated from LLM call or tool execution
    guardrail,  // originated from a harness guardrail
};

pub const ErrorDetails = struct {
    code: ErrorCode,
    source: ErrorSource = .llm,
    /// false = advisory warning; loop continues. true = loop stops.
    is_fatal: bool = true,
    message: []const u8,
    // Existing fields unchanged:
    tool_code: ?[]const u8 = null,
    provider_code: ?[]const u8 = null,
    provider_message: ?[]const u8 = null,
};
```

`AgentEvent.isTerminal()` is updated to return `true` only when `agent_error.is_fatal == true`.

Guardrail warnings use `is_fatal = false` and `source = .guardrail`. The loop emits the event and continues normally. UI consumers (TUI, web) can render guardrail warnings distinctly from fatal errors using the `source` and `is_fatal` fields.

## §7 — Integration Architecture

```
Agent (src/agent/agent.zig)
  └── guardrails: GuardrailState          // new field

    agentLoop (src/agent/loop.zig)
      │
      ├── After each tool execution (after_tool_call hook):
      │   ├── guardrails.stuck_detector.afterToolCall(tool, call_id, result)
      │   └── guardrails.compilation_guard.bumpIfMutation(tool_name)
      │
      ├── Between turns (between_turns hook):
      │   ├── if guardrails.compilation_guard.shouldCompile():
      │   │     run_build_stages() → inject harness tool result
      │   └── if guardrails.stuck_detector.betweenTurns(transcript):
      │         inject hint as harness tool result
      │
      └── On finish_task tool:
          ├── Run compilation guard (final pass, blocking)
          ├── If passes and --auto-commit:
          │     git add -A && git commit -m "<message>"
          └── Else: print summary, wait for next prompt
```

### Q7.1 — New files

**Decision:** Guardrails directory, hybrid hook-struct pattern. Files in `src/agent/guardrails/` expose `afterToolCall` and `betweenTurns` functions that the loop wires in directly — no new vtable or interface type.

```
src/agent/guardrails/
  stuck_detector.zig       — StuckDetector struct + afterToolCall + betweenTurns
  compilation_guard.zig    — CompilationGuard struct + bumpIfMutation + runBuildStages
  finish_task.zig          — finish_task tool definition + auto-commit logic
  guardrails.zig           — GuardrailState aggregate owning all of the above
```

`GuardrailState` is a struct (not a vtable). The loop's `Config` gains a `guardrails: ?*GuardrailState = null` field. When non-null, the loop delegates hook calls to it.

## §8 — Implementation Roadmap

### Phase 1 — Stuck Detector (highest impact, lowest effort)
1. `StuckDetector` in `guardrails/stuck_detector.zig`
2. Wire into `after_tool_call` + `between_turns` hooks
3. Extend `ErrorDetails` with `ErrorCode`, `ErrorSource`, `is_fatal`
4. Tests: fake provider with repeated error patterns

### Phase 2 — Compilation Guard
5. `.frank-workflow.yaml` parser
6. `CompilationGuard` in `guardrails/compilation_guard.zig`
7. Auto-detection fallback (zig/make/cargo)
8. Edit/write counter, configurable threshold
9. Tests: fake workspace with broken build

### Phase 3 — finish_task + Auto-Commit
10. `finish_task` tool in `guardrails/finish_task.zig`
11. Session-start branch creation (default-branch guard)
12. `git add -A && git commit` integration
13. Final compilation pass before commit
14. Tests: auto-commit flow with and without compilation failure

### Phase 4 (deferred)
- Secret scanner → v2.15
- Background compilation (non-blocking compile during LLM call)
- Hard-stop escalation in stuck detector (hint at 5, stop at N)
- Arg-hash similarity already in v1 (§3 Q3.1) — no separate phase needed

## §9 — Configurability

**Threshold settings:** live in `settings.json` only. `.frank-workflow.yaml` is for workflow commands and `auto_commit` only.

| Setting | Default | Location | Description |
|---------|---------|----------|-------------|
| `compilation_guard_threshold` | `5` | `settings.json` | Edit/write mutations before compilation trigger |
| `compilation_guard_timeout_ms` | `120000` | `settings.json` | Max time per build stage |
| `stuck_detector_hint_threshold` | `5` | `settings.json` | Consecutive matching errors before hint |
| `auto_commit` | `false` | `settings.json` / `.frank-workflow.yaml` / CLI `--auto-commit` | Enable auto-commit on `finish_task` |

Hard-stop threshold (`stuck_detector_stop_threshold`) is not implemented in v1; the setting is reserved for v2.
