# v2.16 — Multi-Model Code Review

**Status:** decided — ready for implementation
**Affects:** `src/coding/tools/subagent.zig`, `src/coding/skills.zig`, `src/coding/settings.zig`
**Related:** [v2.14 — Agentic Engineering](./v2.14-agentic-engineering.md)

---

## 1. Problem

`/simplify` already parallelises review across three agents, but each agent runs on the **same model** with a **different focus dimension** (reuse / quality / efficiency). The diversity comes from the prompt, not from the model.

Different models have different training distributions, different code-pattern priors, and different failure modes. A bug that one model consistently overlooks (e.g. a subtle lifetime issue that a code-completion model was trained to accept) will be caught by a model that was trained on a different corpus. Conversely, a finding that appears across three different models independently is significantly more likely to be real than a finding raised by only one.

**The gap:** there is no mechanism today to run the same review across multiple models and surface findings by cross-model confidence.

---

## 2. What already exists

| Primitive | What it provides |
|---|---|
| `subagent` tool | Spawn N isolated agents in parallel, each with its own model profile and system prompt. Already used by /simplify via Claude Code's Agent tool. |
| `code-audit` preset | Read-only workspace agent, structured finding output. Default profile `ollama-gemma4`. |
| Profile system (`settings.json`) | Named model configs with provider, API key, timeout overrides. E.g. `ollama-deepseek-flash`, `ollama-gemma4`, `claude-sonnet`. |
| franky skill files (`~/.franky/skills/`) | Markdown files injected into the system prompt. Can instruct the parent model to invoke tools in a specific pattern. |

The `subagent` tool supports a `profile` override per call, so the parent model can already spawn three subagents on three different profiles with one tool-call fan-out. v2.16 is mostly about making this pattern **discoverable, configurable, and aggregation-aware**.

---

## 3. Proposed mechanism

### 3.1 Trigger

A franky skill file (`~/.franky/skills/multimodel-review.md`) that the parent model activates on `/review`. The skill instructs the model to:

1. Collect the review target (default: `git diff HEAD`).
2. Read the configured review profiles (`settings.json` → `review.profiles`).
3. Spawn one `code-audit` subagent per profile **in a single parallel tool fan-out**.
4. Wait for all results.
5. Synthesise a unified report with per-finding confidence labels.

### 3.2 Model selection

Three sources, priority order:

1. **Invocation argument** — `/review --profiles gemini,deepseek,claude` (highest priority, per-run override).
2. **`review.profiles` in `settings.json`** — user's declared set of models for review use.
3. **Auto-discovery fallback** — if neither is set, use all profiles that have `capabilities.code_generation = true` and a configured API key, up to a max of 4.

Default value for `review_profiles` when no profiles are explicitly configured: the 3 built-in presets whose API keys are present at session start.

### 3.3 Review dimensions

Each model gets an identical "find all code issues" prompt. The diversity comes purely from the model. Aggregation is purely about finding overlap.

### 3.4 Aggregation

Each subagent returns a structured finding list (the `code-audit` preset already produces `file:line — description` output). The parent model:

1. Reads all N reports.
2. Groups findings that are semantically equivalent (same file, same concern, possibly different line references).
3. Labels each group:
   - **`[all N]`** — found by every model → high confidence, surface prominently.
   - **`[N-1 of N]`** — found by all but one → medium-high confidence.
   - **`[1 of N]`** — only one model flagged it → low confidence / disputed, still shown but clearly marked.
4. Produces a single unified report sorted by confidence descending.

The parent model does the aggregation — it already has the full context and the aggregation prompt is small.

### 3.5 Failure handling

If one subagent errors or times out, the run **does not abort** — the remaining reports are still aggregated. The unified report notes which profiles produced no output. A `min_models` setting (default `2`) aborts the whole run if fewer than that many models completed.

## 6. What's explicitly out of scope

- **Auto-fix** — v2.16 is read-only. Applying fixes is a separate `/fix` flow or a manual step. Defer it to v3 specification
- **Per-model scoring** — tracking which model has the best hit rate over time. Interesting, but requires persistent storage; deferred.
- **Cross-session learning** — using past review outcomes to tune which models to invoke. v3 territory.
- **Non-code review** — reviewing design docs, PRs, commit messages. The `code-audit` preset and `git diff` target make this code-specific. Generalisation deferred.

---

## 7. Decisions

### Q7.1 — Same prompt vs. rotated dimensions?

**Decided:** Same prompt (Option A). Model diversity already provides dimension diversity implicitly. Dimension rotation deferred to a future iteration once we have data on which models are strongest on which dimensions.

### Q7.2 — LLM aggregation vs. rule-based dedup?

**Decided:** LLM aggregation using the parent model. The parent already has the full context and the aggregation prompt is small. No rule-based fallback planned.

### Q7.3 — Skill file vs. built-in command?

**Decided:** Skill file (`~/.franky/skills/multimodel-review.md`). User-editable and keeps the core lean. Promote to built-in command if the skill's parameter surface gets complex enough to warrant a parser.

### Q7.4 — Where does `review` config live?

**Decided:** `review: { profiles, min_models, max_models, timeout_ms }` namespace in `settings.json` — avoids cluttering the top level.

### Q7.5 — What is the default review target?

**Decided:** `git diff HEAD` (staged + unstaged) — same as `/simplify`. Caller can pass an explicit path or diff via argument.

#### `applyLayer` JSON parser — add after the `tools` block

```zig
if (obj.get("review")) |rev_v| if (rev_v == .object) {
    if (rev_v.object.get("profiles")) |p| if (p == .array) {

```

### 8.4 Phase 3 — `src/coding/skills.zig` + skill file (~70 lines)

#### Builtin skill mechanism

Add at the top of `skills.zig`:

```zig
const BuiltinSkill = struct {
    name: []const u8,
    content: []const u8,  // full file content including frontmatter
};

const builtins: []const BuiltinSkill = &.{
    .{
        .name = "multimodel-review",
        .content = @embedFile("skills/multimodel-review.md"),
    },
};
```

Modify `loadAll` to append builtins after the existing roots loop — only if the name was not already loaded by a higher-precedence source (the `seen` map already tracks this):

```zig
// After the existing roots loop:
for (builtins) |b| {
    if (seen.contains(b.name)) continue;
    const skill = parseSkill(allocator, "(builtin)", b.content) catch |err| {
        ai_log.log(.warn, "skills", "builtin.skip", "name={s} err={s}", .{ b.name, @errorName(err) });
        continue;
    };
    try out.append(allocator, skill);
    try seen.put(skill.meta.name, {});
}
```

This means: user placing `~/.franky/skills/multimodel-review.md` overrides the builtin. Workspace `skills/multimodel-review.md` also overrides it. The builtin only fires when no file exists in any root.

`parseSkill` in the builtin path gets `"(builtin)"` as the path argument. The `Skill.source_path` field is used only for logging, so a synthetic path is fine.

#### `src/coding/skills/multimodel-review.md` — new file

```markdown
---
name: multimodel-review
description: |
  Runs an identical code-review prompt across N models in parallel and
  aggregates findings by cross-model confidence. TRIGGER when: user
  invokes /review --multimodel.
auto_apply: []
---

# Multi-Model Code Review

Activate on `/review --multimodel` or multimodel is enabled in settings.json

## Process

1. **Collect the diff.**
   Default: `git diff HEAD` (staged + unstaged). Accept an explicit path
   or revision if the user provided one (e.g. `/review src/agent/loop.zig`).

2. **Select profiles.**
   Priority order:
   a. `--profiles A,B,C` in the invocation argument — split on comma, use as-is.
   b. The `## Review configuration` block in this system prompt (populated from
      `settings.json → review.profiles` at session start).
   c. Auto-discovery fallback: call `list_subagent_presets` and select up to
      `max_models` profiles (default 4) whose names suggest code capability.

3. **Fan out in parallel — single tool-call batch.**
   Spawn one `code-audit` subagent per selected profile. All subagents receive
   the identical prompt:

   > "Review the following diff for correctness, security, and performance
   > issues. Report each finding on its own line as:
   > `file:line — description`
   > Include only real issues; omit style nits and formatting comments."
   > [paste full diff]

   Use `timeout_ms` from the Review configuration (default 180000).
   Use `preset: "code-audit"` and `profile: "<profile-name>"` for each call.

4. **Check quorum.**
   Count how many subagents returned output. If fewer than `min_models`
   (default 2), abort: "Multi-model review aborted: only N/M models
   responded (minimum: min_models required)."

5. **Aggregate.**
   Group findings semantically — same file, same concern, possibly different
   line numbers. Label each group:
   - `[all N]`  — every model flagged it → high confidence
   - `[N-1 of N]` — all but one → medium confidence
   - `[1 of N]`  — one model only → low confidence / disputed

6. **Render the unified report** (see Output format below).

## Output format

```
## Multi-Model Review  (N models · M findings)

### High confidence  [found by N/N]
- file:line — description

### Medium confidence  [found by N-1/N]
- file:line — description

### Low confidence  [found by 1/N]
- file:line — description  (model-name only)

### Profiles that timed out or errored
- profile-name: reason
```

Sort sections high → low confidence. Within each section, sort by file path.
```

### 8.5 Phase 4 — `src/coding/modes/print.zig` (~40 lines)

Two changes: populate `cfg.review_profiles_csv`, then use it in `buildSystemPromptIo`.

#### Populate `review_profiles_csv` in `runPrint` (and equivalent session-init in other modes)

After settings are loaded, before calling `buildSystemPromptIo`:

```zig
if (settings.review_profiles.len > 0) {
    // Join with ", " for readability in the injected block.
    var buf: std.ArrayList(u8) = .empty;
    defer buf.deinit(allocator);
    for (settings.review_profiles, 0..) |p, i| {
        if (i > 0) try buf.appendSlice(allocator, ", ");
        try buf.appendSlice(allocator, p);
    }
    cfg.review_profiles_csv = try buf.toOwnedSlice(allocator);
}
```

The same pattern applies in the session-init path of `interactive.zig`, `proxy.zig`, and `rpc.zig`.

#### Inject `## Review configuration` block in `buildSystemPromptIo`

After the existing `with_skills` block is built, check if the multimodel-review skill was activated AND `cfg.review_profiles_csv` is set:

```zig
// After the skills section is built:
if (cfg.review_profiles_csv) |csv| {
    // Only inject if multimodel-review skill is actually active.
    const skill_active = for (active.items) |idx| {
        if (std.mem.eql(u8, loaded.items[idx].meta.name, "multimodel-review")) break true;
    } else false;

    if (skill_active) {
        const block = try std.fmt.allocPrint(allocator,
            "\n\n## Review configuration\n" ++
            "profiles: {s}\n" ++
            "min_models: {d}\n" ++
            "max_models: {d}\n" ++
            "timeout_ms: {d}\n",
            .{ csv, settings_resolved.review_min_models,
               settings_resolved.review_max_models,
               settings_resolved.review_timeout_ms });
        defer allocator.free(block);
        // append to with_skills
        ...
    }
}
```

Because `buildSystemPromptIo` currently only receives `cfg`, not `settings`, the `min_models` / `max_models` / `timeout_ms` need to be plumbed through `cli.Config` as well (three more optional fields, same pattern as `review_profiles_csv`). Alternatively, collapse all four into a single pre-rendered block built by the mode driver:

```zig
// In runPrint, after loading settings:
if (settings.review_profiles.len > 0) {
    cfg.review_config_block = try std.fmt.allocPrint(allocator,
        "## Review configuration\n" ++
        "profiles: {s}\n" ++
        "min_models: {d}\nmax_models: {d}\ntimeout_ms: {d}",
        .{ csv, settings.review_min_models,
           settings.review_max_models, settings.review_timeout_ms });
}
```

Then in `buildSystemPromptIo`, if `cfg.review_config_block != null` and the skill is active, append it after the `## Active skills` section. This keeps all four values out of `cli.Config` (single opaque string) and `buildSystemPromptIo` stays unaware of review semantics.

**Preferred:** Use a single `review_config_block: ?[]const u8 = null` field on `cli.Config`. Replace the separate `review_profiles_csv` field — it's the same concept, more complete.
