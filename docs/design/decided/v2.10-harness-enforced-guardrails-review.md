# Review Findings — v2.10 Implementation Audit

> **Context:** This section documents findings from the detailed implementation review of
> [v2.10-harness-enforced-guardrails.md](./decided/v2.10-harness-enforced-guardrails.md)
> against the code in `src/agent/guardrails/` and all four mode drivers. The design doc
> is marked `complete — ready for implementation`; the code was reviewed for conflicts,
> gaps, and bugs. Review date: 2025-07-17.

## 🔴 Issue 1 — RPC mode has zero guardrails (missing implementation)

**File:** `src/coding/modes/rpc.zig`

**What:** The RPC mode driver does not create a `GuardrailState`, does not set
`loop.Config.guardrails`, does not call `setupAutoCommitBranch`, and does not add
`finish_task` to its tool list. All three other modes (print, interactive, proxy)
wire guardrails identically; RPC is the outlier.

| Mode | GuardrailState.init | .guardrails = &… | finishTaskTool | setupAutoCommitBranch |
|------|---------------------|-------------------|----------------|-----------------------|
| print.zig      | ✅ line 418 | ✅ line 551 | ✅ line 431 | ✅ line 424 |
| interactive.zig | ✅ line 1791 | ✅ line 1094 | ✅ line 1861 | ✅ line 1797 |
| proxy.zig      | ✅ line 558  | ✅ line 2002 | ✅ line 686  | ✅ line 564 |
| **rpc.zig**    | ❌ missing   | ❌ missing   | ❌ missing   | ❌ missing |

**Impact:** A runaway model in RPC mode gets no stuck-detection, no compilation guard,
and no auto-commit. The model can burn turns on repeated identical errors indefinitely.

**Fix:** Copy the pattern from print/interactive/proxy — create a `GuardrailState`
field on the RPC `Session`, init it with the workspace dir, call
`setupAutoCommitBranch`, add `finishTaskTool()` to the tools slice, and wire
`&session.guardrail_state` into `loop.Config.guardrails`.

## 🔴 Issue 2 — Stuck detector `resetState` leaks `pending_hint` (bug)

**File:** `src/agent/guardrails/stuck_detector.zig`, lines 214–221

**What:** `resetState()` clears `consecutive_count` and `last_*` fields on a
successful tool call, but never clears `pending_hint`, `pending_tool_name`, or
`pending_tool_code`. When the model hits the hint threshold (5 consecutive errors)
and the *next* call succeeds, `resetState` fires, `consecutive_count` goes to 0,
but `pending_hint` stays `true`. On the following `betweenTurns`, a stale hint
about "5 times in a row" is injected into the transcript even though the model
has successfully recovered.

```zig
fn resetState(self: *StuckDetector) void {
    self.consecutive_count = 0;
    if (self.last_tool_name) |s| self.allocator.free(s);
    if (self.last_tool_code) |s| self.allocator.free(s);
    self.last_tool_name = null;
    self.last_tool_code = null;
    self.last_arg_hash = std.mem.zeroes([32]u8);
    // BUG: pending_hint not cleared
    // BUG: pending_tool_name / pending_tool_code not freed
}
```

**Impact:** False-positive stuck hints injected into the transcript after the model
has already recovered. Can confuse the model into changing strategy unnecessarily.

**Fix:** Add `self.pending_hint = false;` and free the pending strings.

## 🔴 Issue 3 — Event model `ErrorCode` variants are dead code

**Files:** `src/ai/errors.zig` (lines 65–66, 82–87, 155–156)
`src/agent/guardrails/compilation_guard.zig` (line 195)
`src/agent/guardrails/stuck_detector.zig` (line 188)

**What:** The design doc §6.1 defines:

- `ErrorCode.compilation_failed`
- `ErrorCode.stuck_pattern`
- `ErrorSource.guardrail`
- `is_fatal: bool = true` (on `ErrorDetails`)

The spec says guardrail warnings use `is_fatal = false` and `source = .guardrail`.

All of these exist in the type system but are **never populated at runtime**. The
guardrails bypass `agent_error` entirely and emit harness-synthesised
`tool_execution_start` / `tool_execution_end` events with `is_error=true` and a
bare string `tool_code`:

- `compilation_guard.zig:195` → `tool_code = "compilation_failed"`
- `stuck_detector.zig:188` → `tool_code = "stuck_pattern"`

These string values match the enum variant names, but the typed enum values,
`ErrorSource.guardrail`, and `is_fatal` are never set. The `fromToolResult`
helper in `errors.zig` maps everything to `.tool_runtime`.

**Decide:**

Option A — Update the guardrails to emit real `agent_error` events (typed enum,
`source = .guardrail`, `is_fatal = false`) alongside the tool-execution events.
This matches the spec §6.1 exactly.

Option B — Update the spec to say the ErrorCode/ErrorSource/is_fatal additions
are for "future fatal escalation" (v2) and the v1 path is tool-result injection
only. The current state (type system says one thing, runtime does another) is
misleading.

## 🟡 Issue 4 — `setupAutoCommitBranch` uses timestamp instead of ULID

**File:** `src/agent/guardrails/guardrails.zig`, lines 93–94

**Spec says:** `franky/<session-ulid>` (Q5.3)

**Implementation:** `franky/{millisecond_timestamp}`

```zig
const ts: u64 = @intCast(ai.stream.nowMillis());
const branch_name = std.fmt.allocPrint(allocator, "franky/{d}", .{ts}) catch return;
```

**Impact:** Two sessions started in the same millisecond share the same branch
name. `git checkout -b` fails silently (the error is warn-logged and swallowed).
No `session_ulid` concept exists on `GuardrailState`.

**Fix options:**

1. Append `getpid()`: `"franky/{d}-{d}"` with timestamp + pid.
2. Accept a `session_id` parameter in `GuardrailState.Config` and use that
   instead of computing a timestamp.
3. Use a short random suffix (4 hex chars) to avoid collisions.

## 🟡 Issue 5 — `finish_task` clears state on compilation failure, preventing easy retry

**File:** `src/agent/guardrails/guardrails.zig`, lines 142–144

```zig
if (compile_failed) {
    self.finish_task_state.reset();  // clears commit_message and summary
    return true;
}
```

**What:** When the final compilation guard fails (Q5.2), the task state is reset:
`triggered` becomes `false`, `commit_message` and `summary` are freed and set
to `null`. The model must call `finish_task` again with a fresh message and
summary.

**Impact:** Works correctly but discards the user's original commit message.
The model receives no guidance on what it previously committed to say — it may
produce a different message on retry.

**Suggestion:** Keep `commit_message` across the retry, or inject the previous
commit message in the compilation-failure hint so the model can re-use it.

## 🟡 Issue 6 — RPC mode's `loop.Config` omits `.guardrails` entirely

**File:** `src/coding/modes/rpc.zig`, lines 625–653

**What:** Even if someone adds a `GuardrailState` to the RPC `Session`, the
`loop.Config` literal at line 634 explicitly sets many fields but does not set
`.guardrails` at all. The loop.Config.guardrails default is `null`, so the
state would be created but never wired into the loop hooks.

**Fix:** Add `.guardrails = &session.guardrail_state,` to the config literal
once the GuardrailState field exists (see Issue 1).

## 🟡 Issue 7 — `GuardrailState.Config` defaults are never overridden from settings/CLI

**Files:** All three wired modes:
- `src/coding/modes/print.zig` line 418
- `src/coding/modes/interactive.zig` line 1791
- `src/coding/modes/proxy.zig` line 558

**Spec says (§9):** Settings like `auto_commit`, `compilation_guard_threshold`,
`stuck_detector_hint_threshold` live in `settings.json` and can be overridden
via CLI flags.

**Implementation:** All three modes create `GuardrailState` with default
`Config{}`:

```zig
try agent.guardrails.GuardrailState.init(allocator, .{ .workspace_dir = wr_dir }, io);
```

No `auto_commit` flag is passed from CLI args or settings. The `auto_commit`
default is `false` — correct per spec's "off by default" — but there is no code
path to enable it other than editing the source.

**Impact:** The `--auto-commit` CLI flag exists in the spec but doesn't work;
neither does `"auto_commit": true` in settings.json or `.frank-workflow.yaml`.
The settings plumbing was deferred (the §9 config table says the settings exist
but implementation is incomplete).

## 🟢 Issue 8 — Agent SDK class doesn't wire guardrails

**File:** `src/agent/agent.zig`, lines 483–501

**What:** The `Agent` class builds its own `loop.Config` in `workerFn` but never
sets `.guardrails`. The `loop.Config.guardrails` field exists (line 289 of
loop.zig), but `agent.zig` doesn't expose it — no `guardrails` parameter on
`Agent.Config`, no assignment in the config literal.

**Context:** This is semi-intentional — the mode driver layer creates and owns
the `GuardrailState`, not the generic `Agent` class. Mode drivers call
`agentLoop` directly (print, proxy) or pass their own `Config` to the loop
(interactive passes `&session.guardrail_state` via `loop.Config.guardrails`).
RPC goes through `agentLoop` directly too. Only the `Agent` class (which wraps
agentLoop with stateful subscriber management) misses the field.

**Decision:** If SDK consumers should get guardrails, add a `guardrails` field
to `Agent.Config` and thread it into `workerFn`'s `loop.Config`. If guardrails
are a mode-level concern, leave as-is and document that SDK users must set
`loop.Config.guardrails` themselves.

## 🟢 Issue 9 — Spec signatures vs. actual signatures diverge

**File:** `docs/design/decided/v2.10-harness-enforced-guardrails.md`, §3.4
`src/agent/guardrails/stuck_detector.zig`, line 148
`src/agent/guardrails/compilation_guard.zig`, line 159

**What:** The spec (§3.4) shows these function signatures as the loop hook
shape:

```zig
pub fn afterToolCall(userdata: ?*anyopaque, tool: *const at.AgentTool, call_id: []const u8, result: *at.ToolResult) void
pub fn betweenTurns(userdata: ?*anyopaque, transcript: *Transcript) bool
```

The actual code uses `*StuckDetector` / `*CompilationGuard` as `self` and takes
`allocator`, `io`, and `out` parameters too. This is because the spec describes
the *loop hook signature* but the guardrails expose *lower-level methods* that
`GuardrailState.betweenTurns` calls internally.

**Impact:** No runtime bug, but the doc comments on the guardrail structs
reference the spec's hook-shaped signatures, which is misleading for anyone
reading those files in isolation.

**Fix:** Update the doc comments on `stuck_detector.zig:148` and
`compilation_guard.zig:159` to accurately describe the method as a helper
called by `GuardrailState.betweenTurns`, not a loop hook.

## 🟢 Issue 10 — `pending_*` fields in stuck detector leak on pattern-change + reset

**File:** `src/agent/guardrails/stuck_detector.zig`, lines 30–33 vs 214–221

**What:** The `StuckDetector` struct owns three `pending_*` fields that are
freed in `deinit()` but never freed in `resetState()`:

- `pending_tool_name`
- `pending_tool_code`
- `pending_hint` (just a bool, no leak)

When the model triggers a stuck hint (sets `pending_hint = true`, allocates
`pending_tool_name` / `pending_tool_code` at lines 132–142), then makes a
successful call, `resetState()` clears `last_*` fields but leaves `pending_*`
allocated. On the next error cycle, new values overwrite the pointers in lines
132–142 via `if (self.pending_tool_name) |s| self.allocator.free(s);` — so in
practice the leak is **at most one allocation per lifetime**. The `deinit()`
then frees whatever is left. This is low-severity: a single leaked allocation
that deinit cleans up.

**But** the `pending_hint` not being cleared (Issue 2) is the real problem here;
the memory leak is a secondary symptom of the same root cause.
