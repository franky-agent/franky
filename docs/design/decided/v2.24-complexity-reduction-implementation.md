### 3.3 `src/coding/config/cli.zig:parse` (CCN 159 → target ≤ 30)

**Status:** ✅ Complete (2025-07-16)

**Extracted:** `dispatchFlag`, `applyBoolFlag`, `applyValuedFlag`, `takeValue`.
`parse` now separates flag iteration from flag dispatch. Boolean flags are handled by `applyBoolFlag`, valued flags (string, int, enum) by `applyValuedFlag`. The main function is reduced to iteration + positional handling.

`resolveProvider` now sequences configuration loading, credential resolution, provider selection, and per-provider config building as discrete steps. A `ResolvedCredentials` struct carries the resolved API keys and availability flags.

**Risk:** Low. Provider resolution is already conceptually layered.

### 3.5 `src/coding/modes/print.zig:resolveProviderIo` (CCN 98 → target ≤ 25)

**Status:** ✅ Complete (2025-07-16)

**Extracted:** `resolvePrintSettings`, `applyPrintThinkingOverride`, `loadPrintAuthState`, `loadPrintModelsExtras`, `resolvePrintCredentials`, `choosePrintProvider`, `buildPrintFauxConfig`, `buildPrintAnthropicConfig`, `buildPrintOpenaiConfig`, `buildPrintGatewayConfig`, `buildPrintGeminiConfig`.
`resolveProviderIo` now mirrors the structure of `config.zig:resolveProvider` but retains print-specific IO handling (nullable `?std.Io`, `std.process.Environ` instead of `Environ.Map`). Both share the same `ResolvedCredentials`-like struct and per-provider builder pattern.

**Risk:** Medium. Print mode has subtle differences in IO handling; preserved by keeping the helpers print-specific rather than creating a shared abstraction.

## 4. Phase 2 — AI provider refactoring (target: 5 files, avg CCN 17.5 → ≤ 12)

The AI provider files are consistently complex, with `chat()` / `handle()` functions that encode the entire provider protocol in one function.

### 4.1 Extract protocol layers

Each provider file (`anthropic.zig`, `google_gemini.zig`, `openai_chat.zig`, `openai_responses.zig`) follows the same pattern. For each:

1. Extract `buildRequestBody` — constructs the JSON request payload (CCN contributor: model-config switch, tool definitions, system prompt).
2. Extract `parseResponseBody` — parses the JSON response body (CCN contributor: error handling, content-block extraction, usage stats).
3. Extract `handleStreamEvent` — SSE event handler for streaming (CCN contributor: event-type switch, delta accumulation).
4. The main `handle()` / `chat()` function becomes: build request → send → parse response (or handle stream).

**Target functions:**

| File | Function | Current CCN | Target CCN |
|---|---|---|---|
| `src/ai/providers/anthropic.zig` | `handle` | 72 | ≤ 20 |
| `src/ai/providers/openai_responses.zig` | `handle` | 56 | ≤ 20 |
| `src/ai/providers/openai_responses.zig` | `appendInputItem` | 52 | ≤ 15 |
| `src/ai/providers/google_gemini.zig` | `handle` | 55 | ≤ 20 |
| `src/ai/providers/google_gemini.zig` | `appendContent` | 51 | ≤ 15 |
| `src/ai/providers/openai_chat.zig` | `chat` | 41 | ≤ 20 |

### 4.2 Common protocol template

After extraction, the provider files will converge on a common shape:

```zig
pub fn handle(self: *Provider, req: *Request, ctx: *Context) !Response {
    const body = try self.buildRequestBody(req, ctx);
    const response = try self.sendRequest(body, ctx);
    return self.parseResponse(response, ctx);
}
```

This makes adding a new provider (e.g., DeepSeek, Mistral) a matter of implementing 3 functions instead of writing one 400-line function.

## 5. Phase 3 — Agent/proxy refactoring (target: 2 files, CCN ≤ 30)

### 5.1 `src/agent/proxy.zig:encodeEventJson` (CCN 103 → target ≤ 25)

**Current:** Encodes all possible proxy events (tool_calls, tool_results, errors, streaming chunks, etc.) into JSON lines, all in one function.

**Plan:**
1. Extract per-event-type encoders: `encodeToolCallEvent`, `encodeToolResultEvent`, `encodeStreamChunk`, `encodeErrorEvent`.
2. Define a formal `ProxyEvent` tagged union that the encoder takes as input.
3. `encodeEventJson` becomes a dispatch over the event tag.

**Risk:** Medium. The proxy protocol has strict ordering and timing requirements. Must ensure event types are not missed in the refactoring.

### 5.2 `src/coding/modes/proxy.zig:initSession` (CCN 68 → target ≤ 25)

**Current:** Initializes a proxy session — config loading, state setup, connection handling, all in one function.

**Plan:**
1. Extract `loadProxyConfig` — config loading for proxy mode.
2. Extract `setupConnection` — IO and listener setup.
3. Extract `initProxyState` — state initialization.
4. `initSession` sequences these three.

**Risk:** Low. Session init is already conceptually layered.

## 7. Phase 5 — CI gate (1 day)

After Phases 0–4 are complete, add the CI gate to prevent regression.

### 7.1 Install lizard in CI

```yaml
- name: Install lizard
  run: pip install lizard
```

### 7.2 Complexity check step

```yaml
- name: Complexity check
  run: |
    lizard --csv -l zig -C 15 src/ | awk -F',' '$2 > 30 {print "CCN="$2" "$5" ("$4")"; fail=1} END {exit fail+0}'
  if: github.event_name == 'pull_request'
```

This is a simpler check than the full baseline-comparison design in the open doc — it fails if any function exceeds CCN 30. As the codebase improves, the threshold can be tightened.

### 7.3 Baseline capture (periodic)

```yaml
- name: Capture complexity baseline
  run: |
    lizard --csv -l zig src/ > baselines/complexity/latest.csv
    git add baselines/complexity/latest.csv
  if: github.ref == 'refs/heads/main'
```

### 7.4 Per-PR diff (optional, Phase 5b)

After the baseline exists, add a step that diffs the PR's lizard output against the baseline and comments on the PR with any regressions.

## 10. Measuring success

After each phase, re-run:

```sh
lizard --csv -l zig src/ > /tmp/lizard-new.csv
python3 -c "
import csv
ccns = [int(r[1]) for r in csv.reader(open('/tmp/lizard-new.csv')) if r]
print(f'Functions: {len(ccns)}')
print(f'Avg CCN: {sum(ccns)/len(ccns):.2f}')
print(f'CCN > 30: {sum(1 for c in ccns if c > 30)} ({sum(1 for c in ccns if c > 30)/len(ccns)*100:.1f}%)')
print(f'CCN > 15: {sum(1 for c in ccns if c > 15)} ({sum(1 for c in ccns if c > 15)/len(ccns)*100:.1f}%)')
print(f'Max CCN: {max(ccns)}')
"
```

The existing `lizard.xml` and `lizard_metrics_report.md` serve as the baseline for comparison.
