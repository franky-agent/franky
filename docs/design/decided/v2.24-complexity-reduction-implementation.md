## 2. Phase 0 ‚Äî Immediate fixes (high-value, low-risk extractions)

These functions have the highest CCN but are **isolated** ‚Äî they don't share complex internal state with neighboring code. They can be decomposed with simple Extract Method refactorings.

### 2.1 `src/coding/config/settings.zig:applyLayer` (CCN 96 ‚Üí target ‚â§ 30)

**Current:** One `applyLayer` function handles all setting types via a giant switch.

**Plan:**
1. Extract per-type appliers: `applyStringField`, `applyIntField`, `applyBoolField`, `applyArrayField`, `applyObjectField`.
2. The main `applyLayer` becomes a dispatch table mapping field-type ‚Üí applier function.
3. Each applier is independently testable.

**Risk:** Low. Settings have no complex inter-field dependencies.

### 2.2 `src/coding/config/profiles.zig:applyToCfg` (CCN 74 ‚Üí target ‚â§ 25)

**Current:** Applies profile overlays to config with inline enum/union matching for every config section.

**Plan:**
1. Extract `applyProvider`, `applyTools`, `applyPermissions`, `applyGuardrails` section handlers.
2. Use the same dispatch-table pattern as `applyLayer`.

**Risk:** Low. This is a straightforward mapping function called by `ConfigResolver` (v2.22).

### 2.3 `src/ai/stream.zig:snapshotJson` (CCN 76 ‚Üí target ‚â§ 20)

**Current:** Snapshot serialization with inline switch over all possible event/state types.

**Plan:**
1. Extract per-event-type serializers: `snapshotDelta`, `snapshotDone`, `snapshotError`, etc.
2. The main function becomes a type-dispatch that delegates to the correct serializer.

**Risk:** Low. Pure data transformation, no side effects.

### 2.4 `src/coding/model_catalog/render.zig:render` (CCN 68 ‚Üí target ‚â§ 20)

**Current:** Single `render` function with a massive switch over model provider types.

**Plan:**
1. Extract per-provider renderers: `renderOpenAi`, `renderAnthropic`, `renderGemini`, `renderOllama`, etc.
2. Main `render` selects the correct renderer by provider tag.

**Risk:** Low. Pure formatting code.

### 2.5 `src/coding/diagnostics.zig:toJson` (CCN 58 ‚Üí target ‚â§ 20)

**Current:** Inline JSON serialization for every diagnostic variant.

**Plan:**
1. Extract per-type serializers matching the diagnostic union variants.
2. Main `toJson` dispatches by type tag.

**Risk:** Low. Pure data transformation.

## 3. Phase 1 ‚Äî Structural decompositions (moderate effort, high impact)

These functions are entangled with their enclosing module's control flow but can be decomposed with well-defined interfaces.

### 3.1 `src/agent/loop.zig:runTurn` (CCN 92 ‚Üí target ‚â§ 25)

**Current:** The agent loop's main turn function ‚Äî handles tool dispatch, guardrail checks, streaming, and error recovery in one 247-NCSS function.

**Plan:**
1. Extract `prepareTurn` ‚Äî allocates session context, loads state from persistence.
2. Extract `executeTools` ‚Äî runs the tool-call sub-loop (already partially extracted; finish the job).
3. Extract `handleTurnError` ‚Äî error classification and recovery logic.
4. Extract `finalizeTurn` ‚Äî post-turn bookkeeping (append to transcript, update persistence).
5. `runTurn` becomes: `prepareTurn ‚Üí executeTools ‚Üí finalizeTurn` with error handling around `executeTools`.

**Risk:** Medium. `runTurn` shares in-memory state (`LoopState`, transcript buffer) with many fields. Need to pass explicit structs rather than relying on closure capture.

### 3.2 `src/coding/modes/interactive.zig:runInteractive` (CCN 171 ‚Üí target ‚â§ 35)

**Current:** The interactive REPL ‚Äî handles stdin reading, agent invocations, streaming output, /slash commands, restart logic, all in one function.

This is the **single most complex function** in the project. It needs a multi-step decomposition.

**Plan:**
1. Extract `readLine` ‚Äî stdin reading with line-editor integration (already exists as `editor.feedKey`; wire through).
2. Extract `dispatchSlashCommand` ‚Äî /slash command parsing and dispatch (already partially exists; consolidate).
3. Extract `runTurnWithOutput` ‚Äî agent turn + streaming output rendering.
4. Extract `handleRestart` ‚Äî restart-requested flag check and spawn sequence.
5. `runInteractive` becomes a top-level loop calling these helpers.

**Risk:** High. This is the most complex function; every extraction carries risk of breaking the REPL flow. Mitigation: write integration tests that simulate stdin sequences before refactoring.

### 3.3 `src/coding/config/cli.zig:parse` (CCN 159 ‚Üí target ‚â§ 30)

**Current:** CLI argument parser with inline flag handling for dozens of flags.

**Plan:**
1. Define a flag registry: `const Flag = struct { name: []const u8, short: ?u8, kind: enum { bool, string, int, repeated_string }, handler: *const fn(...) void }`.
2. Build a static array of all flags with their handlers.
3. `parse` iterates over `args`, looks up the flag in the registry, and calls the handler.
4. This eliminates the giant switch and makes adding a flag a one-line addition.

**Risk:** Medium. The flag handler signatures vary (some flags set fields, some call into ConfigResolver). Need a uniform handler interface.

### 3.4 `src/coding/config.zig:resolveProvider` (CCN 89 ‚Üí target ‚â§ 25)

**Current:** Provider resolution ‚Äî selects the correct provider config given a model name, with inline handling for each provider type.

**Plan:**
1. Extract `matchProviderByModel` ‚Äî model-name pattern matching.
2. Extract `resolveProviderConfig` ‚Äî per-provider config resolution.
3. Extract `applyProviderOverrides` ‚Äî env-var and profile overrides.
4. `resolveProvider` orchestrates these three in sequence.

**Risk:** Low. Provider resolution is already conceptually layered; the extraction follows the existing mental model.

### 3.5 `src/coding/modes/print.zig:resolveProviderIo` (CCN 98 ‚Üí target ‚â§ 25)

**Current:** The print-mode equivalent of provider resolution, duplicated from `config.zig:resolveProvider` with print-specific IO handling mixed in.

**Plan:**
1. Extract the common provider-resolution logic into `config.zig` (or a shared helper).
2. `resolveProviderIo` calls the common resolver, then adds print-specific IO setup.
3. This also advances v2.22 (config simplification) by reducing duplication.

**Risk:** Medium. Print mode has subtle differences in how it handles IO (no terminal, stdout/stderr to files). Must not break CI/tests.

## 4. Phase 2 ‚Äî AI provider refactoring (target: 5 files, avg CCN 17.5 ‚Üí ‚â§ 12)

The AI provider files are consistently complex, with `chat()` / `handle()` functions that encode the entire provider protocol in one function.

### 4.1 Extract protocol layers

Each provider file (`anthropic.zig`, `google_gemini.zig`, `openai_chat.zig`, `openai_responses.zig`) follows the same pattern. For each:

1. Extract `buildRequestBody` ‚Äî constructs the JSON request payload (CCN contributor: model-config switch, tool definitions, system prompt).
2. Extract `parseResponseBody` ‚Äî parses the JSON response body (CCN contributor: error handling, content-block extraction, usage stats).
3. Extract `handleStreamEvent` ‚Äî SSE event handler for streaming (CCN contributor: event-type switch, delta accumulation).
4. The main `handle()` / `chat()` function becomes: build request ‚Üí send ‚Üí parse response (or handle stream).

**Target functions:**

| File | Function | Current CCN | Target CCN |
|---|---|---|---|
| `src/ai/providers/anthropic.zig` | `handle` | 72 | ‚â§ 20 |
| `src/ai/providers/openai_responses.zig` | `handle` | 56 | ‚â§ 20 |
| `src/ai/providers/openai_responses.zig` | `appendInputItem` | 52 | ‚â§ 15 |
| `src/ai/providers/google_gemini.zig` | `handle` | 55 | ‚â§ 20 |
| `src/ai/providers/google_gemini.zig` | `appendContent` | 51 | ‚â§ 15 |
| `src/ai/providers/openai_chat.zig` | `chat` | 41 | ‚â§ 20 |

### 4.2 Common protocol template

After extraction, the provider files will converge on a common shape:

```zig
pub fn handle(self: *Provider, req: *Request, ctx: *Context) !Response {
    const body = try self.buildRequestBody(req, ctx);
    const response = try self.sendRequest(body, ctx);
    return self.parseResponse(response, ctx);
}
```

This makes adding a new provider (e.g., DeepSeek, Mistral) a matter of implementing 3 functions instead of writing one 400-line function.

## 5. Phase 3 ‚Äî Agent/proxy refactoring (target: 2 files, CCN ‚â§ 30)

### 5.1 `src/agent/proxy.zig:encodeEventJson` (CCN 103 ‚Üí target ‚â§ 25)

**Current:** Encodes all possible proxy events (tool_calls, tool_results, errors, streaming chunks, etc.) into JSON lines, all in one function.

**Plan:**
1. Extract per-event-type encoders: `encodeToolCallEvent`, `encodeToolResultEvent`, `encodeStreamChunk`, `encodeErrorEvent`.
2. Define a formal `ProxyEvent` tagged union that the encoder takes as input.
3. `encodeEventJson` becomes a dispatch over the event tag.

**Risk:** Medium. The proxy protocol has strict ordering and timing requirements. Must ensure event types are not missed in the refactoring.

### 5.2 `src/coding/modes/proxy.zig:initSession` (CCN 68 ‚Üí target ‚â§ 25)

**Current:** Initializes a proxy session ‚Äî config loading, state setup, connection handling, all in one function.

**Plan:**
1. Extract `loadProxyConfig` ‚Äî config loading for proxy mode.
2. Extract `setupConnection` ‚Äî IO and listener setup.
3. Extract `initProxyState` ‚Äî state initialization.
4. `initSession` sequences these three.

**Risk:** Low. Session init is already conceptually layered.

## 7. Phase 5 ‚Äî CI gate (1 day)

After Phases 0‚Äì4 are complete, add the CI gate to prevent regression.

### 7.1 Install lizard in CI

```yaml
- name: Install lizard
  run: pip install lizard
```

### 7.2 Complexity check step

```yaml
- name: Complexity check
  run: |
    lizard --csv -l zig -C 15 src/ | awk -F',' '$2 > 30 {print "CCN="$2" "$5" ("$4")"; fail=1} END {exit fail+0}'
  if: github.event_name == 'pull_request'
```

This is a simpler check than the full baseline-comparison design in the open doc ‚Äî it fails if any function exceeds CCN 30. As the codebase improves, the threshold can be tightened.

### 7.3 Baseline capture (periodic)

```yaml
- name: Capture complexity baseline
  run: |
    lizard --csv -l zig src/ > baselines/complexity/latest.csv
    git add baselines/complexity/latest.csv
  if: github.ref == 'refs/heads/main'
```

### 7.4 Per-PR diff (optional, Phase 5b)

After the baseline exists, add a step that diffs the PR's lizard output against the baseline and comments on the PR with any regressions.

## 10. Measuring success

After each phase, re-run:

```sh
lizard --csv -l zig src/ > /tmp/lizard-new.csv
python3 -c "
import csv
ccns = [int(r[1]) for r in csv.reader(open('/tmp/lizard-new.csv')) if r]
print(f'Functions: {len(ccns)}')
print(f'Avg CCN: {sum(ccns)/len(ccns):.2f}')
print(f'CCN > 30: {sum(1 for c in ccns if c > 30)} ({sum(1 for c in ccns if c > 30)/len(ccns)*100:.1f}%)')
print(f'CCN > 15: {sum(1 for c in ccns if c > 15)} ({sum(1 for c in ccns if c > 15)/len(ccns)*100:.1f}%)')
print(f'Max CCN: {max(ccns)}')
"
```
