# Configuration simplification — unified loader

**Status:** Phase 2 (print mode migration) complete; Phase 3 (proxy + rpc migration) next

## Problem

The tool, hook (guardrail), skills, preset, and general configuration
(tools.retry, tools.bash.timeoutMs, permissions, etc.) can each be set from
multiple sources — built-in defaults, settings.json layers
(project → FRANKY_HOME → HOME), environment variables
(FRANKY_LOG, FRANKY_MAX_TURNS, provider credentials, etc.), CLI flags,
profiles, and extensions.

Currently each of the four modes (print, interactive, proxy, rpc)
duplicates the initialization/layering/merge logic inline, leading to:

- **~200 lines of duplicated ceremony** per mode (provider resolution,
  settings overlay, auth.json loading, models.json loading, registry
  setup, tool filtering, permission store init, preset registry init,
  extension loading, skills loading)
- **Bugs when one mode gets a fix the others don't** (e.g. log-level
  init was moved to `print.run()` and the other modes silently missed
  it until v1.7.5)
- **No single place to add a new config source or field** — every mode
  must be touched
- **Partial/inconsistent layering** — settings.json loading is done
  separately from profile loading which is done separately from
  env-var resolution, and each mode picks its own subset

## Goal

A single `ConfigResolver` (or similar) that the four mode entry points
call with their `cli.Config` (already parsed from argv) and get back a
fully-resolved, ready-to-use configuration bundle — provider info,
tool list, permission store, preset registry, extension manager, skills,
guardrails, session state, workspace state, etc.

The resolver is the **only** place where layering precedence is
implemented. Changing precedence, adding a source, or adding a field
touches exactly one file.

As of Phase 1 (v2.22.0), the resolver lives at `src/coding/config.zig`
and is accessible as `franky.coding.config.resolver`. No callers
have been migrated yet — that happens in Phases 2–5.

## Precedence (consolidated)

From highest to lowest — each layer only fills in fields that are still
at their default/unset value from the layer above (or, for
**concatenated arrays** like tool allow/deny lists, appends):

```
  CLI flags (--provider, --model, --thinking, …)
    ↑
  Profile (--profile <name>, from settings.json or built-in catalog)
    ↑
  Environment variables (FRANKY_LOG, FRANKY_MAX_TURNS, API key vars, …)
    ↑
  settings.json — project layer  (<cwd>/.franky/settings.json)
    ↑
  settings.json — home layer    ($FRANKY_HOME/settings.json)
    ↑
  settings.json — xdg/canonical ($HOME/.franky/settings.json)
    ↑
  Built-in defaults (hardcoded consts in ConfigResolver)
```

Note: env vars currently sit **above** settings.json but **below**
profiles and CLI. This matches the v2 §5.3 spec
("defaults < env vars < settings.json (top-level) < profile < CLI flags")
for settings.json fields that overlap with env-var names
(e.g. `max_turns` in settings.json vs `FRANKY_MAX_TURNS` env var).

Exception — API keys / auth tokens:
```
  CLI --api-key / --auth-token
    ↑
  Provider-specific env var (e.g. ANTHROPIC_API_KEY, MISTRAL_KEY)
    ↑
  auth.json ($FRANKY_HOME/auth.json or $HOME/.franky/auth.json)
    ↑
  Profile-level api-key / api-key-env / auth-token / auth-token-env
```
This is already implemented in `resolveProviderIo` and `auth.zig`; the
resolver just calls the same helper.

## What the resolver owns

### Input

- `allocator: std.mem.Allocator` (arena or GPA)
- `io: std.Io`
- `cfg: *cli.Config` — already parsed from argv; CLI values are
  authoritative. Profile has already been applied by the caller
  (see "Logger init lives outside the resolver" below).
- `environ: std.process.Environ`
- `environ_map: *std.process.Environ.Map`
- `argv: []const []const u8` — needed only for proxy mode restart

The resolver accepts `io` as a parameter for file I/O during resolution
(settings.json, auth.json, skills scanning, system.md template). It does
**not** store `io` in `ResolvedConfig` — callers already have it. The exception
is `subagent_ctx`, which needs `io` for the sub-agent's event forwarding; the
resolver sets `io` on the `Ctx` before returning (the `Ctx.io` field was added
in v2.22 for this purpose).

### Output: `ResolvedConfig`

```zig
pub const ResolvedConfig = struct {
    // ── Provider (resolved) ───────────────────────────────────────
    provider_name: []const u8,
    model_id: []const u8,
    api_key: ?[]const u8,
    auth_token: ?[]const u8,
    api_tag: []const u8,
    base_url: ?[]const u8,
    thinking_level: ai.types.ThinkingLevel,
    context_window: u32,
    max_output: u32,
    capabilities: ai.types.Capabilities,
    /// Per-provider timeout overrides (from profile or env)
    connect_timeout_ms: u64,
    upload_timeout_ms: u64,
    first_byte_timeout_ms: u64,
    event_gap_timeout_ms: u64,
    /// Text-tool-call-fallback flag (default false)
    text_tool_call_fallback: bool,

    // ── Registry (populated) ──────────────────────────────────────
    registry: ai.registry.Registry,

    // ── Tool set (filtered by role + tools_filter, merged with extensions) ─
    tools: []const at.AgentTool,

    // ── Permission store ──────────────────────────────────────────
    permission_store: permissions_mod.Store,
    session_gates: permissions_mod.SessionGates,

    // ── Preset registry ───────────────────────────────────────────
    preset_registry: tools_mod.subagent.PresetRegistry,

    // ── Extension manager ─────────────────────────────────────────
    ext_manager: extensions_mod.Manager,

    // ── Skills ────────────────────────────────────────────────────
    skills: SkillsState,  // lightweight — skills loaded + rendered separately

    // ── Guardrails (per-mode wiring differs slightly) ─────────────
    guardrail_state: agent.guardrails.GuardrailState,

    // ── Settings overlay values ───────────────────────────────────
    /// These are baked into the tool ctxs (bash_state, read_ctx) at
    /// resolve time — no per-mode overlay needed.
    bash_default_timeout_ms: ?u64,
    read_max_bytes_without_limit: ?usize,
    retry_policy: ai.retry.Policy,
    max_turns: u32,
    prompts_enabled: bool,
    review_config_block: ?[]const u8,  // pre-rendered review system-prompt block

    // ── Workspace ─────────────────────────────────────────────────
    workspace: ?tools_mod.workspace.Workspace,

    // ── Bash state (arena-allocated, address stable) ──────────────
    bash_state: tools_mod.bash.SessionBashState,

    // ── Read ctx (arena-allocated, address stable) ────────────────
    read_ctx: tools_mod.read.ReadCtx,

    // ── Subagent context (io set here, parent_session_dir late-bound) ──
    subagent_ctx: tools_mod.subagent.Ctx,

    // ── Logging ───────────────────────────────────────────────────
    /// Resolved log level. Logger is initialized by the caller
    /// **before** calling resolve(), using this value.
    log_level: ai.log.Level,
    log_file: ?[]const u8,
    http_trace_dir: ?[]const u8,
    log_per_session: bool,

    // ── Session ───────────────────────────────────────────────────
    session_state: ?SessionState,

    // ── Role gate ─────────────────────────────────────────────────
    role_gate: role_mod.RoleGate,
    active_role: role_mod.Role,

    // ── Arena ─────────────────────────────────────────────────────
    /// Everything that outlives `resolve()` is allocated on this arena.
    arena: std.heap.ArenaAllocator,

    pub fn deinit(self: *ResolvedConfig) void;
};
```

### Single entry point

```zig
pub fn resolve(
    allocator: std.mem.Allocator,
    io: std.Io,
    cfg: *cli.Config,
    environ: std.process.Environ,
    environ_map: *std.process.Environ.Map,
    argv: []const []const u8,  // needed only for proxy mode restart
) !ResolvedConfig;
```

All four modes call this, then use the result directly. No mode
duplicates the merge logic.

### What the resolver does (step by step)

1. **Apply env-var overrides via a single comptime loop** — iterate a
   `comptime` array of `EnvOverride` structs `(env_name, field_name)` and
   set the matching `cli.Config` field when it is still at its default.
   This replaces the current N-if-block sprawl (15+ env vars dotted
   across 4 modes in inconsistent order). Adding a new env var is a
   one-line addition to the array.

   Env vars that map to **non-CLI.Config** values (FRANKY_HOME, PWD)
   are handled directly by the resolver body, not by this loop.

2. **Parse settings.json layers** — project (<cwd>/.franky/settings.json)
   → FRANKY_HOME → HOME/.franky (call existing `settings_mod.loadLayered`).
   Note: profile is applied by the **caller** before calling `resolve()`
   (see "Logger init" below), so the resolver receives a cfg that
   already has CLI + profile values baked in.

3. **Apply remaining settings overlays uniformly** — bash timeout, read
   max bytes, max_turns, retry policy, permissions, and prompts_default.
   These are currently applied inconsistently (rpc and proxy skip
   bash/read overlays; rpc and interactive skip retry overlays). The
   resolver applies **all** overlays, fixing these gaps.

4. **Check for --version / --help / --list-profiles / --save-profile**
   These short-circuits are mode-agnostic and stay in `print.run()` as
   they are; the resolver only handles "real" config resolution.

5. **Resolve provider** — call the internal `config.resolveProvider`
   (settings overlay for thinking, auth.json for credentials, model
   alias resolution, models.json overlay, provider selection). Returns
   provider info including name, model id, api key, auth token, base url,
   context window, max output, capabilities.

6. **Register providers** in the registry (faux, anthropic, openai,
   gateway, google-gemini — same as today).

7. **Resolve tool set** — this is where tool ctx wiring happens:
   - Build workspace state from PWD (if available).
   - Create `bash_state` and `read_ctx` (stable addresses).
   - Start with the full built-in list, applying the correct ctx
     variants (workspace-aware when workspace is non-null).
   - Apply `--role` gate (role.zig → `role_gate.filterTools`).
   - Load extensions (manager applies its tool contributions).
   - The resolver builds the final tool slice on its arena, leaving room
     for subagent/listPresets/finishTask tools added below.

8. **Set up permission store** — seed from settings.json
   `permissions.*` fields, apply CLI `--yes` / `--ask-tools` /
   `--allow-tools` / `--deny-tools`, attach persistence.

9. **Initialize preset registry** — register built-in subagent presets,
   let extensions register their own.

10. **Load extensions** — Tier-1 extensions opt-in via `--extensions <csv>`.
    Each activated extension registers slash commands, tools, presets, and
    event subscriptions through a `Host` view.

11. **Build guardrail state** — init stuck detector, compilation guard,
    finish_task tool config.

12. **Build subagent context** — link registry, presets, filtered tools,
    role, permission store, etc. Set `io` on the context here (the
    resolver received it as a parameter). `parent_session_dir` is
    late-bound (set by the mode after session init).

13. **Add subagent/listPresets/finishTask tools** — rebuild the final
    tool slice on the arena with these three extra entries.

14. **Build review config block** — pre-render the multi-model review
    system-prompt injection from `settings.review_profiles`. Done once
    here instead of being duplicated 5× across the modes.

15. **Resolve retry policy + max_turns** — from CLI, settings, env vars
    with the correct precedence chain.

Note: **skills loading** is deferred to `buildSystemPromptIo` (called by
modes after config resolution), not done inside `resolve()`. This keeps
skills rendering in the system-prompt builder where it already lives,
avoiding a dependency cycle with the mode's session lifecycle.

### Logger init lives outside the resolver

The top-level entry point (`print.run()`) resolves the log level **before** calling
the resolver:

```
print.run():
  1. cli.parse()
  2. Apply profile (so --profile can set log level)
  3. Check for --version/--help/--list-profiles/--save-profile (short-circuit)
  4. Resolve log level + init logger with `ai.log.init()`
  5. config.resolve()  ← resolver uses already-initialized logger
  6. Mode dispatch
```

The resolver does **not** call `ai.log.init()` — it is pure data flow (assemble
config from sources, no side effects beyond file reads). The resolved
`log_level` in `ResolvedConfig` is the same value the caller already used, but
is available for modes that need it after resolution (e.g. for per-session log
reinit).

### Mode-specific post-processing

The resolver handles **everything shared**. Some mode-specific differences
remain and are applied after the resolver returns:

| Aspect | Mode | Post-resolve handling |
|--------|------|----------------------|
| **Logger init** | all | Done in `print.run()` **before** `config.resolve()` — see above |
| **Log per session** | print, proxy | After session id known → `maybeReinitLoggerForSession()` |
| **Session init** | print, proxy | `session_state.init()` with resolved config, then `session_dir_path` → `subagent_ctx.parent_session_dir` |
| **Session init** | interactive, rpc | Session is internal to the mode; no disk persistence needed |
| **Restart wiring** | proxy | Pass `original_argv`; register `/restart` endpoint |
| **Alt screen / TUI** | interactive | Terminal setup, scrollback, editor — nothing for resolver |
| **RPC frame loop** | rpc | JSON-RPC dispatch loop — nothing for resolver |
| **Slash commands** | interactive, proxy | Slash-registry is per-mode (different handlers) |

These are **thin wrappers** — not re-implementations of the merge logic.

## What this replaces (file-by-file changes)

### New file: `src/coding/config.zig`

The `resolve()` function + `ResolvedConfig` struct. This is the
central loader. It imports from:

- `settings.zig` — `loadLayered` / `defaults`
- `profiles.zig` — `applyProfile`
- `auth.zig` — `load` / credential resolution
- `models.zig` — `parseFromSlice`
- `modes/print.zig` — `resolveProviderIo`, `resolveLogLevel`,
  `loadSettingsForOverlay`, `applyBashSettingsOverlay`,
  `applyReadSettingsOverlay`, `applyMaxTurnsSettingsOverlay`,
  `applyRetrySettingsOverlay`, `applyPermissionSettingsOverlay`
  (these functions MOVE into `config.zig` and are then deleted from
  `print.zig`)
- `role.zig` — `Role` / `RoleGate`
- `permissions.zig` — `Store` / `SessionGates`
- `skills.zig` — `loadAll`, `activateForWorkspace`
- `extensions.zig` — `Manager`, `loadFromConfig`
- `tools/` — all tool modules
- `tools/subagent.zig` — `PresetRegistry`, `Ctx`, `registerBuiltinPresets`
- `guardrails/` — `GuardrailState`
- `workspace.zig` — `Workspace`

### Modified: `modes/print.zig`

1. Remove `resolveProviderIo` (function body moves to config.zig —
   keep a thin public wrapper that calls `config.zig`'s resolver for
   backward compat with proxy & rpc modes during migration)
2. Remove `loadSettingsForOverlay`, `applyBashSettingsOverlay`,
   `applyReadSettingsOverlay`, `applyMaxTurnsSettingsOverlay`,
   `applyRetrySettingsOverlay`, `applyPermissionSettingsOverlay`
3. Remove `resolveLogLevel` (moves to config.zig)
4. Remove `resolveLogFileFromMap` (moves to config.zig or a shared utility)

   post-processing)
6. `runPrint` becomes ~50 lines: call `config.zig.resolve()`, then
   session init, session-dir-path plumbing, then the agent loop.

### Modified: `modes/interactive.zig`

### Modified: `modes/proxy.zig`

duplicated init. The proxy-specific fields (subscribers, replay
ring, restart flag) are added after the resolver returns.

### Modified: `modes/rpc.zig`

`initSession()` calls `config.zig.resolve()` instead of its own
duplicated init. RPC-specific state (baseline tools, session

### Deleted from `modes/print.zig` (moved to `config.zig`)

- `resolveProviderIo` → `config.resolveProvider`
- `loadSettingsForOverlay` → inline in `resolve`
- `applyBashSettingsOverlay` → inline in `resolve`
- `applyReadSettingsOverlay` → inline in `resolve`
- `applyMaxTurnsSettingsOverlay` → inline in `resolve`
- `applyRetrySettingsOverlay` → inline in `resolve`
- `applyPermissionSettingsOverlay` → inline in `resolve`

- `resolveLogFileFromMap` → `config.resolveLogFileFromMap`

The public API surface of `modes/print.zig` (`ProviderInfo`,
`resolveProviderIo`, `default_anthropic_model`, etc.) stays as
thin re-exports or wrappers until all callers (proxy, rpc,
subagent tool) are migrated.

## Migration plan (safe, incremental)

### Phase 1 — Create `config.zig` with `resolve()`

Write the new module alongside existing code. No callers yet.
`config.resolve()` is the unified entry point that:

1. Calls `settings_mod.loadLayered()`
2. Calls `profiles_mod.applyProfile()`
3. Applies env-var overrides
4. Calls `resolveProviderIo()` (extracted from print.zig)
5. Sets up registry, tools, permissions, presets, extensions,
   skills, guardrails, bash_state, workspace, subagent_ctx, etc.

### Phase 2 — Migrate one mode (print)

- `runPrint()` calls `config.resolve()` instead of doing it inline.
- The functions that moved from print.zig to config.zig are kept as
  thin wrappers in print.zig (delegating to config.zig) so proxy, rpc,
  and subagent tool imports don't break yet.
- Full test pass.

### Phase 3 — Migrate rpc + proxy

- `initSession()` in both modes calls `config.resolve()`.
- Remove the duplicated init code from both modes.
- Full test pass.

### Phase 4 — Migrate interactive

- `SessionBinding.init()` calls `config.resolve()`.
- Remove the duplicated init code.
- Full test pass.

### Phase 5 — Remove thin wrappers from `modes/print.zig`

Once no external module imports `print.resolveProviderIo()` or
`print.loadSettingsForOverlay()` (check: subagent tool, proxy,
rpc), delete the wrappers and import directly from `config.zig`.

## Key invariants preserved

1. **No hidden allocations** — `config.resolve()` takes an explicit
   allocator; everything it returns is either arena-allocated or
   borrows from the input.
2. **CLI always wins** — the resolver never overwrites a non-default
   `cli.Config` field.
3. **Settings overlay is idempotent** — missing files are silent;
   malformed JSON surfaces as `SettingsError.MalformedJson`.
4. **Per-mode differences are thin** — session init, slash commands,
   TUI setup, restart wiring are ~20 lines per mode, not 200.
5. **Extensions still see the same `Host` view** — the resolver
   creates the `extensions.Manager` and calls `loadFromConfig`;
   extensions register tools, presets, subscriptions normally.
6. **Skills are still evaluated once** — the resolver scans skills
   roots, evaluates `auto_apply` globs against the workspace, and
   builds the active set. Modes just inject the rendered block.
7. **Subagent presets are still shared** — the `PresetRegistry` lives
   in `ResolvedConfig`, populated by `registerBuiltinPresets` +
   extension-contributed presets.
8. **Guardrails are still initialized at session-scope** — the resolver
   creates `GuardrailState` once; each mode wires it into the loop
   config identically.

## Resolved decisions

These decisions emerged from review and are now reflected in the body text above.

| # | Question | Decision | Where addressed |
|---|----------|----------|-----------------|
| 1 | Env-var resolution: single loop or field-by-field? | **Single comptime loop** — iterate a `comptime` array of `(env_name, field_offset, parse_fn)` triples. Non-CLI.Config vars (FRANKY_HOME, PWD) are handled directly in the resolver body. | Step 3 under "What the resolver does" |
| 2 | `config.zig` or `settings.zig`? | **Separate `config.zig`** — `settings.zig` is already 677+ lines of schema/loader. A dedicated module keeps responsibilities separated. | "New file: src/coding/config.zig" |
| 3 | Skills scanning timing vs session init | **Before session init** — no dependency cycle. Skills need PWD (available at resolver time) and io+allocator (both available). | Step 11 under "What the resolver does" |
| 4 | Should the resolver own `io`? | **Parameter only, not stored in ResolvedConfig.** Resolver accepts `io` for file I/O, sets it in `subagent_ctx` before returning. | "What the resolver owns — Input" section |
| 5 | Where does logger init live? | **Outside the resolver, in `print.run()`** before calling `resolve()`. The resolver returns `log_level` for modes that need it post-resolution. | "Logger init lives outside the resolver" section |
| 6 | How are tool ctxs (bash_state, read_ctx) wired? | **Resolver allocates them on the arena with stable addresses** — eliminates the two-pass rebuild rpc/proxy currently do. Settings overlay and tool list are built in one pass. | Step 8 under "What the resolver does" |
| 7 | Who renders the review config block? | **Resolver does it once** — eliminates the 5x duplication (print, proxy, rpc, and twice in interactive due to a merge bug). | Step 15 under "What the resolver does" |
| 8 | Settings overlay gaps (bash/read in rpc/proxy; retry in rpc/interactive) | **Resolver applies all overlays uniformly** — fixes the gaps that exist today. | Step 7 under "What the resolver does" |
| 9 | Memory ownership model | **Single arena in `ResolvedConfig`** — everything the resolver allocates lives on this arena. `deinit()` frees it all. Borrowed fields (cfg strings) are owned by `cli.Config` and freed separately. | `ResolvedConfig.arena` field description |
| 10 | Extensions loading: mode-specific customization? | **Resolver loads extensions uniformly** — modes don't intervene. If future extensions need mode context, the resolver passes `cfg.mode`. | Step 8 under "What the resolver does" |
| 11 | Error type design | **Shared `ResolveError` set** for resolver failures; modes add their own error variants (e.g. `BindFailed` for proxy) on top. | (new) |
| 12 | Subagent tool import of `print_mode.resolveProviderIo` | **Migrated to `config.resolveProvider`** in Phase 5 (alongside print.zig wrapper removal). Subagent tool currently depends on print.zig; this is an explicit migration step. | Phase 5 under Migration plan |
