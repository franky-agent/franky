# v2.13 — Hosted Model Service Unavailability

**Status:** decided — ready for implementation
**Affects:** `src/ai/retry.zig`, `src/ai/errors.zig`, `src/ai/providers/*.zig`, `src/coding/settings.zig`, `src/agent/loop.zig`
**Related:** [v2.14 — Agentic Engineering](./v2.14-agentic-engineering.md) (depends on this), [v1 §F.1 — retry policy](../spec/v1.md)

---

## 1. Problem (seed)

Hosted providers go down. Reproduction from a real session:

```
HTTP body: {"error":"Server overloaded, please retry shortly (ref: 37400032-…)"}
```

Today the agent surfaces this as a terminal error after the
transport-layer retry budget exhausts. For an interactive user this is
mildly annoying; for an unattended sibling app (v2.14) running while
nobody is watching, the run dies and the work item ends in
"failed — provider error."

## 2. Goal (seed, from the doc author)

> Let the agent loop back off for a while and then retry the
> request. The agent should be able to handle this gracefully without
> crashing or getting stuck in an infinite loop. The user should also
> be informed about the issue and the retry attempts.
>
> Also we could define a model priority in `settings.json` or a
> fallback model provider to use when the primary model provider is
> unavailable. This way, the agent can switch to a different model if
> the primary one is overloaded.

Two distinct asks:

1. **Better retry behaviour** — longer-running, more visible, doesn't
   crash on sustained outages.
2. **Fallback model chain** — declared in `settings.json`; on primary
   exhaustion, switch to next provider.

---

## 3. What franky already has

Substantial infrastructure exists. v2.13 is mostly *tighten +
observe + extend*, not a from-scratch design.

### `src/ai/retry.zig` — transport-layer retry

- `Policy{ max_retries: u32 = 3, base_delay_ms: u32 = 500, max_retry_delay_ms: u32 = 60_000 }`.
- Decorrelated-jitter exponential backoff:
  `delay = min(max_retry_delay_ms, random(base, prev * 3))`.
- `Retry-After` header takes precedence when ≤ `max_retry_delay_ms`.
- If `Retry-After` exceeds the cap → terminal `rate_limited_hard`
  with `retry_after_ms` propagated for the caller to surface.
- Hard invariant: **never retry once response bytes have started
  flowing** (§F.1) — partial-stream failures are terminal.
- Cancel-fired short-circuit before each attempt and each sleep.

### `src/ai/errors.zig` — error classification

```zig
pub const Code = enum {
    auth, request_invalid, model_unavailable, context_overflow,
    payload_too_large, rate_limited, rate_limited_hard, transient,
    timeout, transport, safety_refusal, empty_response, aborted,
    max_turns_exceeded, tool_arg_validation, tool_runtime,
    tool_blocked, protocol_violation, internal,

    pub fn isRetryable(self: Code) bool {
        return switch (self) {
            .rate_limited, .transient, .timeout, .transport => true,
            else => false,
        };
    }
};

pub const ErrorDetails = struct {
    code: Code,
    message: []const u8,
    provider_code: ?[]const u8 = null,
    provider_message: ?[]const u8 = null,
    http_status: ?u16 = null,
    retry_after_ms: ?u64 = null,
    // ...
};
```

Every provider already maps its native error shape into a `Code`.
HTTP 503 currently maps to `transient` for Anthropic; needs auditing
for the OpenAI-Chat / OpenAI-Responses / OpenAI-Gateway / Gemini /
Vertex providers.

## 4. The gap — what's actually missing

### 4.1 Retry budget too small for unattended use

`max_retries: 3` with `max_retry_delay_ms: 60_000` is bounded by
~3 minutes of total wall time even in the worst case. For an
interactive user this is right (fail fast, let me retry). For an
unattended sibling running while a 5-minute provider outage rolls,
the agent dies.

The cap needs to be **policy-driven**, not hard-coded:

| Mode | max_retries | wall-time cap | rationale |
|---|---|---|---|
| Interactive (today's default) | 3 | ~3 min | fail-fast, user is watching |
| Print one-shot | 5 | ~5 min | small tolerance for transient blips |
| Sibling/unattended | 10–20 | 30 min | survive a real outage |
| Test / CI | 0 | 0 | no retry — fail surfaces the bug |

### 4.2 Retry is invisible to the harness

The retry loop sleeps and retries silently inside `ai/retry.zig`.
The agent loop, the SSE event stream, and any subscribed UI sees
*nothing* until either the retry succeeds (event flow resumes) or
the budget exhausts (terminal `error_ev` arrives).

For interactive users this is mildly OK — they wonder if franky is
hung but eventually the retry succeeds and they see output. For
sibling apps, "silence for 3 minutes then a crash" is
indistinguishable from a network deadlock.

### 4.3 No fallback-model concept

`settings.json` carries a single `model` field per profile. There's
no way to declare "if Anthropic is down, fall back to Gemini" or
"if sonnet 503s, drop to haiku."

### 4.4 Total-wall-time cap missing

The current policy bounds *number* of retries, not *total time
spent retrying*. A pathological 503 storm where each response
carries `Retry-After: 60` would (correctly) honour each header but
could spend ~3 minutes blocked even with `max_retries: 3`. With a
bigger retry budget per §4.1 this becomes "30 minutes blocked" —
fine if intended, terrible by accident.

A wall-time cap is the right ceiling; attempt count is the floor.

## 5. Design — three phased layers

### Phase 1 — Observability (no behaviour change)

Make the existing retry visible. Cheapest win, unblocks v2.14
without changing any retry semantics.

**New stream event** (in `src/ai/stream.zig`):

```zig
pub const StreamEvent = union(enum) {
    // ... existing variants ...
    provider_retry: struct {
        attempt: u32,           // 1-indexed; attempt 1 = first retry
        max_attempts: u32,
        delay_ms: u32,
        reason: errors.Code,    // .transient / .rate_limited / etc.
        provider_code: ?[]const u8 = null,
        provider_message: ?[]const u8 = null,
        http_status: ?u16 = null,
    },
};
```

Emitted by `retry.run()` between the `.retryable` verdict and the
sleep. Propagates up through the existing channel/Reducer paths so:

- The TUI / `--mode interactive` shows "retrying in 4s (attempt
  2/3)…" inline.
- The proxy `/events` SSE stream carries it for web-UI rendering.
- Sibling apps subscribe and route it to the source channel
  ("Agent paused — Anthropic 503, retrying in 4s").

No retry policy changes in this phase — the user just gets a
window into what's happening.

**Acceptance:** `--mode print` against a stub provider that returns
two 503s then success prints two `provider_retry` lines before the
final assistant text.

### Phase 2 — Tunable retry budget

Make the policy configurable per-session via the existing
`Settings` overlay (already wired for `bash_timeout_ms`,
`read_max_bytes`, etc. in v1.19.0).

**New `Settings` fields** (under `tools.retry` or top-level):

- `retry_max_attempts: u32 = 3` — replaces the hard-coded default.
- `retry_max_total_ms: u64 = 180_000` — wall-time cap (3 min default).
- `retry_max_delay_ms: u32 = 60_000` — single-attempt delay cap (today).

**CLI flags** (mirror naming):

- `--retry-max-attempts N`
- `--retry-max-total-ms N`

**Profile presets:**

```json
"profiles": {
  "ollama-deepseek-flash": {
    "model": "deepseek-v4-flash:cloud",
    "retry_max_attempts": 5,
    "retry_max_total_ms": 600000
  },
  "agent-unattended": {
    "model": "claude-sonnet-4-6",
    "retry_max_attempts": 15,
    "retry_max_total_ms": 1800000
  }
}
```

**Wall-time cap implementation:** `retry.run()` tracks accumulated
`total_delay_ms`. Before each sleep, if `total_delay_ms +
next_delay_ms > retry_max_total_ms` → emit terminal with code
`rate_limited_hard` (existing) and `retry_after_ms` set to the
remaining budget so the caller can decide whether to wait or fail.

**Acceptance:** A profile with `retry_max_attempts: 10` and a stub
provider that 503s 5 times before succeeding completes the request
without surfacing terminal.

### Phase 3 — Fallback model chain

Bigger lift. Cross-provider switching has real complexity (auth
divergence, cost asymmetry, capability mismatch). Treat as a
separate phase that ships independently of phases 1+2.

**`Settings`/profile additions:**

```json
{
  "model": "claude-sonnet-4-6",
  "fallback_models": [
    "claude-haiku-4-5",
    "gemini-2.5-pro"
  ],
  "fallback_scope": "session"   // or "turn"
}
```

**Switching trigger:** when phase-2 retry budget on the primary
exhausts → instead of surfacing terminal, walk the fallback list
in order and retry with the next model that was marked available
at startup. Each fallback gets its own retry budget (bounded by
the same `retry_max_total_ms`).

**`fallback_scope` semantics:**

- `"turn"` — fall back for this turn only; next turn tries the
  primary again. Best when the outage is transient and you want
  to recover automatically.
- `"session"` — once switched, stay on the fallback for the rest
  of the session. Avoids flapping when the primary is unstable.

**Cross-provider concerns to design through:**

1. **Auth.** API key management (rotation, storage, issuance) is out
   of scope — handled outside franky. However, at session startup
   franky checks which models in `settings.json` have credentials
   configured and prints a single line:

   ```
   Available models: claude-sonnet-4-6, gemini-2.5-pro  [claude-haiku-4-5: no credentials]
   ```

   Models without credentials are excluded from the fallback walk for
   that session. Full credential lifecycle (rotation, re-check mid-session)
   is deferred to v3.
2. **Tool-call shape.** Already abstracted via the registry —
   tools translate per-provider. Should "just work" but needs a
   round-trip test.
3. **Thinking blocks.** Some providers use them, some don't. The
   transcript carries thinking blocks as `Message.content[]` items;
   fallback model that doesn't support thinking ignores them on
   send but the transcript stays intact.
4. **Cost asymmetry.** Fallback may be drastically more or less
   expensive. The `provider_retry` event (phase 1) extended with a
   `using_fallback: ?[]const u8` field surfaces the switch
   prominently so the human (or sibling) notices.
5. **Quality degradation.** A haiku fallback for a sonnet task
   might silently produce worse outputs. Phase 3 emits a one-time
   `agent_warning{using_fallback}` event so the surface is loud.

**Acceptance:** `model: claude-sonnet-4-6` + `fallback_models:
[gemini-2.5-pro]` + a stub that simulates Anthropic 503 storm:
agent successfully completes the run via Gemini and the transcript
records which provider answered each turn.

## 6. Open questions

### Q1 — Default `retry_max_total_ms` for the OOTB profile?

Today's *implicit* total budget is ~3 min (3 retries × ≤60s caps +
`Retry-After` honoring). What's the right number for the default
non-profiled invocation?

**Proposal:** keep ~3 min default; bump to 30 min only on
explicitly unattended profiles. The interactive-default case
shouldn't change — fail fast is correct when a human is watching.

**Decision:** Keep ~3 min default for non-profiled interactive invocations. Unattended profiles explicitly opt in to longer budgets (e.g. `retry_max_total_ms: 1_800_000`). Interactive default unchanged.

### Q2 — Should `provider_retry` events fire BEFORE the sleep or AFTER?

Before-sleep gives the UI advance notice ("retrying in 4s"); after
gives concrete proof the retry happened.

**Proposal:** before-sleep, at the moment the verdict flips to
`.retryable`. The UI gets a chance to display "retrying in 4s
(attempt 2/3)…" while the sleep runs.

**Decision:** Fire before-sleep, at the moment the verdict flips to `.retryable`. The UI and sibling apps get advance notice ("retrying in 4s") while the sleep runs.

### Q3 — Phase-3 fallback: which providers in the OOTB list?

If we ship default fallback chains per provider:

- `claude-sonnet-4-6` → `claude-haiku-4-5` (same vendor, cheap).
- `gemini-2.5-pro` → `gemini-2.5-flash` (same vendor, cheap).
- `deepseek-v4-flash:cloud` → `deepseek-v4-flash:local` (same model
  via local Ollama if available).

**Proposal:** Don't ship default fallbacks — too opinionated. Phase
3 is opt-in only via explicit `fallback_models` config.

**Decision:** lets use the fallback as you proposed\

### Q4 — How to handle `rate_limited` (HTTP 429 with usage limits) vs `transient` (HTTP 503 server overload)?

429 is "you've spent too much" — retry won't help if your account is
out of quota. 503 is "the server is unhappy" — retry helps. Should
v2.13 differentiate?

**Proposal:** Honour `Retry-After` for both today (already happens).
Phase 3 fallback triggers on EITHER exhausting the primary's retry
budget — fallback to a different vendor sidesteps both quota and
overload issues.

**Decision:** Unified handling. Honour `Retry-After` for both 429 and 503. Phase-3 fallback triggers on exhausting the primary's retry budget regardless of whether it was quota or overload — a different vendor sidesteps both.

### Q5 — Mid-stream 503 (`§F.1` "no retry after first byte")

Today the invariant is: once response bytes flow, terminal. But some
providers send a 200, start streaming, then send a 503 mid-body.
That terminal can't safely retry (tool calls may have already fired
client-side). Should we surface this differently?

**Proposal:** Keep the §F.1 invariant. Mid-stream failures emit
`error_ev{code: transport, mid_stream: true}` (new field) so siblings
can decide whether to abort the whole task or report partial.

**Decision:** Keep §F.1. Add `mid_stream: true` field to `error_ev` so sibling apps and the TUI can distinguish a mid-stream failure from a clean pre-stream one and decide whether to abort the whole task or report partial results.

## 7. Phased implementation roadmap

```
Phase 1   Observability — provider_retry event              (1-2 days)
Phase 2   Tunable budget — settings + CLI + wall-time cap   (2-3 days)
Phase 3   Fallback model chain — cross-provider switching   (1-2 weeks)
```

**Phase 1 alone unblocks v2.14** for the "report retries to the
source channel" use case. Phase 2 makes unattended runs robust to
real outages. Phase 3 is the polish for cross-provider resilience
and can ship later.

---

## 8. Out of scope (explicit non-goals for v2.13)

- **API key / credential management.** Storage, rotation, and issuance
  are solved outside franky and deferred to v3. v2.13 only checks at
  startup whether a credential is present (not valid) for each declared
  model and uses that to build the available-model list for fallback.
- **Per-tool retry** (e.g. retry `bash` on transient exit codes).
  v2.13 is about *provider* unavailability; tool retry is a
  separate concern that v2.10 § stuck-detector partially addresses.
- **Caching responses** to dodge repeat provider calls. Different
  problem (latency / cost), different design.
- **Provider health dashboard** in the web UI. Useful but decoupled.
- **Automatic switch back to primary** after fallback. Phase 3
  proposes `fallback_scope: turn|session`; auto-promotion ("primary
  is healthy again, switch back") is a v2.13.x or later feature.

## 9. Summary — what to decide

- [x] Q1 — Default `retry_max_total_ms` stays ~3 min (interactive default unchanged).
- [x] Q2 — `provider_retry` fires before-sleep (advance notice), not after.
- [x] Q3 — No OOTB fallback chains; phase 3 is opt-in only.
- [x] Q4 — 429 vs 503 unified by phase-3 fallback (no special-casing).
- [x] Q5 — Mid-stream failures surface a `mid_stream: true` field, no retry.
- [x] Approve the 3-phase roadmap (observability → tunable → fallback).

Once these resolve, phase 1 is small enough to land alongside the
v2.10/v2.11/v2.12 work that v2.14 is also waiting on. Phase 3 can
follow on its own track.
