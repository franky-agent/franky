## 1. What this design achieves

The model (franky) produces code changes; those changes land in files that may already be hotspots — high-complexity, high-churn, single-author files that are at risk of knowledge loss. Today the guardrails check **compilation** (`zig build`) and **stuck loops** (repeated identical edits), but not **whether the file being edited is already a quality problem**.

The Soteria guardrail adds:

1. **Pre-next-edit advisory check** — after a tool call that mutates a file, the guardrail checks whether the target is a known hotspot or has a negative complexity trend. If so, it queues a refactoring hint for injection at the next `betweenTurns` — the model sees the hint before its *next* mutation on that file. (The check cannot run *before* the edit because the hook point is `afterToolCall`; the guardrail has no `beforeToolCall` slot. This is acceptable because the hint is advisory, not a block.)
2. **Post-edit regression check** — after a mutation pass, the guardrail runs a new scan and compares changed-file metrics against the baseline. If complexity has increased, it injects a warning.
3. **Session quality delta** — on `finish_task`, the guardrail appends a quality-delta message to the transcript (before the forced compilation step). The model sees it before writing its final summary. The delta is **not** retroactively inserted into the model's own summary string — it is a separate transcript message owned by the guardrail.

All checks are advisory — they inject hints into the transcript, not hard blocks. The model can ignore a hint, but seeing the signal tends to shift its behaviour toward simpler code.

### Non-goals

- Do not replace `CompilationGuard`. Compilation is a hard gate; quality is a soft gate.
- Do not block the model from editing a hotspot file. The hint is informational.
- Do not add a new tool. The guardrail communicates through transcript hints, same as the stuck detector.
- Do not vendor the upstream **binary**; do vendor the upstream **source** as a git submodule and build from source at build time (see §7).

## 2. Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│ GuardrailState                                                  │
│  ┌──────────────────────┐   ┌────────────────────────────────┐  │
│  │ StuckDetector        │   │ SoteriaGuard (NEW)             │  │
│  │  → afterToolCall     │   │  → .soteria/thresholds.json   │  │
│  │  → betweenTurns hint │   │  → .soteria/history.jsonl     │  │
│  ├──────────────────────┤   │  → soteria scan --out /tmp/   │  │
│  │ CompilationGuard     │   │  → checkHotspot(path)         │  │
│  │  → afterToolCall     │   │  → checkRegression(path)      │  │
│  │  → betweenTurns hint │   │  → qualityDelta()             │  │
│  ├──────────────────────┤   └────────────────────────────────┘  │
│  │ FinishTaskState      │                                       │
│  │  → auto-commit       │                                       │
│  │  → quality delta msg │                                       │
│  └──────────────────────┘                                       │
│  GuardrailState.betweenTurns() — runs soteria hints after       │
│  compilation / stuck checks, same priority slot.                │
└─────────────────────────────────────────────────────────────────┘
```

### Priority in `betweenTurns`

The existing `betweenTurns` order is:

1. finish_task pending → schedule compilation
2. finish_task compilation → run forced compilation → commit
3. **Compilation guard** triggered → run build → inject hint
4. **Stuck detector** pending hint → inject hint
5. **[NEW] Soteria quality hint** → inject regression/hotspot hint

Soteria runs last so it never competes with compilation or stuck handling. It is purely informational.

### Soteria binary lifecycle

The `soteria` binary lives at the workspace root, built from vendored source (see §7). The guardrail invokes it as a child process:

```sh
./soteria --out /tmp/soteria-scan-<ts>.json \
          --thresholds .soteria/thresholds.json \
          --history .soteria/history.jsonl
```

The JSON output is parsed into a per-file record array. The guardrail then runs queries against the parsed data.

## 3. SoteriaGuard struct

```zig
const SoteriaGuard = struct {
    allocator: std.mem.Allocator,
    config: Config,
    /// Parsed thresholds from last scan (cached).
    thresholds: ?types.Thresholds,
    /// Scan result from session start (baseline for qualityDelta).
    baseline_report: ?[]ScanRecord,
    /// Paths that have been flagged as hotspots this session
    /// (to avoid repeat-hinting the same file). Keys are owned.
    flagged_paths: std.StringHashMap(void),
    /// Arena for hint strings produced by checkHotspot/checkRegression.
    /// Reset on each betweenTurns cycle after hints are injected.
    hint_arena: std.heap.ArenaAllocator,
    /// Whether a soteria hint is pending injection.
    pending_hint: ?[]const u8 = null,

    pub const Config = struct {
        /// Path to the soteria binary (default: "./soteria").
        binary_path: []const u8 = "./soteria",
        /// Path to thresholds file (default: ".soteria/thresholds.json").
        thresholds_path: []const u8 = ".soteria/thresholds.json",
        /// Path to history DB (default: ".soteria/history.jsonl").
        history_path: []const u8 = ".soteria/history.jsonl",
        /// Whether to run a full scan on session start (seed data).
        scan_on_start: bool = true,
        /// Whether to check hotspot status after every edit/write.
        check_before_edit: bool = true,
        /// Whether to check regression after mutations.
        check_after_mutation: bool = true,
        /// Max wall-clock time for a single soteria scan.
        scan_timeout_ms: u64 = 30_000,
    };

    pub fn init(allocator: std.mem.Allocator, cfg: Config, io: std.Io) !SoteriaGuard { … }
    pub fn deinit(self: *SoteriaGuard) void {
        // Frees hint_arena, flagged_paths keys,
        // baseline_report, thresholds.
    }

    /// Run soteria scan and parse report. Called on session start and
    /// optionally after mutation passes. Respects scan_timeout_ms.
    pub fn runScan(self: *SoteriaGuard, io: std.Io) ![]ScanRecord { … }

    /// Load thresholds from JSON (fast, no scan needed).
    pub fn loadThresholds(self: *SoteriaGuard, io: std.Io) !void { … }

    /// Check whether `path` is a hotspot (orange/red zone) in the last
    /// scan. Returns a formatted hint string allocated from hint_arena,
    /// or null. The returned string lives until the next hint_arena reset.
    pub fn checkHotspot(self: *SoteriaGuard, path: []const u8) ?[]const u8 { … }

    /// After editing `path`, check whether indent_mean or other
    /// metrics moved in the wrong direction compared to the baseline.
    /// Hint string lives in hint_arena, or null.
    pub fn checkRegression(self: *SoteriaGuard, path: []const u8, io: std.Io) ?[]const u8 { … }

    /// Compute a quality delta summary for finish_task.
    /// Returns a message string in hint_arena, or null if no changes.
    /// Uses git diff to find changed files (Q10.4).
    pub fn qualityDelta(self: *SoteriaGuard, io: std.Io) !?[]const u8 { … }

    /// Queue a hotspot check for the given tool args.
    fn queueHotspotCheck(self: *SoteriaGuard, tool_name: []const u8, args_json: []const u8) !void { … }

    /// betweenTurns entry point: inject pending hint if any.
    /// Returns true if hint was injected (caller should run another turn).
    pub fn betweenTurns(self: *SoteriaGuard, allocator: std.mem.Allocator, io: std.Io, transcript: *at.Transcript, out: *at.AgentChannel) !bool { … }

    /// Reset hint_arena + pending_hint after each injection cycle.
    fn resetHints(self: *SoteriaGuard) void { … }
};
```

### Hooks into GuardrailState

#### Interface change: `afterToolCall` gains `arguments_json`

The existing `GuardrailState.afterToolCall` receives only `(tool, call_id, result)`. The soteria guardrail needs the tool's arguments to extract the target `path`. The signature is updated to:

```zig
pub fn afterToolCall(
    self: *GuardrailState,
    tool: *const at.AgentTool,
    call_id: []const u8,
    arguments_json: []const u8,   // NEW
    result: *const at.ToolResult,
) void;
```

The loop already has `tc.arguments_json` at both call sites (serial path and parallel path) so this is a mechanical wiring change. The stuck detector continues to use its own independent `args_cache`; it ignores the new parameter.

#### afterToolCall: queue hotspot check

If the tool is `edit` or `write` and `check_before_edit` is true, parse `arguments_json` to extract `"path"`, then call `checkHotspot`. If a hint is returned, queue it for injection at the next `betweenTurns`.

```zig
// In GuardrailState.afterToolCall (revised):
pub fn afterToolCall(self: *GuardrailState, tool: *const at.AgentTool, call_id: []const u8, args_json: []const u8, result: *const at.ToolResult) void {
    self.stuck_detector.afterToolCall(tool, call_id, result);
    self.compilation_guard.bumpIfMutation(tool.name);

    // NEW: queue hotspot check for edit/write targets
    if (self.soteria_guard.config.check_before_edit) {
        self.soteria_guard.queueHotspotCheck(tool.name, args_json) catch {};
    }
}
```

#### betweenTurns: flush queued hints

In `betweenTurns`, after the stuck detector check, the guardrail flushes any queued hints:

```zig
// NEW arm in betweenTurns, after stuck detector check:
// ── soteria quality hints ──────────────────────────────
if (try self.soteria_guard.betweenTurns(allocator, io, transcript, out)) {
    ai.log.log(.debug, "guardrails", "soteria_fired", "", .{});
    self.guardrail_fire_count += 1;
    return true;
}
```

#### Contextignore guard (skipped for v1)

Per Q10.5, the guardrail does **not** check `.contextignore` before processing hotspot/regression checks. The tools already refuse contextignored paths, so the guardrail will never receive an edit/write call for one. The safety net is unnecessary complexity for v1.

## 4. Hint format

### Hotspot warning (queued after edit)

Injected at the next `betweenTurns` when the model edited a file in orange or red zone:

> **Code quality note:** `src/agent/loop.zig` is a known hotspot
> (hotspot score: 42.3 — 🔴 red zone, complexity trend: rising 🔺).
> Consider whether this change can be done in a smaller,
> lower-complexity file, or whether extracting a helper
> would reduce the complexity of this file.

### Regression warning (post-mutation)

Injected when a file's complexity increased after a batch of edits:

> **Code quality note:** `src/agent/loop.zig` complexity increased
> from 7.2 → 8.1 (indent mean). If this change adds new logic,
> consider extracting it to a separate module.

### Session quality delta (finish_task — separate transcript message)

The quality delta is **not** embedded into the model's own summary string (the model sets that before the guardrail runs). Instead, the guardrail injects a user-role message into the transcript between the finish_task tool result and the forced compilation step. The model sees it before writing its final summary. Format:

> **Quality delta (Soteria):** 3 files improved (complexity ↓), 1 file
> regressed (src/agent/loop.zig — hotspot score +5.1).

This message is owned by the guardrail (allocated from hint_arena) and freed after injection.

## 7. Upstream binary — platform compatibility

The existing `soteria` binary in the workspace root is built for arm64. On x86_64 hosts (CI or other dev machines) it will not execute (`Exec format error`).

Two strategies:

| Strategy | Trade-off |
|---|---|
| **A: Rebuild from source** via `zig build` in `vendor/soteria-guard/` | Always matches the target arch, but adds a build step. Requires Zig 0.15.2+. |
| **B: Use any binary with matching ABI** (download release artifact) | Faster, but needs a download step and version pinning. Upstream has no CI releases yet. |

**Decision: Strategy A — vendor as a git submodule at `vendor/soteria-guard/`, add a `zig build soteria` step that produces `./soteria`.** This resolves the platform-compatibility problem (the binary is always built for the host), avoids manual binary management, and keeps the workspace self-contained.

The `SoteriaGuard` config has a `binary_path` field (default `"./soteria"`). If the binary is missing or the build step hasn't been run, `runScan` returns an error that is logged at `info` level and the check is skipped — the guardrail degrades gracefully.

**This replaces the previous non-goal "Do not vendor or fork the upstream soteria binary" with: "Vendor the upstream **source** (not the binary), build from source at build time."** The non-goals section is updated accordingly.

## 8. Data file layout & ScanRecord schema

The existing `.soteria/` files are written by the upstream binary. The guardrail only **reads** them:

- `.soteria/thresholds.json` — JSON object with p60/p85/p95 thresholds for each metric.
- `.soteria/history.jsonl` — JSON-lines records from each scan, appended by the binary.
- `.soteria/coupling.json` — JSON array of `CouplingPair` objects (read by the guardrail but not used for hints — exposed for debugging).

The guardrail writes nothing to `.soteria/`. Temporary scan outputs go to the system temp directory.

### Scan output record (per-file JSON line)

The upstream soteria binary produces JSON-lines output when invoked with `--out`. Each line is a per-file record:

```json
{
  "scan_id": "live",
  "file": "src/agent/loop.zig",
  "date": "unknown-date",
  "loc": 1591,
  "indent_mean": 6.53,
  "indent_max": 28,
  "revisions": 25,
  "authors": 2,
  "main_dev_pct": 0.56,
  "churn": 2424
}
```

**Note:** The current `.soteria/history.jsonl` data (from an older run) does **not** include a `hotspot_score` field. **Phase 1 is blocked** until the upstream soteria binary emits `hotspot_score` natively (Q10.2 — upstream will implement it). The guardrail uses the upstream-scanned hotspot_score directly rather than computing it from raw metrics.

### Hotspot score formula (NOT USED — see Q10.2)

**Note:** The following formula was proposed during design but is **not adopted**. Per Q10.2, the upstream soteria binary will emit `hotspot_score` natively. The guardrail reads that field. The formula is preserved here as reference for calibration/validation:

```text
hotspot_score = indent_mean × 4.0 + revisions × 1.5 + authors × 5.0 + churn × 0.01
```

Calibrated against the current thresholds (Appendix A):
- p85: 61.6  → typical hotspot threshold (orange zone).
- p95: 113.0 → severe hotspot threshold (red zone).

The weights are tuned so that `src/agent/loop.zig` (indent_mean=6.53, rev=25, auth=2, churn=2424) lands in orange/red:

6.53 × 4.0    = 26.12
25   × 1.5    = 37.5
2    × 5.0    = 10.0
2424 × 0.01   = 24.24
**Total**     ≈ **97.9** (orange, near red at 113).

This formula is provisional and may be used as a fallback if the upstream field is absent.

  "p60_complexity": 5.78,

  "p85_revisions": 10,

Zone assignment for hotspot check:
- **Green:** hotspot_score < p60 (17.5) — no hint.
- **Orange:** p60 ≤ hotspot_score ≤ p85 (17.5–61.6) — hint with 🟠 orange warning.
- **Red:** hotspot_score > p85 (61.6) — hint with 🔴 red warning.

(Using p85 as the red threshold rather than p95 is more conservative — more files get flagged, which is appropriate for an advisory system.)

### Baseline-report caching for qualityDelta

`qualityDelta` compares two full-scan results: the session-start scan (`baseline_report`) and a new scan run at finish_task. Both are parsed into `[]ScanRecord`. The diff is computed by running `git diff --name-only HEAD` at finish_task (reusable utility, shared with future guardrails) and comparing metrics for files that actually changed. This avoids tracking edited paths in memory and gives an accurate change set even across reverted edits.

## 9. Decisions

All open questions from the initial draft and the implementation analysis are now resolved:

### From initial draft (original §9)

1. **How often to re-scan during a session?** **Decision:** only on `finish_task` for now. A scan takes ~2 seconds; running it more frequently would add unacceptable latency. Revisit if empirical data shows the model frequently regresses files mid-session.

2. **Should the hotspot check use the static thresholds file or run a full scan?** **Decision:** thresholds file for pre-edit hotspot check (fast — no scan needed), full scan for finish_task quality delta.

3. **Should the guardrail track flagged paths to avoid repeat hints?** **Decision:** yes — `flagged_paths` in `SoteriaGuard` prevents hinting the same file twice. Keys are allocated from the struct's allocator and freed on `deinit`. Reset on `finish_task`.

4. **How to handle new files (no git history)?** Metrics default to 0 (revisions, authors, churn). Hotspot score will be low → green zone, so no spurious hint. Regression check compares session-end scan vs. session-start scan — a new file's "baseline" is its own initial metrics, so the delta is zero.

5. **What happens if the soteria binary is absent or wrong arch?** **Decision:** check binary availability at session start; if absent or unexecutable, log at `info` level and disable all soteria checks for the session. No crash, no warnings.

6. **Should soteria be vendored as a git submodule or fetched at build time?** **Decision:** vendor as a git submodule at `vendor/soteria-guard/` (Strategy A from §7). Build with `zig build soteria`. This resolves the arm64-only binary problem and ensures reproducible builds.

7. **What metrics should trigger a regression warning?** **Decision:** `indent_mean` (Δ ≥ +0.5) and `hotspot_score` (Δ ≥ +5). If either metric increases beyond the noise threshold, fire the hint.

8. **Should the coupling data be used for hints?** **Decision:** not in v1. Expose in debug log only.

### From implementation analysis (original §10)

**Q10.1 — Path extraction: how does `afterToolCall` get `arguments_json`?**
**Decision:** Thread `arguments_json` into `afterToolCall`. Change the signature to accept `arguments_json: []const u8`. The loop already has the value at both call sites (serial and parallel paths). This is a mechanical one-line change per call site.

**Q10.2 — Hotspot score: who computes it?**
**Decision:** Upstream soteria binary will emit `hotspot_score` natively. Phase 1 is **blocked** until this is available. The provisional formula in §8 is preserved as reference/fallback but is not the primary approach.

**Q10.3 — Quality delta: how to communicate to the model on finish_task?**
**Decision:** Approach A — inject a separate user-role transcript message between the finish_task tool result and the forced compilation step. Do not modify the model's summary string.

**Q10.4 — What file changes should be tracked for regression?**
**Decision:** Approach B — run `git diff --name-only HEAD` at finish_task. Implement as a reusable utility function so future guardrails (e.g., diff-review) can share it.

**Q10.5 — Should the guardrail skip checks for `.contextignore` paths?**
**Decision:** Skip the check for v1. The tools already refuse contextignored paths, so the guardrail would never see an edit/write call targeting one. The safety net is unnecessary complexity.

**Q10.6 — Scan timeout: what happens if the scan hangs?**
**Decision:** Yes, add `scan_timeout_ms` (default 30s). Use `std.process.run` with a wall-clock timeout. Log a warning and skip the scan if the timeout fires.

## 11. Future directions (post-v1)

- **Auto-refactoring:** When soteria flags a function with complexity trend 🔺, the model could be prompted to refactor before adding new code. This is already partially covered by the v2.24 complexity reduction work.
- **CI gate:** A `.franky-workflow.yaml` stage that runs `soteria --out-markdown` and fails the build if any new file hits red zone.
- **Per-author quality trends:** Use the history DB to show whether a model session is improving or degrading the codebase over time (visible in the proxy dashboard).
- **Multi-language support:** soteria's `source_extensions` already covers .zig, .rs, .go, .py etc. The guardrail works unchanged for any language soteria supports.

## Appendix B — Soteria source structure (upstream v0.1.0)

```
src/
├── main.zig          — CLI entry, orchestration
├── types.zig          — FileMetrics, EvolutionMetrics, Signals, Thresholds, Report
├── scan.zig           — Static file scan (LOC, indent, comments)
├── git_log.zig        — Git history extraction
├── churn.zig          — Evolutionary metric computation
├── trend.zig          — Complexity trend over time
├── signals.zig        — Derived signals (hotspot, risk, congestion)
├── thresholds.zig     — Percentile calibration + zone assignment
├── coupling.zig       — Co-change coupling (git log --name-only)
├── report.zig         — JSON / Markdown report writers
├── ignore.zig         — .guardrailignore glob matching
└── complexity.zig     — Indentation-based complexity proxy
```

The guardrail only needs the JSON output format (parsed into `ScanRecord`). It does not import any upstream Zig code.
