# §2.4 — Richer browser UI (`franky-web-ui`)

**Status: plan (not started).** Design document for the
`franky-web-ui` sibling project — the richer Lit-based browser
UI that v2.md §2.4 defers.

---

## 1. Rationale

The v1.5.0 in-binary MVP web UI (`src/coding/modes/web/` —
`index.html`, `app.js`, `style.css`) covers the "browser as
front-end" goal with zero supply-chain surface. It is **good
enough** for daily use and will remain the default UI embedded
in the `franky` binary.

`franky-web-ui` is a **separate sibling project** that consumes
the same `streamProxy` API but layers on features that would
unacceptably bloat the in-binary MVP. Being a sibling project
keeps the one-way `ai → agent → coding` layering invariant
(v1 §9) intact — the web UI only depends on `franky.sdk` and
the proxy's wire protocol.

**Trigger / unlock.** Independent. The richer UI ships if/when
a specific deploy needs it — a team using franky daily wants
in-chat PDF viewing, a product demo needs artifact rendering, or
a dev env needs multi-pane layouts.

## 2. Architecture

```
┌──────────────────┐     SSE (GET /events)      ┌──────────────────────┐
│  franky binary   │◄───────────────────────────│  franky-web-ui       │
│  --mode proxy    │     REST (POST /prompt,     │  (Lit + TS, sibling) │
│                  │           GET /transcript,  │                      │
│                  │           POST /interrupt,  │  Served by:          │
│                  │           ...)              │  - `franky proxy`    │
│                  ├───────────────────────────►│    as static files    │
│                  │     serves ./dist/          │  - OR standalone     │
│                  │     at GET /                │    dev server (Vite) │
└──────────────────┘                            └──────────────────────┘
```

The proxy listener already serves the in-binary MVP at `GET /`.
`franky-web-ui` is served via one of:

1. **`--web-ui-root <path>`** — a new flag on `franky --mode proxy`
   that overrides the built-in `@embedFile` files with a directory
   of static assets. The proxy serves these instead of the defaults.
2. **Standalone dev server** — during development, `vite dev` on
   a separate port. The franky proxy runs on its port, and the
   Vite dev server proxies `/events` + REST endpoints to the franky
   port (standard Vite proxy config).
3. **Production build** — `vite build` outputs to `./dist/`. The
   directory is placed behind any static file server (nginx, S3,
   franky's `--web-ui-root`).

**Wire format unchanged.** Every SSE event kind, every REST
endpoint shape, every JSON field — all identical to what the
in-binary MVP consumes. The proxy doesn't know (or care) which
UI is connected.

## 3. Technology choices

| Concern | Choice | Rationale |
|---------|--------|-----------|
| Framework | **Lit** | Web Components natively; <5 kB gzipped. Follows the original v1 §7 vision. No framework lock-in — components degrade gracefully in any HTML context. |
| Language | **TypeScript** (strict mode) | Catch null/undefined bugs at build time. Lit's decorator syntax is well-supported. |
| Build tool | **Vite** | Fastest DX, native TS support, esbuild under the hood, proxy config for dev, outputs static `dist/`. |
| Markdown | **unified/remark** + **DOMPurify** | Modular pipeline; install only the CommonMark plugins needed. Smallest targeted bundle (<15 kB gzipped). |
| Syntax highlight | **Shiki** | VS Code-quality highlighting; generates themed tokens. Smaller than highlight.js for a full language set; no runtime parser overhead. |
| PDF viewer | **pdfjs-dist** | Standard; embed as a `<canvas>` in chat messages. Accept the ~2 MB cost — it's loaded lazily on first PDF attachment. |
| XLSX viewer | **SheetJS** (Community Edition) | Parse `.xlsx`/`.csv` to HTML tables or JSON. Rendered inline in the chat. Lazy-loaded. |
| Artifact system | Custom Lit component | Expandable/collapsible card that wraps any of the above renderers. Header shows filename + size + type icon. |
| Multi-pane layout | CSS Grid + Lit | Resizable panels using CSS `grid-template-columns` with a drag handle — no library. Left pane for sidebar/conversation list, center for active chat, right for artifact preview. |
| Editor (tiptap-style) | **ProseMirror** (via tiptap or bare) | Rich text + slash commands + @-mentions + code blocks in the composer. Tiptap wraps ProseMirror nicely for Vue/React; for Lit we use bare ProseMirror or `@prosemirror-adapter/lit`. |
| State management | Lit reactive properties + URL hash | No external store. Session state lives on root-level Lit elements; conversation list from `GET /sessions`; active transcript from `GET /transcript`. SSE events update properties reactively. |

## 4. Feature catalog

### 4.1 Must-have (MVP for the sibling project)

These match parity with the in-binary MVP + meaningful
improvements that justify a sibling project:

- [ ] **Full CommonMark rendering** (tables, task lists, footnotes,
      strikethrough, autolinks) via remark + DOMPurify.
- [ ] **Syntax-highlighted code blocks** with language label,
      copy-button, and line numbers (Shiki + custom element).
- [ ] **Sidebar** with session list, search, delete, export.
- [ ] **Composer improvements**: slash-command palette (port from
      MVP), multi-line editing, code-block insertion toolbar,
      drag-and-drop file upload.
- [ ] **Tool cards** with expandable args JSON, execution time,
      and "view full output" toggle.
- [ ] **Thinking blocks** collapsed by default with a click-to-
      expand toggle.
- [ ] **Responsive layout** — works on tablet and desktop.
- [ ] **SSE reconnect** with `Last-Event-ID` replay (port from
      v1.16.0 proxy logic).

### 4.2 Nice-to-have (phase 2)

- [ ] **Artifact system** — PDF, XLSX, image, and large-text
      artifacts rendered inline in the chat as expandable cards.
      Files uploaded via the composer are shown as artifacts.
- [ ] **Multi-pane layout** — left: sessions, center: chat, right:
      artifact preview. Panes resizable via drag handle.
- [ ] **Rich text composer** (ProseMirror) — inline formatting,
      slash commands, code blocks, @-file references.
- [ ] **Dark/light theme toggle** (override `prefers-color-scheme`).
- [ ] **Conversation search** — full-text search across all
      sessions via a new `GET /search?q=` endpoint (requires
      proxy-side addition).
- [ ] **Keyboard shortcuts** — `Cmd+K` for command palette, `Cmd+S`
      for session switcher, `Cmd+Shift+F` for search.

### 4.3 Future (phase 3+)

- [ ] **Offline mode** — ServiceWorker caching of recent sessions,
      PWA manifest, install prompt.
- [ ] **Multi-model dashboard** — if the proxy exposes multiple
      provider backends, a pane showing live cost/token stats.
- [ ] **Audio/video call** — WebRTC-based pair session sharing
      (very speculative).

## 6. Implementation phases

### Phase 0 — Scaffold

1. `mkdir franky-web-ui && cd franky-web-ui && npm create vite@latest . -- --template lit-ts`
2. Wire `vite.config.ts` with proxy for `/events`, `/prompt`, etc.
   pointing at `http://localhost:9090` (the franky proxy).
3. Add `deps`: `lit`, `unified`, `remark-parse`, `remark-gfm`,
   `remark-rehype`, `rehype-stringify`, `dompurify`, `shiki`.
4. Replicate the in-binary MVP's SSE event dispatch + REST calls
   in TypeScript (port from `app.js`).
5. Achieve **functional parity** with the in-binary MVP.

**Deliverable:** The sibling project loads in a browser, connects
to a running `franky --mode proxy`, and renders the same chat UX
as the in-binary MVP. All three files (`app.ts`, `style.css`, `index.html`)
are replaced.

### Phase 1 — Markdown + highlighting

1. Replace the hand-rolled `Markdown` renderer with a unified/remark
   pipeline:
   - `remark-parse` + `remark-gfm` (tables, task lists, strikethrough).
   - `remark-rehype` + `rehype-stringify` → HTML string.
   - Run through DOMPurify before `innerHTML`.
2. Add `code-block.ts` — a Lit element that detects `<pre><code>`
   in rendered content, extracts the language class, and re-renders
   with Shiki highlighting.
3. Wire Shiki lazy-loading: only fetch grammar for languages that
   appear in the document (default: `typescript`, `zig`, `json`, `bash`,
   `python`, `yaml`).

**Deliverable:** Rendered assistant messages have proper tables,
task lists, linked references, and beautiful code blocks.

### Phase 2 — Artifacts + file upload

1. Add `artifact-viewer.ts` — detects file uploads (drag-and-drop
   on the composer) and renders known types:
   - `.pdf` → embedded `<canvas>` (pdfjs-dist).
   - `.xlsx` / `.csv` → parsed to HTML table (SheetJS).
   - `.png` / `.jpg` / `.gif` / `.svg` → inline `<img>`.
   - `.json` / `.log` / `.txt` → syntax-highlighted pre block.
2. File upload flow: POST to `/upload` (new proxy endpoint returning
   an artifact ID), attach the artifact ID to the next `/prompt`.
   Fallback: base64-inline for small files.

**Deliverable:** Users can drag a PDF onto the composer, and it
renders inline in the chat as an artifact card.

### Phase 3 — Multi-pane layout + polish

1. Implement `app-shell.ts` with CSS Grid three-pane layout:
   - Left: sidebar (collapsible).
   - Center: chat pane.
   - Right: artifact preview (opens when an artifact is clicked;
     closed by default).
2. Add drag handle for pane resizing.
3. Add dark/light theme toggle (user override of `prefers-color-scheme`).
4. Add conversation search (requires new `GET /search` endpoint).

**Deliverable:** A full-featured multi-pane chat UI.

### Phase 4 — Rich composer + ProseMirror

1. Replace `<textarea>` with a ProseMirror editor (via `@prosemirror-adapter/lit`).
2. Wire slash commands (/help, /clear, /retry, /edit, /compact)
   with the existing `/command` endpoint.
3. Add formatting toolbar (bold, italic, code, link).
4. Add @-file references (autocomplete file paths in the workspace).

**Deliverable:** A modern rich-text composer indistinguishable
from a native app.

## 7. Testing strategy

- **Unit tests:** Vitest for `lib/` modules (markdown pipeline,
  sanitize, API client, SSE client).
- **Component tests:** `@web/test-runner` (or Playwright) for
  Lit component rendering — each component renders in isolation
  with mock SSE events.
- **Integration tests:** Playwright driving a real browser against
  a Vite dev server proxied to a franky test instance
  (`franky --mode proxy --no-session` with canned transcripts).
- **Fixture replay:** Reuse the `test/fixtures/` directory from
  franky-core (the `franky replay` output). The UI connects to a
  proxy that replays a fixture's events via
  `franky replay <trace> --sse` (new mode that emits fixture
  events as SSE over a real HTTP connection).

## 8. Integration with franky-core

`franky-web-ui` does **not** modify franky-core. The only change
needed (if desired) is a small flag on the proxy:

```
--web-ui-root <path>    # serve custom static files instead of
                        # the built-in @embedFile UI
```

This is ~20 LOC in `proxy.zig` (check if the flag is set; if so,
serve files from that directory; otherwise fall back to embedded
files). It's a **phase 0 optional** — the sibling project works
fine standalone via Vite proxy.

## 9. Open questions

1. **Should `franky-web-ui` live in the franky monorepo or a
   separate repo?** The sibling projects `franky-do` and
   `franky-pods` are separate repos. `franky-web-ui` could be a
   directory inside the main franky repo (under `sibling/` or
   `web/`) for easier cross-repo CI, or fully separate.
   Recommendation: **separate repo** (`github.com/franky-ai/franky-web-ui`)
   to match the sibling convention.

2. **Who maintains it?** The MVP UI will remain in franky-core
   forever. The sibling project is maintained by whoever deploys
   it. Franky-core maintainers review the `--web-ui-root` flag
   but don't own the sibling code.

3. **Shiki vs highlight.js build step.** Shiki requires a build
   step to pre-generate highlighted themes. For a Vite project
   this is handled by a Vite plugin (`vite-plugin-shiki`). If
   that's unavailable, fall back to highlight.js with a curated
   language subset.

4. **ProseMirror + Lit integration maturity.** `@prosemirror-adapter/lit`
   may be immature. Fallback: keep `<textarea>` with enhanced
   slash-commands and code-block insertion (the MVP approach,
   but in TypeScript). Rich-text ProseMirror becomes phase 4+
   only if the adapter is stable.
