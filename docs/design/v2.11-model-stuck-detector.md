# Stuck in a loop Model Detector

- Repeat-error detector: 4 consecutive Zig API errors with similar shape ("no member named 'X' in 'Y'") should flag as STUCK_PATTERN, distinct from individual TOOL_ERROR. LAo consider general stuck in loop detection of all avialbe tools, e.g. 5 consecutive errors of similar shape, or 3 consecutive errors with the same tool_code or 10 successive calls to the same tool with same args in short amount of time. ...

How the agent harness can help to detect this and give hints or instructions to the model to get out of the loop? E.g. "It seems you are stuck in a loop of calling tool X with args Y, which is causing errors. Please check if you are using the tool correctly or if there is an issue with your logic."

Also it should consider to stop the model loop and instruct it to document the state and the issue to start a fresh new loop with a clean state. E.g. "It seems you are stuck in a loop of calling tool X with args Y, which is causing errors. To prevent further issues, I will stop the current loop. Please document the current state and the issue you are facing, and then we can start a fresh new loop with a clean state." (The new loop could even be started automatically after the documentation step.)


## §3 — Repeat-Error / Stuck-Pattern Detector (§2 of the original doc + v2.11)

Detect when the model calls the same tool with similar args and gets the same error repeatedly. Inject a structured hint or, on escalation, stop the loop and ask for a fresh start.

### Q3.1 — What constitutes "similar"?

**Proposed for v1:** Same tool name + same `tool_code` value on the error result (e.g. `edit_no_match`, `path_escape_workspace`, `bash_timeout`). This is the strongest signal and avoids fuzzy matching.

**v2 addition:** Add arg-hash similarity — SHA-256 of the canonical JSON args. If the model passes the same file path with slightly different edit content, that's still the same *kind* of error.

### Q3.2 — Threshold for hint

**Proposed:** After 4 consecutive same-error calls (matching tool_code) within a single turn or across consecutive turns. The counter resets when the model makes a successful tool call (any tool, no error).

**Decision:** _____

### Q3.3 — Escalation: from hint to hard stop

**Proposed for v1 (mirrors v2.11):**
1. After 4 consecutive same-error calls → inject a hint: "It seems you are stuck in a loop of calling tool X with args Y, which is causing errors. Please check if you are using the tool correctly or if there is an issue with your logic."
2. After 8 consecutive same-error calls → hard stop. Emit a turn-end, push a user message: "It seems you are stuck in a loop. To prevent further issues, I will stop the current loop. Please document the current state and the issue you are facing, and then we can start a fresh new loop with a clean state." The loop continues after the model's documentation response.

**v2 aspiration:** After the documentation step, automatically start a new loop (no user intervention needed).

**Decision:** _____

### Q3.4 — Where does the detector live?

**Proposed:** A new module `src/agent/stuck_detector.zig`. The `Agent` struct holds a `StuckDetector` instance. The `after_tool_call` hook calls `.record(tool_name, tool_code)` on it. The `between_turns` hook calls `.check()` which returns an optional hint message. The loop injects the hint into the transcript if non-null.

**Decision:** _____

### Q3.5 — What about non-error stuck patterns?

v2.11 mentions detecting "10 successive calls to the same tool with same args in a short amount of time" even without errors.

**Proposed:** Defer to v2. Error-only detection covers the most common and most harmful cases. Adding non-error pattern detection (e.g. the model keeps reading the same file over and over) is lower priority and risks false positives.
