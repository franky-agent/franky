# Smartphone Notification for Franky Agent Events

**Status:** draft — open for discussion  
**Target:** Android (primary)  
**Version:** v2.16

## Problem

Franky runs as a long-lived agent on a Linux/macOS server. When the agent finishes a task (`finish_task` tool fires) or hits a non-retryable error (`is_fatal: true`), there is currently no mechanism to notify the user outside of the terminal session. The user may be away from the keyboard and wants a push notification on their Android phone.

## Goal

Send a push notification to an Android phone when:
1. A `finish_task` event is emitted (task completed successfully).
2. A non-retryable (`is_fatal = true`) `agent_error` event is emitted.
3. (Stretch) A `max_turns_exceeded` event is emitted even though it is not fatal — the task is stuck without explicit failure.

The solution must be **free of per-message costs** (no SMS) and **as dead-simple as possible** — no custom Android app, no extra daemon to run if avoidable. The user just wants to see a notification pop up on their phone when franky finishes or fails.

## Existing Architecture (Franky Event System)

Franky already has an **observer/subscriber pattern** that can be leveraged:

- `Agent` exposes `subscribe(handler, userdata) → u32` and `unsubscribe(id)`.
- The worker thread drains events from `AgentChannel` and broadcasts each `AgentEvent` to all subscribers via the `broadcast` method.
- `AgentEvent` is a tagged union with variants including `.agent_error` (carries `ErrorDetails` with `is_fatal`, `message`, `code`) and other variants for turn/message/tool lifecycle events.

The `finish_task` tool is a guardrail tool that sets `terminate = true` on the loop. It is **not currently emitted as a distinct `AgentEvent`**. Instead, the loop sees `terminate` on a tool result and closes the channel with a final `turn_end`. This means for finish detection, we need to either:

- **a)** Add a new `AgentEvent.task_finished` variant that is emitted when the tool fires, or  
- **b)** Detect finish by subscribing to `tool_execution_end` and inspecting the tool name and result.

**Recommendation:** Option (a) — add a dedicated `AgentEvent.task_finished` variant. It is cleaner, avoids string-compare on tool names, and gives future event hooks a stable type to match on.

## Simpler Options (No Extra Daemon Needed)

### Option A: Email via SMTP (GMail) — The Simplest

**Concept:** Send an email to your own GMail address. The GMail Android app already has push notifications enabled by default. No extra app to install, no server to run, no WebSocket battery drain — the phone buzzes for email already.

**How it works:**

```
franky ──subprocess──> msmtp ──SMTP──> smtp.gmail.com:587 ──> GMail push ──> Android phone
```

Franky calls `msmtp` (a 100 KB static binary) to send the email. Or franky opens an SMTP connection directly using its vendored HTTP+TLS client.

**Trigger from franky:** The simplest path — spawn a subprocess:

```zig
// Using msmtp (installed separately)
const argv = [_][]const u8{
    "msmtp", "--from=default", "-t",
};
const input = try std.fmt.allocPrint(allocator,
    \\To: {s}\n\
    \\Subject: ✅ Franky: Task completed\n\
    \\From: {s}\n\
    \\feat(api): handle null pointer\n\
    \\Summary: Fixed edge case in user lookup\n\
, .{ to_email, from_email });
_ = try std.process.Child.run(.{ .argv = &argv, .stdin = .{ .string = input } });
```

Or send directly using franky's vendored HTTP+TLS client to connect to `smtp.gmail.com:587`, STARTTLS, and send the email without any external binary.

**Setup complexity:** Very Low. Two steps:
1. Create a [Google App Password](https://support.google.com/accounts/answer/185833) for your GMail account (16-char password, app-specific).
2. Configure msmtp or inline SMTP with that password.

Minimal `~/.msmtprc`:
```
defaults
tls on
tls_trust_file /etc/ssl/certs/ca-certificates.crt

account default
host smtp.gmail.com
port 587
auth on
from yourname@gmail.com
user yourname@gmail.com
passwordeval echo 'your-app-password'
```

**Cost:** Free. GMail accounts are free. Sending via SMTP is free and unlimited (GMail's send limit is 500 recipients/day for automated senders, but for personal use to your own inbox it's effectively unlimited).

**Reliability:** Very high. Google's SMTP infrastructure is world-class. The email arrives in your GMail inbox within seconds, and GMail's push notification delivers it to your phone instantly.

**Notification on Android:** When the email arrives, GMail's push notification shows the subject line and snippet on your lock screen — same as any other email. No extra app needed.

**Richness:**
- Subject line = notification title.
- Body = notification content (first ~2 lines show in the notification shade).
- No priority levels, tags, or action buttons — but you get the email in your inbox for later reference.
- Full message history searchable in GMail.

**Pros:**
- **No extra app** — every Android phone already has GMail with push enabled.
- **No daemon to run** — no self-hosted server, no Docker, no maintenance.
- **Already works** — the user's phone already buzzes for email.
- **Inbox history** — every notification is stored and searchable.
- **Zero new infrastructure** — uses an existing GMail account.

**Cons:**
- Requires a Google account (most people have one).
- App Password needed (2FA must be enabled on the GMail account, which is recommended anyway).
- The app password must be stored in franky's secret management system.
- Requires internet access to smtp.gmail.com (SMTP port 587).
- Less notification richness — no priority levels, no tags/emojis.
- Email subject/body format is plain text (can use HTML for basic formatting).
- Notifications are mixed with all other GMail notifications (can mitigate with a filter + label).

### Option B: ntfy.sh Public Cloud (No Server, Just an App)

**Concept:** Use the free public ntfy.sh server instead of self-hosting. Send a `curl`-style HTTP POST to `ntfy.sh/your-secret-topic`. Install the ntfy Android app, subscribe to the same topic, and receive native push notifications. **No server to run.**

**How it works:**

```
franky ──HTTP POST──> ntfy.sh:443 ──WebSocket──> Android ntfy app
```

The public ntfy.sh server is free, requires no account, and has a reasonable free tier (250 messages/day). Topics are created on the fly — the topic name is your secret (like a password).

**Trigger from franky:** One HTTP POST using the existing vendored HTTP client:

```zig
const url = try std.fmt.allocPrint(allocator,
    "https://ntfy.sh/franky-{s}", .{secret_topic});
const body = try std.fmt.allocPrint(allocator,
    "✅ Task completed: {s}\n\n{s}",
    .{ commit_message, summary });
// Use franky's vendored HTTP client to POST
```

Or from shell:
```bash
curl -d "Task completed" https://ntfy.sh/franky-mysecret123
```

**Setup complexity:** Minimal.
1. Install ntfy Android app from F-Droid or Play Store.
2. Pick a long random topic name (acts as password).
3. Subscribe to the topic in the app.
4. Send HTTP POSTs from franky.

**Cost:** Free. Public ntfy.sh has a free tier: 250 messages/day, 15 MB attachments. No credit card. No account sign-up. Pro tier ($10/mo) for 20k messages/day if needed (unlikely for task notifications).

**Reliability:** Depends on the public ntfy.sh service (operated by the project author). The service has been running since 2022 with high uptime. Messages are ephemeral — if the Android app is offline, it misses messages (no persistent queue unless you self-host).

**Richness:** Same as self-hosted ntfy (Option C): priority levels, tags/emojis, title, click actions.

**Pros:**
- **No server to run** — uses the free public ntfy.sh.
- **No Docker, no maintenance** — just the Android app.
- Single HTTP POST from franky — 3 lines of code.
- Rich notifications (priority, tags, emoji, click actions).
- Works on degoogled phones (no Play Services needed).

**Cons:**
- Third-party dependency: ntfy.sh is operated by the project author (Philipp Heckel). It's a free service, could change terms or go down.
- Topic name is your only secret — if someone guesses it, they can read your notifications. Use a long random string.
- Free tier limit: 250 messages/day. Likely fine for task_finish + error events (probably <10/day).
- Android app needs background WebSocket connection (some battery drain).
- Messages are ephemeral — no history if the app was offline.

## Self-Hosted Options (More Control)

### Solution 1: ntfy (Self-Hosted)

**Architecture:** Self-hosted Go server on the same machine as franky (or a nearby LAN host). Publishes messages via HTTP PUT/POST to a topic. Android app subscribes to the topic over WebSocket and shows a system notification.

**Pricing** https://ntfy.sh/#pricing

**How it works:**

```
franky ──HTTP POST──> ntfy server (localhost:8080) ──WebSocket──> Android ntfy app
```

- ntfy server runs as a Docker container or native binary.
- Topics are URI paths: `POST http://ntfy:80/franky-alerts` with message body.
- Android app connects to the self-hosted server URL, subscribes to `franky-alerts`, and gets native push notifications with priority, title, tags/emojis, action buttons.

**Trigger from franky:** The notification subscriber in franky uses the existing vendored HTTP client to `POST` to `http://127.0.0.1:8093/franky-{event_type}` with the event payload as the body.

**Setup complexity:** Low. Single `docker run` command. Create a `server.yml` with auth if desired.

**Cost:** Free. Open source (Apache 2.0 / GPLv2). No per-message fees.

**Reliability:** Depends on network connectivity between franky server and ntfy server. Can run on same machine (localhost). If ntfy server is down, notifications are queued only briefly (cache TTL configurable). No third-party dependency when self-hosting.

**Richness:**
- 5 priority levels (min, low, default, high, urgent) → map to Android vibration/sound levels.
- Title, message body, tags/emojis, click actions (open URL, etc.).
- Attachments (file URLs up to 15 MB).
- E2E encryption (client-side) for message bodies.
- Scheduled delivery (not needed here but available).

**Android app:** Available on F-Droid and Play Store (100k+ downloads, 4.8★). Works on degoogled ROMs. Foreground service with persistent notification for connection status (minimizable). Battery optimization must be disabled.

**UnifiedPush support:** ntfy is the reference UnifiedPush distributor, meaning any app that supports UnifiedPush can use ntfy as its push backend. This is a future-proofing benefit.

**Community:** Very active. 28.5k+ GitHub stars. 160+ contributors to Android app. Recent releases (v1.24.0 Android app 2026-03-07, v2.19.2 server).

**Pros:**
- Trivial to trigger (curl or HTTP POST from any language).
- Self-hosted, no third-party dependency.
- Rich notification features (priority, click actions, tags).
- Active community and well-maintained.
- Works without Google Play Services (degoogled phones).

**Cons:**
- Need to run and maintain another server daemon.
- Android app needs a persistent background WebSocket connection (uses some battery).
- Topic-based (no auth by default; should enable auth for production).

**Auth/security for MVP:**
```yaml
# server.yml
auth-file: "/var/lib/ntfy/auth.db"
auth-default-access: "deny-all"
```
Then add a user: `ntfy user add --role=admin franky_bot`
Set perms: `ntfy access franky_bot 'franky-*' read-write`

**Docker Compose snippet:**
```yaml
services:
  ntfy:
    image: binwiederhier/ntfy:latest
    container_name: ntfy
    command: serve
    volumes:
      - ntfy-cache:/var/cache/ntfy
      - ./ntfy-server.yml:/etc/ntfy/server.yml:ro
    ports:
      - "8093:80"
    restart: unless-stopped
volumes:
  ntfy-cache:
```

**When to choose: Self-hosted ntfy over public ntfy.sh:**
- You need message history/persistence (messages survive app offline).
- You want control over the data (no third-party server).
- You need more than 250 messages/day.
- You want auth for the topic.
- You work air-gapped / no internet.

### Solution 2: Gotify

**Architecture:** Self-hosted Go server. Uses an application-token model: create an "app" in the web UI, get a token, POST messages with the token. Android app connects via WebSocket.

**How it works:**

```
franky ──HTTP POST──> Gotify server (localhost:8080) ──WebSocket──> Android Gotify app
```

**Trigger from franky:** `POST http://gotify:80/message?token=APP_TOKEN` with `title`, `message`, `priority` form fields.

**Setup complexity:** Medium. Docker run, then web UI to create app + token. Auth is always required (username/password).

**Cost:** Free. MIT License.

**Reliability:** Self-hosted, no third-party dependency. Messages are stored in SQLite/MySQL/PostgreSQL so they persist if the Android client is offline.

**Richness:**
- Priority 0–10 (0 = silent, 8–10 = sound+vibration).
- Title and message body.
- No native tags/emojis or click actions (must use message extras).
- No E2E encryption.
- No scheduled delivery.

**Android app:** Available on F-Droid and Play Store (10k+ downloads, 4.9★). Shows a foreground service notification for connection status (minimizable). Battery optimization must be disabled.

**Cons vs ntfy:**
- No UnifiedPush support.
- Requires app token per notification source (more setup).
- Less notification richness (no tags, no click actions, no attachments natively).
- Smaller community (1.3k stars on Android repo).

**Pros:**
- Messages persist in db — history accessible from web UI.
- Web dashboard with message management.
- Plugin system (Go-based).

**Recommendation:** Gotify is viable but strictly inferior to ntfy for this use case. Only choose if you already run Gotify.

### Solution 3: Telegram Bot API

**Architecture:** Create a Telegram bot (via @BotFather). Use the bot token to send messages to a specific chat (your own Telegram account) via the Bot API. No server daemon needed on your side — the Telegram servers handle delivery.

**How it works:**

```
franky ──HTTP POST──> api.telegram.org/bot<TOKEN>/sendMessage ──> Telegram servers ──> Android Telegram app
```

**Trigger from franky:** `POST https://api.telegram.org/bot<TOKEN>/sendMessage` with JSON body `{ "chat_id": "...", "text": "...", "parse_mode": "HTML" }`.

**Setup complexity:** Low-medium. Create bot with @BotFather (2 minutes), get chat_id by sending a message to the bot and fetching updates, store the token in franky's secret management system.

**Cost:** Free. No per-message charges. Requires a Telegram account (free).

**Reliability:** Depends on Telegram's infrastructure (highly reliable). No server to maintain. However, it requires internet access to api.telegram.org, which may be blocked in some environments. Rate limits apply (~30 messages/second per chat).

**Richness:**
- Full markdown/HTML formatting.
- Inline keyboards, buttons, images, files, polls.
- No native priority levels (but can use Telegram's "silent send" parameter).
- Message threading, editing, deletion.

**Android app:** The official Telegram app is arguably the most reliable notification channel — Telegram push notifications are delivered via FCM (or its own persistent connection on degoogled phones). The user is likely to have Telegram installed already.

**Pros:**
- No server to self-host — Telegram handles everything.
- Extremely reliable delivery.
- Very rich message formatting.
- Existing Telegram app (user likely already has it).
- Can be extended to interactive features (approve/deny actions via inline keyboards).

**Cons:**
- **Third-party dependency.** Telegram is a centralized service. The message passes through Telegram's servers.
- Requires internet access to api.telegram.org.
- Bot token must be kept secret — if leaked, anyone can send messages as your bot.
- Rate-limited (30 msg/s, but also per-chat limits).
- Can't work offline / air-gapped.
- Notification is delivered to the Telegram app, not as a native system notification from a franky-specific app (though Telegram's notification is itself a system notification).

**Secret management:** The bot token is the only secret. It should be stored in franky's secret management system (v2.15 — zero-trust, in-memory only, not accessible by bash tool).

### Solution 4: Signal CLI (signal-cli)

**Architecture:** Install `signal-cli` on the franky server, register a phone number, and use it to send messages to your own Signal account. The Signal protocol uses end-to-end encryption.

**How it works:**

```
franky ──JSON-RPC──> signal-cli daemon (localhost:8080) ──Signal Protocol──> Signal servers ──> Android Signal app
```

**Trigger from franky:** `POST http://localhost:8080/api/v1/rpc` with JSON-RPC payload calling `send` method with account and recipient.

**Setup complexity:** High.
1. Need a phone number for registration (receives SMS/voice verification).
2. Requires Java Runtime Environment (JRE 25+).
3. Registration may require CAPTCHA solving.
4. Must keep signal-cli updated (older than 3 months may break due to Signal server changes).
5. The Signal protocol expects regular message receiving for encryption efficiency.

**Cost:** Free. Open source (GPLv3).

**Reliability:** Moderate. Depends on Signal servers. Registration is a barrier. Releases older than 3 months may stop working due to Signal protocol changes.

**Richness:**
- End-to-end encryption.
- Text messages only (no rich formatting like Telegram).
- Can send to usernames or phone numbers.
- No priority levels, tags, or click actions.

**Pros:**
- End-to-end encryption (privacy-conscious).
- No third-party notification service — uses Signal's existing infrastructure.
- Works on degoogled phones (Signal app uses its own persistent connection).

**Cons:**
- Requires a phone number and SMS/voice verification.
- Java dependency (JRE 25+).
- signal-cli must be kept very up-to-date.
- Registration complexity includes potential CAPTCHA.
- Limited message richness (text only).
- No priority levels for Android notifications.
- signal-cli daemon must run continuously for encryption efficiency.

**Recommendation:** Too heavyweight for an MVP. The registration burden and Java dependency make this a poor fit for the initial implementation. Revisit if E2E encryption is a hard requirement.

### Solution 5: Firebase Cloud Messaging (FCM)

**Architecture:** Build a custom Android app (or use an existing FCM client) that receives messages from franky via Firebase Cloud Messaging. franky sends HTTP v1 API requests to FCM with the device registration token.

**How it works:**

```
franky ──HTTP POST──> fcm.googleapis.com/v1/projects/<PROJECT>/messages:send
    ──> FCM servers ──> Android FCM app (custom)
```

**Trigger from franky:** `POST https://fcm.googleapis.com/v1/projects/{project}/messages:send` with OAuth 2.0 bearer token and JSON payload targeting the device token or topic.

**Setup complexity:** Very high.
1. Create a Firebase project and enable Cloud Messaging.
2. Generate a service account JSON key.
3. Build and deploy a custom Android app (or modify an existing one) that registers for FCM and displays notifications.
4. Handle OAuth 2.0 token refresh on the server side.
5. Manage device registration tokens.

**Cost:** Free tier is generous (first 20,000 messages/day free, then $0.25/million on Spark plan). Requires a Google account and Firebase project.

**Reliability:** Very high (Google's infrastructure). Works even if the app is in the background or the phone is in doze mode.

**Richness:**
- Maximum — fully customizable notification payloads.
- Priority levels, sounds, vibration patterns.
- Data messages (app processes in background) vs notification messages (system handles).
- Click actions, deep links, images.

**Android app:** Requires building a custom app (or using a generic FCM receiver). This is the highest-friction aspect.

**Pros:**
- Most reliable delivery (FCM has persistent connection to Google Play Services).
- Works when app is killed, phone is in doze mode.
- Highest notification richness.
- No persistent background service on Android (FCM handles the connection).

**Cons:**
- **Requires Google Play Services** (does not work on degoogled phones like GrapheneOS).
- **Requires building and maintaining a custom Android app.**
- Third-party dependency on Google infrastructure.
- Need to manage OAuth 2.0 tokens and service account keys.
- Device registration token management (tokens can change).

**Recommendation:** Overkill for MVP. The custom Android app requirement is a significant barrier. Consider only if all other solutions are unacceptable or if a custom Android app is desired for other reasons.

## Recommendation

**For the truly simplest MVP: Email via GMail.** Zero infrastructure. Zero extra apps. The phone already buzzes for email. It's 20 lines of Zig code to send an SMTP message using franky's existing HTTP+TLS client.

**If you want richer notifications (priority, tags, emoji): Use ntfy.sh public cloud.** One HTTP POST, install one Android app, no server to run. The free tier (250 msg/day) is more than enough.

**If you want full control or work air-gapped: Self-host ntfy.** Single Docker container, same HTTP POST API.

Tiebreaker table:

## Integration Design

### 1. New `AgentEvent` variant: `task_finished`

To cleanly detect `finish_task`, add a new event variant:

```zig
// in agent/types.zig, inside AgentEvent union:
task_finished: struct {
    commit_message: []const u8,
    summary: []const u8,
},
```

Emit this from `finish_task.zig`'s `execute` function after setting state and before returning the tool result with `terminate=true`.

### 2. Notification subscriber module

New file: `src/coding/notifications.zig`

```zig
pub const NotificationConfig = struct {
    /// The notification provider (ntfy, gotify, telegram)
    provider: enum { ntfy, gotify, telegram },
    /// Base URL for self-hosted services (e.g. "http://192.168.1.50:8093")
    base_url: []const u8,
    /// Authentication token / password
    auth_token: ?[]const u8 = null,
    /// Topic name (ntfy) or app token (gotify) or chat_id (telegram)
    destination: ?[]const u8 = null,
    /// Minimum severity level to notify
    min_severity: enum { all, errors_only, finish_only } = .all,
};
```

### 3. Wiring into the Agent

The notification subscriber registers itself via `Agent.subscribe()`:

```zig
// In print.zig or wherever the Agent is initialized:
if (cfg.notification) |notif_cfg| {
    const state = try Notifier.init(allocator, notif_cfg);
    // state lives as long as the session
    _ = try agent.subscribe(Notifier.handler, state);
}
```

The handler receives `AgentEvent` callbacks and dispatches HTTP requests for the events we care about.

### 4. Settings integration

In `settings.zig`, add:

```zig
notification: ?struct {
    provider: []const u8,   // "ntfy", "gotify", "telegram"
    base_url: []const u8,
    destination: []const u8,
    min_severity: []const u8 = "all", // "all", "errors_only", "finish_only"
} = null,
```

Settings file format in `~/.franky/settings.json`:

```json
{
  "notification": {
    "provider": "ntfy",
    "base_url": "http://192.168.1.50:8093",
    "destination": "franky-alerts",
    "min_severity": "all"
  }
}
```

### 5. CLI flags (future)

```
--notify-provider ntfy
--notify-url http://192.168.1.50:8093
--notify-destination franky-alerts
--notify-on errors
```

### 7. Notification message format

**ntfy (MVP):**

When `finish_task` fires:
```
POST /franky-finish HTTP/1.1
Content-Type: text/plain

✅ Task completed: feat(api): handle null pointer

What: Fixed null pointer in user lookup API
Summary: 3 files changed, 42 insertions
```

When `agent_error` (fatal) fires:
```
POST /franky-error HTTP/1.1
Content-Type: text/plain

❌ Agent error: rate_limited

Provider returned 429 — rate limit exceeded.
Model: claude-sonnet-4-6
```

**With JSON API (richer):**
```json
POST /franky-finish HTTP/1.1
Content-Type: application/json

{
  "title": "✅ Task completed",
  "message": "feat(api): handle null pointer",
  "priority": 5,
  "tags": ["white_check_mark"],
  "click": "http://franky.local/logs/session-123"
}
```

### 8. Notification event flow diagram

```
┌─────────────────────────────────────────────────────┐
│                    Agent Worker                       │
│                                                       │
│  agentLoop() → events → drain loop → broadcast(ev)   │
│                                          │            │
└──────────────────────────────────────────┼────────────┘
                                           │
                              ┌────────────┴────────────┐
                              │    Notifier (handler)    │
                              │                         │
                              │  match ev:               │
                              │    .task_finished → POST │
                              │    .agent_error          │
                              │      if fatal → POST     │
                              │    .turn_end             │
                              │      if finish_task was  │
                              │      pending → POST      │
                              │                         │
                              └────────────┬────────────┘
                                           │ HTTP POST
                                           ▼
                              ┌─────────────────────────┐
                              │   ntfy server (:8093)    │
                              │   (or Gotify / Telegram) │
                              └────────────┬────────────┘
                                           │ WebSocket / FCM
                                           ▼
                              ┌─────────────────────────┐
                              │   Android Phone          │
                              │   (notification popup)   │
                              └─────────────────────────┘
```

### 9. Security considerations

- For self-hosted solutions (ntfy/Gotify), use authentication (token or username/password).
- The ntfy auth token or Gotify app token should be stored in franky's secret management system (v2.15), not in settings.json.
- For Telegram, the bot token must be in secret management.
- ntfy topics are essentially passwords — use a long, random, unguessable topic name like `franky-a3f8c2b1-9e4d` if not using auth.
- Consider HTTPS for the ntfy/Gotify server if not on localhost, with a self-signed cert or Let's Encrypt.

### 10. Future enhancements

- **Action buttons:** ntfy supports action buttons — "View log", "Acknowledge" could be added to notifications.
- **Notification queue:** If the server is offline and notifications accumulate, ntfy's cache stores them for configurable TTL.
- **Multiple destinations:** Send to multiple phones (team notifications).
- **Notification filtering:** Per-event-type opt-in/opt-out via settings.
- **Sub-agent notifications:** When a sub-agent completes or fails independently.
- **Two-way interaction:** Telegram inline keyboards could allow "Approve next dangerous tool call" from the phone.
- **Dedicated app:** If the team wants maximum control, build a custom Android app powered by FCM + ntfy as fallback. But this is well beyond MVP.

### Email via GMail (Simplest)

```bash
# Option A: Using msmtp (install once)
sudo apt install msmtp msmtp-mta  # Debian/Ubuntu
brew install msmtp                 # macOS

# Configure ~/.msmtprc:
cat > ~/.msmtprc << 'EOF'
defaults
tls on
tls_trust_file /etc/ssl/certs/ca-certificates.crt

account default
host smtp.gmail.com
port 587
auth on
from yourname@gmail.com
user yourname@gmail.com
passwordeval echo 'YOUR_16_CHAR_APP_PASSWORD'
EOF
chmod 600 ~/.msmtprc

# Test:
echo -e "Subject: Franky test\n\nHello from franky" | msmtp yourname@gmail.com

# Or Option B: Inline in Zig (no msmtp needed) — use franky's vendored
# HTTP+TLS client to connect to smtp.gmail.com:587 with STARTTLS.
```

**Getting a Google App Password:**
1. Go to https://myaccount.google.com/security
2. Enable 2-Step Verification (if not already on).
3. Go to "App Passwords" (search in Google Account settings).
4. Generate a new app password for "Mail" on your device.
5. Copy the 16-character password — this is your SMTP password.
