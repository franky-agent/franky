# To guide the Model

The harness could enforce some guardrails like:
* after 3 -5 edits run compilation (if success full run test) and if they fail then give hints to the model to fix the compilation issue before making more edits. This can help to prevent the model from making too many edits that are not working and getting stuck in a loop of making edits without testing them.

* after 3-5 edits with the same tool or similar args that are causing errors, give hints to the model to check if they are using the tool correctly or if there is an issue with their logic. This can help to prevent the model from getting stuck in a loop of calling the same tool with the same args that are causing errors.

* running gitleaks or other secret scanner after a certain number of edits to check if the model is accidentally introducing secrets into the codebase. If secrets are detected, give hints to the model to remove them before making more edits. This can help to prevent the model from introducing security vulnerabilities into the codebase.
