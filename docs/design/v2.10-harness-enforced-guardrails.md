# To guide the Model — Harness-Enforced Guardrails

The harness could enforce some guardrails like:

- after 3-5 edits run compilation (if success full run test) and if they fail then give hints to the model to fix the compilation issue before making more edits. This can help to prevent the model from making too many edits that are not working and getting stuck in a loop of making edits without testing them.

- after 3-5 edits with the same tool or similar args that are causing errors, give hints to the model to check if they are using the tool correctly or if there is an issue with their logic. This can help to prevent the model from getting stuck in a loop of calling the same tool with the same args that are causing errors.

- running gitleaks or other secret scanner after a certain number of edits to check if the model is accidentally introducing secrets into the codebase. If secrets are detected, give hints to the model to remove them before making more edits. This can help to prevent the model from introducing security vulnerabilities into the codebase.
  (mark the session as unsecure visual warning in the ui with a reason (secret detected)
  ** how to handle false positives ?

- also after a task is completed and the guardrails are passing auto commit (should be opt-in) and start a new task with a clean state. This can help to prevent the model from getting stuck in a loop of making edits without a clear goal or task.
  - create a new branch for the new task to keep the history clean and organized.
  - then branch of from this branch for the next task and so on.
  - or just use one branch per session and only have indiviual commits for each task.

## §2 — Compilation Check Trigger (§1 of the original doc)

Run compilation (and tests on success) after a threshold of source-code mutations. On failure, inject a structured hint suggesting the model fix the issue before making more edits.

### Q2.1 — Where does the check fire?

**Proposed:** The `between_turns` hook in `agent/loop.zig` (line 243). After a turn ends (tools executed, results collected) but before the next LLM call, check whether the edit/write count has crossed the threshold. If so, run the guard.

**Rationale:** Between turns is the natural pause point. Running during a turn would block tool execution; running after every tool call would be too frequent.

**Decision:** _____

### Q2.2 — What counts as "an edit"?

**Proposed:** Both `edit` and `write` tool invocations count as source-code mutations. `bash` invocations do not count (even if they run a compiler). The counter resets after a successful compilation.

**Rationale:** `write` is as much a mutation as `edit` — it creates/replaces files. `bash` could be anything (install deps, run tests, etc.) and is not reliably a mutation.

**Decision:** _____

### Q2.3 — Which build command to run?

**Proposed:** Configurable per-session/project. Default heuristic:
1. If `build.zig` exists in workspace root → run `zig build`
2. Else if `Makefile` or `makefile` exists → run `make`
3. Else if `Cargo.toml` exists → run `cargo build`
4. Else → skip compilation (no recognized build system)

The command should be overridable via a project-level config (e.g. a `.franky-build` file or a `settings.json` field).

**Decision:** _____

### Q2.4 — How are failures presented to the model?

**Proposed:** The compiler stdout/stderr is injected as a user-role message (or a tool_result message) into the transcript. The message is prefixed with a hint like:

> "The workspace no longer compiles. Here is the compiler output:\n\n```\n<output>\n```\n\nPlease fix the issue before continuing with further edits."

**Rationale:** Keeping it in the conversation as a regular message means the model sees it as context, not as a special magic signal. No new event types needed.

**Decision:** _____

### Q2.5 — Compilation run: blocking or background?

**Proposed for v1:** Blocking, with a 120-second timeout. If it times out, emit "compilation timed out" and let the model proceed.

**Rationale:** Non-blocking compilation that finishes after the next LLM call has already started would inject stale results. Blocking is simpler and correct. 120s covers most incremental builds.

**v2 aspiration:** Run compilation in a background thread. If it completes before the next LLM call finishes, inject results between turns as above. If not, let the call proceed and warn that compilation is still running.

**Decision:** _____

### Q2.6 — Threshold: fixed or configurable?

**Proposed:** Fixed at 5 mutations for v1. Promoted to a `settings.json` field (`compilation_guard_threshold`) once the feature stabilizes.

**Decision:** _____

### Q2.7 — Auto-recompile after fix?

Once the model fixes the issue (makes edits that satisfy the compiler), does the harness auto-recompile or wait for the next threshold?

**Proposed:** The harness re-runs compilation automatically after the *next* batch of edits crosses the threshold again, not after every individual tool call. This avoids a "compile → fix one line → compile → fix one line" ping-pong.

**Decision:** _____

### Q2.8 — False-positive: mid-refactor states

Some intermediate states are intentionally broken (e.g. renaming a symbol: half the call sites use the old name, half the new). How do we avoid pestering the model mid-refactor?

**Proposed:** The hint is advisory, not blocking. The message says "FYI: the workspace doesn't compile. Here's the error: ... You can keep editing or fix it now." The model decides urgency. We do not block further tool calls.

**Decision:** _____

## §4 — Secret Scanner (§3 of the original doc)

Run a secret scanner (gitleaks or built-in) after a threshold of mutations. If secrets are detected, warn the user and optionally hint the model.

### Q4.1 — Which scanner engine?

**Proposed for v1:** Built-in regex scanner for common patterns (AWS keys `AKIA[0-9A-Z]{16}`, GitHub tokens `gh[pousr]_[A-Za-z0-9_]{36,}`, generic `-----BEGIN.*PRIVATE KEY-----`, etc.). No external binary dependency.

**v2 aspiration:** Integrate `gitleaks` as an optional scanner when available. Parse its JSON output. Add a `.gitleaks.toml` allowlist mechanism for project-level false-positive management.

**Rationale:** A built-in scanner works offline, has zero installation friction, and covers the 80% case. Gitleaks is more thorough but adds a dependency and a false-positive management burden.

**Decision:** _____

### Q4.2 — How to handle false positives?

This is the hardest problem. Scanners flag example API keys in test files, long hex strings, etc.

**Proposed:**
- Findings are shown to the **user** via a UI warning, **not** injected into the model's context. The model should not waste time "fixing" example keys.
- The UI renders a warning banner: "🔓 Potential secrets detected in 3 files. Review with `gitleaks` or configure allowlist."
- A per-project allowlist (`.franky/secret-allowlist.json`) lists regex patterns or file paths to exclude. The harness checks findings against this list before surfacing the warning.

**Rationale:** Injecting false positives into the conversation would confuse the model and waste turns. Showing them to the user only keeps the model focused on the actual task.

**Decision:** _____

### Q4.3 — UI integration: "unsecure" marking

**Proposed:** A new `agent_warning` event kind:

```zig
agent_warning: struct {
    code: WarningCode,  // .secret_detected, .compilation_failed, .stuck_pattern
    message: []const u8,
    details_json: ?[]const u8,
}
```

The TUI (interactive mode) subscribes to this event and renders a persistent warning bar. The proxy mode's web UI shows a badge in the session sidebar. The warning persists until the finding is resolved or dismissed.

**Decision:** _____

### Q4.4 — Run frequency & performance

**Proposed:** Scan after every 10th mutation (edit/write call). Cache the scan result by git tree hash (if the workspace is a git repo) — skip re-scan if the tree hasn't changed since the last scan.

**Rationale:** Secret scanning is relatively fast (<1s for most projects with the built-in scanner), but scanning on every single edit would be wasteful for rapid iterators.

**Decision:** _____

## §5 — Auto-Commit on Task Completion (§4 of the original doc)

When a task is done and all guardrails pass, optionally auto-commit (opt-in) and start a new task with a clean state.

### Q5.1 — How does the harness know a task is done?

**Proposed:** A new `finish_task` tool (not a magic text signal). The tool schema:

```json
{
  "type": "object",
  "required": ["commit_message", "summary"],
  "properties": {
    "commit_message": {"type": "string", "description": "Git commit message describing this task's changes."},
    "summary": {"type": "string", "description": "What was done, what's still unknown, what to do next."}
  },
  "additionalProperties": false
}
```

The tool has `terminate: true` — the loop stops after it fires. The harness captures the commit_message and summary from the tool call.

**Alternative considered:** Magic `@@DONE@@` text marker in the model's output. Rejected because it's fragile (model may not emit it consistently) and unstructured (no commit message).

**Decision:** finish_task tool

### Q5.2 — When does the commit happen?

**Proposed:** After the `finish_task` tool fires, the harness runs all guardrails one final time:
1. Compilation check (must pass)
2. Secret scan (warnings only, not blocking)
3. If both pass → `git add -A && git commit -m "<commit_message>"`
4. If compilation fails → do not commit, inject the compiler error into the transcript and let the model fix it (loop continues)
5. If secrets detected → do not commit, surface the warning to the user, let the model fix it

**Rationale:** Committing broken code or secrets defeats the purpose of the guardrails. Final validation before commit is essential.

**Decision:** _____

### Q5.3 — Branching strategy

**Option A (proposed):** One branch per session (`franky/<session-ulid>`). Sequential auto-commits on that branch. Simple, clean, maps naturally to the existing session persistence (which already has `active_branch`).

**Option B:** Each task forks from `main` (or the previous task's branch). Branch name: `<session-ulid>/<task-index>`. More powerful for parallel task exploration, but complex to manage.

**Proposed for v1:** Option A. One branch per session. The branch is created (`git checkout -b franky/<ulid>`) at session start if `--auto-commit` is enabled.

**Decision:** A

### Q5.4 — Opt-in mechanism

**Proposed:** Opt-in via:
1. CLI flag: `--auto-commit`
2. Settings: `"auto_commit": true` in `settings.json`
3. Project config: `{"auto_commit": true}` in `.franky/config.json`

The interactive mode prompts on first task completion: "Auto-commit on task completion? [y/N]" if none of the above are set.

**Rationale:** Auto-commit is destructive (it writes to git history). Defaulting to off is the safe choice.

**Decision:** Off by default opt-in via CLI flag / settings / project config

### Q5.5 — Commit message generation

**Proposed:** The model provides the commit message in the `finish_task` tool call. The harness validates it (non-empty, printable ASCII) and uses it verbatim. No auto-generation.

**Rationale:** The model knows best what it changed. Auto-generated commit messages from diffs tend to be generic ("fix bugs") and miss context.

**Decision:** Use model-provided commit messages verbatim, validated for non-empty and printable ASCII.

## §6 — Event Model: New `agent_warning` Kind

Multiple guardrails need to surface non-fatal diagnostics to the UI (secret detected, compilation warning, stuck pattern). Rather than overloading `agent_error` (which implies the loop is stopping), add a new event kind.

### Q6.1 — Schema

**Proposed:**

```zig
pub const WarningCode = enum {
    secret_detected,
    compilation_failed,
    stuck_pattern,
};

// Added to AgentEvent union:
agent_warning: struct {
    code: WarningCode,
    message: []const u8,
    details_json: ?[]const u8,
    turn_index: u32,
}
```

The `agent_warning` event does not change the loop state — it's purely informational. The loop continues normally after emitting it.

**Decision:** Could we assume this this different secret detected, compilation warning, stuck pattern harness guardrails are just virtual
subagents of the main agent and then we can just use the existing `agent_error` event with a new `error_code` enum that includes these cases? This would avoid adding a new event kind and keep all diagnostics in one stream. The loop can decide to treat certain error codes as warnings (non-blocking) vs fatal errors (blocking). Also we still can add a subagent `type` field with `llm` or `guardrail` to distinguish the source of the event.

## §7 — Integration Architecture

How the guardrails wire into the existing codebase.

```
Agent (src/agent/agent.zig)
  └── guardrails: GuardrailState          // New field

    agentLoop (src/agent/loop.zig)
      │
      ├── After each tool execution (after_tool_call hook):
      │   ├── guardrails.stuck_detector.record(tool_name, tool_code)
      │   └── guardrails.compilation_counter.bump() // for edit/write
      │
      ├── Between turns (between_turns hook):
      │   ├── if guardrails.compilation_counter.should_compile():
      │   │     run_compile_and_tests() → inject result
      │   ├── if guardrails.stuck_detector.check()) |hint|:
      │   │     inject hint into transcript (or hard-stop if escalated)
      │   ├── if guardrails.secret_scanner.should_scan():
      │   │     run_scan() → emit agent_warning if findings
      │   └── return keep_going or stop
      │
      └── On finish_task tool:
          ├── Run all guardrails one final time
          ├── If --auto-commit and guardrails pass:
          │     git add -A && git commit
          └── Else: print summary, wait for next prompt
```

### Q7.1 — New files

**Proposed:**
- `src/agent/guardrails/stuck_detector.zig` — `StuckDetector` struct
- `src/agent/guardrails/compilation_guard.zig` — `CompilationGuard` struct + build-command detection
- `src/agent/guardrails/secret_scanner.zig` — `SecretScanner` struct with built-in regex patterns
- `src/agent/guardrails/finish_task.zig` — `FinishTaskTool` and auto-commit logic
- `src/agent/guardrails/guardrails.zig` — `GuardrailState` aggregate struct that owns all four

**Decision:** can we maybe use the existing hook system to avoid adding new files for each guardrail? For example, the stuck pattern detection logic could be integrated directly into the `after_tool_call` and `between_turns` hooks in `agent/loop.zig`, without needing a separate `StuckDetector` struct. Similarly, the compilation check could be a function called directly from the `between_turns` hook. This would reduce the number of new files and keep the guardrail logic closer to where it's executed. We can still organize the code in the guardrails directory just for readability technical those are just special predefined hooks rather than separate modules with state.

## §8 — Implementation Roadmap

### Phase 1 (highest impact, lowest effort)
1. **`StuckDetector`** — record error patterns, inject hints between turns (no hard stop yet)
2. Wire via `after_tool_call` hook + `between_turns` hook
3. Tests: fake provider with repeated error patterns

### Phase 2
4. **`CompilationGuard`** — detect build system, run zig/go/make, inject results
5. Edit/write counter with configurable threshold
6. Tests: fake workspace with broken build

### Phase 3
7. **`SecretScanner`** — built-in regex patterns for common secret formats
8. `agent_warning` event kind + TUI warning bar rendering + web UI badge
9. Allowlist mechanism for false-positive management

### Phase 4
10. **`finish_task` tool** — structured task completion
11. **Auto-commit** — git integration, branch management
12. Guardrail final validation pass before commit
13. Interactive mode prompt: "Auto-commit? [y/N]"

### Phase 5 (optional, deferred)
14. Gitleaks integration (external binary, JSON output parsing)
15. Background compilation (non-blocking compile during LLM call)
16. Arg-hash similarity in stuck detector
