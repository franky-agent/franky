# Agent API OpenAPI Specification — v3.2

Drive-by summary: Write and maintain a single OpenAPI 3.1.1 YAML spec
(`docs/api/agent-api-v1.yaml`) that describes every HTTP endpoint the
franky agent exposes in `--mode proxy`. The orchestrator consumes this
spec to generate Go client code and validate its assumptions.

## 1. Motivation

The franky agent and the franky-orchestrator are separate projects in
different languages (Zig vs Go). They communicate exclusively over HTTP.
Today the contract is documented in a Markdown design doc
(`v3.1-franky-orchestrator.md`, §3) as ad-hoc request/response blocks.
This works for human readers but:

- The orchestrator author must manually read the markdown and write Go
  structs + deserialization — a fragile, error-prone translation.
- There is no machine-readable source of truth. A field rename in Zig
  (`messageCount` → `message_count`) silently breaks the Go client.
- There is no versioning. If the agent API evolves, there is no way to
  tell which version the running agent implements.

An **OpenAPI 3.1.1** specification solves all three:

1. **Single source of truth.** Every endpoint, request body, response
   schema, and error code is defined once in `docs/api/agent-api-v1.yaml`.
2. **Code generation.** The orchestrator feeds the spec into
   `oapi-codegen` (Go) and gets typed client stubs, request/response
   structs, and server skeletons for free. No manual deserialization.
3. **Versioned contract.** The spec has an `info.version` field (e.g.
   `"1.0.0"`). The agent exposes it in `GET /health`. The orchestrator
   checks it at registration and warns on mismatch.

## 2. Scope

The spec covers every HTTP endpoint that a franky `--mode proxy` process
exposes. It does **not** cover the orchestrator's own HTTP API (that is a
separate Go project), nor does it cover orchestrator-side endpoints that
the agent calls (`/register`, `/unregister`).

**In scope** (Agent API — §3 of the orchestrator design doc):

| # | Method | Path | Purpose |
|---|--------|------|---------|
| 1 | `GET` | `/health` | Liveness probe |
| 2 | `GET` | `/events` | SSE event stream |
| 3 | `POST` | `/prompt` | Submit a user prompt |
| 4 | `POST` | `/abort` | Cancel in-flight turn |
| 5 | `POST` | `/interrupt` | Graceful stop after current turn |
| 6 | `POST` | `/restart` | Exit and re-spawn |
| 7 | `POST` | `/permission/resolve` | Resolve a pending tool permission |
| 8 | `POST` | `/command` | Execute a slash command |
| 9 | `GET` | `/transcript` | Current session transcript |
| 10 | `GET` | `/session` | Active session info |
| 11 | `GET` | `/sessions` | List available sessions |
| 12 | `GET` | `/sessions/{id}/transcript` | Session transcript by id |
| 13 | `POST` | `/session/new` | Create a new session |
| 14 | `POST` | `/session/activate` | Switch to a session |
| 15 | `GET` | `/role` | Role + provider + model info |
| 16 | `GET` | `/usage` | Tool call counters |
| 17 | `GET` | `/design-docs` | List design documents |
| 18 | `POST` | `/design-docs/archive` | Archive a design document |
| 19 | `GET` | `/` | Web UI (HTML) |
| 20 | `GET` | `/app.js` | Web UI script |
| 21 | `GET` | `/style.css` | Web UI stylesheet |
| 22 | `GET` | `/prism.js` | Syntax highlighter script |
| 23 | `GET` | `/prism-tomorrow.css` | Syntax highlighter theme |

**Out of scope** (orchestrator-side endpoints that the agent calls):

| Endpoint | Defined in |
|----------|-----------|
| `POST /register` | `v3.1-franky-orchestrator.md` §4.1 |
| `POST /unregister` | `v3.1-franky-orchestrator.md` §4.3 |

These are the orchestrator's own API — the orchestrator repo should
define them in its own OAS spec. The agent simply implements the HTTP
client calls.

## 3. File layout

```
franky/
├── docs/
│   ├── design/
│   │   ├── v3.1-franky-orchestrator.md   # existing — still the high-level design doc
│   │   └── v3.2-agent-api-openapi-spec.md  # this file
│   └── api/
│       ├── agent-api-v1.yaml               # THE spec — single file, no external refs
│       └── CHECKSHA                         # pinned SHA from orchestrator repo (see §5)
```

**One file, no splitting.** At ~23 endpoints, a single `agent-api-v1.yaml`
is manageable. Splitting into `paths/`, `schemas/`, `responses/` adds
tooling complexity without benefit at this size. If the spec grows beyond
50 endpoints we can split — but cross that bridge when we reach it.

**Naming convention:** `agent-api-v{major}.{minor}.yaml`. A breaking
change bumps the major version and produces a new file
(`agent-api-v2.0.yaml`). The old file stays for orchestrators that
haven't upgraded yet (see §5).

## 4. Spec structure

### 4.1 Top-level metadata

```yaml
openapi: "3.1.1"
info:
  title: "Franky Agent API"
  version: "1.0.0"
  description: >
    HTTP API exposed by a franky `--mode proxy` process. Consumed by
    the franky-orchestrator (Go) and by the built-in web UI.

servers:
  - url: "http://localhost:8787"
    description: "Default proxy port"
```

### 4.2 Paths

Every endpoint from §2 appears under `paths:`. Each path entry includes:

- `operationId` — stable identifier for codegen (e.g. `getHealth`,
  `postPrompt`, `getTranscript`).
- `parameters` — path/query/header params (e.g. `Last-Event-ID` for
  `GET /events`).
- `requestBody` — for POST endpoints with JSON bodies.
- `responses` — every status code the endpoint can return, each with a
  `$ref` to a `components.responses` entry.

### 4.3 Schemas — `components.schemas`

Reusable JSON Schema types. Each maps to a Go struct via `oapi-codegen`:

| Schema name | Purpose | Used by |
|-------------|---------|---------|
| `HealthResponse` | `{"ok":true}` | `GET /health` |
| `PromptResponse` | `{"ok":true}` | `POST /prompt` |
| `AbortResponse` | `{"ok":true,"aborted":true}` | `POST /abort` |
| `InterruptResponse` | `{"ok":true,"interrupted":true}` | `POST /interrupt` |
| `RestartResponse` | `{"ok":true,"restarting":true}` | `POST /restart` |
| `PermissionResolveRequest` | `{"call_id":"...","resolution":"..."}` | `POST /permission/resolve` |
| `PermissionResolveResponse` | `{"ok":true}` | `POST /permission/resolve` |
| `CommandResponse` | `{"ok":true,"output":"...","sideEffect":"...","data":...}` | `POST /command` |
| `TranscriptResponse` | `{"messages":[...]}` | `GET /transcript` |
| `Message` | single transcript message | `TranscriptResponse.messages[]` |
| `MessageBlock` | text / thinking / tool_call block | `Message.blocks[]` |
| `Usage` | token counters | `Message.usage` |
| `SessionResponse` | `{"id":"...","messageCount":42,"persisted":true}` | `GET /session` |
| `SessionListResponse` | `{"sessions":[...],"active":"...","persisted":true}` | `GET /sessions` |
| `SessionEntry` | single session in list | `SessionListResponse.sessions[]` |
| `NewSessionResponse` | `{"id":"...","activated":true}` | `POST /session/new` |
| `ActivateSessionRequest` | `{"id":"..."}` | `POST /session/activate` |
| `ActivateSessionResponse` | `{"id":"...","activated":true}` | `POST /session/activate` |
| `RoleResponse` | `{"role":"...","sandboxed":...,"permittedTools":[...],...}` | `GET /role` |
| `UsageResponse` | `{"guardrails":2,"tools":{...}}` | `GET /usage` |
| `DesignDocListResponse` | `{"docs":[...]}` | `GET /design-docs` |
| `ArchiveDesignDocRequest` | `{"src":"..."}` | `POST /design-docs/archive` |
| `ArchiveDesignDocResponse` | `{"ok":true,"archived":"..."}` | `POST /design-docs/archive` |
| `ErrorResponse` | `{"ok":false,"error":"...","errorCode":"..."}` | All endpoints on error |

### 4.4 SSE — `GET /events`

SSE does not map cleanly onto OpenAPI's request/response model. The spec
documents it as `text/event-stream` with a **description** of the event
shapes and a `oneOf` schema for the data payload:

```yaml
/events:
  get:
    operationId: getEventStream
    parameters:
      - name: Last-Event-ID
        in: header
        schema:
          type: integer
          format: int64
        required: false
        description: >
          Monotonic event id for replay. Omit or 0 to receive all
          buffered events before switching to live mode.
    responses:
      "200":
        description: "SSE event stream (text/event-stream)"
        content:
          text/event-stream:
            schema:
              type: object
              properties:
                event:
                  type: string
                  enum: [turn_start, turn_end, message_start, message_update,
                    message_end, tool_execution_start, tool_execution_update,
                    tool_execution_end, tool_permission_request, agent_interrupted,
                    provider_retry, agent_error, replay_gap, ping]
                data:
                  $ref: "#/components/schemas/SseEventPayload"
```

The `SseEventPayload` is a `oneOf` of every event kind. This is
descriptive, not generative — `oapi-codegen` won't produce a perfect SSE
parser. But it documents the contract rigorously and can be used by
custom codegen or manual parsing.

### 4.5 Security

The proxy currently binds without auth (default `0.0.0.0`). The spec
defines a placeholder `securitySchemes`:

```yaml
components:
  securitySchemes: {}
  # No auth is implemented yet. When token-based auth is added (v2),
  # define a BearerAuth scheme here and apply it to every path
  # except /health and static assets.
```

## 5. Contract pinning — Approach 1 (git SHA)

The franky orchestrator pins the exact spec revision it was built
against using a **git SHA** from the franky repository.

### 5.1 How it works

```
# In the orchestrator repo: Makefile or go.mod comments

# File: <orchestrator-root>/FRANKY_SPEC_SHA
# Pinned SHA of the agent API spec in the franky repo.
# Bump this when upgrading to a newer spec version.
a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0
```

The orchestrator's build or CI:

1. Reads `FRANKY_SPEC_SHA`.
2. Checks out the franky repo at that SHA (or fetches the single file
   via GitHub raw URL):
   ```
   gh api repos/franky/franky/contents/docs/api/agent-api-v1.yaml?ref=a1b2c3d4
   ```
3. Runs `oapi-codegen` to regenerate Go client code.
4. If the generated code differs from what's committed, CI fails.

### 5.2 How to bump

1. A change to the agent API is proposed (PR in franky repo).
2. The spec `docs/api/agent-api-v1.yaml` is updated in the same PR.
3. CI in the franky repo validates the spec against the implementation
   (see §6).
4. The orchestrator author reviews the spec diff, updates
   `FRANKY_SPEC_SHA` in the orchestrator repo, and regenerates.
5. If the change is backward-compatible (new field, new endpoint), the
   `info.version` patch number is bumped (`1.0.0` → `1.0.1`).
6. If the change is breaking (field removal, type change), the minor
   version is bumped (`1.0.0` → `1.1.0`) and the old major version
   file stays in the franky repo for legacy orchestrators.

### 5.3 Why git SHA instead of semver pinning

- **No registry.** There's no npm/pypi/crates.io for API specs. A git SHA
  is the most primitive, most reliable pin primitive.
- **Git blame tells the story.** `git log FRANKY_SPEC_SHA` in the
  orchestrator repo answers "when did we last update the API contract?"
  and `git diff <old-sha> <new-sha> -- docs/api/agent-api-v1.yaml` shows
  every schema change at once.
- **Atomicity.** A SHA points to an exact revision — there is no
  "oops, 1.0.1 was republished" ambiguity.
- **No version negotiation.** The orchestrator either speaks the pinned
  version or doesn't. No runtime content negotiation headers.

## 6. CI validation

The franky repo CI must ensure the implementation matches the spec.
Without this, the spec drifts silently and the orchestrator breaks at
an unpredictable time.

### 6.1 Agent CI — schema validation against running proxy

A CI job that:

1. Builds `franky --mode proxy` and starts it on a random port.
2. Iterates over every endpoint in `docs/api/agent-api-v1.yaml`.
3. For each endpoint, sends a valid request and validates the JSON
   response against the spec's response schema using a JSON Schema
   validator (`yajsv`, `check-jsonschema`, or a Zig-native one).
4. For `POST /prompt`, sends a dummy prompt and validates that
   subsequent `GET /transcript` returns a valid `TranscriptResponse`.
5. If any validation fails, the CI step fails.

```yaml
# .github/workflows/ci.yml (new job)
job:
  validate-agent-api-spec:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: zig build
      - name: Start proxy
        run: |
          ./zig-out/bin/franky --mode proxy --port 8787 &
          sleep 2
      - name: Validate spec
        run: |
          pip install check-jsonschema
          # Validate every endpoint defined in the spec
          check-jsonschema \
            --schemafile docs/api/agent-api-v1.yaml \
            --base-uri http://localhost:8787 \
            /health /session /sessions /role /usage /transcript
```

### 6.2 Breaking change detection

When `info.version` changes, CI must also run a diff between the old and
new spec:

```yaml
# On PRs that modify docs/api/agent-api-v1.yaml:
- name: Check for breaking changes
  run: |
    # Fetch the base version of the spec
    git show main:docs/api/agent-api-v1.yaml > /tmp/old.yaml
    # Use openapi-diff to detect breaking changes
    npx openapi-diff /tmp/old.yaml docs/api/agent-api-v1.yaml
```

If `openapi-diff` detects a breaking change (required field removed,
type narrowed), the PR is flagged for review. The version must be bumped
accordingly.

## 7. Orchestrator consumption

The orchestrator repo workflow:

```
# 1. Bootstrap (once)
git clone https://github.com/franky/franky-orchestrator
cd franky-orchestrator
# Add the SHA pin file
echo "a1b2c3d4e5..." > FRANKY_SPEC_SHA
# Generate Go client code
make generate-agent-client

# 2. Update (on agent API change)
# Edit FRANKY_SPEC_SHA to the new SHA
echo "e5f6a7b8c9..." > FRANKY_SPEC_SHA
make generate-agent-client
# Review the diff on generated/*.go
git diff
# Commit both files
```

The orchestrator's `Makefile` has:

```makefile
# <orchestrator-root>/Makefile
FRANKY_REPO ?= https://raw.githubusercontent.com/franky/franky
FRANKY_SPEC_SHA := $(shell cat FRANKY_SPEC_SHA)

generate-agent-client:
	curl -sL "$(FRANKY_REPO)/$(FRANKY_SPEC_SHA)/docs/api/agent-api-v1.yaml" \
	  -o /tmp/agent-api-v1.yaml
	oapi-codegen -package agentapi -generate types,client /tmp/agent-api-v1.yaml \
	  > internal/agentapi/client.gen.go
```

## 8. Change workflow

A step-by-step flow for when the agent API needs to change:

```
┌─ franky repo ───────────────────────────────────────────────────┐
│                                                                  │
│  1. PR author edits proxy.zig (the implementation)               │
│  2. PR author also edits docs/api/agent-api-v1.yaml (the spec)  │
│  3. CI runs: spec ↔ proxy validation (§6)                       │
│  4. CI passes → PR merges                                       │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
                           │
                           │ new SHA on main
                           ▼
┌─ franky-orchestrator repo ──────────────────────────────────────┐
│                                                                  │
│  5. Orchestrator author reads spec diff (git blame, spec diff)   │
│  6. Updates FRANKY_SPEC_SHA to the merge commit SHA             │
│  7. Runs make generate-agent-client                              │
│  8. Reviews generated code diff                                  │
│  9. If generated code needs manual adjustment, adjusts           │
│ 10. Commits FRANKY_SPEC_SHA + generated code                     │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

**Guiding principle:** Never commit a spec change without the
corresponding implementation change in the same PR, and never commit an
implementation change without updating the spec. The CI gate in step 3
enforces this.

## 9. Open questions

1. **`GET /events` codegen.** OpenAPI doesn't natively model SSE. Should
   we write a small custom Go parser for the SSE frames and validate
   using the `oneOf` schema manually, or accept that `GET /events` is
   documented but not code-generated?

   *Lean: Accept manual parsing for SSE. The spec documents the event
   shapes; `oapi-codegen` generates the event payload structs, but the
   SSE framing (lines, `event:`, `data:`) must be handwritten in Go.*

2. **Spec split threshold.** If the spec grows beyond ~50 endpoints,
   should we split into `agent-api-v1-paths.yaml` +
   `agent-api-v1-schemas.yaml`?

   *Lean: Wait until it's actually unwieldy. One file is easier to
   review, copy, and pin.*

3. **Rate-limiting / pagination.** Some endpoints (`GET /sessions`) could
   benefit from pagination. Should we spec it now even if the current
   implementation returns all results?

   *Lean: No. Spec what exists. Add pagination params when the
   implementation adds them.*

4. **Web UI static assets.** Should `GET /`, `/app.js`, `/style.css` be
   in the spec? They return non-JSON content and the orchestrator doesn't
   consume them directly (it opens new browser tabs to the agent's URL).
   But they ARE part of the agent's API surface.

   *Lean: Include them as `text/html` / `text/javascript` paths with
   minimal schema (just `type: string` for the response body). They
   document the full surface even if the orchestrator never accesses
   them over HTTP.*

5. **Spec linting.** Should we enforce a spec style guide (operationId
   naming, path casing, response uniformity) via CI?

   *Lean: Yes — `spectral lint docs/api/agent-api-v1.yaml` in CI with
   a minimal ruleset. Catches inconsistent naming before reviews.*
