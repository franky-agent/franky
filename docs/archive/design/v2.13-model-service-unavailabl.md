# v2.13 — Hosted Model Service Unavailability (Phases 1+2)

**Status:** implemented — Phases 1 (observability) and 2 (tunable retry budget) complete
**Phase 3** (fallback model chain) is tracked separately in
[`docs/design/decided/v2.13b-fallback-model-chain.md`](../decided/v2.13b-fallback-model-chain.md)
**Affects:** `src/ai/retry.zig`, `src/ai/stream.zig`, `src/coding/settings.zig`, `src/coding/cli.zig`
**Related:** [v2.14 — Agentic Engineering](../decided/v2.14-agentic-engineering.md) (depends on this)

---

## 1. Problem

Hosted providers go down. Reproduction from a real session:

```
HTTP body: {"error":"Server overloaded, please retry shortly (ref: 37400032-…)"}
```

Before v2.13, the agent surfaced this as a terminal error after the transport-layer retry
budget exhausted. For an interactive user this was mildly annoying; for an unattended
sibling app running while nobody watched, the run died and the work item ended in
"failed — provider error."

## 2. Goal

> Let the agent loop back off for a while and then retry the request. The agent should
> be able to handle this gracefully without crashing or getting stuck in an infinite
> loop. The user should also be informed about the issue and the retry attempts.

Two distinct asks:

1. **Better retry behaviour** — longer-running, more visible, doesn't crash on sustained outages.
2. **Fallback model chain** — declared in `settings.json`; on primary exhaustion, switch to next
   provider. _(Phase 3 — tracked separately.)_

## 3. What franky already had

### `src/ai/retry.zig` — transport-layer retry

- `Policy{ max_retries: u32 = 3, base_delay_ms: u32 = 500, max_retry_delay_ms: u32 = 60_000 }`.
- Decorrelated-jitter exponential backoff:
  `delay = min(max_retry_delay_ms, random(base, prev * 3))`.
- `Retry-After` header takes precedence when ≤ `max_retry_delay_ms`.
- Hard invariant: **never retry once response bytes have started flowing** (§F.1).
- Cancel-fired short-circuit before each attempt and each sleep.

### `src/ai/errors.zig` — error classification

```zig
pub const Code = enum {
    auth, request_invalid, model_unavailable, context_overflow,
    payload_too_large, rate_limited, rate_limited_hard, transient,
    timeout, transport, safety_refusal, empty_response, aborted,
    max_turns_exceeded, tool_arg_validation, tool_runtime,
    tool_blocked, protocol_violation, internal,

    pub fn isRetryable(self: Code) bool {
        return switch (self) {
            .rate_limited, .transient, .timeout, .transport => true,
            else => false,
        };
    }
};
```

## 4. Gaps addressed in this doc

### 4.1 Retry budget too small for unattended use

`max_retries: 3` with `max_retry_delay_ms: 60_000` bounded total wall time to ~3 minutes.
The cap is now policy-driven via `retry_max_attempts` and `retry_max_total_ms` in settings.

### 4.2 Retry invisible to the harness

The retry loop slept and retried silently. The agent loop, SSE stream, and any subscribed UI
saw nothing. Phase 1 adds a `provider_retry` stream event emitted before each sleep so the
UI and sibling apps get advance notice.

## 5. Implementation

### Phase 1 — Observability (`provider_retry` stream event)

**`src/ai/stream.zig`** — new variant:

```zig
provider_retry: struct {
    attempt: u32,
    max_attempts: u32,
    delay_ms: u32,
    reason: errors.Code,
    provider_code: ?[]const u8 = null,
    provider_message: ?[]const u8 = null,
    http_status: ?u16 = null,
},
```

Emitted by `retry.run()` **before the sleep**, at the moment the verdict flips to `.retryable`.
Propagates through the existing channel/Reducer paths so:

- The TUI / `--mode interactive` shows "retrying in 4s (attempt 2/3)…" inline.
- The proxy `/events` SSE stream carries it for web-UI rendering.
- Sibling apps subscribe and route it to the source channel.

### Phase 2 — Tunable retry budget

**`src/coding/settings.zig`** — new fields:

```zig
retry_max_attempts: ?u32 = null,   // overrides Policy.max_retries
retry_max_total_ms: ?u64 = null,   // wall-time cap on total retry delay
```

Parsed from `settings.json` under `tools.retry.{maxAttempts, maxTotalMs}`.

**`src/coding/cli.zig`** — new CLI flags:

```
--retry-max-attempts N
--retry-max-total-ms N
```

CLI flags win over settings.json (consistent with existing overlay precedence).

**`src/ai/retry.zig`** — `Policy.max_total_delay_ms`:

`retry.run()` tracks accumulated `total_delay_ms`. Before each sleep, if
`total_delay_ms + next_delay_ms > max_total_delay_ms` → emit terminal with
`rate_limited_hard` and `retry_after_ms` set to the remaining budget.

**Profile presets (example):**

```json
"profiles": {
  "agent-unattended": {
    "model": "claude-sonnet-4-6",
    "retry_max_attempts": 15,
    "retry_max_total_ms": 1800000
  }
}
```

## 6. Decisions

| Q | Decision |
|---|---|
| Q1 — Default `retry_max_total_ms` | Keep ~3 min for interactive default; unattended profiles opt in explicitly |
| Q2 — `provider_retry` timing | Before-sleep, at verdict flip. UI gets advance notice while sleep runs |
| Q4 — 429 vs 503 | Unified: honour `Retry-After` for both; phase-3 fallback handles either exhaustion |
| Q5 — Mid-stream 503 | Keep §F.1 invariant; add `mid_stream: true` field to `error_ev` for sibling disambiguation |

---

## 7. Out of scope

- API key / credential management (deferred to v3).
- Per-tool retry (separate concern from provider unavailability).
- Caching responses.
- Provider health dashboard in web UI.
- Fallback model chain (Phase 3 — see `v2.13b-fallback-model-chain.md`).
