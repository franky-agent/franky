# v2.27 — Nudge stalled models (Option A)

**Status:** ✅ Implementation complete  
**Implemented:** 2026-05-13  

## Problem

GPT-family models and some other providers occasionally stop generating
mid-thought — the SSE stream ends with `stop_reason=length` (token cap)
or `stop_reason=stop` with very little / no content. The model considers
its response complete, but the output is truncated or vacuous.

Standard "go on" / "continue" nudges help, but there is no standard way
to inject a nudge **mid-stream** (the HTTP request is fire-and-forget).
The only option is to inject a synthetic user message **between turns**
and call the LLM again.

## Approach — Option A

Inject a nudge in the agent loop, between `runTurn()` returning `false`
and the existing `between_turns` hook. The nudge is a `.user`-role
message in the transcript that says "continue from where you left off."
The loop then runs another turn transparently.

### Detection

After `runTurn` returns `false`, inspect the last assistant message in
the transcript:

| Signal | Action |
|---|---|
| `stop_reason == .length` | **Strong** — model hit token cap mid-generation |
| `stop_reason == .stop` AND no tool calls AND total text < ~50 chars | **Moderate** — stopped with near-empty output |
| `stop_reason == .stop` AND zero text content (degenerate) | **Moderate** — empty response, no explanation |
| `stop_reason == .tool_use` | **Skip** — model is productive (calling tools) |
| `stop_reason == .refusal` / `.err` / `.aborted` | **Skip** — intent was to stop or error |

### Nudge messages

| Trigger | Nudge text |
|---|---|
| `stop_reason == .length` | `"You were cut off mid-response (token limit reached). Continue from where you left off."` |
| Degenerate / near-empty `.stop` | `"Your response was very brief. Please continue or elaborate."` |

### Guard — nudge limit

At most **2 nudges per stall episode**. A counter `nudge_count` lives on
the agent loop's stack and resets implicitly on agent loop restart. After
2 nudges the loop lets the turn end naturally (falls through to the
`between_turns` hook, which may still request another turn via steer /
followUp).

### Config

Add to `loop.Config`:

```zig
/// v2.27 — when true, the loop detects stalled models and injects
/// a synthetic "please continue" user message into the transcript
/// before calling the LLM again. Caps at 2 nudges per episode.
nudge_on_stall: bool = false,
```

Default `false` — no behavioural change for existing callers. Mode
drivers (print, proxy, TUI) opt in.

## Integration sketch

In `agentLoop`, right before the `between_turns` hook:

```
runTurn returns false
├─ guardrails.betweenTurns → false
├─ v2.27 [nudge_on_stall]:
│  ├─ inspect last transcript message
│  ├─ if stalled AND nudge_count < 2:
│  │  ├─ nudge_count += 1
│  │  ├─ append "continue" user message to transcript
│  │  └─ continue (run another turn)
│  └─ else: fall through
├─ between_turns hook
├─ out.close / return
```

## Test plan

Test cases in `test/agent_loop_test.zig`:

1. **`stop_reason=length` triggers nudge** — faux emits text + `.length`,
   loop config `nudge_on_stall=true`. Expect: nudge injected, second turn
   runs, transcript has 3 messages (assistant1, user-nudge, assistant2).

2. **Degenerate `.stop` triggers nudge** — faux emits zero text blocks,
   `stop_reason=.stop`. Same expected outcome.

3. **Nudge cap (max 2)** — faux emits three scripts all with
   `stop_reason=length`. Loop nudges twice, third turn ends naturally.

4. **`nudge_on_stall=false` (default) — no nudge** — faux emits `.length`,
   default config. Loop ends after 1 turn, no extra message.

5. **Tool calls skip nudge** — faux emits tool_call + `.tool_use`,
   `nudge_on_stall=true`. Loop executes the tool and returns, no nudge.

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| Infinite nudge loop (model keeps stalling) | Cap at 2 nudges then fall through |
| False positive on short but complete answers | Threshold of ~50 chars + no terminal punctuation check |
| Nudge message visible in UI | It's a `.user` transcript message — currently user messages aren't streamed as events by the loop, only the assistant's reply is. If rendering changes in the future, the nudge uses standard `.user` role so it renders naturally |
| Nudge delays steer/followUp by up to 2 turns | Acceptable — nudge is fast and steer queue is drained on the next non-nudge turn |
