# LLM Agent Standards — Franky Support (Implemented Parts)

This document collates the portions of the [v2.30 design](./v2.30-llm-agent-standards.md)
that have been implemented in code. The remaining items live in
[v2.30-not-implemented.md](./v2.30-not-implemented.md).

## 1. Implemented: InstructionForest + AGENTS.md Loading (§4.2, §4.8)

**Files:**
- `src/coding/instructions.zig` — NEW module.
- `src/coding/modes/print.zig` — `buildSystemPromptIo()` extended.
- `src/coding/config/cli.zig` — `--no-standards` flag.

**What works:**
- `InstructionForest` scans `AGENTS.md`, `CLAUDE.md`, and `.agents/agents.md` at
  workspace root and recursively in subdirectories (depth-limited to 8 levels,
  max 50 files).
- `InstructionsState.scan()` returns the forest + a pre-rendered root-level block.
- `pickRootBlock()` precedence: `AGENTS.md` > `.agents/agents.md` > `CLAUDE.md`
  at the same directory depth.
- `buildSystemPromptIo()` in print mode injects the rendered instructions block
  after the subagent hint and before the skills section.
- `--no-standards` CLI flag suppresses instruction scanning entirely.
- 6 tests covering: empty workspace, AGENTS.md only, CLAUDE.md only, AGENTS.md
  beats CLAUDE.md, `.agents/agents.md` content, multi-depth root-level picking.

**API:**

```zig
pub const InstructionBlock = struct {
    source_path: []const u8,
    content: []const u8,
    kind: InstructionKind,  // agents_md | claude_md | dot_agents
    dir: []const u8,
};

pub const InstructionForest = struct {
    blocks: []InstructionBlock,
    any_found: bool,
};

pub const InstructionsState = struct {
    forest: InstructionForest,
    rendered: ?[]const u8,
    pub fn scan(allocator, io, workspace_root) !InstructionsState;
};
```

**Known limitation — deferred to Phase 2:**
`resolveClosest()` (per-file proximity resolution) returns `null` — only the
root-level block is rendered at session start.

## 2. Implemented: Skill Root Unification (§4.3, §4.6, §5)

**File:** `src/coding/skills.zig`

**What works:**
- `Roots.standard_roots` field holds additional skill directories from standard
  locations.
- `Roots.resolveStandard()` computes paths for:
  - `<workspace>/.agents/skills/`
  - `<workspace>/.claude/skills/`
  - `<workspace>/.cursor/rules/`
  - `$FRANKY_HOME/.agents/skills/` (or `$HOME/.franky/.agents/skills/`)
  - `$FRANKY_HOME/.claude/skills/` (or `$HOME/.franky/.claude/skills/`)
- Frontmatter extended with `id:` field (`.agents/skills/*/skill.md` uses `id:`
  when `name:` is absent; `name` wins when both present, `id` stored as alias).
- Frontmatter extended with `enabled:` field (defaults to `true`; skills with
  `enabled: false` are excluded from selection regardless of `auto_apply` or
  `--skill` flag).
- Unknown frontmatter keys are silently ignored (forward-compat).

**Skill root precedence:**
```
1. --skills <dir>              [CLI, highest]
2. <workspace>/.franky/skills/ [franky project]
3. <workspace>/.agents/skills/ [.agents protocol]
4. <workspace>/.claude/skills/ [Claude interop]
5. <workspace>/.cursor/rules/  [Cursor interop — converted]
6. <workspace>/skills/         [franky workspace]
7. $FRANKY_HOME/skills/        [franky user]
8. ~/.agents/skills/           [global .agents]
9. ~/.claude/skills/           [global Claude]
```

**Known limitation — no Cursor parser:**
The `.cursor/rules/` path is scanned as a skill root, but there is no
`parseCursorRule()` converter yet. Cursor rules use a different frontmatter
schema than franky skills; a converter must be written before skills from that
root are usable (tracked in [v2.30-not-implemented.md](./v2.30-not-implemented.md)).

## 3. Implemented: CLI Flags (§4.1, §5)

**File:** `src/coding/config/cli.zig`

**What works:**
- `--no-standards` — boolean flag; when set, `buildSystemPromptIo()` skips
  instruction scanning entirely. Franky-native paths (`--skills`, `.franky/`,
  `$FRANKY_HOME/`) are unaffected.
- `--agents <dir>` — parsed into `cfg.agents_path`. **Parsed but not yet
  wired into any scanning logic** — see not-implemented doc.

## 4. Current System Prompt Assembly Order (§4.8, partial)

The runtime assembly order in `buildSystemPromptIo()`:

1. Base system prompt (disk or built-in default)
2. `Current folder:` injection
3. Subagent hint block (built-in default only)
4. **AGENTS.md / CLAUDE.md instructions** (new — v2.30)
5. Active skills section (franky + standard roots merged)
6. Review config block
7. `--append-system-prompt` content (always last)

Items not yet implemented:
- `.agents/system-prompt.md` override (step 3 in the §4.8 design)
- CLI `--system-prompt` override (already existed before v2.30 — see §4.8 step 2)
