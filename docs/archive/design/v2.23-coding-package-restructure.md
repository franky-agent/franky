## 1. Problem

`src/coding/` has grown to **48 files** across 16 directories (including sub-dirs) with **~41K lines of Zig**. The module root (`mod.zig`) re-exports 38 distinct namespaces. Symptoms:

- **Top-level namespace pollution** — every module sits side-by-side regardless of domain. A developer looking for "how sessions work" must mentally navigate 30+ flat entries to find the 6 that are session-related.
- **Oversized files** — `session.zig` (1542 lines), `permissions.zig` (1499), `profiles.zig` (1332), `improvement.zig` (1658), `diagnostics.zig` (1608), `compaction.zig` (1115). Each mixes multiple concerns.
- **Duplicated import boilerplate** — every tool in `tools/*.zig` independently aliases `ai/types.zig`, `ai/stream.zig`, `agent/types.zig` with near-identical `const` blocks.
- **No clear domain boundaries** — `path_safety.zig` and `env_denylist.zig` are both security policies used by `workspace.zig` but live at the top level alongside session and config modules.

---

## 2. Goals

1. **Reduce the mod.zig re-export count** from 38 to ~15-18 by grouping domains into subdirectories.
2. **Split oversized files** (>800 lines) into focused sub-modules.
3. **Eliminate import boilerplate duplication** by centralizing shared type aliases.
4. **Preserve every public API** — no consumer (`root.zig`, `sdk.zig`, `agent/loop.zig`) should need changes beyond updated import paths (and those should be minimal).
5. **Keep `tools/` and `modes/` untouched** — they are already well-structured per-domain.

### 3.1 Rationale for each grouping

| Group | Files | Rationale |
|-------|-------|-----------|
| **`session/`** | session (split), compaction, branching, object_store, replay | All co-evolve around transcript persistence. Compaction reads/writes branches and uses object_store for blob spill. Replay reads session traces. |
| **`config/`** | cli, settings, profiles | All answer "what configuration values are in effect?" — layered merge (CLI > profile > settings > env > defaults). |
| **`security/`** | role, permissions, auth, path_safety, env_denylist | All enforce access boundaries: which tools the model sees (role), which calls actually run (permissions), which credentials are available (auth), what paths/env are reachable (path_safety, env_denylist). |
| **`model_catalog/`** | catalog, render, fetch | The three faces of the models registry: the data model, the JSON renderer, and the provider endpoint parsers. Renamed to avoid collision with `coding.models` struct. |
| **Flat utilities** | terminal, slash, regex, gitignore, skills, rpc | Each is standalone, imported by at most 1-2 consumers, and small enough to not warrant a subdirectory. |

### 3.2 Splits of oversized files

#### `session.zig` (1542 lines → `session/mod.zig` + `session/persistence.zig`)

- **`session/persistence.zig`** (913 lines) — all I/O functions: write/read session header, write/read transcript, atomic write helpers, schema migration, content-address write-back, JSON helpers, helpers used by sibling modules (`sha256Hex`).
- **`session/mod.zig`** (664 lines) — `Session` struct + `Ulid` + `SessionHeader` types, convenience `save()`/`load()`, re-exports of every `pub` symbol from `persistence.zig`, and all tests.

> `session/transcript.zig` was **not created** — the `Transcript` type lives in `agent/loop.zig`, and separating message-list operations from the current file was deemed unnecessary for v2.23. Compaction + branching were moved into `session/` as whole files without splitting.

#### `permissions.zig` (1499 lines)

**Deferred** — kept as one file. Has clean seams (`Decision`, `Resolution`, `Gate`, `Store`, `Prompter`) but splitting would require intermediate types that don't exist yet.

#### `profiles.zig` (1332 lines)

**Deferred** — kept as one file. Cohesive: `Profile` struct, JSON parsing, field-level overlay, env interpolation. Splitting would separate parsing from application logic but they're tightly coupled.

#### `improvement.zig` (1658 lines)

**Deferred** — kept as one file. `Finding` types, aggregation heuristics, and rendering are tightly coupled through the `findings()` pipeline output.

#### `diagnostics.zig` (1608 lines)

**Deferred** — kept as one file. Single-pass analyzer; splitting into `analyze.zig` + `render.zig` would introduce a `Report` struct that's currently unnamed.

#### `compaction.zig` (1115 lines)

**Deferred** — moved to `session/compaction.zig` as-is. Has clean boundaries (`selectSpan` vs `run` vs `renderSpanAsPrompt`) but splitting would touch the compaction call site.

## 4. Redundancy elimination

### 4.1 Central type-import module (`src/coding/types.zig`)

Every tool file begins with:

```zig
const ai = struct {
    pub const types = @import("../../ai/types.zig");
    pub const stream = @import("../../ai/stream.zig");
};
const at = @import("../../agent/types.zig");
```

This is copy-pasted across **12 tool files** (subagent, grep, web_search, bash, ls, web_fetch, edit, common, write, read, find, workspace ≈ 40 lines each). Consolidate into:

```zig
// src/coding/types.zig
pub const ai = struct {
    pub const types = @import("../ai/types.zig");
    pub const stream = @import("../ai/stream.zig");
    pub const registry = @import("../ai/registry.zig");
    pub const log = @import("../ai/log.zig");
    pub const utils = @import("../ai/utils.zig");
    pub const http = @import("../ai/http.zig");
    pub const partial_json = @import("../ai/partial_json.zig");
};
pub const agent = struct {
    pub const types = @import("../agent/types.zig");
    pub const mod = @import("../agent/mod.zig");
    pub const loop = @import("../agent/loop.zig");
};
```

Then each tool writes:
```zig
const ct = @import("../types.zig");
const at = ct.agent.types;
const ai = ct.ai;
```

**Savings:** ~350 lines of import boilerplate removed; one import path to update when the stdlib moves.

### 4.2 Tool error pattern (already done)

`tools/common.zig` already centralizes `toolError()`, `contextIgnoreError()`, and `tool_code_contextignored`. No further consolidation needed here — this was already the v1.3.0 R1 refactor.

### 4.3 `workspace.zig` + `path_safety.zig` + `env_denylist.zig` coupling

`workspace.zig` imports `path_safety` and `env_denylist`. These three are now co-located under `security/`. No code duplication to remove, but the move makes the relationship obvious.

### 4.4 `gitignore.zig` usage pattern

`gitignore.zig` is imported by `tools/common.zig`, `tools/ls.zig`, `tools/find.zig`, and `tools/grep.zig`. Each tool constructs a `Stack` and calls `isIgnored`. This is clean — no duplication. **Leave as-is.**

### 4.5 `truncate.zig` usage

`truncate.zig` is imported by `agent/loop.zig` (crossing the `coding/` boundary) and by individual tools. Constants like `default_max_lines`, `default_max_bytes`, and `tool_result_max_bytes` are the single source of truth via `pub const`. **Leave as-is.**

## 6. Migration plan

### Phase 1 — create subdirectories + move files (mechanical)

1. `mkdir -p src/coding/{session,config,security,model_catalog}`
2. Move files per table in §5 without content changes.
3. Update import paths in every moved file.
4. Update import paths in every file that references a moved file.

### Phase 2 — create sub-module re-export files

1. Create `src/coding/session/mod.zig` — re-export `Session`, `Ulid`, `compaction`, `branching`, `object_store`, `replay`.
2. Create `src/coding/config/mod.zig` — re-export `Config`, `Settings`, `Profile`.
3. Create `src/coding/security/mod.zig` — re-export roles, permissions, auth, path_safety, env_denylist.
4. Create `src/coding/model_catalog/mod.zig` — re-export `Entry`, `builtins`, `render`, `fetch`.

### Phase 3 — split `session.zig`

1. Extract Ulid + session metadata → `session/mod.zig` (stays as the file created in Phase 2).
2. Extract `persistence.zig` — `saveSession`, `loadSession`, `saveTranscript`, `loadTranscript`, atomic write, migration, content-address spill.
3. Extract `transcript.zig` — message list ops, branch views, pinned-message tracking.

### Phase 4 — create `types.zig`

1. Write `src/coding/types.zig` with centralized `ai` and `agent` type aliases.
2. Update each tools/*.zig and modes/*.zig to use `const ct = @import("../types.zig")`.
3. Remove the private `const ai = struct { ... }` blocks.

### Phase 5 — finalize `mod.zig`

Rewrite `mod.zig` to match the new structure. Update `root.zig` and `sdk.zig` if any re-export name changed (they should not — we preserve every public name).

## 7. Risks and mitigations

| Risk | Mitigation |
|------|------------|
| **Import path rot** — missed internal imports cause compile errors | Run `zig build` after each phase. There are ~3 external consumers (`root.zig`, `sdk.zig`, `agent/loop.zig`) and ~30 internal files. The compiler catches every missed path. |
| **Split introduces behavioral change** — the refactor touches live code | Phase 1+2 are pure moves (no semantic change). Phase 3 splits session.zig but the new modules re-export the same functions with the same signatures. Phase 4 changes only the import alias style, not the called functions. |
| **Merge conflicts with in-flight branches** | Schedule this as a single focused PR. The design doc serves as the commit message. |
| **`models` name collision** — `coding.model_catalog` is verbose | Alternative: call it `catalog/` instead of `model_catalog/`. Checked: `catalog/` doesn't collide with anything. But `model_catalog/` is more descriptive. Decision: **`model_catalog/`**. |
