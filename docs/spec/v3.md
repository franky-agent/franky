## 1. Mid-session `max_turns` extension prompt

### 1.1 Context

`agent.loop.Config.max_turns` (default 50) caps the loop's turn count
per prompt. When the cap is reached the loop emits
`agent_error{code: max_turns_exceeded}` and closes (see
[CHANGELOG vN entry for "Configurable max_turns"](../../CHANGELOG.md)).

The agent-loop layer already supports an `on_max_turns` hook that
returns either `.stop` or `.extend(N)` to add credits additively. SDK
consumers (`Agent.Config.max_turns_hook`) can wire this to whatever UX
their surface allows. The CLI / settings / profile layers all wire
the cap correctly.

**The interactive mode currently does not prompt** mid-session. When
the cap hits, scrollback shows the error + a hint line ("raise the
cap with `--max-turns N`, `FRANKY_MAX_TURNS`, or `max_turns` in
settings.json / profile") and the user is dropped back at the input
box. Continuing the conversation requires a fresh `franky --max-turns
<bigger>` invocation.

### 1.2 Proposed UX (v3.x)

When the loop reaches `max_turns` mid-session in interactive mode,
suspend the worker and render an in-place modal (parallel to v1.11.2's
permission modal):

```
┌─────────────────────────────────────────────────────────┐
│ Max turns reached (50). Extend?                         │
│   [Y] yes (+50)    [n] no (terminate)                   │
└─────────────────────────────────────────────────────────┘
```

Default is `n` (terminate) so an absent user doesn't accidentally
budget-burn. `Y` extends additively; subsequent hits show the same
modal (the user can extend repeatedly).

### 1.3 Implementation plan

Mirror the v1.11.x `PermissionPrompter` pattern:

1. **New event variant**: `AgentEventKind.max_turns_request` carrying
   `{ turns_used: u32, current_cap: u32 }`. Non-terminal — the channel
   stays open across the prompt.
2. **`MaxTurnsPrompter`** in `coding/permissions.zig` (or a new
   `coding/prompters.zig`): mutex + condvar, single-slot. The `on_max_turns`
   callback emits the event then `wait()`s on the condvar; the mode
   driver calls `resolve(decision)` to unblock the worker.
3. **Mode wiring**:
   - **Interactive** owns the modal UX. Reuse `ModalState`'s arming
     pattern; bind `Y` / `Enter` to `.extend(50)` and `n` / `Esc` to
     `.stop`. The default-on-unrecognized-key is `.stop` (safer).
   - **RPC** routes through `notifications/maxTurnsRequest` and a new
     `loop/resolveMaxTurns` method shape.
   - **Proxy** emits an SSE `event: max_turns_request` and accepts
     `POST /loop/max_turns_resolve` (mirrors `/permission/resolve`).
   - **Print** stays no-op (no UX surface; just emit the error).
4. **Configurable extension amount**: a `--max-turns-extend N` flag
   and matching settings/profile field for users who want a different
   credit size per Y press. Default 50.
5. **Multi-amount picker** (stretch): replace `[Y/n]` with `[+10] [+50]
   [+100] [no]` for power users. Only worth doing if anyone asks.

### 1.4 Anchor cross-refs

When this lands in v3, the loop / Agent / SDK fields stay where they
already are; only the mode driver gets new code. The current
`agent_error{max_turns_exceeded}` event remains the SDK signal — the
new `max_turns_request` event is layered on top, used only when an
`on_max_turns` hook is wired AND the mode wants the in-session prompt.

## 2. Subagent intermediate progress recovery on reload

### 2.1 Context

The proxy web UI rehydrates the conversation from `/transcript` on page
load (`src/coding/modes/web/app.js` `rehydrate()`). The v2.x rehydrate
fix restores **final** state of every tool call and subagent on
reload — result panel bodies are repopulated from `toolResult` blocks,
subagent cards re-attach the inline log + side-panel row, and the
overlay shows the subagent's `final_text` when opened.

What the v2.x fix does **not** restore is the subagent's
**intermediate progress**: the per-step `text_delta`, `thinking_delta`,
`tool_start`, `tool_end`, and `error` events the subagent emitted
during its run. Those events live only in the SSE replay ring
(`replay_ring_capacity = 256`, `proxy.zig`) and are not persisted in
the parent transcript. After a reload — or once the ring rolls past
their ids — that timeline is lost; the overlay shows only the final
answer with no trace of how the subagent got there.

### 2.2 Proposed UX (v3.x)

When the user opens the overlay for a rehydrated subagent card, the
overlay should reconstruct the full intermediate timeline (thinking
block, text block, every tool call with args + status) from the
subagent's **own** persisted transcript, identical in structure to
what the live SSE flow would have rendered.

Loading is on-demand (overlay open) so the home view stays cheap; an
error or missing transcript falls back gracefully to the v2.x
final-text-only view.

### 2.3 Implementation plan

The data is already on disk — `src/coding/tools/subagent.zig`'s
`successResult()` writes the subagent's own session to
`<parent>/subagents/<call_id>/transcript.json` and embeds the path in
the parent's tool_result body as `transcript_path`.

1. **New proxy endpoint**: `GET /subagents/<call_id>/transcript` returns
   the subagent's transcript JSON. Path is resolved by joining
   `parent_session_dir + "subagents" + call_id + "transcript.json"` and
   guarded against path traversal (call_id is `[A-Za-z0-9_-]+`).
2. **Transcript shape mapping**: the subagent transcript contains the
   same `messages[]` shape `/transcript` already emits. The web UI's
   rehydrate already knows how to render that shape into bubbles +
   tool cards. Reuse the same passes inside the overlay body, but
   targeting `saOverlayBody` instead of `conversation`.
3. **Trigger point**: in `openSubagentOverlay()`, when `state` exists
   from a rehydrated card and `state.eventLog` is empty (i.e. the
   live event log was not captured because the page reloaded),
   `fetch('/subagents/' + callId + '/transcript')`, build a
   minimal `eventLog` + `textBlocks` + `thinkingBlocks` from the
   response, then call the existing `renderSubagentOverlayContent`.
4. **Cache**: store the reconstructed state on the section so a
   re-open is instant. Invalidate when the section is removed.
5. **Failure modes** (graceful):
   - `transcript_path` missing from result body (subagent persistence
     was disabled or failed) → no fetch, overlay shows the v2.x
     final-text view.
   - Endpoint returns 404 (file deleted) → log to console, fall back.
   - Endpoint returns 500 / network error → log + fall back.

### 2.4 Anchor cross-refs

This is layered on top of the v2.x rehydrate fix; nothing in v2 needs
to change to make room for it. The producer side
(`subagent.zig:successResult`) already emits the path; the recovery
path is a pure consumer addition.

## 3. Stuck-loop recovery

### 3.1 Context

Some models — especially smaller / weaker reasoners — fail to emit
`end_turn` and instead grind through tool calls indefinitely. Three
reinforcing pressures make this hard to break out of: (a) tasks with
no observable "done" predicate, (b) post-training that rewards
finishing tasks over abandoning them, (c) `end_turn` competes with
tool calls in the same sampling step (no separate "should I stop?"
decision), and (d) in proxy / sub-agent runs there's no
between-turn user signal that would naturally invite a stop.

Three interventions were proposed (see Deepseek diagnostics analysis
in repo conversation history):

- **Hard budget + forced summarization** — guarantees a final
  `end_turn` artifact when `max_turns` is hit.
- **Repetition-triggered escalation** — detect repeated tool error
  shapes and inject "don't retry the same approach" mid-stream.
- **Checkpoint injection** — every N turns, inject a meta-prompt
  asking the model to enumerate hypotheses and remaining options.

### 3.2 What shipped (v2.x)

**Forced summarization on cap hit.** New `loop.Config` field
`max_turns_summarize: bool` (default `false`). When set, the
agent loop, after exhausting `max_turns` and any
`on_max_turns.extend(N)` credits, runs ONE final tool-disabled
turn with a synthetic user message asking the model to produce a
final report from what it has so far. The `agent_error{max_turns_exceeded}`
event still fires after the summary turn so callers that
programmatically gate on it stay backwards-compatible.

For the sub-agent path this composes naturally with §2 of `v1.md`
and the `no_final_text` guard in `subagent.zig`: the forced summary
populates the last assistant message's text block, so
`lastAssistantText` succeeds and the parent receives a real
`final_text` instead of an empty answer or a `no_final_text` error.

Off-by-default for callers that want the strict-failure semantics.
Sub-agent presets opt in via the `Agent.Config.max_turns_summarize`
plumbing.

### 3.3 Deferred to v3.x — repetition-triggered escalation

When the same tool-error shape (regex on the error message:
`"no member named 'X' in 'Y'"`, `"file not found at <path>"`, etc.)
fires N times within a sliding window, inject a synthetic
system-role nudge before the next turn:

> You've hit this same error N times. Don't retry the same approach.
> Either change strategy or stop and report what's blocking you.

Implementation sketch:

1. **Error-shape clustering.** A small fingerprint extractor over
   `tool_execution_end{is_error: true}` events. Strip volatile bits
   (paths, line numbers, hex addresses) and hash. Window: last 10
   turns. Threshold: 3 matching fingerprints triggers.
2. **Injection point.** Same `between_turns` hook surface used by
   followUp queues. Loop currently has `between_turns: ?BetweenTurnsFn`
   on `Config`; the escalation logic wraps an existing hook (so
   followUp + escalation can both fire).
3. **Backoff.** Once injected, suppress the same fingerprint for
   the rest of the run. Otherwise the nudge fires every turn the
   error keeps happening.

### 3.4 Deferred — checkpoint injection

Every N turns without an `end_turn`, inject a meta-prompt asking
the model to list hypotheses ruled out, current hypothesis,
evidence, and one alternative. Lower priority than §3.3 — false-
positive injections degrade healthy long sessions, and the
forced-summarization path in §3.2 already catches the worst case
(unbounded grinding).

Open design questions before scheduling:
- Should N be fixed (e.g. 30) or dynamic (% of `max_turns`)?
- Suppress when the model is making visible progress (e.g. each
  recent turn produced a successful tool result)?
- Surface as a system message vs. a synthetic user message?

### 3.5 Anchor cross-refs

§3.2 ships in v2.x and is implemented at `src/agent/loop.zig`'s
cap-exhaustion branch + a new `Config.max_turns_summarize` field.
§3.3 and §3.4 stay deferred until §3.2 has signal in production
about whether the simple intervention is sufficient.

## 4. Server-side thinking-delta coalescing

### 4.1 Context

Reasoning models (DeepSeek, Kimi, OpenAI o-series, …) can emit
*thousands* of `reasoning` chunks in a single turn. Observed in the
wild: deepseek-v4-flash:cloud producing 10,615 reasoning chunks
(~2.6 MB stream, 4-minute duration) before the assistant's first
structured tool_call.

Each chunk currently turns into its own SSE frame in
`src/coding/modes/proxy.zig`'s `fanOutLocked`, which acquires
`events_mutex` once per frame and writes a JSON envelope per frame.
That's:

- 10K+ mutex round-trips on the broadcast path.
- 10K+ small SSE writes through the kernel, each with TCP framing
  overhead.
- Multi-MB JSON-encoding work on the server.
- 10K+ message-update events delivered to the browser, where the
  v2.x UI fix in §A throttles render but still has to parse each
  event individually.

### 4.2 Proposed change (v3.x)

Coalesce consecutive same-block `thinking_delta` events into a
single SSE frame on a small time/byte budget (e.g. flush whenever
≥16 KiB accumulated *or* ≥50 ms have passed since the first
buffered chunk *or* a non-thinking event arrives). The browser
accumulator already keys deltas by `block_index`, so receiving one
big delta vs. many small ones is transparent.

Implementation sketch:

1. **Per-subscriber buffer.** `SseSubscriber` gains a small
   per-block buffer (`thinking_buf: ArrayList(u8)`) plus a
   `flush_deadline_ms`. Caps: 16 KiB and 50 ms (tunable).
2. **Coalesce on push.** When `broadcastEvent` enters with a
   `message_update.thinking` event, append into the per-subscriber
   buffer instead of emitting a frame; only flush when one of the
   thresholds trips.
3. **Force-flush on barriers.** Any non-thinking event
   (`text_delta`, `tool_execution_*`, `turn_end`, etc.) must
   flush all pending thinking buffers first so order is preserved
   relative to other events.
4. **Replay-ring interaction.** Coalesced frames go into the ring
   the same way single frames do — but a thoughtful re-stamp:
   each coalesced frame is one event_id rather than re-using the
   first chunk's id, otherwise reconnect-replay would skip ahead.

### 4.3 Why deferred

The v2.x UI throttle (§A in this conversation, implemented at
`src/coding/modes/web/app.js` `appendThinkingDelta`) addresses the
most visible symptom (browser tab freeze on long reasoning streams)
without protocol changes. Server-side coalescing is the durable
fix — it cuts bandwidth, mutex pressure, and browser parse cost by
~10× — but it's a wire-format change that affects the SSE replay
ring and any external `/events` consumers, so it deserves spec
review before shipping. Leave deferred until §A's signal in
production tells us whether server coalescing is needed.

### 4.4 Anchor cross-refs

This is independent of §3 (stuck-loop recovery). It's a pure
plumbing improvement on the SSE broadcast path. Files in scope:
`src/coding/modes/proxy.zig` (broadcast pipeline), maybe a small
addition to `src/agent/types.zig` if a new agent-event variant is
needed for "coalesced" thinking. No web-UI changes required if
existing per-block accumulation is preserved.
