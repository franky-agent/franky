## 1. Mid-session `max_turns` extension prompt

### 1.1 Context

`agent.loop.Config.max_turns` (default 50) caps the loop's turn count
per prompt. When the cap is reached the loop emits
`agent_error{code: max_turns_exceeded}` and closes (see
[CHANGELOG vN entry for "Configurable max_turns"](../../CHANGELOG.md)).

The agent-loop layer already supports an `on_max_turns` hook that
returns either `.stop` or `.extend(N)` to add credits additively. SDK
consumers (`Agent.Config.max_turns_hook`) can wire this to whatever UX
their surface allows. The CLI / settings / profile layers all wire
the cap correctly.

**The interactive mode currently does not prompt** mid-session. When
the cap hits, scrollback shows the error + a hint line ("raise the
cap with `--max-turns N`, `FRANKY_MAX_TURNS`, or `max_turns` in
settings.json / profile") and the user is dropped back at the input
box. Continuing the conversation requires a fresh `franky --max-turns
<bigger>` invocation.

### 1.2 Proposed UX (v3.x)

When the loop reaches `max_turns` mid-session in interactive mode,
suspend the worker and render an in-place modal (parallel to v1.11.2's
permission modal):

```
┌─────────────────────────────────────────────────────────┐
│ Max turns reached (50). Extend?                         │
│   [Y] yes (+50)    [n] no (terminate)                   │
└─────────────────────────────────────────────────────────┘
```

Default is `n` (terminate) so an absent user doesn't accidentally
budget-burn. `Y` extends additively; subsequent hits show the same
modal (the user can extend repeatedly).

### 1.3 Implementation plan

Mirror the v1.11.x `PermissionPrompter` pattern:

1. **New event variant**: `AgentEventKind.max_turns_request` carrying
   `{ turns_used: u32, current_cap: u32 }`. Non-terminal — the channel
   stays open across the prompt.
2. **`MaxTurnsPrompter`** in `coding/permissions.zig` (or a new
   `coding/prompters.zig`): mutex + condvar, single-slot. The `on_max_turns`
   callback emits the event then `wait()`s on the condvar; the mode
   driver calls `resolve(decision)` to unblock the worker.
3. **Mode wiring**:
   - **Interactive** owns the modal UX. Reuse `ModalState`'s arming
     pattern; bind `Y` / `Enter` to `.extend(50)` and `n` / `Esc` to
     `.stop`. The default-on-unrecognized-key is `.stop` (safer).
   - **RPC** routes through `notifications/maxTurnsRequest` and a new
     `loop/resolveMaxTurns` method shape.
   - **Proxy** emits an SSE `event: max_turns_request` and accepts
     `POST /loop/max_turns_resolve` (mirrors `/permission/resolve`).
   - **Print** stays no-op (no UX surface; just emit the error).
4. **Configurable extension amount**: a `--max-turns-extend N` flag
   and matching settings/profile field for users who want a different
   credit size per Y press. Default 50.
5. **Multi-amount picker** (stretch): replace `[Y/n]` with `[+10] [+50]
   [+100] [no]` for power users. Only worth doing if anyone asks.

### 1.4 Anchor cross-refs

When this lands in v3, the loop / Agent / SDK fields stay where they
already are; only the mode driver gets new code. The current
`agent_error{max_turns_exceeded}` event remains the SDK signal — the
new `max_turns_request` event is layered on top, used only when an
`on_max_turns` hook is wired AND the mode wants the in-session prompt.
