# v1.24 design — sub-agent tool

This document captures the open design questions and proposed
defaults for the next milestone: a `subagent` tool that lets the
top-level agent spawn isolated child agents — each with its own
model, provider, transcript, and tool sandbox — driven by the
parent LLM via tool calls.

**Review and edit inline.** Mark each question with one of:

- `✓ accept` — go with the proposed answer.
- `→ <new answer>` — override with what you want instead.
- `?` — needs more discussion before deciding.

Once every question has a decision, I'll fold the resolved design
into `franky-spec-v1.md` (or `franky-spec-v2.md` for any
deferred bits) and start implementation.

---

## 0. Why sub-agents

The agent loop today is single-LLM, single-context. Real
multi-step tasks ("audit this codebase, summarize the auth
flow, then propose three migration paths") could be decomposed
into specialist sub-tasks, each running on a model better suited
to its work — Sonnet for reasoning + planning, Flash for
high-throughput grep/summarize, GPT-5-thinking for math-heavy
sections. Today franky's only knob for that is `/model` to
hot-swap mid-conversation, which forces a single context window
to absorb every sub-task's intermediate state.

Sub-agents fix this by letting the parent agent **delegate** —
hand a fresh sub-agent a focused prompt, let it run to
completion in its own transcript, get back the final answer.
The parent's context only sees the input and the result, not
the sub-agent's intermediate tool calls. Net win on context-
window pressure, latency (parallelism), and cost (per-task
model selection).

## 1. Goal + non-goals

### 1.1 In scope (v1.24.0)

- A built-in `subagent` tool registered in print/proxy/rpc/interactive.
- Argument: `{profile, prompt, timeout_ms?, max_turns?, role?}`.
- Each sub-agent inherits the parent's tool set MINUS `subagent`
  itself (depth-1 only).
- Sub-agents run concurrently when the LLM emits multiple
  `subagent` tool calls in the same assistant turn (reuses §4.4
  parallel-homogeneous execution).
- Per-sub-agent transcript + session persistence under
  `<parent_session_dir>/subagents/<call_id>/`.
- Per-sub-agent timeout, max_turns, max_token spend cap.

### 1.2 Out of scope (deferred to v1.25+ or v2)

- **Multi-level recursion** (sub-agents spawning sub-agents).
  Hard-coded depth limit = 1. Reasons: (a) blast-radius / cost
  control, (b) debugging quality drops sharply with depth, (c)
  most real workflows are 1-2 levels. Multi-level lands behind
  an explicit `--enable-recursive-subagents` flag in v2 if
  demand surfaces.
- **Bidirectional streaming** (parent watches sub-agent's
  intermediate tokens). v1.24.0 ships buffered-only — parent
  gets the final assistant message as the tool result, no
  intermediate forwarding. The §3.3 follow-up below proposes
  the v1.25 streaming layer.
- **Inter-sub-agent communication** (agent A messages agent B
  mid-run). Sub-agents are isolated for v1.24. Coordination is
  the parent LLM's job: it gathers A's result, decides what to
  pass to B.
- **Persistent sub-agents** that survive across parent turns
  ("the researcher agent stays warm and accumulates context
  across the whole conversation"). Pure stateless spawn-and-die
  in v1.24; persistence is v2 territory under an extension API.

### 1.3 What "full feature set" means in practice

A sub-agent gets:
- the whole `coding/` tool layer (read/write/edit/bash/ls/find/grep)
  filtered by the inherited `--role` (or per-call override),
- session persistence, transcript, branching/object-store,
- the same provider + auth resolution path as the parent,
- streaming events on its own internal channel,
- `Cancel` flag + per-phase HTTP timeouts.

What it does NOT get:
- the `subagent` tool itself (depth-1 enforcement).
- access to the parent's transcript or the other sibling
  sub-agents' transcripts.

## 2. Surface (proposed)

### 2.1 The tool itself

Registered alongside `read/write/edit/bash/ls/find/grep`. JSON
Schema (sketched, will be refined during impl):

```json
{
  "type": "object",
  "required": ["profile", "prompt"],
  "properties": {
    "profile": {
      "type": "string",
      "description": "Name of a profile defined in settings.json or a built-in (gemini, groq, ollama, …). Resolves provider + model + auth."
    },
    "prompt": {
      "type": "string",
      "description": "The instruction the sub-agent will run. Written as a self-contained task — the sub-agent has no access to the parent's transcript."
    },
    "timeout_ms": {
      "type": "integer",
      "minimum": 1000,
      "maximum": 7200000,
      "default": 1800000,
      "description": "Wall-clock timeout. Default 1,800,000 ms (30 minutes); clamped to [1s, 2h]. On expiry the sub-agent is cancelled and the tool returns a structured timeout error."
    },
    "max_turns": {
      "type": "integer",
      "minimum": 1,
      "default": 20,
      "description": "Hard cap on agent-loop turns. Default 20. Parent's max_turns is unaffected."
    },
    "role": {
      "type": "string",
      "enum": ["read", "plan", "code", "full"],
      "description": "Capability role for this sub-agent. Defaults to the parent's role; sub-agents can be DEMOTED (parent=full → sub=plan) but never PROMOTED. Caller errors if a sub asks for higher role than parent has."
    },
    "system_prompt": {
      "type": "string",
      "description": "Optional override for the sub-agent's system prompt. Defaults to franky's standard system prompt minus the subagent-tool-availability paragraph."
    }
  },
  "additionalProperties": false
}
```

Tool result shape on success: `{final_text, turn_count, tool_call_count, tokens_in, tokens_out, duration_ms, session_id}`.

Tool result shape on timeout / abort / error: `{error_kind, error_message, partial_text?, turn_count, …}`.

### 2.2 Concurrent execution

When the LLM emits multiple `subagent` tool_use blocks in the
same assistant turn, they run **in parallel** via the existing
parallel-homogeneous tool execution path (§4.4). Tool results
are collected in source order regardless of completion order
(deterministic transcript invariant — preserved).

```
parent assistant turn:
  ↳ tool_use { name: subagent, args: {profile: gemini-pro, prompt: "audit auth.zig"} }    → spawn A
  ↳ tool_use { name: subagent, args: {profile: groq, prompt: "audit session.zig"} }      → spawn B
  ↳ tool_use { name: subagent, args: {profile: ollama, prompt: "audit oauth/*"} }        → spawn C

  (A, B, C run concurrently)

parent next turn:
  tool_result { call_id: a, result: {final_text: "...", ...} }
  tool_result { call_id: b, result: {final_text: "...", ...} }
  tool_result { call_id: c, result: {final_text: "...", ...} }
```

### 2.3 Per-sub-agent log + transcript

```
<session_dir>/
├── transcript.json          (parent)
├── session.json             (parent)
└── subagents/
    ├── <call_id_a>/
    │   ├── transcript.json
    │   ├── session.json
    │   └── meta.json        (profile, prompt, parent_call_id, started_at, ended_at, exit_kind)
    ├── <call_id_b>/
    │   └── …
    └── <call_id_c>/
        └── …
```

`meta.json` makes after-the-fact debugging tractable: which
profile, which prompt, how it ended (success / timeout / error
/ aborted-by-parent).

## 3. Open questions

### 3.1 — Who drives the sub-agents — LLM, code, or both?

**Proposal: LLM-driven only for v1.24.0.** The `subagent` tool
is a regular tool. The LLM emits `tool_use{name: "subagent",
args: …}`, the agent executor spawns the sub-agent, runs it to
completion, returns the result back as a `tool_result`. No
other code path.

**Why not also code-driven**: a code-driven graph (LangGraph-
style "the developer pre-wires that agent A runs, then agent B
runs on A's output, then agent C runs on B's output") is a
different abstraction — a *pipeline*, not a *tool*. franky's
philosophy is LLM-as-driver, tools-as-actions; pipelines belong
in a higher-level orchestrator that USES franky, not inside
franky itself. Building both into v1.24 would conflate the two
abstractions (tool-call vs. pipeline-step) and bloat the spec.

**Why this is sufficient**: an LLM that wants pipeline behavior
can already emit a sequence of tool calls in successive turns
("call sub_a, see result, then call sub_b based on that"). The
LLM IS the orchestrator. If a developer wants deterministic
ordering, they write the prompt to make it so. Deterministic
DAGs that a developer wants without involving an LLM at the
top → out of scope; that's franky-as-library territory and
deserves its own design.

**Decision:** ✓ accept

---

### 3.2 — How does `profile` resolve at sub-agent spawn time?

**Proposal:** Reuse `profiles.applyProfile(...)` end-to-end.
Sub-agent gets a fresh `cli.Config` seeded with the sub-agent's
arguments → `applyProfile(profile_name)` overlays
provider/model/api_key/auth_token → `resolveProvider` returns a
`ProviderInfo` → sub-agent's worker thread reads from it. Same
chain print mode runs at process startup, just per-call instead
of once-at-startup.

**Catalog:** the same precedence print mode uses — built-ins
(in `profiles.zig`'s `builtin_catalog`) + user-defined in
`settings.json[profiles]`. Project `.franky/settings.json` wins
over `$FRANKY_HOME/settings.json` wins over
`$HOME/.franky/settings.json`, then built-in. This is the
existing `loadFromSettings` order.

**Error path:** `error.ProfileNotFound` returns a structured
tool error: `{error_kind: "profile_not_found", error_message:
"profile 'fizzbuzz' not found in settings.json or built-in
catalog. Try one of: …", available_profiles: [...]}`. The LLM
can self-correct on retry.

**Key constraint per user request:** the `subagent` tool's
`profile` arg only accepts profile names — NOT raw provider /
model / base_url. This keeps blast radius tight (operator
controls the catalog of valid configurations; LLM can't, e.g.,
point a sub-agent at an attacker-controlled gateway URL).

**Decision:** ✓ accept

---

### 3.3 — Result shape: buffered vs. streamed?

The user-facing question: does the parent UI (CLI / web / Slack)
see "[sub-A: thinking…] [sub-A: calling read(…)] [sub-A: 1234
tokens]" while the sub-agent runs, or does it just see "calling
subagent…" until completion?

**Proposal for v1.24.0: buffered.** The sub-agent runs to
completion on a worker thread inside the tool's executor; the
tool doesn't return until the sub-agent's `waitForIdle` returns;
the tool result is the final assistant message + a metadata
struct (turn_count, token usage, duration). No event forwarding.

**Why buffered first**: streaming-back-to-parent introduces a
non-trivial event-forwarding layer (§3.3.1 below) that needs
its own design pass. Buffered ships value immediately and the
streaming follow-up is purely additive.

**§3.3.1 — Streamed forwarding (deferred to v1.25.0 follow-up):**
the sub-agent's events would be re-emitted as a new
`AgentEvent.subagent_event { call_id, inner: AgentEvent }`
variant. UI consumers (web / Slack) opt-in to render these by
checking for the wrapping; CLI print-mode stays clean by
default. Pin: only forward `turn_start` / `turn_end` /
`agent_error` / final `message_end` — not the raw
`message_update` deltas (would explode the parent's event
volume; let users open the sub-agent's transcript file instead
if they want full detail).

**Decision:** ✓ accept

---

### 3.4 — Recursion / depth limit

**Proposal: hard-coded depth = 1.** Sub-agents do NOT register
the `subagent` tool. Implementation: when the parent agent
prepares the sub-agent's tool list, it filters out the subagent
tool from `cfg.tools`. The sub-agent's LLM sees a tool set
without `subagent` and cannot emit it. Even if it tried (e.g.
hallucinated the call), the registry would refuse the unknown
tool.

**Why not "configurable max depth N"**: each level of recursion
multiplies the failure surface AND the cost. Real audits of
LLM-orchestrated systems consistently show that "spawn a
sub-agent that spawns a sub-agent" is a bug pattern, not a
feature. Defer to v2 with explicit opt-in if real demand
surfaces.

**Edge case:** what if a sub-agent's prompt asks "spawn a
sub-agent"? With the tool absent, the LLM returns a textual
explanation that it can't. Acceptable — the parent LLM (or
user) sees the response and can rephrase or call a different
sub-agent themselves.

**Decision:** ✓ accept

---

### 3.5 — Permission inheritance

The parent has a `SessionGates` (role + permission store +
prompter). Three options for what the sub-agent inherits:

**Option A — share by reference.** Sub-agent's `tool_gate`
points at the SAME `SessionGates` heap allocation as the
parent. "Always allow `read`" decisions made by the user
during the parent's first prompt apply automatically to all
sub-agents.

**Option B — fresh per sub-agent.** Each sub-agent gets its own
`SessionGates` (same `Store`, same `Prompter`, fresh state).
Independent decisions.

**Option C — fresh `SessionGates`, bot-shared `Store` and
`Prompter`.** Like B, but with the underlying always-allow /
always-deny database shared via the Store pointer.

**Proposal: C.** Same shape franky-do v0.3.2 uses for its
per-Agent gates. Always-allow/always-deny decisions persist
across sub-agents (the user said "always allow `read`" —
they shouldn't have to re-say it for each sub-agent), but
per-call `ask` resolutions stay per-sub-agent (a sub-agent's
`write` to a config file shouldn't auto-resolve another
sub-agent's `bash`). The prompter is shared so the user sees
ALL pending prompts in one queue rather than one queue per
sub-agent.

**Role override:** if the sub-agent's `role` arg requests a
HIGHER role than the parent has (parent = `plan`, sub asks
`code`), the spawn fails with `error_kind:
"role_promotion_denied"`. Demotion (parent = `full`, sub asks
`plan`) is allowed and is a useful pattern: "let the
research-only sub-agent loose with read-only tools, even though
my main agent has full access".

**Decision:** ✓ accept

---

### 3.6 — Timeout enforcement

**Proposal:** Per-sub-agent timeout via the existing `Cancel`
mechanism. The subagent tool spawns a timer thread that
`cancel.fire()`s the sub-agent's cancel flag after `timeout_ms`.
The agent loop catches it, unwinds with `agent_error{aborted}`,
the sub-agent's worker exits, the tool returns
`{error_kind: "timeout", error_message: "…", partial_text:
"…what the sub-agent had emitted before the timer fired"}`.

**Default:** 1,800,000 ms (30 minutes) — locked in per the
override below. Sub-agents doing real codebase audits routinely
push past 10 minutes; 30 leaves headroom without inviting
infinite loops.

**Maximum:** clamp at 7,200,000 ms (2 hours) — locked in per
the override below. The spec note: "if your sub-agent
legitimately needs more than 2 hours, decompose the work into
smaller chunks the parent LLM can re-spawn as separate
sub-agents."

**Minimum:** 1,000 ms (1 second). Catches typos like
`timeout_ms: 100` that would auto-cancel before the first HTTP
request even completed.

**Decision:** → Default 30 minutes (1,800,000 ms), max 2 hours
(7,200,000 ms). Schema in §2.1 updated.

---

### 3.7 — Token / spend caps

A runaway sub-agent can burn money. Three layers:

1. **`max_turns`** (default: 20 — locked in below). Hard cap on
   agent-loop iterations. Already implemented in `agentLoop`.
2. **`max_input_tokens`** + **`max_output_tokens`**. Provider
   rejects on its own when the context window is exceeded; the
   agent loop already returns that as an
   `agent_error{context_overflow}`. Not exposed as tool args
   for v1.24.
3. **Wall-clock cost budget**. Deferred to v2 — requires
   per-provider pricing introspection (already partly in
   `coding/pricing.zig`'s franky-do helper, but not exposed
   here). Add when real users complain about bills.

**Decision:** → `max_turns` default 20. Defer token + spend caps
to v2. Schema in §2.1 updated.

---

### 3.8 — Failure modes + LLM-friendly error reporting

A sub-agent can fail in many ways. The tool result must give
the parent LLM enough context to recover, retry, or apologize:

| `error_kind` | Trigger | LLM remediation hint |
|---|---|---|
| `profile_not_found` | unknown profile name | retry with one of `available_profiles` |
| `role_promotion_denied` | sub asked higher role | retry with parent's role or lower |
| `timeout` | wall-clock expired | summarize task more, retry; partial_text contains what was produced |
| `max_turns_exceeded` | loop hit cap | task too complex; decompose further |
| `agent_error` | provider/transport/etc. | retry; details in nested error |
| `aborted` | parent agent's own cancel fired | parent is being shut down; don't retry |
| `tool_error` | sub-agent's tool call failed unrecoverably | usually reflects an unfixable workspace state |

**Proposal:** structured tool result, never raw stack traces.
Each error_kind gets a one-line remediation hint as a `hint`
field so an LLM that doesn't read the error tag still has a
path forward. Mirrors franky's existing `ai/errors.zig`
structured-error philosophy.

**Decision:** ✓ accept

---

### 3.9 — Where the sub-agent's `cwd` lives

Sub-agents call `bash`, `read`, `write`. Two options:

**Option A — same cwd as parent.** The sub-agent inherits the
parent's working directory, sees the same workspace.

**Option B — sandboxed cwd.** Each sub-agent gets a fresh
tempdir that's deleted on completion.

**Proposal: A (shared cwd) for v1.24.0.** Real workflows want
sibling sub-agents to operate on the same workspace ("you grep
the auth code, you grep the session code, then I'll merge your
findings"). Sandboxing each one would force the sub-agents to
duplicate workspace state, which is both expensive and breaks
the "agents are pair programmers on one repo" mental model.

**Risk:** parallel sub-agents writing the same file race each
other. Mitigation: `coding/tools/edit.zig` is atomic; concurrent
`write` to the same path is the LLM's bug to avoid (the parent
prompts the sub-agents with non-overlapping work). v2 may add
a per-sub-agent rwlock on file paths if real bugs emerge.

**Decision:** ✓ accept but defer rwlock mitigation to v2 if needed.

---

### 3.10 — Cancellation of sub-agents when the parent aborts

Parent gets `agent.abort()` (Ctrl-C from CLI, `❌` reaction in
Slack, browser close in proxy). What happens to in-flight sub-
agents?

**Proposal: cascade abort.** Each in-flight sub-agent gets its
own `Cancel`; the parent's tool executor is responsible for
tracking the per-call cancel and firing it when its own
`Cancel` fires. The cascade is one level deep (since recursion
is forbidden), so this is just "parent fires N child cancels".

**Implementation:** the subagent tool keeps a heap-allocated
`Cancel` per pending sub-agent + a watcher fiber/thread that
listens for the parent's cancel and propagates. Simple.

**Decision:** ✓ accept

---

### 3.11 — Sub-agent's system prompt

**Proposal:**
- **Default:** franky's standard system prompt MINUS the
  paragraph that mentions the `subagent` tool (since the sub-
  agent doesn't have it). The standard prompt is otherwise
  untouched — sub-agents act like fresh franky sessions.
- **Override:** `system_prompt` arg lets the parent LLM steer
  the sub-agent ("You are a security auditor. Only use read-
  only tools…"). Useful for differentiating sub-agents by
  role.
- **Append:** maybe also `append_system_prompt` arg? Pin it as
  a v2 follow-up if the override-only approach proves
  insufficient — overlap-with-default is YAGNI for now.

**Decision:** ✓ accept with a v2 follow-up

---

### 3.12 — UI / observability

How do users SEE that sub-agents are running?

**Print mode:** stderr logs at `info` level — `subagent: spawn
profile=gemini-pro call_id=…`, `subagent: turn 1`, `subagent:
done turns=4 duration=8.2s`. Default-on for visibility.

**Proxy / web UI:** a new event variant `subagent_status_update
{call_id, profile, turn_count, state}` emitted at each
sub-agent turn boundary. The web UI renders a small
"Sub-agent: gemini-pro · turn 3 · 8s elapsed" badge per active
sub-agent. (See §3.3.1 streaming follow-up — same plumbing.)

**RPC mode:** same event, JSON-RPC clients opt-in by
subscribing to the `subagent_status_update` topic.

**Slack / franky-do:** a single status reaction per parent
mention works fine for v1.24. Per-sub-agent reactions belong
in v0.5 alongside other UX work.

**Decision:** ✓ accept the print mode logs; defer the event-based UI updates to v2 follow-up that adds streaming.

---

### 3.13 — Naming

The tool name itself. Options:
- `subagent` — current proposal. Clear, short, no ambiguity.
- `delegate` — verb-based, reads naturally in LLM tool calls
  ("I'll delegate this to a Gemini sub-agent").
- `task` — Anthropic's swarm/Claude Code's name for the
  similar tool. Familiar to Claude users.
- `agent` — concise but ambiguous (agent of what?).

**Proposal: `subagent`.** Most explicit, hardest to confuse
with anything else in franky's tool catalog. `task` would be
better if the goal were familiarity for Claude Code users —
but franky has its own identity and `task` could collide with
future task-management tooling.

**Decision:** ✓ accept subagent

## 4. Wire shape

```zig
// src/coding/tools/subagent.zig — new

pub fn tool() at.AgentTool {
    return .{
        .name = "subagent",
        .description = "Spawn an isolated sub-agent with its own model + provider …",
        .parameters_json = subagent_params_json,
        .execute = execute,
    };
}

fn execute(ctx: at.ToolExecuteCtx) anyerror!at.ToolResult {
    // 1. Parse args (profile, prompt, timeout_ms, max_turns, role, system_prompt).
    // 2. Build a fresh cli.Config seeded with parent's environ_map.
    // 3. profiles.applyProfile(&cfg, name) — fails with profile_not_found.
    // 4. resolveProvider → ProviderInfo.
    // 5. Build sub-Agent with the parent's tools[] MINUS subagent itself.
    // 6. Wire SessionGates: same Store, same Prompter, fresh role gate.
    // 7. Allocate per-sub-agent Cancel + spawn timer thread.
    // 8. Hook parent's Cancel into the sub-agent's via a watcher.
    // 9. agent.prompt(args.prompt); agent.waitForIdle();
    // 10. Persist transcript under <session>/subagents/<call_id>/.
    // 11. Build structured ToolResult — final_text, turn_count, etc.
}
```

**Affected modules:**

- `src/coding/tools/subagent.zig` (new) — the tool itself.
- `src/coding/tools/mod.zig` — re-export.
- `src/coding/modes/print.zig` (and proxy/rpc/interactive) —
  register the tool by default; add `--no-subagent` CLI flag
  for users who want to disable it.
- `src/agent/loop.zig` — no changes (parallel-homogeneous
  execution already in place).
- `src/coding/profiles.zig` — exists, no changes.
- `src/coding/role.zig` — add a `canDemoteTo` helper.
- `src/coding/session.zig` — add a `subagents/<call_id>/` sub-
  directory layout. Mostly auto-handled by reusing the same
  session-save logic with a different parent dir.

## 5. Tests

1. **Happy path** — parent spawns one sub-agent with a faux
   profile, faux returns scripted text, parent gets the
   `tool_result` with the right `final_text` and turn_count.
2. **Concurrent spawn** — parent emits 3 `subagent` tool calls
   in one turn; all three run in parallel; results ordered
   deterministically (source-order, §4.4).
3. **Profile-not-found** — sub-agent with an invalid profile
   name; tool returns `{error_kind: "profile_not_found",
   available_profiles: […]}`.
4. **Depth-1 enforcement** — sub-agent's tool list does NOT
   include `subagent`; if the model hallucinates a call to it
   the registry returns `error.ModelUnavailable`-style result.
5. **Timeout** — sub-agent with `timeout_ms: 1000`; the
   underlying provider is a faux that never completes; the
   timer fires, the sub-agent unwinds cleanly, the tool
   result has `error_kind: "timeout"` + a `partial_text`.
6. **Cascade abort** — parent's cancel fires while a sub-agent
   is mid-turn; sub-agent unwinds with `aborted`; no
   leaked threads or unfreed transcripts.
7. **Role demotion** — parent role=full, sub asks role=plan;
   the sub-agent's tool list excludes `bash` / `write` /
   `edit`.
8. **Role promotion denied** — parent role=plan, sub asks
   role=full; tool returns `{error_kind:
   "role_promotion_denied"}`.
9. **Permission inheritance** — parent's `Store` has
   `always_allow_tools = ["read"]`; sub-agent's `read` calls
   bypass the prompter automatically.
10. **Persistence** — after a sub-agent run, the parent
    session dir contains `subagents/<call_id>/{transcript,
    session, meta}.json`.
11. **Subagent-tool-availability paragraph in system prompt**
    — sub-agent's prompt does NOT mention the `subagent` tool
    (since it doesn't have it).

## 7. Decisions summary

All 13 questions resolved. Two overrides on numeric defaults
(timeouts §3.6, max_turns §3.7). Three accept-with-deferrals
that scope the streaming UI (§3.3) and event-based observability
(§3.12) into a v1.25 follow-up; the rwlock mitigation in §3.9
becomes a v2 contingency.

|---|---|---|

| 3.2 | profile resolution | ✓ `profiles.applyProfile`; profile name only (no raw provider/url). |
| 3.3 | buffered vs streamed | ✓ Buffered for v1.24.0; streaming `subagent_event` wrapper deferred to v1.25.0. |
| 3.4 | recursion depth | ✓ Hard-coded depth = 1; sub-agent's tool list excludes `subagent`. |

| 3.6 | timeouts | → Default **30 min** (1,800,000 ms), max **2 h** (7,200,000 ms), min 1 s. |
